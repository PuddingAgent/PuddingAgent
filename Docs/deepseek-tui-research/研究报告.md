# DeepSeek TUI 设计研究报告

> 研究目标：[deepseek-tui](https://github.com/Hmbown/deepseek-tui)（v0.8.x）  
> 研究日期：2026-05-13  
> 目的：分析其架构设计与工程亮点，提炼对 Pudding Agent 的启发

---

## 1. 项目概述

DeepSeek TUI 是一个**终端原生的 AI 编程代理**，面向 DeepSeek V4 模型优化。它提供基于 ratatui 的 TUI 界面，支持文件操作、Shell 执行、Git 管理、Web 搜索、子代理协调等功能。

**核心特征：**
- 单一二进制分发（Rust 编译，npm/cargo/brew 安装）
- 三种运行模式：Plan（只读探索）、Agent（交互审批）、YOLO（自动批准）
- 流式推理：实时 SSE 流 + Thinking Block 流式渲染
- 1M Token 上下文支持 + 前缀缓存感知
- 持久化会话、后台任务队列、工作区快照回滚

## 2. 技术栈

| 维度 | 选型 |
|------|------|
| 语言 | Rust (95.5%) |
| TUI 框架 | ratatui + crossterm |
| 异步运行时 | tokio |
| HTTP 客户端 | reqwest (SSE 流) |

## 4. 设计亮点深度分析

### 4.1 流式渲染优先 (Streaming-First)

**机制：**
- 所有 LLM 响应通过 SSE 流式传输
- 自定义 `StreamingState` 状态机，支持 Smooth（~10字符）和 Aggressive（~100字符）两种分块策略
- Thinking Block 实时流式展开（可折叠/展开，Ctrl+O 切换）
- 增量文本提交防止流错误丢失内容
- 低运动模式（30 FPS vs 120 FPS）减少终端闪烁

**对 Pudding 的启发：**
Pudding 目前使用 DirectLlmClient → OpenAiLlmGateway 流式路径，但在前端渲染上可借鉴：
- Thinking/Reasoning 块的可折叠流式渲染
- 分块策略可配置（适应不同网络条件）
- 增量提交 + 修订缓存避免整屏重绘

### 4.2 三级安全模式 (Plan / Agent / YOLO)

**机制：**

| 模式 | 行为 | 适用场景 |
|------|------|---------|
| Plan | 只读探索，工具执行前需切换到 Agent | 代码理解、调研 |
| Agent | 交互式，每个工具调用需人工审批 | 日常开发 |
| YOLO | 自动批准所有操作 | 可信工作区 |

**对 Pudding 的启发：**
这是 Pudding 目前缺失的设计。可以引入类似的模式切换机制：
- 在 Web UI 顶部添加模式指示器
- Plan 模式下仅允许只读工具（read_file、search、grep）
- Agent 模式保留审批弹窗
- YOLO 模式用于自动化场景

### 4.3 Thinking-Mode 回放机制

**背景：** DeepSeek V4 的 reasoning 模型要求后续 Turn 必须携带之前的 `reasoning_content`，否则返回 HTTP 400。

**解决方案：**
- 自动检测 reasoning 模型（V4 系列）
- 在 tool-call 消息上强制注入占位 `reasoning_content`
- Final sanitizer 在 API 调用前修正请求 body

**对 Pudding 的启发：**
- 模型特化处理模式：不同模型有不同的协议约束，应该有适配层
- 当前 Pudding 的 `OpenAiLlmGateway` 可扩展模型适配器注册机制

### 4.4 工作区快照（Side-Git，无侵入回滚）

**机制：**
- 每次 Turn 前后对工作区做 Git 快照
- 快照存储在 `~/.deepseek/snapshots/<project_hash>/<worktree_hash>/.git`
- 完全独立于用户的 `.git`，不污染 git 历史
- `/restore N` 和 `revert_turn` 工具回滚文件状态

**对 Pudding 的启发：**
这是非常巧妙的设计。Pudding 目前没有文件操作回滚能力。可考虑：
- 在 Agent 执行文件编辑工具时自动创建旁路 Git 快照
- 提供回滚 UI（类似"撤销上次编辑"按钮）
- 完全不影响用户项目的 Git 历史

### 4.5 持久化子代理编排 (Durable Sub-Agent)

**机制：**
- `agent_open` / `agent_eval` / `agent_close` 管理命名子代理
- 默认 10 个并发子代理，可配置到 20
- 大量结果通过 `var_handle` 引用停放，模型按需调用 `handle_read` 切片读取
- 完成通知：结构化 `<deepseek:subagent.done>` 事件

**与 Pudding P2P 模型的对比：**
- Pudding 的多 Agent 是**进程级 P2P 组网**（mDNS 发现 + 直连通信）
- deepseek-tui 的子代理是**进程内并发编排**（同一进程，异步并发）
- 两者互补：Pudding 可以既支持跨进程 P2P 协作，也支持进程内子代理并发

### 4.6 多层上下文压缩与容量护栏

**机制：**
```
容量状态机：
  Normal → RefreshingContext (警告) 
        → VerifyingRecentWork (干预) 
        → ResettingPlan (重置)
```

- **手动压缩：** `/compact` 或 `/clear`
- **自动压缩：** 上下文 > 95% 容量时触发
- **选择性保留：** 项目上下文包、用户记忆、Skills 保持静态不压缩
- **前缀缓存感知：** 分别追踪缓存命中/未命中 Token
- **Turn Meta 去重：** 重复的环境元数据替换为 `<turn_meta_unchanged />`

**对 Pudding 的启发：**
Pudding 目前缺乏系统性的上下文管理。可借鉴：
- 引入容量监控与分级响应机制
- 压缩时选择性保留关键上下文（记忆、Skill 指令）
- Turn 间去重优化（环境描述不重复发送）

### 4.7 LSP 编辑后诊断集成

**机制：**
- 每次 `edit_file` / `apply_patch` / `write_file` 成功后自动触发
- 收集配置好的 LSP 诊断（rust-analyzer、pyright、gopls、clangd 等）
- 将诊断结果作为**合成用户消息**注入到下一轮 API 请求前
- Agent 看到自己的编辑引入的错误，自动修正

**对 Pudding 的启发：**
这是极具价值的反馈闭环。Pudding 可以：
- 利用 .NET Roslyn Analyzer 做编译后诊断
- 将编译错误作为合成消息注入对话上下文
- 实现"编辑 → 诊断 → 自动修正"的自愈循环

### 4.8 工具结果去重与上下文节省

**机制：**
- 超过 1KB 的工具结果计算 SHA-256 哈希
- 重复出现时替换为 `<TOOL_RESULT_REF>` 引用块
- 大幅减少上下文浪费

**对 Pudding 的启发：**
简单但高效的优化，可直接在 Pudding 的 `AgentExecutionService` 中实现。

### 4.9 Schema Gate（持久化版本门禁）

**机制：**
- `session_manager.rs`、`runtime_threads.rs`、`task_manager.rs` 在持久化记录中嵌入 `schema_version`
- 加载时检测版本不兼容，**显式报错而非静默截断**
- 防止二进制与存储状态不同步时的数据损坏

**对 Pudding 的启发：**
Pudding 使用 EF Core 迁移管理 schema 变更，但缺少运行时版本校验。可考虑在 `MigrateAsync()` 后写入当前 schema 版本号，启动时做版本兼容性检查。

### 4.10 配置分层策略

**文件优先级：**
```
/etc/deepseek/managed_config.toml    # 托管默认值（Unix）
/etc/deepseek/requirements.toml      # 策略约束（Unix）
~/.deepseek/config.toml              # 用户配置
环境变量                              # 运行时覆盖
CLI 参数                              # 最高优先级
```

**对 Pudding 的启发：**
Pudding 目前以 `data/conf/pudding-config.json` 为唯一配置源。可考虑引入分层覆盖机制（如环境变量覆盖 JSON 中的 API Key）。

## 5. 终端 UI 渲染策略

### 5.1 布局分区

```
┌────────────────────────────────────────────┐
│ Plan | model:flash | context: 12% / 1M     │  ← 状态头（模式/模型/上下文占比/费用）
├────────────────┬───────────────────────────┤
│   文件树        │                           │
│   (可选)       │   对话记录                  │  ← 主区域（聊天历史 + 流式渲染）
│               │   (活跃 Cell 渲染)          │
├────────────────┴───────────────────────────┤
│ [pending input preview]                     │  ← 待发送预览
├─────────────────────────────────────────────┤
│ > @file /command ...                        │  ← 输入栏（多行、自适应高度）
├─────────────────────────────────────────────┤
│ 🤖 working · Cache: 45% | Esc cancel       │  ← 状态脚（动画喷口）
└─────────────────────────────────────────────┘
```

### 5.2 交互设计亮点

| 功能 | 实现 |
|------|------|
| @file 提及 | 输入 @ 时弹出文件选择，支持多选和移除 |
| /slash 命令 | 输入 / 时弹出命令菜单，可发现所有可用命令 |
| 草稿暂存 | Ctrl+S 保存当前输入，`/stash list` 恢复 |
| 会话恢复 | Ctrl+R 弹出历史会话选择器 |
| Thinking 展开 | Ctrl+O 在流式过程中展开/折叠推理块 |
| 鼠标支持 | 滚动条拖拽、边缘自动滚动 |

### 5.3 状态可见性

- **信号栏颜色渐变：** 上下文使用率 绿 → 黄 → 红
- **Footer 芯片：** 缓存命中率、推理回放 Token、会话费用、Git 分支
- **Toast 消息：** 排队状态消息以 1-2 行条带显示在 Footer 上方
- **活跃 Cell 渲染：** 进行中的工具组在对话中显示实时状态（Running → Success/Failed）

## 6. 与 Pudding Agent 的对比分析

| 维度 | DeepSeek TUI | Pudding Agent |
|------|-------------|---------------|
| **UI 形态** | 终端 TUI（ratatui） | Web UI（React） |
| **语言/运行时** | Rust / tokio | .NET / ASP.NET Core |
| **数据库** | SQLite + JSON 文件 | SQLite（EF Core） |
| **分发方式** | 单二进制（npm/cargo/brew） | 单二进制 / Docker |
| **多 Agent** | 进程内子代理并发编排 | 跨进程 P2P 组网（mDNS） |
| **流式渲染** | 高度优化的 SSE + 分块策略 | 基础 SSE 流式 |
| **安全模式** | Plan / Agent / YOLO 三级 | 暂未分层 |
| **文件回滚** | Side-Git 快照 | 无 |
| **编辑后诊断** | LSP 集成（多语言） | 无 |
| **上下文管理** | 多层压缩 + 容量护栏 | 基础 |
| **任务队列** | 持久化后台任务 + Worker 池 | 无 |
| **配置分层** | 托管配置 → 用户配置 → 环境变量 → CLI | 单一 JSON 文件 |
| **Schema Gate** | 持久化版本校验 | EF 迁移（无运行时校验） |

## 7. 对 Pudding 的具体行动建议

### 高优先级（可直接落地）

1. **三级安全模式**
   - Web UI 顶部添加 Plan / Agent / YOLO 模式切换器
   - 各模式控制工具白名单和审批策略
   - Plan 模式仅允许只读工具

2. **文件操作回滚（Side-Git）**
   - 在文件编辑工具执行前后创建旁路 Git 快照
   - 存储于 `data/snapshots/`，不影响用户 `.git`
   - 提供"撤销上次编辑"交互

3. **工具结果去重**
   - 在 `AgentExecutionService` 中缓存大结果的 SHA-256
   - 重复结果替换为引用标记

### 中优先级（需要一定重构）

4. **编辑后诊断反馈闭环**
   - 利用 Roslyn Analyzer 在文件编辑后收集编译诊断
   - 注入为合成用户消息，让 Agent 自动修正

5. **上下文容量护栏**
   - 追踪 Token 使用量，分阶段预警
   - 压缩时选择性保留记忆和 Skill 指令
   - Turn 间环境描述去重

6. **会话快照与恢复**
   - 每个 Turn 前自动保存会话检查点
   - 提供会话恢复 UI（类似 Ctrl+R 选择器）

### 低优先级（长期演进）

7. **持久化后台任务队列**
   - 支持 `/task add` 式的异步任务提交
   - Worker 池并发执行，重启后可恢复

8. **模型适配器注册机制**
   - 不同模型提供商有不同协议约束（如 DeepSeek 的 reasoning 回放）
   - 在 `OpenAiLlmGateway` 中引入适配器模式

9. **配置分层覆盖**
   - 环境变量 → 配置文件 → 默认值 的优先级链

---

## 8. 总结

DeepSeek TUI 是一个工程成熟度很高的终端 AI Agent 实现。其最值得 Pudding 借鉴的设计模式包括：

1. **三级安全模式** — 以极低成本提升安全性
2. **Side-Git 快照** — 无侵入的文件回滚方案
3. **编辑→诊断→自愈闭环** — LSP 反馈机制让 Agent 真正"看到"自己的错误
4. **分层上下文管理** — 容量护栏 + 选择保留 + 去重
5. **持久化 Schema Gate** — 防止跨版本数据损坏的防御性设计

这些设计并非终端 UI 专属，完全可以迁移到 Web UI 形态的 Pudding Agent 中。
