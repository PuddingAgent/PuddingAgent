# DeepSeek-Reasonix 前缀缓存研究报告

> 研究目标：[esengine/deepseek-reasonix](https://github.com/esengine/deepseek-reasonix)
> 研究日期：2026-05-25
> 研究版本：`main` 分支 `1d6a30d11b7600f2acc535009db3fc5896191f2b`
> 目的：分析 Reasonix 在 prefix-cache stability、前缀缓存优化、缓存自动诊断和统计分析上的工程设计，并提炼 Pudding Agent 可落地方案。

---

## 1. 项目概述

DeepSeek-Reasonix 是一个 DeepSeek 原生的终端 AI 编程代理，项目定位不是通用多模型框架，而是围绕 DeepSeek 的前缀缓存机制做端到端工程化。其 README 明确把 "prefix-cache stability" 作为核心卖点，架构文档也把 Cache-First Loop 列为第一支柱。

**核心判断：**

- Reasonix 的低成本不是单点优化，而是把提示词、工具规格、历史消息、压缩、诊断和统计都纳入同一套缓存不变量。
- 前缀缓存的有效性来自 "字节稳定"：系统提示词、工具列表和 few-shot 组成不可变前缀，历史消息只追加，不重排、不重写。
- 项目把 prefix hash 写入 transcript，把缓存命中/未命中 token 写入 usage，把这些数据再用于 replay、diff、stats 和 UI 状态展示，形成可审计闭环。

## 3. Prefix-Cache Stability 机制

### 3.1 三段式上下文

Reasonix 把模型请求上下文拆成三段：

```text
IMMUTABLE PREFIX = system + tool_specs + few_shots
APPEND-ONLY LOG  = user / assistant / tool messages in order
VOLATILE SCRATCH = reasoning、临时 plan 状态、本地 notes
```

核心不变量：

1. `ImmutablePrefix` 在会话开始时构建，正常 turn 中不重建。
2. `AppendOnlyLog` 只追加消息，历史顺序保持稳定。
3. `VolatileScratch` 每轮重置，不进入上游请求。
4. 需要改变系统提示词或工具规格时显式 invalidates fingerprint，并接受下一轮缓存 miss。

### 3.2 `ImmutablePrefix` 指纹

`ImmutablePrefix` 对 `system`、`tools`、`fewShots` 做 JSON 序列化并取 SHA-256 前 16 位作为 fingerprint。它不是安全哈希用途，而是工程诊断用途：如果 prefix hash 在会话中变多，说明缓存前缀发生漂移。

值得借鉴的细节：

- `toolSpecs` 保持数组顺序，因为工具顺序也是字节前缀的一部分。
- `addTool`、`removeTool`、`replaceSystem` 都清空 fingerprint cache。
- `verifyFingerprint()` 用于开发/测试期发现绕过 API 的内部突变。
- 测试覆盖了新增工具、删除工具、描述变化、参数 schema 变化、工具重排等 prefix drift 场景。

### 3.3 工具热插拔的缓存成本模型

Reasonix 没有假装 MCP 工具热插拔是无成本的。新增或移除工具会改变工具 schema 前缀，因此项目把它建模为 "下一轮缓存 miss"，并通过 fingerprint 表示这类变化。

对 Pudding 的启发：Pudding 如果支持运行中增删 MCP 工具、技能工具或平台工具，必须让 UI 和统计知道这是一次预期的 prefix churn，而不是把命中率下降误判为模型或服务商异常。

## 4. 前缀缓存优化策略

### 4.1 缓存优先循环

Reasonix 的 `CacheFirstLoop.step()` 每轮先追加用户消息，再构造 `prefix.toMessages() + log.toMessages()` 发送给 DeepSeek。工具调用结果按声明顺序追加回 log，即使并行执行，也保持模型看到的历史形状稳定。

关键做法：

- 只读工具可以并行执行，但结果按原始工具调用顺序 append。
- mutating / side-effect 工具默认串行，避免 read-after-write 顺序污染。
- tool result 进入下一轮上下文前会做 shrink / compact，降低后续请求成本。
- `ContextManager` 在 turn start 和 response 后都检查上下文比例，超过阈值时折叠历史。

### 4.2 折叠时复用 live prefix

`ContextManager.summarizeForFold()` 不是用一个完全独立的简短 summarizer prompt，而是复用主 agent 的 system prompt、few-shot 和 tool list。这样折叠摘要请求仍能共享主循环已经缓存的 prefix bytes。

这是一个高价值设计：上下文压缩通常会被当作"额外成本"，但 Reasonix 把压缩调用本身也纳入前缀缓存优化范围。

### 4.3 成本定价与缓存 savings

`SessionStats` 和 `usage.ts` 直接区分：

- `prompt_cache_hit_tokens`
- `prompt_cache_miss_tokens`
- `completion_tokens`
- input cache hit 单价
- input cache miss 单价
- output 单价

成本计算不是粗略按 prompt token 统一计价，而是使用：

```text
cost = hit_tokens * hit_price
     + miss_tokens * miss_price
     + completion_tokens * output_price
```

这让 `/cost`、`/status`、`stats` 能展示真实的缓存收益，而不是只展示 token 数。

## 5. 缓存自动诊断设计

Reasonix 的诊断不是单一 doctor 命令，而是一组分层工具。

| 入口 | 诊断内容 |
|---|---|
| `reasonix doctor` / `/doctor` | API Key、配置、DeepSeek 可达性、代理路由、tokenizer、session、hooks、项目状态 |
| `/status` | 模型、stream、effort、session cost、cache hit、context 使用、MCP 工具数 |
| `/cost` | 最近一轮成本和按当前命中率估算的下一轮成本 |
| `reasonix stats` / `/stats` | today/week/month/all-time 的 turns、cache hit、cost、cache saved、vs Claude |
| `reasonix replay` | 离线重放 transcript，并展示累计 cache hit、cost、prefix hashes |
| `reasonix diff` | 对比两个 transcript，输出 cache hit、cost、prefix hashes 和首个分歧 turn |
| `/feedback` | 收集版本、平台、终端、模型、effort、模式、MCP 数量、session id 等诊断块 |

### 5.1 Prefix churn 诊断

`transcript/log.ts` 在 assistant final 事件上记录 `prefixHash`。`transcript/replay.ts` 会汇总 distinct prefix hashes；`transcript/diff.ts` 会判断：

- `prefixHashes.length <= 1`：prefix byte-stable。
- `prefixHashes.length > 1`：发生 prefix churn，属于 cache-hostile session。
- A/B transcript prefix hash 相同但命中率不同：问题更多来自 log stability 或请求尾部增长，而不是系统前缀变化。

这比单看 cache hit percentage 更可靠，因为 cache hit 下降可能来自三类原因：

1. prefix 变了；
2. prefix 没变，但 log 被重写/压缩/重排；
3. 请求新增尾部过大，导致短期 miss token 上升。

### 5.2 自动诊断可迁移方案

Pudding 可把每次 LLM 请求记录成结构化 `LlmTurnTelemetry`：

| 字段 | 用途 |
|---|---|
| `SessionId` / `ConversationId` / `TurnId` | 聚合维度 |
| `Provider` / `Model` | 定价和特性维度 |
| `PrefixHash` | 判断 prefix stability |
| `SystemPromptHash` | 区分系统提示词变化 |
| `ToolSpecHash` | 区分工具列表/schema 变化 |
| `MemoryHash` | 区分注入记忆变化 |
| `PromptTokens` | 上下文压力 |
| `PromptCacheHitTokens` | DeepSeek 命中 token |
| `PromptCacheMissTokens` | DeepSeek 未命中 token |
| `CompletionTokens` | 输出成本 |
| `InputCost` / `OutputCost` / `TotalCost` | 账务统计 |
| `CompactionReason` | 判断压缩是否导致命中率波动 |

自动诊断规则建议：

```text
if distinct(PrefixHash, last N turns) > 1:
  report "prefix churn"
  compare SystemPromptHash / ToolSpecHash / MemoryHash to identify source

if PrefixHash stable and cacheHitRatio drops below threshold after warmup:
  report "log instability or large tail growth"
  inspect compaction events and prompt token deltas

if ToolSpecHash changes mid-session:
  report "tool schema drift"
  mark next turn cache miss as expected

if cacheHitRatio is unavailable:
  report "provider does not expose prompt cache telemetry"
  fall back to synthetic prefix stability only
```

## 6. 统计分析模型

Reasonix 统计有三层：

1. **实时状态**：TUI 状态栏和 `/status` 使用 `SessionStats.summary()`，显示 session 级 cache hit。
2. **跨会话 usage log**：`~/.reasonix/usage.jsonl` 只记录 token、成本、模型、session，不记录 prompt/completion 内容，隐私边界清晰。
3. **离线 transcript**：transcript 是审计凭证，包含 turn、usage、cost、model、prefixHash、tool args，可被 replay/diff 复盘。

设计优势：

- 实时 UI 和长期统计共享同一套公式，避免状态栏、`/cost`、dashboard 数字不一致。
- usage log 不保存对话正文，适合长期留存。
- transcript 保存足够的复盘信息，可解释一次任务为什么贵、哪里开始分歧、prefix 是否漂移。
- `SessionStats` 对内存中的 turn 数做上限，老 turn 折叠进 carryover，避免统计对象随长会话无限增长。

## 7. 对 Pudding Agent 的落地建议

### 7.1 建立 Provider Feature Profile

建议在 Pudding 的服务商/模型层显式描述 DeepSeek 能力：

```text
ProviderFeatureProfile
  SupportsPromptCacheTelemetry = true
  SupportsPrefixCacheOptimization = true
  PromptCacheHitUsageField = "prompt_cache_hit_tokens"
  PromptCacheMissUsageField = "prompt_cache_miss_tokens"
  RequiresByteStablePrefix = true
```

这样缓存优化不会散落在 `AgentExecutionService` 或网关实现里，而是作为模型特性驱动请求构建、统计展示和诊断规则。

### 7.2 引入 Pudding 版 Immutable Prefix

Pudding 可把以下内容纳入稳定前缀：

- Agent system prompt
- 可用工具 schema
- 固定技能说明
- 固定项目记忆/约束
- 模型适配器必须携带的 few-shot 或协议说明

注意事项：

- 工具 schema 按稳定 key 排序，避免字典遍历顺序导致 hash 抖动。
- 所有动态字段（时间戳、随机 id、实时状态、运行目录快照）不得进入 immutable prefix。
- 如果项目记忆或工具列表改变，必须产生新的 `PrefixHash` 并记录 `PrefixChangeReason`。

### 7.3 日志 append-only，压缩以摘要消息替代重写

Pudding 当前如需做上下文窗口管理，建议采用：

```text
prefix 固定
log 只追加
compact 时把旧 head 替换为一条 synthetic summary message + tail
compact event 写入 telemetry
```

不要在每轮请求前重排历史消息、重写 tool result、插入动态 system note，否则 prefix cache 的收益会被隐性破坏。

### 7.4 建立缓存诊断 UI

可在 Pudding 管理端或聊天调试面板加入：

- Session cache hit ratio
- Last turn hit/miss tokens
- Prefix hash / tool hash / memory hash
- Prefix stability 状态：stable / expected churn / unexpected churn
- Cache saved 估算
- Top cache miss causes

最小可用版本只需在开发面板展示最近 20 turn 的表格；后续再做 dashboard。

### 7.5 把测试写成不变量

建议新增测试用例：

- 相同 system + tool schema 构建出的 prefix hash 一致。
- 工具新增、删除、重排、描述变更、参数 schema 变更都会改变 hash。
- 用户消息追加不会改变 prefix hash。
- 压缩历史不会改变 prefix hash。
- DeepSeek usage 缺少 miss 字段时，用 `prompt_tokens - hit_tokens` 防御性补齐。

## 8. 风险与差异

- DeepSeek 的前缀缓存收益依赖服务端实际返回的 usage 字段；如果经由第三方 OpenAI-compatible 网关，字段可能缺失或语义不同。
- Pudding 是 Web + 平台化多 Agent 系统，不能照搬 Reasonix 的终端 UI 形态，但可以直接借鉴数据模型与不变量。
- Pudding 的工具、技能、MCP 可能比 Reasonix 更动态，因此需要区分 "预期 prefix churn" 与 "异常 prefix churn"。
- 如果 Pudding 支持多服务商，缓存诊断必须允许 provider-specific fallback，不能把 DeepSeek 规则强行套到所有模型。

---

## 9. 结论

DeepSeek-Reasonix 最值得 Pudding 借鉴的是：把前缀缓存从"服务商能力"提升为 Agent Loop 的架构不变量，并通过 prefix hash、append-only log、usage 统计、transcript replay/diff 构成可审计闭环。

Pudding 的优先落地顺序建议：

1. 增加 DeepSeek cache telemetry 解析与持久化。
2. 引入 prefix hash / tool hash / memory hash。
3. 在调试面板展示 session 级 cache hit 和 prefix stability。
4. 把异常 prefix churn 变成自动诊断告警。
5. 再做上下文压缩复用 live prefix 的成本优化。
