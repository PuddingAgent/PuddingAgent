# 27 ADR-026 终验阻塞修复与证据链强制闭环方案

> 状态：**partially-implemented**（代码修复完成，Playwright 未在运行中服务上通过 evidence 全链路）
> 日期：2026-05-20
> 范围：ADR-026 终验阻塞项、E2E evidence 强断言、子代理 facade 接入、LLM profile 真实解析、文档状态回滚与验收口径修正
> 前置：[26ADR-025验收阻塞修复与执行闭环方案](26ADR-025验收阻塞修复与执行闭环方案.md)、[25运行时契约闭环与可观测性验收基线ADR](25运行时契约闭环与可观测性验收基线ADR.md)

---

## 1. 背景

ADR-026 已完成一轮修复，但终验发现“全部完成”的结论仍不成立：

1. Playwright evidence test 没有强制断言 `traceId` 非空，且调用了错误的 evidence API 路由。
2. 后端 `done` frame 未携带 `traceId`，前端 debug write 依赖不存在的字段。
3. `ISubAgentInvocationService` 已注册，但子代理触发仍经 `SubAgentTool -> ISubAgentManager.SpawnAsync`，没有经过子代理 facade。
4. `ILlmProfileResolver` 已存在，但当前实现仍只是把 `profile.ModelId` 转为 `LlmConfig.ModelId`，provider/profile/role 未参与真实配置解析。
5. `AgentExecutionService` 仍保留多处直连 fallback。当前生产 DI 下多数路径会走 facade，但源码层面仍没有形成强边界。
6. `Docs/07架构/26ADR-025验收阻塞修复与执行闭环方案.md` 已标记为 `implemented`，但其验收条件并未全部满足。

本 ADR 定义最终修复路线。目标是把 ADR-026 从“部分实现”推进到“可验收实现”，并防止后续继续把“注册了 facade”误判为“执行路径闭环”。

## 2. 终验口径

### ADR-027-A：E2E evidence 必须强制失败，不允许条件跳过

**决定**：E2E 证据链测试中，`sessionId`、`traceId`、evidence API、timeline 关键组件全部是硬断言。

禁止：

```ts
if (traceId) {
  // call evidence API
}
```

要求：

```ts
expect(sessionId).toBeTruthy();
expect(traceId).toBeTruthy();
const evidenceResp = await page.request.get(`/api/diagnostics/e2e/evidence/${encodeURIComponent(traceId!)}`);
expect(evidenceResp.ok()).toBeTruthy();
const evidence = await evidenceResp.json();
expect(evidence.timeline.length).toBeGreaterThan(0);
expect(components).toContain('agent_execution');
expect(components.some(c => c === 'llm_gateway' || c === 'llm_invocation' || c === 'direct_llm')).toBeTruthy();
```

### ADR-027-B：Debug trace 来源以后端为准

**决定**：前端不能凭空推断 trace。traceId 必须来自后端返回或后端可查询数据。

优先级：

1. `metadata` 或 `done` frame 显式包含 `traceId`。
2. 如果帧没有 `traceId`，前端用 `sessionId` 调用 timeline API，取最新或当前消息相关 trace。
3. 如果仍无法获得 traceId，E2E 必须失败。

最小后端契约：

```csharp
var doneFrame = ServerSentEventFrame.Json(
    SseEventTypes.Done,
    new
    {
        reply,
        usage = finalUsage,
        traceId = execTrace.TraceId,
        sessionId = request.SessionId,
        messageId = request.MessageId
    });
```

如果当前 request 没有 `MessageId`，必须至少包含 `traceId` 和 `sessionId`。

### ADR-027-C：子代理 facade 必须进入子代理工具执行路径

**决定**：`spawn_sub_agent` 是子代理能力入口，不应绕过 `ISubAgentInvocationService`。

允许两种实现，推荐方案 1：

| 方案 | 描述 | 评价 |
|------|------|------|
| 方案 1 | `SubAgentTool` 依赖 `ISubAgentInvocationService`，由 facade 包装 spawn/sync 生命周期 | 推荐，改动局部，保持工具模型 |
| 方案 2 | `AgentExecutionService` 识别 `spawn_sub_agent`，直接调用 `ISubAgentInvocationService` | 不推荐，会把工具特例塞回执行引擎 |

选择：**方案 1**。

目标结构：

```text
AgentExecutionService
  -> IToolInvocationService
    -> SkillRuntime
      -> SubAgentTool
        -> ISubAgentInvocationService
          -> ISubAgentManager
```

这样主执行引擎仍只知道 tool facade，而子代理生命周期也确实经过 sub-agent facade。

### ADR-027-D：LLM profile resolver 必须接入配置源

**决定**：`ILlmProfileResolver` 不再只是日志和 `ModelId` 透传。它必须根据 provider/profile/role 解析到完整 `LlmConfig`。

解析输入：

```csharp
public sealed record LlmInvocationProfile
{
    public required string ProviderId { get; init; }
    public required string ProfileId { get; init; }
    public required string ModelId { get; init; }
    public string Role { get; init; } = "conscious";
}
```

解析输出：

```csharp
public sealed record ResolvedLlmInvocationProfile
{
    public required string ProviderId { get; init; }
    public required string ProfileId { get; init; }
    public required string ModelId { get; init; }
    public required string Role { get; init; }
    public required LlmConfig Config { get; init; }
    public IReadOnlyDictionary<string, string> Metadata { get; init; } = new Dictionary<string, string>();
}
```

Resolver 契约调整：

```csharp
public interface ILlmProfileResolver
{
    Task<ResolvedLlmInvocationProfile> ResolveAsync(
        string workspaceId,
        string agentInstanceId,
        LlmInvocationProfile profile,
        CancellationToken ct = default);
}
```

过渡期实现可以依赖现有 `DbLlmResolver` 或 JSON LLM 配置服务，但必须满足：

- provider/profile/role 参与选择。
- `ModelId` override 必须记录到 metadata。
- 找不到 provider/profile 时返回明确错误，不得静默降级到 `"default"`。
- `LlmInvocationService` 记录 resolved provider/profile/model/role 到结果和 timeline metadata。

### ADR-027-E：fallback 只能作为测试兼容，不作为 done 口径

**决定**：生产 DI 已注册 facade 不等于源码闭环完成。保留 fallback 时必须满足：

- fallback 分支必须标记 `[Obsolete]` 不适用于局部语句，因此用注释和日志双标记 `ADR-027 legacy fallback for tests only`。
- 对应 facade 缺失时必须产生 warning。
- `AgentExecutionService` 的生产注册必须覆盖所有 facade。
- QA 结论中不得写“所有外部动作经 facade”，只能写“生产 DI 默认路径经 facade，legacy fallback 仍保留”。

### Phase 4：子代理 facade 接入到 `SubAgentTool`（P0）

目标：子代理生命周期必须经过 `ISubAgentInvocationService`。

修改文件：

```text
Source/PuddingRuntime/Services/Skills/SubAgentTool.cs
Source/PuddingRuntime/Services/SubAgentInvocationService.cs
Source/PuddingCore/Runtime/SubAgentInvocationContracts.cs
Source/PuddingRuntime/DependencyInjection.cs
Source/PuddingCoreTests/Runtime/RuntimeContractTests.cs
```

契约补强：

```csharp
public sealed record SubAgentInvocationRequest
{
    public required string ParentSessionId { get; init; }
    public required string ParentAgentId { get; init; }
    public required string WorkspaceId { get; init; }
    public required string Task { get; init; }
    public required string TemplateId { get; init; }
    public required LlmInvocationProfile LlmProfile { get; init; }
    public required LlmConfig LlmConfig { get; init; }
    public int? MaxRounds { get; init; }
    public CapabilityPolicy? CapabilityPolicy { get; init; }
    public bool IsAsync { get; init; } = true;
    public RuntimeTraceContext? Trace { get; init; }
}
```

`SubAgentTool` 改造：

```csharp
var invocation = serviceProvider.GetService<ISubAgentInvocationService>();
if (invocation is null)
    return new SkillResult { Success = false, Error = "Sub-agent invocation service is not registered", ExitCode = 1 };

var spawnResult = await invocation.InvokeAsync(new SubAgentInvocationRequest
{
    ParentSessionId = request.SessionId,
    ParentAgentId = request.AgentInstanceId,
    WorkspaceId = request.WorkspaceId ?? "",
    Task = task!,
    TemplateId = template.TemplateId,
    LlmProfile = childLlmRoute.Profile,
    LlmConfig = childLlmRoute.Config,
    MaxRounds = 10,
    CapabilityPolicy = childCapability,
    IsAsync = true,
}, ct);
```

`ILlmResolver` 是子代理 LLM Provider/Model 选择的唯一解析边界，
`SubAgentTool` 是子代理调用语义的组装边界：

1. Smart 工具只从父 Agent manifest 读取角色字段，并传入 `providerId/modelId`。
2. `ILlmResolver.ResolveRouteAsync()` 通过 `ILlmConfigService` 从
   `data/config/llm.providers.json` 原子解析出 `ProviderId + ModelId + LlmConfig`。
   默认路由必须由 `ILlmConfigService.GetDefaultProfile()` 显式返回身份，禁止按 model/endpoint/key 反推 Provider。
3. `SubAgentTool.ResolveChildLlmRouteAsync()` 只补充
   `ProfileId=subagent.conscious`、`Role=conscious`，形成 `LlmInvocationProfile`。
4. `SubAgentInvocationService` 只映射为 `SubAgentSpawnRequest`，不得重新解析或反推 Provider。
5. `SubAgentManager` 必须在写状态、事件或运行归档前校验 Provider/Profile/Model 非空，
   且 `LlmProfile.ModelId == LlmConfig.ModelId`。
6. `SubAgentSpawnRequest` 不再保留第三份冗余 `ModelId`；诊断统一读取
   `LlmProfile.ModelId`。

验收：

```powershell
rg "subAgentManager\\.SpawnAsync|ISubAgentInvocationService" Source\PuddingRuntime\Services\Skills\SubAgentTool.cs -n
```

期望：

- `SubAgentTool` 不再直接调用 `subAgentManager.SpawnAsync`。
- `SubAgentTool` 调用 `ISubAgentInvocationService.InvokeAsync`。
- `SubAgentInvocationRequest`、`SubAgentBatchInvocationRequest`、`SubAgentSpawnRequest`
  的 `LlmProfile` 与 `LlmConfig` 均为 `required`。
- `SubAgentManager.DispatchChildAgentImpl` 原样把两者写入 `RuntimeDispatchRequest`。
- 空路由或 Profile/Config 模型错配在 Manager 产生任何副作用前失败。

## 5. 完成标准

ADR-027 只有在以下条件全部满足时才可标记为 `implemented`：

1. ADR-026 状态不再误标为 implemented，除非 ADR-027 全部通过。
2. E2E 对 `traceId`、evidence API、timeline 组件都是强断言。
3. E2E 使用 `/api/diagnostics/e2e/evidence/{traceId}`。
4. 后端 done 或 metadata frame 稳定提供 traceId，或者前端有强制反查 trace 的实现。
5. `SubAgentTool` 不再直接调用 `ISubAgentManager.SpawnAsync`。
6. LLM resolver 返回完整 resolved profile，provider/profile/role 参与解析。
7. `AgentExecutionService` 直连 fallback 全部标记为 test-only，未标记直连调用为 0。
8. 后端 build、核心 tests、diagnostics TS filter、E2E TS 全部通过。
9. Playwright evidence test 在运行中服务上通过，QA 报告记录证据。

## 6. 非目标

1. 本 ADR 不要求清理全量 chat TypeScript 历史错误。该问题应另立 UI 类型债务任务。
2. 本 ADR 不要求完成所有多 provider 配置目录治理，但要求 resolver 不再只是 `ModelId` 透传。
3. 本 ADR 不要求删除所有 legacy fallback，但要求 fallback 不再被计入“全部经 facade”的完成口径。
4. 本 ADR 不处理 `external/github.hyfree.GM` 子模块 `.lscache` 卫生项，仅要求 QA 报告继续单独标记。
