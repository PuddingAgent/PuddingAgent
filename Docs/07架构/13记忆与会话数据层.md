## 1. 背景与现状评估

### 1.1 当前状态（基于源码核对）

| 维度 | 现状 | 文件证据 |
|---|---|---|
| 会话历史 | 单表 `ChatMessageEntity`：`SessionId/Role/Content/ThinkingJson/UsageJson/CreatedAt(long ms)` | `Source/PuddingPlatform/Data/Entities/ChatMessageEntity.cs` |
| 工具调用 | 未持久化（仅在 SSE 流中以 step 帧出现，落盘时被丢弃） | `ChatApiController.cs#L693-L730` |
| 记忆引擎 | 全内存：`SessionMemoryStore`/`WorkspaceMemoryStore` 用 `ConcurrentDictionary`，进程重启即丢失 | `Source/PuddingMemoryEngine/SessionMemoryStore.cs` |
| 召回机制 | 文本拼接 + 写回正则 `REMEMBER[tag]: ...`，无语义检索 | `MemoryEngine.cs#L11-L13` |
| 上下文窗口 | 硬编码 `maxMessages = 40`，超出直接截断 | `AgentExecutionService.cs#L1198-L1202` |
| 全文检索 | 无 | — |
| 向量检索 | 无 | — |
| 知识图谱 | 无 | — |
| 数据库 | SQLite + EF Core（`PlatformDbContext`、`ControllerDbContext`），单数据库文件 `data/pudding_platform.db` | `Source/PuddingAgent/Program.cs#L130-L155` |

### 1.2 核心痛点

1. **失忆**：进程重启后 `SessionMemoryStore` 全部丢失；超出 40 条的历史在内存上下文中被丢弃，且数据库里也不存工具调用细节。
2. **不可回放**：`ChatMessageEntity` 只存最终 `Content`，思维链/工具调用/工具结果/子代理转录都不在里面，QA 与审计无据可查。
3. **不可搜索**：日积月累的对话变成"垃圾堆"，找不到 3 天前那次让 Agent 学到东西的对话。
4. **不可关联**：用户在 A 会话告诉 Agent "我喜欢深色主题"，B 会话里 Agent 完全不知道——`Workspace` 级记忆只是个 in-memory hashset，且 LLM 自助"REMEMBER"标记触发率极低。
5. **分支无表达**：Agent 自我纠错（重试一个失败的工具调用）、子代理 fork、压缩边界后的 summary 与原始消息——这些"非线性"对话拓扑被强行平铺为按时间排序的列表。

### 1.3 设计目标

| 目标 | 量化指标 |
|---|---|
| 完整可回放 | 任意 Session 100 条消息（含工具调用/结果/思维链）恢复 < 500ms |
| 关键词搜索 | 10K 消息 FTS5 BM25 搜索 < 200ms |
| 语义检索 | Workspace 级 50K 记忆条目 top-K=10 < 300ms |
| 跨会话关联 | 实体级（人/项目/概念）三元组查询 < 100ms |
| 零运维 | 仍是单 SQLite 文件；扩展按需加载，加载失败有降级路径 |
| 向后兼容 | 现有 `ChatMessageEntity`/`MemoryEntry` 数据可无损迁入新表，旧 API 继续工作直到下游切换 |

## 2. 架构决策

### ADR-013-A：存储引擎选型——继续 SQLite，新增专用 db 文件

**方案对比**

| 方案 | 优点 | 缺点 | 评分 |
|---|---|---|---|
| A. 继续 SQLite，新增 `pudding_memory.db` | 零运维、与现有架构一致、FTS5 内置、WAL 多读单写够用 | 单写并发上限低、向量检索依赖扩展 | ✅ |
| B. SQLite + DuckDB（OLAP 旁路） | 解析查询强 | 双引擎心智负担、跨库 join 麻烦 | ❌ |
| C. 引入 PostgreSQL + pgvector | 成熟、向量原生 | 违反"零外部依赖"产品定位（详见 01 总览） | ❌ |
| D. 嵌入式向量库（Qdrant embedded / Lance） | 向量性能好 | 多个数据文件、备份复杂 | ❌ |

**决定**：方案 A。新增 `data/pudding_memory.db`，与 `pudding_platform.db`、`pudding_controller.db` 同级。**不混入现有 db**，原因：
- FTS5 与 vec0 虚拟表会膨胀主库 page cache；
- 记忆库写入频率高（每条工具调用都落盘），WAL 增长快，分库便于独立备份/清理；
- 失败隔离：扩展加载失败只影响记忆能力，不影响登录、Agent 配置等核心功能。

### ADR-013-B：访问层——EF Core + Dapper 双轨

**方案对比**

| 方案 | 优点 | 缺点 |
|---|---|---|
| A. 全 EF Core | 一致、迁移工具齐 | FTS5/vec0 虚拟表 EF 不直接支持，需大量 `FromSqlRaw` |
| B. 全 Dapper / ADO.NET | 灵活、性能好 | 失去 EF 迁移、模型变更易漂移 |
| C. **EF Core 管 schema + Dapper 管热路径查询** | 各取所长 | 两套心智 |

**决定**：方案 C。
- **EF Core 负责**：常规实体（Session/Message/Memory/Entity/Edge）的 schema 定义、迁移、CRUD。
- **Dapper（或原生 `SqliteCommand`）负责**：FTS5 `MATCH` 查询、vec0 `vss_search` 查询、批量插入、`json_extract` 复杂查询。
- 共用同一 `SqliteConnection`，避免连接池碎片。

### ADR-013-C：向量检索——sqlite-vec 为首选 + 可降级

**关键事实**：`sqlite-vec`（asg017/sqlite-vec）截至 2026-Q2 仍处 **pre-1.0**（issue #221 明确 breaking changes 可能），但已被 LangChain、LlamaIndex 等主流栈采纳，作者活跃。**不能直接绑定为强依赖**。

**决定**：三层降级架构。

```
┌─────────────────────────────────────────┐
│  IVectorIndex  接口                       │
├─────────────────────────────────────────┤
│  1. SqliteVecIndex   (sqlite-vec)         │ ← 默认
│  2. InMemoryFlatIndex (cosine 暴力扫描)    │ ← 降级（< 5K 条仍可用）
│  3. NoopVectorIndex                       │ ← 完全禁用，只走 FTS5
└─────────────────────────────────────────┘
```

启动时按顺序尝试加载 sqlite-vec 扩展，失败则降级，记录 WARN 日志。**Embedding 仍照常计算并存储 `BLOB`**，未来扩展可用即可重建索引。

**Embedding 模型**：复用现有 `IRuntimeLlmClient` 的 embeddings 端点（OpenAI 兼容 `/v1/embeddings`）。维度配置化，默认 1024（兼容主流 BGE/M3E 模型）。

### ADR-013-D：记忆树模型——Parent-UUID 链 + 三种边类型

**核心模型**：每条 `Message` 有 `MessageId (UUID)`、`ParentId (UUID?)`、`BranchType (enum)`。  
线性对话即 `Parent → Child` 单链；分支/侧链通过同一 `ParentId` 下多个 `Child` 表达。

| BranchType | 语义 | 谁产生 |
|---|---|---|
| `MAIN` | 主干消息（用户/Agent 一问一答） | 默认 |
| `RETRY` | 同一 Parent 下的重试（如工具调用失败重做） | Runtime |
| `FORK` | 用户/系统主动从某历史点分叉新对话 | UI/Agent |
| `SIDECHAIN` | 子代理转录（subagent 内部 N 轮浓缩为 1 个 Result Message 接回主链） | SubAgent |
| `COMPACTION` | 压缩边界标记：N 条原始消息被一个 summary message 覆盖；原消息保留但不进入默认上下文窗口 | MemoryBoundaryService |

**当前会话视图**：从最新 `MAIN` 叶子向上回溯 Parent 链，跳过被 `COMPACTION` 覆盖的范围；其它分支默认隐藏，UI 可显式展开。

### ADR-013-E：知识图谱——三元组 + 类型化指针

**模型**：实体-关系-实体（SPO 三元组）+ 实体到消息的"出处指针"。

```
Entity (id, type, name, normalized_name, aliases_json, embedding)
Edge   (subject_id, predicate, object_id, weight, evidence_msg_id, valid_from, valid_until)
Pointer(entity_id, target_type, target_id)   -- target: message / memory / file / url
```

- `predicate` 为开放词汇（如 `prefers`, `works_on`, `mentioned_in`），避免一开始过度建模。  
- `evidence_msg_id` 强制每条 edge 必须能溯源到某条消息——这是"可信度"基础。  
- `valid_from/valid_until` 表达时效性（"用户三个月前喜欢深色主题，上周改回浅色"）。

**抽取触发**：不在主链路同步抽取（怕拖慢回复）。改为：消息落盘后投递到 `MemoryExtractionQueue`（进程内 `Channel<T>`），后台 worker 调用 LLM 做抽取，写入 Edge 表。失败重试 3 次，最终失败入死信表。

### ADR-013-F：迁移策略——双写期 + 影子表 + 一次性回填

不做"停机大切换"，分三步：

1. **双写期**（1 个迭代）：新代码同时写入 `ChatMessages`（旧表，保留）和 `Messages`（新表）。读路径仍走旧表。
2. **回填期**：脚本将旧 `ChatMessages` 全量回填到 `Messages`，建立 `MessageId` 映射；同时建立 FTS5 索引、计算 embedding。
3. **切换期**：读路径切到新表，验证 1 周；旧表标记 `[Obsolete]`，下个版本删除。

`MemoryEntry`（内存）→ `Memories`（新表）：直接迁移当前活跃 Workspace，旧 Session 级条目随进程重启自然消失（不值得迁）。

## 3. SQLite Schema 设计

> Schema 写在 `Source/PuddingMemoryEngine/Schema/*.sql`，由启动时的 `MemoryDbInitializer` 幂等执行（`CREATE TABLE IF NOT EXISTS` + `PRAGMA user_version` 管理迁移）。EF Core 仅管理 `Sessions/Messages/Memories/Entities/Edges` 等"非虚拟表"实体；FTS5/vec0 虚拟表用原始 SQL。

### 3.1 会话与消息

```sql
-- 会话主表
CREATE TABLE Sessions (
    SessionId       TEXT PRIMARY KEY,            -- UUID
    WorkspaceId     TEXT NOT NULL,
    AgentId         TEXT NOT NULL,
    Title           TEXT,                        -- 由首条消息生成或用户设置
    Mode            TEXT NOT NULL DEFAULT 'chat',-- chat / task / workflow
    Status          TEXT NOT NULL DEFAULT 'active', -- active / archived / deleted
    Tags            TEXT,                        -- JSON array
    CreatedAt       INTEGER NOT NULL,            -- Unix ms
    LastActivityAt  INTEGER NOT NULL,
    MessageCount    INTEGER NOT NULL DEFAULT 0,
    TokenTotal      INTEGER NOT NULL DEFAULT 0,
    Metadata        TEXT                         -- JSON 扩展位
);
CREATE INDEX IX_Sessions_Workspace_LastActivity ON Sessions(WorkspaceId, LastActivityAt DESC);
CREATE INDEX IX_Sessions_Status ON Sessions(Status, LastActivityAt DESC);

-- 消息主表（含工具调用、思维链、子代理）
CREATE TABLE Messages (
    MessageId       TEXT PRIMARY KEY,            -- UUID
    SessionId       TEXT NOT NULL REFERENCES Sessions(SessionId) ON DELETE CASCADE,
    ParentId        TEXT REFERENCES Messages(MessageId),
    BranchType      TEXT NOT NULL DEFAULT 'MAIN',-- MAIN/RETRY/FORK/SIDECHAIN/COMPACTION
    Sequence        INTEGER NOT NULL,            -- 同 Session 内严格自增（含分支）
    Role            TEXT NOT NULL,               -- user/assistant/system/tool/subagent
    ContentType     TEXT NOT NULL DEFAULT 'text',-- text/tool_call/tool_result/thinking/summary/multimodal
    Content         TEXT,                        -- 主内容（文本/JSON）
    ToolCallsJson   TEXT,                        -- assistant 发起的工具调用列表
    ToolResultJson  TEXT,                        -- tool 角色的执行结果
    ThinkingJson    TEXT,                        -- 思维链片段
    AttachmentsJson TEXT,                        -- 多模态资源指针：[{type, url, sha256, ...}]
    UsageJson       TEXT,                        -- token 用量
    ModelId         TEXT,                        -- 哪个模型产生（assistant 消息）
    AgentId         TEXT,                        -- 哪个 Agent（子代理时与主 Agent 不同）
    Source          TEXT,                        -- agentInstanceId / "system" / "user"
    CompactedBy     TEXT REFERENCES Messages(MessageId), -- 被哪条 summary 覆盖（NULL 表示在窗口内）
    CreatedAt       INTEGER NOT NULL,
    Metadata        TEXT
);
CREATE INDEX IX_Messages_Session_Seq ON Messages(SessionId, Sequence);
CREATE INDEX IX_Messages_Parent ON Messages(ParentId);
CREATE INDEX IX_Messages_Session_Branch_Seq ON Messages(SessionId, BranchType, Sequence);
CREATE INDEX IX_Messages_CompactedBy ON Messages(CompactedBy) WHERE CompactedBy IS NOT NULL;

-- FTS5 全文索引（external content 模式，不重复存内容）
CREATE VIRTUAL TABLE Messages_fts USING fts5(
    Content,
    content='Messages',
    content_rowid='rowid',
    tokenize='unicode61 remove_diacritics 2'   -- 中文回退到 trigram，见 §3.5
);

-- 触发器：保持 fts 与主表同步
CREATE TRIGGER trg_Messages_ai AFTER INSERT ON Messages BEGIN
  INSERT INTO Messages_fts(rowid, Content) VALUES (new.rowid, new.Content);
END;
CREATE TRIGGER trg_Messages_ad AFTER DELETE ON Messages BEGIN
  INSERT INTO Messages_fts(Messages_fts, rowid, Content) VALUES('delete', old.rowid, old.Content);
END;
CREATE TRIGGER trg_Messages_au AFTER UPDATE ON Messages BEGIN
  INSERT INTO Messages_fts(Messages_fts, rowid, Content) VALUES('delete', old.rowid, old.Content);
  INSERT INTO Messages_fts(rowid, Content) VALUES (new.rowid, new.Content);
END;
```

### 3.2 长期记忆（提纯后的事实）

```sql
CREATE TABLE Memories (
    MemoryId        TEXT PRIMARY KEY,            -- UUID
    Scope           TEXT NOT NULL,               -- session / workspace / global

    WorkspaceId     TEXT,                        -- Scope=workspace 必填
    AgentId         TEXT,                        -- 谁的记忆

    Content         TEXT NOT NULL,
    Importance      REAL NOT NULL DEFAULT 0.5,   -- 0~1，用于排序与淘汰

CREATE INDEX IX_Memories_Tag ON Memories(WorkspaceId, Tag);

    content_rowid='rowid',
```

### 3.4 知识图谱

```sql
CREATE TABLE Entities (
    EntityId        TEXT PRIMARY KEY,            -- UUID
    WorkspaceId     TEXT NOT NULL,
    Type            TEXT NOT NULL,               -- person/project/concept/file/url/...
    Name            TEXT NOT NULL,
    NormalizedName  TEXT NOT NULL,               -- lowercase + 去标点，用于去重
    AliasesJson     TEXT,                        -- ["小明","Ming"]
    Description     TEXT,
    CreatedAt       INTEGER NOT NULL,
    UpdatedAt       INTEGER NOT NULL,
    Metadata        TEXT,
    UNIQUE (WorkspaceId, Type, NormalizedName)
);
CREATE INDEX IX_Entities_Workspace_Type ON Entities(WorkspaceId, Type);

CREATE TABLE Edges (
    EdgeId          TEXT PRIMARY KEY,
    WorkspaceId     TEXT NOT NULL,
    SubjectId       TEXT NOT NULL REFERENCES Entities(EntityId) ON DELETE CASCADE,
    Predicate       TEXT NOT NULL,
    ObjectId        TEXT NOT NULL REFERENCES Entities(EntityId) ON DELETE CASCADE,
    Weight          REAL NOT NULL DEFAULT 1.0,
    Confidence      REAL NOT NULL DEFAULT 0.8,
    EvidenceMsgId   TEXT REFERENCES Messages(MessageId),
    ValidFrom       INTEGER NOT NULL,
    ValidUntil      INTEGER,                     -- NULL = 持续有效
    CreatedAt       INTEGER NOT NULL,
    Metadata        TEXT
);
CREATE INDEX IX_Edges_Subject ON Edges(SubjectId, Predicate);
CREATE INDEX IX_Edges_Object  ON Edges(ObjectId, Predicate);
CREATE INDEX IX_Edges_Live    ON Edges(WorkspaceId) WHERE ValidUntil IS NULL;

-- 通用指针表：实体 → 任意目标（消息/记忆/文件/URL）
CREATE TABLE EntityPointers (
    EntityId        TEXT NOT NULL REFERENCES Entities(EntityId) ON DELETE CASCADE,
    TargetType      TEXT NOT NULL,               -- message/memory/file/url/entity
    TargetId        TEXT NOT NULL,
    Relation        TEXT NOT NULL DEFAULT 'mentioned_in',
    CreatedAt       INTEGER NOT NULL,
    PRIMARY KEY (EntityId, TargetType, TargetId, Relation)
);
CREATE INDEX IX_Pointers_Target ON EntityPointers(TargetType, TargetId);
```

### 3.5 中文分词与 FTS5

`unicode61` tokenizer 默认按空格拆词，对中文不友好。三选一：

| 方案 | 说明 | 选择理由 |
|---|---|---|
| A. **trigram 兜底**（`tokenize='trigram'`）| SQLite 3.34+ 内置，无依赖 | ✅ 默认采用，零依赖 |
| B. simple-jieba 扩展 | 中文准确率高 | ❌ 需自编译 |
| C. 应用层预分词（jieba.NET）写入 `ContentTokenized` | 控制力强 | 备选，若 trigram 召回不足时启用 |

**决定**：默认 trigram；保留 `ContentTokenized` 字段（NULL 即可），未来切换零迁移。

## 5. 记忆树工作流示例

### 5.1 线性对话
```
M1(user)─►M2(assistant)─►M3(user)─►M4(assistant) ...   全部 BranchType=MAIN
```

### 5.2 工具调用重试
```
M5(assistant, ToolCalls=[search])──►M6(tool, error)
                                ↓ RETRY
                                M6'(tool, success)─►M7(assistant)
```
读视图：默认取最新 RETRY 链（M6'），M6 仍可在 UI"展开失败尝试"中查看。

### 5.3 子代理侧链
```
主链:  ...M10(assistant)──►M11(SIDECHAIN, Role=subagent, summary)──►M12...
        子代理内部 N 轮全部存储，但 ParentId 链独立挂在 M10 下，
        BranchType=SIDECHAIN，主链上下文构建时跳过子链原始消息，
        只取 M11 的 summary。
```

### 5.4 压缩边界
```
M1..M30 ──►M31(COMPACTION, summary="前 30 条要点...")──►M32...
M1..M30 的 CompactedBy = M31
默认上下文窗口构建：跳过 CompactedBy IS NOT NULL 的消息，但保留 M31 summary。
搜索/审计仍可看到 M1..M30 完整内容。
```

## 6. 知识图谱抽取流程

```
Message saved
     │
     ▼
MemoryExtractionQueue (in-process Channel<MessageId>)
     │
     ▼
背景 Worker (Hosted Service)
     │
     ├─► LLM 抽取 prompt：列出实体/关系/事实
     │     输入: 最近 N 条消息文本
     │     输出: { entities:[...], edges:[...], memories:[...] }
     │
     ├─► 实体归一化（NormalizedName 查找/合并）
     ├─► 写 Edges（带 EvidenceMsgId、Confidence）
     ├─► 写 Memories（用户偏好/事实/关键决策）
     └─► 计算 embedding 写 Vectors / EmbeddingsRaw

失败重试 3 次 → 死信表 ExtractionDeadLetter
```

## 7. 检索 API 设计（Runtime 注入上下文）

```csharp
public interface IMemoryRetriever
{
    // FTS5 关键词检索
    Task<IReadOnlyList<MemoryHit>> SearchKeywordAsync(
        string workspaceId, string query, int topK = 10, CancellationToken ct = default);

    // 向量语义检索
    Task<IReadOnlyList<MemoryHit>> SearchVectorAsync(
        string workspaceId, ReadOnlyMemory<float> embedding, int topK = 10, CancellationToken ct = default);

    // 混合检索（RRF 融合）—— 默认入口
    Task<IReadOnlyList<MemoryHit>> SearchHybridAsync(
        string workspaceId, string query, int topK = 10, CancellationToken ct = default);

    // 实体邻域查询
    Task<EntityNeighborhood> GetEntityNeighborhoodAsync(
        string workspaceId, string entityId, int hops = 1, CancellationToken ct = default);
}
```

`AgentExecutionService.BuildSystemPrompt` 在构建 `MEMORY` 层时调用 `SearchHybridAsync`，取 top-K 注入。

## 11. 未决问题（Open Questions）

1. **Embedding 模型的"漂移"**：用户切换 LLM 提供商后，新老 embedding 不可比。**初步策略**：`EmbeddingsRaw` 记录 `ModelId`，跨模型查询时对老向量置信度衰减；后台触发重算。
2. **多模态消息的存储**：图片/音频是否入 SQLite BLOB？**初步策略**：BLOB 仅存 ≤256KB 的小资源；大文件走 `MinioStorageService`，库内只存 SHA256 + URL。
3. **图谱删除语义**：用户删除某条消息时，证据来自该消息的 Edges 如何处理？**初步策略**：Edge 软删（置 `ValidUntil=now`），保留审计；UI 显示"证据已撤回"。
4. **跨 Workspace 的全局记忆**：是否支持"我"这个用户跨所有 Workspace 的偏好？**初步策略**：`Memories.Scope='global'` 预留字段，Phase 3 之后再启用。
5. **sqlite-vec 1.0 时间表**：若到 Phase 2 仍 pre-1.0，是否考虑替代（Lance / Annoy Native）？**触发条件**：v0.x 出现严重 breaking 时，由 `@architect` 重新评估。

## 附录 B：术语表

| 术语 | 定义 |
|---|---|
| 完整会话历史 | 包含工具调用/结果/思维链的原始消息流，未经压缩或提纯 |
| 提纯记忆 (Memory) | 从会话中抽取出的事实/偏好/决策，可独立于原对话存在 |
| 记忆树 | 由 ParentId 链组织的非线性消息 DAG |
| 压缩边界 | 一组消息被一条 summary 覆盖的标记点 |
| 侧链 (sidechain) | 子代理内部 N 轮对话，对外只暴露 summary |
| 三元组 (SPO) | Subject-Predicate-Object 知识表达单元 |
| 出处指针 | 实体到任意目标对象（消息/文件/URL）的可寻址引用 |
