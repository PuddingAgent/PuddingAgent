# 26 ADR-025 验收阻塞修复与执行闭环方案

> 状态：**implemented**
> 日期：2026-05-20
> 范围：ADR-025 二次验收阻塞修复、执行引擎 facade 闭环、诊断 DTO 契约、E2E evidence 闭环
> 前置：[25运行时契约闭环与可观测性验收基线ADR](25运行时契约闭环与可观测性验收基线ADR.md)、[24核心架构组件边界与执行引擎拆分ADR](24核心架构组件边界与执行引擎拆分ADR.md)、[23运行时可观测性闭环与E2E验证基线ADR](23运行时可观测性闭环与E2E验证基线ADR.md)

## 1. 背景

ADR-025 已完成方向定义，但二次验收发现实现仍未闭环：

1. `PuddingPlatformAdmin` 全量 `npx tsc --noEmit` 失败，且 `diagnostics` 模块存在本轮相关错误。
2. `AgentExecutionService` 仍直接调用 `_llmClient.ChatStreamAsync`、`_skillRuntime.InvokeAsync`、`_ssm.AppendAsync`，未完成 facade 接入。
3. `LlmInvocationProfile` 只是进入请求结构，provider/profile/role 尚未进入真实解析链路。
4. `getSessionTimeline` 前端声明返回分页结果，后端实际返回数组。
5. Debug API 仍缺少稳定写入路径，Playwright 仍不是 evidence test。

本 ADR 不新增架构方向，只定义 ADR-025 的关闭条件、修改步骤、接口契约和验收基线。

---

## 2. 决策

### ADR-026-A：先修可编译契约，再做架构闭环

**决定**：所有后续架构修复必须先恢复可重复验证基线。

顺序固定为：

```text
diagnostics TS compile
  -> DTO contract tests
  -> execution facade integration
  -> LLM profile resolution
  -> debug/evidence E2E
  -> QA closeout
```

原因：

- 当前 diagnostics 页面已经出现类型错误，如果继续接入后端能力，会扩大不确定性。
- DTO contract test 是防止前后端再次漂移的最小防线。
- 执行引擎 facade 接入会触碰核心路径，必须在可验证基线上推进。

### ADR-026-B：执行引擎只能依赖 runtime facade 做外部动作

**决定**：`AgentExecutionService` 仍保留流程编排职责，但所有外部动作必须通过以下 facade：

```csharp
public sealed class AgentExecutionService
{
    private readonly IContextAssemblyService _contextAssembly;
    private readonly ILlmInvocationService _llmInvocation;
    private readonly IToolInvocationService _toolInvocation;
    private readonly ISubAgentInvocationService _subAgentInvocation;
    private readonly ISessionOutputWriter _sessionOutput;
}
```

允许保留 fallback 的条件：

- fallback 只能用于 facade 未注册的兼容期。
- fallback 必须有结构化 warning 日志。
- fallback 必须在 ADR-026 完成后删除或转为测试专用路径。

禁止新增：

- 新的 `_skillRuntime.InvokeAsync` 调用点。
- 新的 `_llmClient.ChatAsync` / `_llmClient.ChatStreamAsync` 调用点。
- 新的 `_ssm.AppendAsync` / `Append()` 直接写会话帧调用点。
- 新的 `_subAgentManager.SpawnAsync` 直接调用点。

### ADR-026-C：LLM Profile 必须成为解析输入，不只是日志字段

**决定**：`LlmInvocationProfile` 必须表达并传递以下语义：

```csharp
public sealed record LlmInvocationProfile
{
    public required string ProviderId { get; init; }
    public required string ProfileId { get; init; }
    public required string ModelId { get; init; }
    public string Role { get; init; } = "conscious";
}
```

新增解析契约：

```csharp
public interface ILlmProfileResolver
{
    Task<LlmConfig> ResolveAsync(
        string workspaceId,
        string agentInstanceId,
        LlmInvocationProfile profile,
        CancellationToken ct = default);
}
```

规则：

- `ProviderId` 决定服务商。
- `ProfileId` 决定该 agent 或 workspace 下的 profile。
- `ModelId` 是 profile 解析后的最终模型，可被请求显式覆盖，但必须记录 override。
- `Role` 决定显意识 LLM 或潜意识 LLM，不允许用字符串默认值绕过配置。

过渡实现：

- 第一阶段可把 `direct` 作为 legacy provider。
- `conscious.default` 可作为默认 profile。
- 但 `LlmInvocationService` 不得只把 `ModelId` 写入 `LlmConfig` 后绕过 provider/profile。

### ADR-026-D：诊断 API DTO 以序列化契约为准

**决定**：诊断 DTO 以后端序列化输出和前端 TypeScript 类型共同定义，必须有测试覆盖。

必须覆盖的 DTO：

```text
EventStatsDto
SubAgentRunSummaryDto
SubAgentRunDetailDto
RuntimeTimelineItemDto
RuntimeTimelineQueryDto
PagedRuntimeTimelineResultDto
```

建议新增后端 DTO：

```csharp
public sealed record EventStatsDto
{
    public required int TotalCount { get; init; }
    public required IReadOnlyList<EventStatusCountDto> ByStatus { get; init; }
    public required IReadOnlyList<EventComponentCountDto> ByComponent { get; init; }
}

public sealed record EventStatusCountDto
{
    public required string Status { get; init; }
    public required int Count { get; init; }
}

public sealed record EventComponentCountDto
{
    public required string Component { get; init; }
    public required int Count { get; init; }
}
```

前端对应类型：

```ts
export type EventStats = {
  totalCount: number;
  byStatus: { status: string; count: number }[];
  byComponent: { component: string; count: number }[];
};
```

Timeline 规则：

- `/api/diagnostics/runtime/timeline` 返回 `PagedTimelineResult`。
- `/api/diagnostics/sessions/{sessionId}/timeline` 要么改为返回 `RuntimeTimelineItem[]`，前端同步改类型；要么后端改为分页结果。不得继续前后端不一致。
- 默认 UI 展示执行顺序用 `sortOrder=asc`。
- 排查最近问题用 `sortOrder=desc`。

### ADR-026-E：E2E 必须验证 evidence，不只是 smoke

**决定**：Playwright E2E 的验收目标从“页面能发送消息”升级为“证据链可追踪”。

Debug API 契约：

```ts
export type PuddingDebugApi = {
  getSessionState(sessionId: string): unknown | null;
  getLastTraceId(): string | null;
  getLastSessionId(): string | null;
  getLastMessageId(): string | null;
  clearDebugEvents(): void;
};
```

写入规则：

- `sendChatMessage` 成功返回后写入 `pudding_last_session_id`。
- 收到 metadata/done frame 后写入 `pudding_last_trace_id` 和 `pudding_last_message_id`。
- 写入逻辑只在 `?debug=1` 或显式 debug config 下启用。

E2E 断言：

```text
1. page.goto('/?debug=1')
2. 登录
3. 发送消息
4. 等待 done frame 或最终 assistant message
5. window.__PUDDING_DEBUG__.getLastSessionId() 非空
6. window.__PUDDING_DEBUG__.getLastTraceId() 非空
7. GET /api/diagnostics/runtime/evidence/{traceId} 返回 200
8. evidence.timeline 包含 agent_execution、context_pipeline、llm_gateway 或 legacy direct_llm 映射
9. timeline 按 StartedAtUtc 正序可解释
```

## 4. 接口契约清单

### 4.1 执行引擎对外调用契约

```csharp
public interface ILlmInvocationService
{
    Task<LlmInvocationResult> InvokeAsync(LlmInvocationRequest request, CancellationToken ct = default);

{

{

{
```

## 5. 验收标准

ADR-026 只有在以下条件全部满足时才可标记为 implemented：

1. `PuddingPlatformAdmin` 全量 `npx tsc --noEmit --pretty false` 不再出现 diagnostics 相关错误。
2. 诊断 DTO 有后端序列化 contract tests。
3. `AgentExecutionService` 的 LLM streaming、tool、sub-agent、session-output 生产路径全部经 facade。
4. LLM 调用不再把 provider/profile/model/role 降级成单个 `ModelId`。
5. Debug API 有写入路径，Playwright 能拿到真实 sessionId/traceId。
6. E2E 调用 evidence API 并断言关键组件链路。
7. QA 报告记录完整命令、结果、残余风险。
8. `external/github.hyfree.GM` 子模块内临时 `.lscache` 文件清理或记录为独立卫生项，不混入 ADR-026 交付。

## 6. 残余风险

1. 全量前端 TypeScript 当前仍有大量 chat 模块历史错误。ADR-026 第一阶段只要求 diagnostics 不再新增错误；是否清理全量前端类型债务应单独立项。
2. 多 provider/profile 的最终配置来源仍依赖后续配置目录治理任务。ADR-026 只要求执行链路不再混淆 provider/profile/model。
3. 如果真实 LLM 不适合 E2E，必须使用 Fake LLM provider，但 evidence 链路仍必须完整。
4. Facade 接入会触碰核心执行路径，必须小步提交，每完成一个 facade 接入就运行后端测试和一次手工/自动对话验证。
