# 总览与分层

## 产品定位

Pudding Agent Network 不是单个智能体应用，而是一个以 Workspace 为治理边界、以 Controller + Runtime 为核心、以多渠道和多 Agent 协作为目标的 Agent OS 与协作网络平台。

它的设计目标不是继续堆高单 Agent 的“思考能力”，而是把 AI 系统工程里的四个难点做成基础设施：

- workflow orchestration
- memory isolation
- event-driven runtime
- distributed collaboration

因此，Pudding 更接近一个 **LLM orchestration kernel**，而不是一个“更会聊天的 Agent 应用”。

## 核心设计动机

- 真实任务更像任务图（Task Graph），而不是自由散漫的思考链，因此 Workflow 必须是一等能力。
- 企业场景中的记忆污染往往比数据库污染更难排查，因此 Workspace 必须成为完整命名空间。
- 心跳轮询会浪费 token、增加延迟，因此 Agent 必须能被事件直接唤醒。
- 真实企业更关心 ERP、GitHub、设备、Webhook、MQTT 如何接入，而不是让 Agent 模拟点网页。
- 多智能体要解决的是协同协议问题，而不只是“多开几个对话窗口”。

## 顶层分层

- PuddingCore：无状态抽象、协议、通用模型。
- PuddingRuntime：执行面，负责 Session、Agent 实例、记忆处理、技能运行与受限执行。
- PuddingController：控制面，负责接入、路由、鉴权、审批、调度、审计与治理。
- PuddingGateway：Controller 内的边界接入模块。
- PuddingPlatform：上层平台层，负责业务语义、产品能力组合与平台编排。
- PuddingAgent：Agent 模板、画像、默认能力与策略定义。
- PuddingCLI / PuddingWeb / PuddingAvalonia：客户端与操作者入口。

## 事件驱动基线

- 平台采用统一事件总线作为控制面与执行面的协作骨架，内部事件总线建议基于 RabbitMQ。
- 事件既支持全局域，也支持 Workspace 域；默认优先在 Workspace 域内隔离、路由和消费。
- 外部世界通过 Gateway Adapter 接入后，统一转换为标准化事件，再交由 Controller 和 Runtime 继续处理。
- 内部世界的任务完成、Agent 完成任务、心跳、告警、审批结果、记忆候选写回、工具调用结果等，也统一视为事件。
- Agent 不应只靠轮询感知变化；在 Runtime 支持下，Agent 可以订阅自己感兴趣的事件，并在命中时被直接唤醒。

## MVP 三核心模块

- `Agent Runtime`：执行与唤醒的最小宿主。
- `Event Bus`：事件驱动、订阅、死信、重放与唤醒的基础骨架。
- `Workspace Memory`：记忆、Agent、工具、工作流和事件域的隔离命名空间。

## 关键边界

- Session 权威在 Runtime。
- 路由、权限、审批、审计权威在 Controller。
- Platform 不直接替代控制面，而是组织整体产品能力。
- Agent 模板归属 Workspace，Agent 实例归属 Session。
- 公开或低信任输入默认不能直接污染长期记忆。
- “万物皆事件”是系统建模原则，但事件是否可见、可订阅、可触发动作，仍受 Workspace 策略、权限与审批约束。

## Controller / Runtime / Agent 分工摘要

为了避免后续设计再次混杂，三层分工应长期保持稳定：

- **Controller**：控制权威。负责路由、权限、审批、审计、事件治理、Workflow 控制、Adapter 治理与 Runtime 节点管理。
- **Runtime**：执行权威。负责 Session 热状态、Agent 执行、事件命中后的唤醒、技能/MCP/记忆/沙箱装配与运行态承载。
- **Agent**：定义权威。负责声明“这个 Agent 是谁、默认具备什么能力、遵循什么角色策略”，而不直接承载执行态。

一个简单判断标准是：

- 如果它决定 **能不能做**，大概率属于 Controller。
- 如果它决定 **现在怎么做**，大概率属于 Runtime。
- 如果它决定 **应该以什么角色和默认策略去做**，大概率属于 Agent。
