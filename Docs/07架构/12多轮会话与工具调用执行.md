## Tool 接口设计：fail-closed 默认值

借鉴 Claude Code 的工具接口设计哲学——默认值是保守的，不声明就意味着不安全：

| 属性 | 默认值 | 含义 |
|------|--------|------|
| `IsEnabled` | `true` | 默认启用 |
| `IsReadOnly` | **`false`** | 默认需要写权限 → 需用户确认 |
| `IsConcurrencySafe` | **`false`** | 默认不安全 → 串行执行 |
| `IsDestructive` | **`false`** | 默认非破坏性 |

核心原则：**忘记声明 = 更严格的安全约束**。一个工具忘记声明 `IsReadOnly = true`，它就会被当作写操作处理，触发权限检查——这是 fail-closed（安全侧关闭）的设计。

### BuildTool() 工厂模式

```csharp
// 工具默认值（Pudding 对应 Claude Code 的 TOOL_DEFAULTS）
public static class ToolDefaults
{
    public const bool IsEnabled = true;
    public const bool IsReadOnly = false;        // fail-closed: 默认需权限
    public const bool IsConcurrencySafe = false; // fail-closed: 默认串行
    public const bool IsDestructive = false;
}

// BuildTool 工厂：只需声明差异部分
public static IPuddingTool BuildTool(ToolDefinition def)
{
    return new PuddingTool
    {
        Name = def.Name,
        Description = def.Description,
        InputSchema = def.InputSchema,
        IsReadOnly = def.IsReadOnly ?? ToolDefaults.IsReadOnly,
        IsConcurrencySafe = def.IsConcurrencySafe ?? ToolDefaults.IsConcurrencySafe,
        IsDestructive = def.IsDestructive ?? ToolDefaults.IsDestructive,
        // ... 其他属性
    };
}
```

### 三层工具过滤

参考 Claude Code 的工具注册策略：
1. **Feature Flag** — 编译时/启动时决定哪些工具可用
2. **Permission Rules** — 根据 Agent 角色/权限级别过滤
3. **Mode Restrictions** — 根据当前模式（Coordinator/Worker/ReadOnly）进一步限制

## SSE 流式路径（Chat，历史兼容/跨进程 Adapter）

```
Platform ChatApiController (SSE endpoint /api/workspaces/{wsId}/chat/message/stream)
  → PlatformApiClient.ReadSseFramesAsync
    → Controller MessageIngressController (stream)
      → SessionRouter.RouteMessageStreamAsync
        → RuntimeDispatcher.DispatchStreamAsync
          → RuntimeExecuteController.ExecuteStream
            → AgentExecutionService.ExecuteStreamAsync
              → IRuntimeLlmClient.ChatStreamAsync
                → LlmProxyController.ChatStream
                  → OpenAiLlmGateway.ChatStreamAsync (OpenAI SSE)
```

关键设计：
- 每层定义独立 SSE 端点，上游消费 `ServerSentEventFrame` 后 relay 给上一级
- 每层独立管理 `CancellationToken`，用户点击停止 → `AbortController` → 逐级 `OperationCanceledException`
- 帧类型通过 SSE `event:` 字段区分：`delta`（增量文本）、`usage`（Token 统计）、`error`、`done`
- Token 用量以 `TokenUsageDto` 全链路透传，不做中途变换
- 公共 SSE 解析逻辑提取到 `StreamingContracts.SseFrameReader`

上述路径只用于服务跨进程部署或迁移兼容。同进程目标路径不再逐层解析和重编码 SSE，而是 `ChatExecutionWorker → IChatExecutionCoordinator → Runtime → ISessionOutputWriter`；浏览器通过独立 Session SSE 读取已经提交的事件。

## 执行模型（同步路径）

- Runtime 模块控制循环
- LLM 每次只做单轮决策
- 工具在进程内直接执行
- 停止信号由 Controller 模块（同进程）下发

## 与之前设计的唯一区别

同步和流式执行在同进程部署下都不再要求 Controller → Runtime HTTP 调用，统一通过领域接口调度。HTTP/SSE 只作为跨进程 Adapter；客户端实时效果由 Session Event Log 的 replay + live SSE 提供。
