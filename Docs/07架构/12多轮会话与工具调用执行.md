# 12多轮会话与工具调用执行

## 定位

本分册定义 Pudding Agent Network 中单 Agent 在 `PuddingRuntime` 内的多轮会话执行模型、工具调用循环、停止控制、完成判定与 Hook 扩展点。

它解决的问题不是“如何让 LLM 自己循环调用自己”，而是：

- Runtime 如何稳定承载一次任务执行
- LLM 如何只承担单轮决策，而不是接管整个执行控制权
- 工具调用如何被纳入可中断、可审计、可恢复的执行链
- 何时应该继续、等待、完成、失败或被外部停止

一句话概括：

> **多轮执行由 Runtime 驱动，LLM 只负责本轮决策。**

这份分册与 [03PuddingRuntime.md](03PuddingRuntime.md)、[10事件系统与事件总线.md](10事件系统与事件总线.md) 配套阅读。

## 为什么不能把循环控制权交给 LLM

如果把多轮工具调用简单实现为：

```text
用户请求 -> LLM -> 工具 -> LLM -> 工具 -> DONE
```

虽然原型阶段容易做出来，但会很快遇到以下问题：

1. **停止信号不可靠**
	用户点击停止、Workspace 被冻结、Controller 下发取消时，不应等待 LLM 下一轮“理解到自己该停下”。

2. **完成判定不稳定**
	LLM 可能在外部副作用尚未完成、审批尚未通过、子任务尚未回收时错误输出 `DONE`。

3. **工具循环容易卡死**
	同一工具同一参数反复重试、无进展对话重复、长尾网络超时都会放大成死循环。

4. **工具调用与生命周期语义被混在一起**
	“我要执行某个工具”和“这次任务已经完成”不是一回事，不应通过一个松散 JSON 结构混为同级决定。

5. **无法自然接入事件驱动模型**
	Pudding 的目标不是单一 chat loop，而是事件唤醒、审批挂起、恢复执行和工作流推进。执行态必须支持 `WAIT_EVENT`、`WAIT_APPROVAL` 等挂起语义。

因此，Pudding 的基线应当是：

- Runtime 控制循环
- LLM 只做单轮结构化决策
- Controller 负责外部治理动作

## 执行总原则

### 原则 1：Runtime 驱动 loop，LLM 只做 reasoning node

LLM 可以提出：

- 继续执行
- 调用某个工具
- 任务完成
- 等待事件/审批
- 任务失败或需要人工介入

但它不能直接决定：

- 总共循环多少轮
- 是否立即停止
- 是否忽略外部 cancel / freeze
- 是否越过审批链继续高风险动作
- 是否在未满足结束条件时擅自收口

### 原则 2：停止属于控制面动作，不属于模型能力

- `CancelExecutionRequest`
- `FreezeExecutionRequest`
- `ResumeExecutionRequest`

这些都属于 Controller -> Runtime 的治理控制语义，而不是业务 Agent 通过 function call 自己决定的动作。

### 原则 3：完成属于“模型请求 + Runtime 裁决”

Agent 可以提出“任务已完成”，但 Runtime 需要根据执行上下文判断：

- 是否还有未完成副作用
- 是否还有待审批项
- 是否还有子任务、sub_agent 或 Workflow Worker 未回收
- 是否当前处于 stop/freeze/cancel 中断链路

只有在 Runtime 的完成策略判定通过后，才能真正进入 `Completed`。

### 原则 4：同一主执行链默认串行

同一 `Session + AgentExecutionContext` 的主执行链不应并发重入，否则会导致：

- 短期上下文被覆盖
- 工具结果串味
- 恢复锚点失真
- stop / resume 语义失效

sub_agent 可以并发，但必须挂靠到主执行链的预算和因果链路之下。

## 推荐的执行总链路

建议把多轮执行建模为如下链路：

```text
Event / User Request
    ↓
PuddingController 路由 / 鉴权 / Session 定位
    ↓
RuntimeDispatchInbox
    ↓
SessionSupervisor
    ↓
AgentExecutionCoordinator
    ↓
[循环]
  1. Stop / Budget / State 检查
  2. 装配上下文、记忆、能力视图
  3. 调用 LLM 生成本轮决策
  4. Runtime 验证决策结构
  5. 若命中工具调用 -> Tool Executor
  6. 记录 ExecutionJournal / Hook / Event
  7. 判断 CONTINUE / WAIT / DONE / FAIL / CANCELLED
    ↓
Finalize / Suspend / Return
```

关键点：

- LLM 在 loop 内部
- Runtime 在 loop 外部掌控边界
- 工具执行是步骤，不是主控制器

## 运行时内核建议

为了避免把所有执行逻辑继续堆进一个 `ExecuteAsync`，建议在 `PuddingRuntime` 内补齐以下最小内核对象。

### `RuntimeDispatchInbox`

职责：

- 接收 `DispatchExecutionRequest`
- 接收 `DispatchWakeupRequest`
- 接收 `CancelExecutionRequest`
- 接收 `FreezeExecutionRequest`
- 接收 `ResumeExecutionRequest`

它是 Runtime 的控制入口，不直接承载执行细节。

### `SessionSupervisor`

职责：

- 定位或创建 `SessionRuntime`
- 检查目标 `AgentExecutionContext` 是否允许继续执行
- 保证同一主执行链串行
- 维护等待态、冻结态、取消态与恢复态

### `AgentExecutionCoordinator`

职责：

- 驱动一次执行任务的多轮 loop
- 在轮次边界检查 cancel / freeze / timeout / budget
- 调用 LLM
- 调用 Tool Executor
- 生成最终结果、等待态或失败态

### `ExecutionControlRegistry`

职责：

- 保存每次执行的控制状态
- 对外暴露 stop / freeze / cancel 标志
- 为工具调用、流式输出和多轮循环提供统一取消源

### `ExecutionJournal`

职责：

- 记录每轮输入摘要
- 记录模型输出摘要
- 记录工具调用、工具结果、失败原因
- 记录中断、等待、恢复与完成原因
- 生成 ResumeAnchor

### `CompletionPolicy`

职责：

- 判断模型提出的 `DONE` 是否真的可以完成
- 判断当前应该 `Completed`、`WaitingEvent`、`WaitingApproval`、`Failed` 或 `Cancelled`

### `HookDispatcher`

职责：

- 统一调度工具级 Hook 与执行生命周期 Hook
- 保障 Hook 失败不会中断主执行链
- 为审计、观测、外部集成提供稳定扩展点

## 执行状态机建议

建议至少区分以下执行状态：

- `Created`
- `Running`
- `Busy`
- `WaitingTool`
- `WaitingEvent`
- `WaitingApproval`
- `Completed`
- `Cancelled`
- `Failed`
- `Frozen`

语义建议：

### `Running`

表示 Agent 在线可用，可以被投递执行，但当前不一定正忙。

### `Busy`

表示当前主执行链正在进行：

- 调用模型
- 调用工具
- 等待工具返回
- 处理工具结果

### `WaitingEvent`

表示本次任务尚未完成，但执行链已显式挂起，等待某个事件或恢复条件命中。

### `WaitingApproval`

表示任务进入审批链或人工确认链，不能继续自动推进。

### `Completed`

表示完成策略判定通过，本次执行收口。

### `Cancelled`

表示外部控制链路要求中止，并已完成必要清理。

### `Frozen`

表示受治理动作影响，当前不接受普通执行、事件唤醒和自动恢复。

## 多轮循环的停止护栏

“最多 10 轮”是必要条件，但不是充分条件。建议至少同时启用以下 5 类护栏。

### 1. 最大轮数

例如：

- `MaxTurns = 10`

用于防止无穷决策循环。

### 2. 最大总耗时

例如：

- `MaxElapsed = 60s` 或 `120s`

用于防止轮次不多但单轮极慢，导致执行长时间挂起。

### 3. 最大工具调用次数

例如：

- `MaxToolCallsTotal = 20`

用于限制高频工具重试与组合调用膨胀。

### 4. 相同工具重复限制

例如：

- 相同工具 + 相同参数哈希连续最多 2~3 次

用于阻断“同一个失败动作无限重试”的局面。

### 5. 无进展轮数限制

例如：

- 连续 2 轮未产生新信息、未改变状态、未推进任务

可通过以下信号近似判断：

- 相同 reply 摘要重复出现
- 相同工具重复调用
- 相同错误重复出现
- 未产生任何新的交付物、状态变化或候选结果

## 停止、冻结与恢复机制

### 外部停止

停止应由 Controller 通过控制链路下发，而不是由模型 function call 自行决定。

建议最小控制语义：

- `CancelExecutionRequest`
- `FreezeExecutionRequest`
- `ResumeExecutionRequest`

### 轮次边界检查点

Runtime 应至少在以下位置检查 stop / cancel / freeze 标志：

1. 调用 LLM 前
2. LLM 返回后
3. 执行工具前
4. 工具返回后
5. 准备进入下一轮前

### 工具级取消

Tool Executor 必须支持：

- `CancellationToken`
- 超时控制
- 中止/kill 语义
- 清理/回滚回调（若工具具备可回滚性）

否则 stop 只会终止 loop，而不会终止真正仍在运行的副作用工具。

### 冻结优先级

冻结是安全治理动作，优先级高于普通任务完成。若执行中收到冻结请求：

- 当前轮应尽量在安全边界停下
- 停止后续普通执行
- 必要时停止对应 sandbox / Docker 容器
- 进入 `Frozen`

## 完成判定策略

Runtime 不应直接把 LLM 输出的 `DONE` 视为任务完成，而应通过 `CompletionPolicy` 二次判定。

建议最小判定项：

1. 是否仍有未完成工具副作用
2. 是否仍有待审批项
3. 是否仍有待回收的 sub_agent / worker
4. 是否仍有强制等待事件
5. 是否收到 cancel / freeze 等更高优先级控制动作

若未全部满足：

- 可转为 `WaitingEvent`
- 可转为 `WaitingApproval`
- 可补一轮澄清
- 可进入 `Failed`

### 推荐思路：完成是请求，不是命令

也就是说：

- LLM 可以说“我建议完成”
- Runtime 需要说“现在是否允许完成”

## 结构化决策协议建议

若模型支持原生 function/tool calling，应优先采用原生结构化调用，而不是完全依赖提示词 JSON。

### 推荐优先级

1. **优先：原生 function calling / tool calling**
2. **兼容：严格 JSON 协议输出**

### 推荐的生命周期工具

除业务工具外，建议引入少量生命周期语义工具：

- `complete_task`
- `wait_for_event`
- `request_approval`
- `request_sleep`

这些工具表示的是运行意图，由 Runtime 判定是否真正落地。

### 若必须用 JSON 协议

不建议只保留 `CONTINUE / DONE` 两个状态。

建议最小协议：

```json
{
  "status": "CONTINUE | DONE | WAIT | FAILED",
  "message": "当前轮说明或交付结果",
  "tool": {
    "name": "工具名或 null",
    "args": {}
  },
  "meta": {
    "reason": "为什么继续/等待/完成/失败",
    "confidence": 0.0
  }
}
```

其中：

- `CONTINUE`：继续执行，可选工具调用
- `DONE`：提出完成请求，`tool` 必须为 `null`
- `WAIT`：进入挂起态，等待事件、审批或恢复条件
- `FAILED`：本轮无法继续，需交给 Runtime 做失败策略判定

### 为什么不建议只保留二元状态

因为 Pudding 是事件驱动、多治理链路系统，而不是单纯的 ChatBot；没有 `WAIT`，后续的事件唤醒与审批挂起语义都会被迫伪装成“还没 done”。

## Hook 机制建议

当前平台已经存在工具级与回复级 Hook 语义，这一命名应保持稳定，不建议 Runtime 另起一套平行命名。

### 兼容保留的 Hook

- `pre_tool_call`
- `post_tool_call`
- `pre_reply`
- `post_reply`

### 建议新增的执行生命周期 Hook

- `on_execution_started`
- `on_turn_started`
- `on_llm_response`
- `on_tool_selected`
- `on_tool_succeeded`
- `on_tool_failed`
- `on_waiting`
- `on_stop_requested`
- `on_cancelled`
- `on_completed`
- `on_failed`
- `on_max_turns_reached`
- `on_memory_writeback`

### 关键原则

1. Hook 失败不能拖垮主执行链
2. Hook 事件名在 CLI、Runtime、审计与外部扩展中尽量一致
3. `on_completed` 应成为稳定的收口扩展点
4. `on_stop_requested` 与 `on_cancelled` 应区分“收到停止请求”与“已经完成取消”

## 与事件总线的协同关系

Pudding 的多轮执行不能只理解为聊天记录累加，还必须理解为“事件触发下的一次执行任务”。

建议区分两层：

### 1. 会话层多轮

表示用户或渠道层面的连续消息往返。

### 2. 执行层多轮

表示某次任务执行内部的多轮推理与工具调用循环。

推荐关系：

- 一个 `SessionRuntime`
  - 可以承载多次 `ExecutionSession`
- 一次 `ExecutionSession`
  - 可以包含多轮 `Turn`
- 一个 `Turn`
  - 可以产生 0..n 个工具调用与事件产物

### 等待与恢复

若某轮决策结果为 `WAIT`：

- Runtime 生成 `ResumeAnchor`
- 写入 `ExecutionJournal`
- 进入 `WaitingEvent` 或 `WaitingApproval`
- 等待 Controller + Event Bus 在条件命中后再次投递 `DispatchWakeupRequest`

这样执行链才能与 [10事件系统与事件总线.md](10事件系统与事件总线.md) 中的直接唤醒模型闭环。

## 建议的最小结果类型

建议把一次执行的结果统一建模为以下几类：

- `ExecutionCompletedResult`
- `ExecutionWaitingResult`
- `ExecutionCancelledResult`
- `ExecutionFailedResult`

而不是只返回一段最终 reply 文本。

### `ExecutionCompletedResult`

至少包含：

- 最终用户可见回复
- 完成摘要
- 本次工具调用摘要
- 记忆候选摘要
- 审计关联 ID

### `ExecutionWaitingResult`

至少包含：

- 等待原因
- 等待类型（事件 / 审批 / 子任务 / 人工确认）
- ResumeAnchor
- 推荐恢复条件

### `ExecutionCancelledResult`

至少包含：

- 取消原因
- 取消来源（用户 / Controller / 审计 / 超时）
- 当前已完成步骤摘要
- 是否存在未清理副作用

### `ExecutionFailedResult`

至少包含：

- 失败分类
- 失败原因
- 是否可重试
- 推荐恢复策略

## MVP 实施建议

若只做最小闭环，建议按以下顺序实现：

### 第一步：先稳定 loop 的控制权

先明确：

- Runtime 驱动 loop
- LLM 只负责单轮决策
- stop 不依赖模型

### 第二步：补齐最小状态机

至少支持：

- `Running`
- `Busy`
- `WaitingEvent`
- `Completed`
- `Cancelled`
- `Failed`

### 第三步：补齐最小护栏

至少支持：

- 最大轮数
- 最大总耗时
- 工具调用超时
- cancel 检查
- 同工具重复限制

### 第四步：保留并扩展 Hook

保留现有：

- `pre_tool_call`
- `post_tool_call`
- `pre_reply`
- `post_reply`

新增至少：

- `on_completed`
- `on_cancelled`
- `on_failed`

### 第五步：引入等待态而不是强行 DONE

在支持事件唤醒前，就先把 `WAIT` 作为合法结果引入，避免后续再推翻现有协议。

## 这份方案不主张什么

为了防止设计膨胀，本分册不主张：

- 让 LLM 完整接管执行主循环
- 让 stop / freeze 依赖 function call 自行触发
- 让 `DONE` 直接等于任务完成
- 让同一主执行链自由并发重入
- 让 Hook 故障拖垮执行链
- 让“等待事件/审批”继续伪装成普通 CONTINUE

## 总结

Pudding 的多轮会话与工具调用执行模型，应当以 Runtime 为执行控制中心，而不是让模型自身充当调度器。

稳定的基线应当是：

- Runtime 驱动 loop
- Controller 掌控停止、冻结、恢复等治理动作
- LLM 只输出本轮结构化决策
- CompletionPolicy 负责最终收口判定
- HookDispatcher 提供稳定扩展点
- Event Bus 为等待与恢复提供闭环

只有这样，多轮工具调用才能从“能跑的 demo”升级为“可中断、可治理、可恢复、可审计的 Agent Runtime 能力”。
