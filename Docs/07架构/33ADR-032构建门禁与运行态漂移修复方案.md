# ADR-032：构建门禁与运行态漂移修复方案

> 状态：**Proposed**
> 日期：2026-05-23
> 范围：PuddingAgent Docker 部署、PuddingPlatform Chat API、PuddingController/MemoryEngine 构建完整性、Admin 聊天错误展示
> 前置：[20会话状态机与事件规范ADR](20会话状态机与事件规范ADR.md)、[31ADR-031聊天历史转录持久化与事件日志回放边界](32ADR-031聊天历史转录持久化与事件日志回放边界.md)

---

## 1. 背景

Admin 聊天发送消息时前端提示 `Request failed with status code 405`。排查发现：

1. `POST /api/workspaces/{workspaceId}/chat/message` 在当前源码中存在。
2. 未登录直接请求该 POST 返回 `401 Unauthorized`，说明路由和方法本身可匹配。
3. 运行中容器日志显示 `ChatApiController` 激活失败：`ChatTranscriptWriter` 未在 DI 注册。
4. 当前源码定义了 `ChatTranscriptWriter`，但**未在 `Program.cs` 中注册为 DI 服务**，导致运行时控制器激活失败。
5. Docker 对外端口为 `5000:8080`，但部分文档仍提示访问 `8080`。
6. 前端 axios 错误处理只显示 `Response status: xxx`，缺少 method、url、后端 body 信息，导致 401/405/500/DI 崩溃难以区分。

因此，本问题不是单一 HTTP 405，而是构建完整性、运行态漂移、错误暴露不足叠加造成的聊天链路不可诊断。

---

## 2. 决策

### ADR-032-A：ChatTranscriptWriter 必须注册 DI

**决定**：在 `Program.cs` 中显式注册 `ChatTranscriptWriter`：

```csharp
builder.Services.AddSingleton<ChatTranscriptWriter>();
```

原因：
- `ChatApiController` 构造函数依赖 `ChatTranscriptWriter`，若未注册，控制器激活时抛出 `InvalidOperationException`，表现为 HTTP 500。
- 此遗漏在构建期不可检测（编译通过），只在运行期暴露。

### ADR-032-B：构建失败时禁止发布或重启业务镜像

**决定**：`PuddingAgent` 镜像构建前必须通过：

```powershell
dotnet build Source\PuddingAgent\PuddingAgent.csproj -c Release --nologo
```

若构建失败，不允许继续执行 `docker compose up -d --build`。

原因：
- 旧镜像仍能运行，但已与源码不一致。
- 继续依赖旧镜像会让线上问题与源码排查脱节。

### ADR-032-C：聊天 API 必须有运行时激活测试

**决定**：新增 Web/API 冒烟测试，覆盖：

```text
POST /api/workspaces/{workspaceId}/chat/message
```

测试目标不是要求 LLM 完整返回，而是验证：
- 路由存在。
- HTTP method 匹配。
- 控制器能被 DI 激活。
- 未授权返回 401，授权后不会因 DI 缺失返回 500。

### ADR-032-D：容器端口以 compose 为准并同步文档

**决定**：单进程 Docker 部署的本机入口固定为：

```text
http://localhost:5000
```

除非 compose 改为 `8080:8080`，否则 README 中的 Docker 命令、compose 注释不得继续提示 `localhost:8080`。

### ADR-032-E：前端错误必须暴露服务端 message

**决定**：Admin 请求错误展示至少包含：

```text
HTTP method
URL
status code
response body message / problem details title
```

`requestErrorConfig.ts` 中 axios 错误处理分支已改为展示 `{METHOD} {URL} — HTTP {status} — {body.message}`。

---

## 3. 验收标准

1. `dotnet build Source\PuddingAgent\PuddingAgent.csproj -c Release --nologo` 通过（已验证 ✓）。
2. `ChatTranscriptWriter` 注册到 DI 后，`ChatApiController` 激活不再失败（已验证 ✓）。
3. `docker compose up -d --build pudding-agent` 后镜像不再因 `ChatTranscriptWriter` 未注册导致控制器 500。
4. 登录后发送 `你好` 不再触发控制器激活异常。
5. `docker logs puddingagent-pudding-agent-1` 中无 `Unable to resolve service`。
6. README / compose / healthcheck 的端口说明一致（已完成 ✓）。
7. 聊天接口失败时，前端能展示后端返回的具体 `message`（已完成 ✓）。
