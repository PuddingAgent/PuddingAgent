# 数据模型与配置

## 核心数据实体

- AgentTemplate：角色、默认能力、运行画像、记忆策略。
- AgentInstance：模板派生出的实际运行实例。
- Session：ServiceSession、ExecutionSession、AuditSession、ChildSession。
- Workflow / WorkflowStep：显式工作流与步骤定义。
- MemoryRecord / MemoryPolicy：正式记忆、候选记忆与记忆治理策略。
- ApprovalRecord：高风险执行与确认链路的核心对象。
- ChannelBinding：渠道绑定到 Workspace 的治理关系。
- EventEnvelope / EventRoute：统一事件对象与路由结果。
- EventSubscription：Agent、Runtime、审计链路对事件的订阅声明。
- EventCursor / ReplayCheckpoint：长连接消费、补偿消费与重放游标。
- DeadLetterEvent：无法投递、无法消费或超限事件的隔离对象。

## 建模原则

- 模板与实例必须分离。
- Workspace 是配置与治理边界，Session 是运行时边界。
- 记忆必须同时建模正式存储与候选提升。
- 路由、审批、权限、审计需要有可查询的显式对象。
- 事件模型必须同时支持外部接入事件与内部状态事件，不能只覆盖消息渠道。
- 订阅关系、事件可见性与重放权限必须可治理、可审计、可查询。

## Workspace 最小配置对象

- AgentTemplate
- ChannelBinding
- Workflow
- MemoryPolicy
- PermissionPolicy
- BudgetPolicy
- PluginBinding
- EventPolicy
- EventSubscriptionPolicy

## 配置来源

第一阶段采用 Workspace YAML 加平台持久化的混合模式。配置变更优先手动 reload，而不是一开始就做自动热更新。

## 事件系统最小配置对象

- `EventPolicy`：声明允许进入本 Workspace 的事件类型、来源范围、信任等级和保留策略。
- `EventSubscriptionPolicy`：声明哪些 Agent / Workflow / Runtime 组件可以订阅哪些事件，以及是否允许直接唤醒。
- `EventRetentionPolicy`：声明事件日志、死信事件、重放窗口和归档策略。
- `AdapterEventMapping`：声明某个 Gateway Adapter 的原始协议字段如何映射到统一事件类型。
