# ADR-031：聊天历史转录持久化与事件日志回放边界

> 状态：**Proposed**  
> 日期：2026-05-21  
> 关联：[16会话状态层与客户端解耦ADR](16会话状态层与客户端解耦ADR.md)、[20会话状态机与事件规范ADR](20会话状态机与事件规范ADR.md)  
> 相关代码：`ChatApiController.cs`、`MessageApiController.cs`、`ChatHistoryService.cs`、`useChatState.ts`

---

## 1. Context

当前发送链路在 `ChatApiController.SendMessage` 中启动后台流，解析 `metadata`，并把每个 SSE frame 通过 `ssm.AppendAsync` 写入 `session_event_log`。`done` 帧仅用于 token 统计，没有物化写入 `ChatMessages`。

历史查询链路 `/api/sessions/{sessionId}/messages` 和 `IChatHistoryService` 只读 `ChatMessages`。因此当 `ChatMessages` 为空时，历史点击必然得到空消息列表。

前端还有一个独立问题：`handleSelectSession` 在 `sid === selectedSessionId` 时直接返回。当前会话已选中但 `turns` 为空时，用户再次点击历史项不会触发重载。

旧会话的事件日志可恢复 AI 输出和 usage/thinking 等过程信息，但由于用户输入未写入事件 payload，不能完整恢复用户原文。

---

## 2. Decision

1. `session_event_log` 保持 append-only 事实日志，负责执行过程、流式恢复、审计和调试。
2. `ChatMessages` 定义为聊天转录的物化表，负责 `/messages`、分页历史、token 统计、工具侧历史查询。
3. 发送消息时必须双写：
   - 收到或确认 `sessionId/messageId` 后，写入 `Role=user`、`Content=req.MessageText`。
   - 收到 `done` 后，写入 `Role=agent`、`Content=reply 或 delta 聚合结果`、`UsageJson`、可选 `ThinkingJson`。
4. 双写不改变 `session_event_log` 的权威性。事件日志仍是执行事实源；`ChatMessages` 是面向 UI 和检索的转录索引。
5. 历史 API 增加旧数据 fallback：若 `ChatMessages` 为空但 `session_event_log` 有 `done/delta/thinking/usage`，合成只含 assistant 的 orphan turn。用户原文显示为空或“历史用户消息未持久化”，不得伪造。
6. 前端移除当前会话空白短路：同一 `sid` 但 `turns.length === 0` 时仍允许加载历史。

---

## 3. Rejected Options

### A. 只改前端从 `/events` 回放历史

不采纳。当前回放依赖 turn/message 映射，旧事件也缺用户输入；会让普通历史加载耦合到复杂 timeline 重放。

### B. 废弃 `ChatMessages`，全部以事件日志为源

不采纳。会破坏 `/messages`、token-stats、`IChatHistoryService` 的简单查询模型，也不利于分页和跨会话摘要。

### C. 只补未来写入，不处理旧会话

不采纳。能修未来，但当前用户点击历史仍显示欢迎页，体验问题没有闭环。

---

## 4. Consequences

正向影响：

- 未来会话可完整恢复 user/agent 问答。
- `/messages`、token 统计、工具历史查询重新有稳定数据源。
- 事件日志仍可用于 timeline、SSE 补偿、审计，不被 UI 历史加载绑死。
- 当前旧会话至少能显示 AI 回复，不再空白。

代价：

- 发送链路需要处理双写幂等，避免重试产生重复消息。
- `done` payload 如果没有完整 reply，需要在后台流处理中聚合 `delta`。
- 旧会话无法恢复用户原文，这是不可逆数据丢失，只能降级展示。

---

## 5. Implementation Notes

1. 修 `useChatState.ts`：`sid === selectedSessionId && turns.length > 0` 才短路。
2. 在 `ChatApiController` 后台流中收集 `delta/thinking/usage/done`。
3. 使用 scoped `PlatformDbContext` 写入 `ChatMessages`，不要复用请求注入的 `db` 跨后台任务。
4. 为用户消息和助手消息做幂等保护。若现有表无 `MessageId/TurnId`，短期用 `(SessionId, Role, CreatedAt窗口/Content)` 降低重复风险；中期建议新增 nullable `ExternalMessageId` 或 `TurnId`。
5. 在 `MessageApiController.List` 增加 fallback，仅在 `ChatMessages` 为空时从事件日志合成 assistant-only 历史。
6. 加测试：发送后 `ChatMessages` 有 user+agent；旧事件日志 fallback 返回 assistant；同一会话空 turns 点击会重新加载。
