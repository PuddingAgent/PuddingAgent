# 记忆图书馆 (Memory Library) — 基础设施层设计

> **日期**: 2026-05-07  
> **状态**: 设计阶段  
> **定位**: PuddingMemoryEngine 的底层存储基础设施，独立于上层策略逻辑

## 1. 定位与边界

### 1.1 核心原则

记忆图书馆是**纯存储基础设施**，类比关系数据库：

| 类比 | 关系数据库 | 记忆图书馆 |
|------|-----------|-----------|
| 存储单元 | 表 (Table) | Book（书籍）|
| 组织单元 | 行 (Row) | Chapter（章节）|
| 索引 | B-Tree / Hash | Tag 树 / FTS5 / 向量 |
| 关联 | Foreign Key | Pointer（指针）|
| 查询 | SQL | IMemoryLibrary API |

**记忆图书馆不关心**：何时写入、粒度控制、Chapter 排序、写入触发条件 —— 这些全部由上层（MemoryEngine / MemoryBoundaryService）决策。

### 1.2 分层架构

```
┌─────────────────────────────────────────────────┐
│  上层（软件架构）                                │
│  MemoryEngine / MemoryBoundaryService /          │
│  BookWriter / BookAwarenessDetector              │
│  → 决定 "什么时候写" "写什么" "拆多细"          │
└──────────────────────┬──────────────────────────┘
                       │ 依赖
┌──────────────────────▼──────────────────────────┐
│  IMemoryLibrary (接口层)                         │
│  - CreateBook / GetBook / UpdateBook             │
│  - AddChapter / GetChapter / UpdateChapter       │
│  - CreatePointer / GetPointers / ResolvePointer  │
│  - SearchBooks / SearchChapters (FTS5 / Vector)  │
│  - GetTagChildren / SearchByTagPrefix            │
│  - BranchBook / MergeChapter                    │
└──────────────────────┬──────────────────────────┘
                       │ 实现
┌──────────────────────▼──────────────────────────┐
│  MemoryLibrary (实现层)                          │
│  底层：SQLite + FTS5 + sqlite-vec + TagTree     │
│  不关心业务语义，只提供存储和检索能力            │
└─────────────────────────────────────────────────┘
```

## 2. 数据模型 (SQLite Schema)

### 2.1 Libraries 表

图书馆容器——一个 Workspace 可以有多个 Library（如"个人知识库""团队共享库"）。

```sql
CREATE TABLE IF NOT EXISTS Libraries (
    LibraryId     TEXT PRIMARY KEY,
    WorkspaceId   TEXT NOT NULL,
    Name          TEXT NOT NULL,
    Description   TEXT,
    CreatedAt     INTEGER NOT NULL,
    UpdatedAt     INTEGER NOT NULL
);

CREATE INDEX IF NOT EXISTS IX_Libraries_Workspace
    ON Libraries(WorkspaceId);
```

### 2.2 Books 表

书籍——独立的完整记忆单元。一个主题一本书（如"MySQL运维手册""PuddingAgent部署指南"）。

```sql
CREATE TABLE IF NOT EXISTS Books (
    BookId        TEXT PRIMARY KEY,
    LibraryId     TEXT NOT NULL REFERENCES Libraries(LibraryId),
    Title         TEXT NOT NULL,
    Summary       TEXT NOT NULL DEFAULT '',      -- 一句话摘要，用于列表展示
    Status        TEXT NOT NULL DEFAULT 'active', -- active | archived
    Version       INTEGER NOT NULL DEFAULT 1,    -- 每次章节更新递增
    AccessCount   INTEGER NOT NULL DEFAULT 0,
    LastAccessedAt INTEGER,
    CreatedAt     INTEGER NOT NULL,
    UpdatedAt     INTEGER NOT NULL
);

CREATE INDEX IF NOT EXISTS IX_Books_Library_UpdatedAt
    ON Books(LibraryId, UpdatedAt DESC);
CREATE INDEX IF NOT EXISTS IX_Books_Status
    ON Books(Status, UpdatedAt DESC);
```

### 2.3 BookIndexes 表

书籍的多路径索引——每本书可以有多个 Tag 路径，实现"一本书可以出现在多个分类下"。

```sql
CREATE TABLE IF NOT EXISTS BookIndexes (
    Id            INTEGER PRIMARY KEY AUTOINCREMENT,
    BookId        TEXT NOT NULL REFERENCES Books(BookId) ON DELETE CASCADE,
    TagPath       TEXT NOT NULL,                 -- eg: "技术/数据库/MySQL/运维"
    Weight        INTEGER NOT NULL DEFAULT 1,    -- 该路径下的权重
    CreatedAt     INTEGER NOT NULL
);

CREATE INDEX IF NOT EXISTS IX_BookIndexes_TagPath
    ON BookIndexes(TagPath);
CREATE INDEX IF NOT EXISTS IX_BookIndexes_BookId
    ON BookIndexes(BookId);
```

### 2.4 Chapters 表

章节——书籍的最小经验单元。每章记录一段完整的经验：问题上下文、解决步骤、代码、结论。

```sql
CREATE TABLE IF NOT EXISTS Chapters (
    ChapterId     TEXT PRIMARY KEY,
    BookId        TEXT NOT NULL REFERENCES Books(BookId) ON DELETE CASCADE,
    Title         TEXT NOT NULL,
    ChapterOrder  INTEGER NOT NULL DEFAULT 0,     -- 书籍内逻辑顺序（由上层控制）
    Content       TEXT NOT NULL,                   -- 正文：经验+步骤+代码+结论
    ContentType   TEXT NOT NULL DEFAULT 'markdown',-- markdown | plain | code
    Importance    REAL NOT NULL DEFAULT 0.5,       -- 重要性评分 (0-1)
    SourceSessionId TEXT,                          -- 来源会话（可回溯完整现场）
    WordCount     INTEGER NOT NULL DEFAULT 0,      -- Content 字数（便于Token预算）
    CreatedAt     INTEGER NOT NULL,
    UpdatedAt     INTEGER NOT NULL
);

CREATE INDEX IF NOT EXISTS IX_Chapters_Book_Order
    ON Chapters(BookId, ChapterOrder);
```

### 2.5 Pointers 表

指针——章节之间的交叉引用。支持指向：其他章节 / 完整会话日志 / 外部URL / 操作数据库记录。

```sql
CREATE TABLE IF NOT EXISTS Pointers (
    PointerId     TEXT PRIMARY KEY,
    ChapterId     TEXT NOT NULL REFERENCES Chapters(ChapterId) ON DELETE CASCADE,
    TargetType    TEXT NOT NULL,                   -- book | chapter | session | url | message | log
    TargetId      TEXT NOT NULL,                   -- 目标标识符（BookId/ChapterId/URL等）
    TargetLabel   TEXT,                            -- 显示标签（人类可读）
    Description   TEXT,                            -- 为什么引用这段
    Relevance     INTEGER NOT NULL DEFAULT 5,      -- 相关性评分 (1-10)
    CreatedAt     INTEGER NOT NULL
);

CREATE INDEX IF NOT EXISTS IX_Pointers_ChapterId
    ON Pointers(ChapterId);
CREATE INDEX IF NOT EXISTS IX_Pointers_TargetType_TargetId
    ON Pointers(TargetType, TargetId);
```

### 2.6 Branches 表

分支——Book 的平行版本。用于"同一主题的不同解题思路"或"多人协作的不同贡献"。

```sql
CREATE TABLE IF NOT EXISTS Branches (
    BranchId      TEXT PRIMARY KEY,
    BookId        TEXT NOT NULL REFERENCES Books(BookId) ON DELETE CASCADE,
    BranchName    TEXT NOT NULL,
    Description   TEXT,
    CreatedBy     TEXT,                            -- 创建者 AgentId / UserId
    MergedInto    TEXT REFERENCES Branches(BranchId), -- 合并目标
    IsDefault     INTEGER NOT NULL DEFAULT 0,      -- 是否默认分支
    CreatedAt     INTEGER NOT NULL
);

CREATE INDEX IF NOT EXISTS IX_Branches_BookId
    ON Branches(BookId);
```

### 2.7 全文索引 (FTS5)

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS Books_fts USING fts5(
    Title, Summary, BookId UNINDEXED,
    content=Books, content_rowid=rowid
);

CREATE TRIGGER IF NOT EXISTS trg_Books_ai AFTER INSERT ON Books BEGIN
    INSERT INTO Books_fts(rowid, Title, Summary, BookId)
    VALUES (new.rowid, new.Title, new.Summary, new.BookId);
END;

CREATE VIRTUAL TABLE IF NOT EXISTS Chapters_fts USING fts5(
    Title, Content, ChapterId UNINDEXED,
    content=Chapters, content_rowid=rowid
);

CREATE TRIGGER IF NOT EXISTS trg_Chapters_ai AFTER INSERT ON Chapters BEGIN
    INSERT INTO Chapters_fts(rowid, Title, Content, ChapterId)
    VALUES (new.rowid, new.Title, new.Content, new.ChapterId);
END;
```

## 4. 索引与检索策略

### 4.1 三层检索架构

```
用户/LLM 查询
    │
    ├─ 第1层：TagTree 前缀匹配（最快，O(log n)，适合"技术/数据库"精确路径）
    │
    ├─ 第2层：FTS5 全文搜索（中等，适合"MySQL 安装失败"模糊语义）
    │
    └─ 第3层：向量语义搜索（最慢，适合跨领域概念联想，Phase 4 实现）
           │
           └─ RRF (Reciprocal Rank Fusion) 融合三层结果
```

### 4.2 TagTree 多路径索引

一本书可以有多个 Tag 路径（通过 `BookIndexes` 表）：

```
Book: "MySQL 运维手册"
  ├── "技术/数据库/MySQL/运维"      (weight=5, 主路径)
  ├── "运维/数据库/故障处理"        (weight=3, 交叉分类)
  └── "经验/基础设施/MySQL"        (weight=2, 按经验类型)
```

检索 `GetTagChildren("技术/数据库")` 时，MySQL 运维手册出现在结果中。

## 5. 与现有系统的关系

### 5.1 现有 `MemoryEntity` 表 → 保留不变

现有的 `Memories` 表（Session/Workspace 级扁平记忆）继续作为**即时记忆层**使用。  
记忆图书馆是**结构化知识层**，两者互补：

| 维度 | Memories 表 | 记忆图书馆 |
|------|------------|-----------|
| 粒度 | 条目级（一句话） | Book+Chapter（一本手册） |
| 生命周期 | Session 内 / Workspace 共享 | 永久，可归档 |
| 结构化 | 扁平 Tag | 层级 Book → Chapter → Pointer |
| 写入 | `REMEMBER[tag]: text` | 上层调用 API 显式写入 |
| 使用场景 | 对话上下文注入 | 知识检索、经验传递、新人引导 |

### 5.2 现有 `MemoryDbContext` → 不变

记忆图书馆使用独立的 `MemoryLibraryDbContext`（共享同一 SQLite 连接，独立 DbContext 实例）。

### 5.3 依赖方向

```
PuddingRuntime (MemoryEngine)  →  IMemoryLibrary (基础设施接口)
                                →  IMemoryEngine (现有接口，保持不变)
```

## 8. 长程 Agent 适用性评估

### 8.1 模型：LLM 操作记忆图书馆的典型场景

```
场景 A：潜意识 LLM 在 Session 结束后自动整理
  "刚才的对话里，用户解决了 MySQL 8.0 主从复制延迟问题，
   我应该把这个经验写进记忆图书馆"
  → 需要：找到或创建 Book → 追加 Chapter → 添加 Pointer → 更新索引

场景 B：显意识 LLM 在对话中被用户问到时检索
  "我记得之前处理过数据库连接池耗尽的问题，
   帮我查查记忆图书馆里有没有相关经验"
  → 需要：FTS5 搜索 → TagTree 导航 → 返回 Chapter 内容 → 连带 Pointer 引用

场景 C：跨 Session 知识积累
  "这是第 5 次遇到 Docker 构建缓存问题了，
   把这次的新发现补充到之前的经验书里"
  → 需要：找到已有 Book → 追加/更新 Chapter → 不重复不覆盖
```

### 8.2 摩擦点分析

| # | 摩擦点 | 严重度 | 根因 | 举例 |
|---|--------|--------|------|------|
| 1 | **UUID 负担** | 🔴 高 | 每次操作需要手动生成/传递 BookId/ChapterId | LLM 写入经验前必须 `var bookId = Guid.NewGuid().ToString("N")`，且后续引用这本书时必须记住或搜索出这个 ID |
| 2 | **Find-or-Create 双步操作** | 🔴 高 | 没有 "如果存在就追加，否则新建" 的原子操作 | LLM 写 MySQL 经验 → 先 `SearchBooksFtsAsync("MySQL")` → 无结果 → 再 `CreateBookAsync(...)` → 二步操作，且中间可能竞态 |
| 3 | **Tag 路径僵化** | 🟡 中 | TagPath 是精确字符串，LLM 必须知道完整的分类体系 | LLM 写到 `"技术/数据库/MySQL/运维"`，但现有分类是 `"技术/计算机/数据库/MySQL"`——路径不匹配导致检索时找不到 |
| 4 | **Chapter 排序需全局感知** | 🟡 中 | `ChapterOrder` 要求 LLM 知道整本书的章节结构 | 追加第 7 章时，LLM 必须先 `ListChaptersAsync` 确认当前有多少章 |
| 5 | **Pointer 需要先查后引** | 🟡 中 | 创建 Pointer 需要目标 ChapterId，必须先搜索 | 引用"之前写过的连接池经验" → 先 `SearchChaptersFtsAsync("连接池")` → 取 ChapterId → 再 `CreatePointerAsync` |
| 6 | **无去重机制** | 🟡 中 | 多次处理同一问题可能创建重复 Book/Chapter | 5 个 Session 都解决了 Docker 问题 → 可能创建 5 本独立的 Docker 书 |

### 8.3 长程适用性判断

**结论：当前设计正确但不够顺手。** 类比：你给了 LLM 一个完整的 SQL 数据库，但没有给它 ORM 和 `upsert`。它可以完成所有操作，但每一步都需要显式编排，对长程 Agent 来说摩擦过大。

长程 Agent 的核心需求是 **"Fire and forget"**——潜意识 LLM 发现问题解决了，丢给记忆图书馆一个经验包，记忆图书馆自己决定是新建 Book 还是追加到已有 Chapter。

### 8.4 解决方案：增加 LLM 友好操作层（仍在基础设施层）

基础设施层可以 —— 也**应该** —— 提供高层封装，只要这些封装不包含业务策略，只是纯粹的"数据智能路由"。

```
基础设施层内部细分：

┌─────────────────────────────────────────┐
│  高层 API（LLM 友好，内部无策略）        │
│  - UpsertExperienceAsync(经验包) → 自动  │
│    路由到合适的 Book/Chapter            │
│  - SmartSearchAsync(自然语言查询) →      │
│    融合 TagTree + FTS5 + 向量           │
│  - GetBookByTitleOrCreateAsync           │
│  - AppendChapterAutoOrder                │
│  - AutoPointerDiscovery                  │
└──────────────┬──────────────────────────┘
               │ 内部委托
┌──────────────▼──────────────────────────┐
│  底层 API（精确控制，当前设计）          │
│  - CreateBook / GetBook / AddChapter    │
│  - CreatePointer / FTS5 Search          │
└─────────────────────────────────────────┘
```

**关键区分**：高层 API 只是"智能路由"，不是策略。例如：
- `UpsertExperienceAsync` 内部做 FTS5 匹配 + 相似度阈值判断 → 找到相似 Book 则追加 Chapter → 否则新建 Book
- 这是**数据层的责任**，因为判断"这两段文字是不是同一个主题"依赖 FTS5 相似度——这是存储引擎的能力，不是上层策略

### 8.5 新增 LLM 友好操作

```csharp
/// <summary>
/// LLM 友好操作层——提供"直接丢内容进来，图书馆自己管路由"的便利方法。
/// 这些方法不包含业务策略，只是存储引擎内部的智能路由。
/// </summary>
public interface IMemoryLibraryConvenience
{
    // ── 经验写入（Upsert 模式）──

    /// <summary>
    /// 写入一段经验。内部自动：
    /// 1. FTS5 搜索相似 Book → 有则追加 Chapter，无则新建 Book
    /// 2. 自动计算 ChapterOrder（追加到末尾）
    /// 3. 自动发现 Content 中引用其他 Book/Chapter 的关键词并创建 Pointer
    /// 返回：被写入的 BookRecord + ChapterRecord
    /// </summary>
    Task<ExperienceWriteResult> UpsertExperienceAsync(
        ExperiencePackage experience,
        CancellationToken ct = default);

    // ── 智能检索（一次调用融合多路）──

    /// <summary>融合 TagTree + FTS5 的自然语言检索（Phase 4 加入向量）。</summary>
    Task<IReadOnlyList<RankedResult>> SmartSearchAsync(
        string naturalLanguageQuery,
        int topK = 20,
        CancellationToken ct = default);

    // ── 便捷操作 ──

    /// <summary>按 Title 精确获取或创建 Book（Title 作为自然键去重）。</summary>
    Task<BookRecord> GetOrCreateBookAsync(
        string title, string? summary, IReadOnlyList<string>? tagPaths, CancellationToken ct);

    /// <summary>追加 Chapter 到 Book 末尾（自动计算 ChapterOrder）。</summary>
    Task<ChapterRecord> AppendChapterAsync(
        string bookId, string title, string content, string? sourceSessionId = null, CancellationToken ct = default);

    /// <summary>扫描 Content 中的已知 Book/Chapter 引用，自动创建 Pointer。</summary>
    Task<IReadOnlyList<PointerRecord>> AutoDiscoverPointersAsync(
        string chapterId, CancellationToken ct = default);

    /// <summary>获取 Tag 树根节点（"图书馆→技术→数据库→MySQL" 的顶层分类）。</summary>
    Task<IReadOnlyList<TagTreeNode>> GetTagRootsAsync(CancellationToken ct = default);
}

/// <summary>经验包——LLM 交给记忆图书馆的最小输入单元。</summary>
public sealed record ExperiencePackage
{
    /// <summary>经验标题（如"MySQL 8.0 主从复制延迟排查"）。</summary>
    public string Title { get; init; } = string.Empty;

    /// <summary>正文（Markdown，含步骤/代码/结论）。</summary>
    public string Content { get; init; } = string.Empty;

    /// <summary>建议的分类路径（如 ["技术/数据库/MySQL"]，可为空由图书馆自动推断）。</summary>
    public IReadOnlyList<string>? SuggestedTags { get; init; }

    /// <summary>来源会话 ID（用于回溯完整现场）。</summary>
    public string? SourceSessionId { get; init; }

    /// <summary>重要性（0-1）。</summary>
    public double Importance { get; init; } = 0.5;

    /// <summary>关联的外部 URL（如 StackOverflow、文档）。</summary>
    public IReadOnlyList<string>? ExternalRefs { get; init; }
}

/// <summary>融合检索结果。</summary>
public sealed record RankedResult
{
    public string BookId { get; init; }
    public string BookTitle { get; init; }
    public string? ChapterId { get; init; }
    public string? ChapterTitle { get; init; }
    public string Snippet { get; init; }      // 高亮摘要片段
    public double Score { get; init; }
    public string MatchSource { get; init; }  // "tag" | "fts5" | "vector" | "pointer"
}
```

### 8.6 操作复杂度对比

| 操作 | 原设计（原子 API） | 新设计（LLM 友好 API） |
|------|-------------------|----------------------|
| 写入经验 | ①生成BookId ②SearchBooks ③CreateBook ④ListChapters ⑤AddChapter(手动order) ⑥SearchChapters ⑦CreatePointer | ① `UpsertExperienceAsync(experience)` |
| 检索知识 | ①SearchByTag ②SearchFts ③手动合并去重 | ① `SmartSearchAsync("MySQL 主从延迟")` |
| 追加章节 | ①ListChapters ②计算max order+1 ③AddChapter | ① `AppendChapterAsync(bookId, title, content)` |
| 搜索后追加 | ①SearchBooks ②取BookId ③ListChapters ④AddChapter | ① `UpsertExperienceAsync(experience)` 自动路由 |

### 8.7 长程 Agent 适用性结论

**适合**。当前设计 + LLM 友好操作层后：

- ✅ **低摩擦**：潜意识 LLM 只需构造 `ExperiencePackage`，一行调用完成写入
- ✅ **自动去重**：FTS5 相似度匹配防止重复 Book
- ✅ **自动索引**：Tag 路径自动推断 + 多路径注册
- ✅ **自动关联**：`AutoDiscoverPointersAsync` 扫描内容中的 Book 引用
- ✅ **渐进式**：底层精确 API 和上层便捷 API 共存，复杂场景可降级
- ✅ **跨 Session 积累**：`UpsertExperience` 幂等——同一经验多次提交不会重复
