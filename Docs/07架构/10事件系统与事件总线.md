# 事件系统与事件总线

## 定位

事件系统不是 `PuddingController` 的一个可选附件，而是整个协作网络的基础协作面。

如果说：

- `PuddingController` 负责治理、路由与控制
- `PuddingGateway` 负责协议接入与边界转换
- `PuddingRuntime` 负责执行与唤醒

那么事件总线负责把它们连成一个真正的事件驱动系统。

一句话原则：**万物皆事件**。

但这句话并不等于“所有事件都可以被任何人看到或消费”。事件始终受 Workspace、权限、审批、信任等级与治理策略约束。

## 为什么必须有统一事件总线

没有统一事件总线时，系统容易退化为：

- Adapter 收到消息后直接特判调用某个 Controller 分支
- Runtime 只能通过轮询发现外部变化
- Agent 只能在收到显式消息时工作，无法订阅环境变化
- 审计、回放、重放、补偿都缺乏统一入口

统一事件总线的价值在于：

- 让外部协议世界统一翻译成平台事件世界
- 让内部状态变化也采用统一事件模型
- 让 Agent、Workflow、Runtime、审计链路消费同一类对象
- 让订阅、重放、死信、审计、回溯都具备统一语义

更重要的是，它直接回应了很多现有个人智能体系统的问题：

- 只有心跳，没有事件队列
- Agent 靠轮询等待任务，延迟高且浪费 token
- 外部系统接入后只能走特判逻辑，难以演进
- GitHub、MQTT、Webhook、WebSocket 这类真实企业触发器很难成为一等输入

Pudding 选择的方向是：**event-driven AI runtime**，而不是“带心跳的聊天机器人宿主”。

## 非目标

Pudding 的事件系统并不以“替代所有外部系统 API”或“做一个通用消息中间件产品”为目标。

它的目标是：

- 为 Agent Runtime 提供统一事件语义
- 为 Workspace 提供可治理的事件命名空间
- 为多智能体协作提供低延迟、低 token 浪费的唤醒机制
- 为企业集成提供稳定协议适配层

## 事件源分类

### 外部事件

来自 Gateway Adapter 或边界接入层：

- MQTT 设备状态变化
- Webhook 回调
- HTTP 渠道消息
- WebSocket 会话消息
- Email 入站邮件
- 嵌入式 Runtime 宿主上送的原生状态事件

### 内部事件

来自 Controller、Runtime、Workflow 或系统基础设施：

- `task.completed`
- `agent.completed`
- `agent.failed`
- `runtime.heartbeat`
- `runtime.capacity.changed`
- `approval.requested`
- `approval.completed`
- `memory.candidate.created`
- `workflow.step.completed`
- `audit.record.created`

## 事件域

事件不是天然全局广播。平台至少需要两层事件域：

- `global`：平台级治理、节点健康、跨 Workspace 协作、基础设施事件
- `workspace`：绝大多数业务事件、渠道事件、Agent 协作事件

默认原则：

- 业务事件优先进入 `workspace` 域
- 只有治理、基础设施或明确跨域协作的事件才进入 `global` 域
- 从 `workspace` 升级到 `global` 必须是显式策略，而不是默认行为
- 默认情况下，绝大多数事件都应被视为 `workspace` 级事件，而不是全局广播事件
- 只有控制层、用户或具备明确特权的 Agent / 系统模块，才允许发布 `global` 事件

也就是说：

- **workspace 是默认事件隔离域**
- **global 是严格限权的治理事件域**

## 统一事件 Envelope

建议统一事件对象至少包含：

- `eventId`：事件唯一标识
- `eventType`：事件类型，采用稳定命名规范
- `scope`：`global` 或 `workspace`
- `workspaceId`：Workspace 域事件的边界标识
- `source`：来源 Adapter、Runtime、Agent 或系统模块
- `subject`：关联对象，例如设备、任务、Session、Agent
- `payload`：结构化负载
- `trustLevel`：来源信任等级
- `correlationId`：跨链路关联 ID
- `causationId`：导致本事件的上游事件 ID
- `occurredAt`：事件发生时间
- `publishedAt`：事件进入总线时间
- `version`：事件契约版本

## 事件命名规范

事件类型建议统一采用小写点分命名：

- `device.sensor.triggered`
- `device.alarm.triggered`
- `task.completed`
- `task.failed`
- `agent.completed`
- `runtime.heartbeat`
- `workflow.step.completed`
- `approval.requested`

命名原则：

- 先领域，后对象，再动作
- 动作使用稳定动词过去式或状态式
- 不把底层协议名写进业务事件名，避免出现 `mqtt.xxx.received` 这类协议泄漏型命名
- 协议信息保留在 `source` 与 `payload` 元数据里，而不是污染事件语义

## 订阅模型

事件总线上的订阅者不应只有基础设施组件，也包括：

- Runtime 内部组件
- Runtime 管理下的 AgentExecutionContext
- Runtime 管理下的 sub_agent 执行上下文
- Workflow Worker
- 审计链路
- 调试/可观测性组件
- 高风险审批与治理模块

但这里有一个非常重要的边界：

> **事件订阅的执行权在 Runtime，而不是在 Agent。**

也就是说：

- AgentTemplate 可以声明订阅偏好
- Controller 可以负责治理准入、权限与唤醒资格判定
- **真正执行订阅、命中事件、插入上下文、恢复会话、决定是否打断的，是 Runtime**

Agent 本身不应把自己实现成“长时间挂在总线上等待消息的消费者”。

更准确的理解应该是：

- Agent 对“事件总线、订阅连接、插入时机、中断策略”整体无感
- Agent 只会在某一轮输入组装时，看见 Runtime 已经决定交给它的事件消息、恢复提示与上下文补丁
- Runtime 负责把“外部事件到来”转换为“下一轮 LLM 可理解的上下文变化”

订阅声明至少需要包含：

- 订阅者身份
- 允许可见的事件域
- 感兴趣的事件类型或模式
- 过滤条件
- 是否允许直接唤醒
- 消费失败策略
- 是否允许重放

## 发布权限模型

事件治理不只包含“谁可以订阅什么”，还必须包含“谁可以发布到哪个事件域”。

建议默认规则：

1. **默认发布到 `workspace` 域**
	- 大多数业务事件、协作事件、任务事件、工具结果事件都应发布到所属 Workspace
	- Runtime、普通 Agent、Workflow Worker 默认只应在 `workspace` 域内发布事件

2. **`global` 域发布必须限权**
	- 只有 `PuddingController`
	- 或显式用户动作
	- 或具备明确特权的 Agent / 审计模块 / 系统治理模块
	才允许发布 `global` 事件

3. **普通 Agent 不得默认发布全局事件**
	- 即使 Agent 能看到某些高价值信息，也不等于它天然拥有全局广播权
	- 若确需发布 `global` 事件，必须经过模板权限、Workspace 策略和 Controller 治理准入

这条规则的意义，是避免系统退化为“谁拿到一点信息，谁就可以把噪声广播给全平台”。

## 直接唤醒语义

平台的关键目标之一，是让 Agent 从“轮询驱动”转向“事件驱动”。

因此，订阅命中后不应只产生一条日志，而应支持：

- 恢复休眠中的 Agent
- 补齐恢复执行所需的上下文
- 根据订阅策略触发 Workflow、Task 或原生能力调用
- 在必要时进入审批或人工确认链路

职责边界：

- Controller 负责判定“这个订阅是否合法”
- Runtime 负责落实“命中后如何恢复并执行”

## 订阅执行权与 Agent 无感原则

为了避免把 Agent 设计成“既要推理，又要自己维护消息队列与订阅连接”的混合体，事件订阅应遵循下面的执行边界：

1. **Agent 只声明偏好，不直接订阅总线**
	- AgentTemplate 可以声明：关心什么事件、是否允许直接唤醒、默认优先级偏好。

2. **Controller 只做治理，不直接执行插入**
	- Controller 负责：权限、Workspace 边界、事件域、审批、是否允许高优先级插入。

3. **Runtime 负责落实命中与消息插入**
	- Runtime 持有 Session 热状态，因此只有 Runtime 知道：
		- 当前 LLM 是否正在一轮会话中
		- 当前执行片段是否允许被打断
		- 当前事件应该立即插入、延迟插入，还是排队等待
		- 被打断后如何恢复原来的任务流程

因此，事件订阅在运行时视角上更像：

- `ApprovedEventSubscription`：已批准的订阅
- `WakeupBinding`：Runtime 本地绑定
- `WakeupDecision`：本次命中后怎么做
- `InterruptDecision`：是否允许插入 / 打断 / 延迟

而不是“Agent 自己开一个消费者线程盯着消息”。那种模型对 demo 很友好，对复杂上下文恢复则通常不太友好，甚至会把 Agent 训练成兼职消息中间件管理员。

## 消息事件优先级：P0 / P1 / P2

并不是所有消息事件都应该以同一种方式插入到 LLM 上下文里。对于 Runtime 来说，至少应把“消息型事件”分成三档优先级：

- `P0`：紧急打断级
- `P1`：当前轮插入级
- `P2`：空闲后投递级

这里的“消息型事件”包括但不限于：

- 用户发给 Agent 的临时指令
- 审批结果或人工控制消息
- 需要 Agent 感知的系统广播
- 来自特定渠道的高优先级协作消息

### P0：紧急打断级

`P0` 代表必须立即让 Runtime 处理的消息事件。

建议语义：

- Runtime 会立即把该消息插入当前执行上下文
- 如果 LLM 当前仍处于活跃会话状态，Runtime **可以直接打断当前轮执行**
- 当前正在运行的主执行片段会被挂起，并生成恢复锚点

适用场景：

- 人工紧急指令
- 高优先级安全控制
- Freeze / Cancel / Stop 之类控制链路消息
- 经过特殊授权的关键协作通知

治理原则：

- `P0` **默认不允许公开使用**
- 只有具备明确权限的用户、系统模块或特殊 Agent 才能发布 `P0`
- `P0` 的发布、命中、打断、恢复都必须进入审计链路

## 冻结 / 恢复作为全局治理事件

冻结与恢复不应只被理解为某个 API 调用，而应被建模为 **全局治理事件**。

建议首批事件类型：

- `workspace.freeze.requested`
- `workspace.resume.requested`
- `agent.freeze.requested`
- `agent.resume.requested`
- `workspace.freeze.applied`
- `workspace.resume.applied`
- `agent.freeze.applied`
- `agent.resume.applied`

建议原则：

- `requested` 事件由 Controller 发布到 `global` 域
- 所有 Runtime 订阅这类全局治理事件
- 每个 Runtime 只处理自己当前承载、且命中 `workspaceId` / `agentId` 的实例
- `applied` 事件由 Runtime 在动作执行完成后回发，供 Platform、审计链路和 Controller 汇总结果

这里也再次体现一条总规则：冻结 / 恢复之所以能进入 `global` 域，不是因为它们特殊，而是因为它们属于控制层治理事件。

### 典型冻结广播链路

以用户点击 `Stop Workspace` 为例：

1. Platform 向 Controller 发送冻结命令
2. Controller 完成权限、审计和目标校验
3. Controller 发布 `workspace.freeze.requested` 全局事件
4. 所有 Runtime 订阅并收到该事件
5. 每个 Runtime 查找自己当前承载的 Agent 中是否存在该 `workspaceId`
6. 命中的 Runtime 将这些 Agent 迁移为 `Frozen`
7. Runtime stop 对应 Docker 容器或 sandbox
8. Runtime 发布 `workspace.freeze.applied` 事件，附带冻结 Agent 数、停止容器数和失败项

恢复链路与此对称，只是把 stop / freeze 替换为 restart / resume。

### P1：当前轮插入级

`P1` 是最常见、最实用的一类“会打断思路，但尽量不打爆现场”的消息事件。

建议语义：

- Runtime 不直接粗暴终止整个会话
- Runtime 会在当前一轮会话的输入组装阶段，把 `P1` 消息注入上下文
- Agent 在下一次推理时可以看到这条消息，并自行决定是否立即转向处理

关键要求：

- Runtime 必须在消息到来时记录一个**恢复锚点**
- 这个恢复锚点应指向记忆引擎中“被打断时我做到哪里”的位置
- 当 Agent 处理完 `P1` 事件后，Runtime 要能提供内置能力帮助其快速恢复原任务

可以把它类比成人类被打断时，先在便签上写一句“做到第 3 步，下一步该检查 xxx”，这样回来时不会从宇宙大爆炸重新思考一遍。

### P2：空闲后投递级

`P2` 是“不必立刻打扰 Agent”的一般消息事件。

建议语义：

- 如果 Agent 当前忙碌，Runtime 不主动打断
- Runtime 会等待当前会话结束，或者 Agent 进入空闲 / Sleeping / WaitingEvent 等合适状态后，再插入该消息
- 适用于延迟可接受、不要求即时处理的消息

适用场景：

- 一般通知
- 低风险协作消息
- 异步结果回传
- 可等下一轮处理的环境变化

## 中断、恢复与上下文锚点

只定义“能不能打断”是不够的。只要允许 `P0` / `P1` 插入，就必须把恢复合同一起设计进去。

Runtime 至少应维护下面几类对象或等价结构：

- `InterruptionRecord`：记录谁在什么时候被什么消息打断
- `ResumeAnchor`：记录原任务恢复点
- `ContextCheckpoint`：记录上下文快照或最小摘要
- `PendingMessageQueue`：延迟投递的消息队列

其中 `ResumeAnchor` 的核心目的不是保存整份上下文 dump，而是回答：

- 当消息到来时，我原来在做哪件事
- 我做到哪一步了
- 下一次恢复时应该先召回哪段记忆 / 哪条摘要 / 哪个执行点

## Runtime 为 Agent 提供的内置恢复能力

为了让 Agent 在处理完打断消息后快速恢复原工作，Runtime 应提供内置恢复能力，而不是要求 Agent 自己凭空回忆：

- `recall_interrupted_work`：查询最近一次被打断的任务锚点
- `list_resume_anchors`：查看当前 Session 可恢复的上下文索引
- `resume_previous_flow`：请求 Runtime 按指定锚点恢复原任务链

这些能力的底层不一定表现为对外显式工具，但在架构语义上，Runtime 必须具备这样的内置支持。

建议最小索引内容：

- `anchorId`
- `sessionId`
- `agentId`
- `workflowRunId?`
- `interruptedByEventId`
- `interruptedAt`
- `memoryPointer`
- `summary`
- `resumePriority`

其中：

- `memoryPointer` 指向记忆引擎或执行日志中的特定恢复位置
- `summary` 记录“打断前做到哪里”
- `resumePriority` 决定后续恢复排序

## 优先级消息的治理约束

消息优先级不能只是一枚字段，还必须和权限与治理绑定：

- `P0`：默认只允许系统控制消息、特殊授权用户、特殊授权 Agent 发布
- `P1`：允许常规协作消息进入当前轮，但仍受 Workspace 策略、频率限制与预算约束
- `P2`：允许广泛使用，但 Runtime 可按忙碌程度延迟投递

同时建议：

- 对 `P0`/`P1` 设置频率限制，避免消息风暴
- 对同一来源在短时间内的重复消息做合并、防抖或折叠
- 对无权限发布高优先级消息的行为进行拒绝和审计

## 路由模型

典型路由链：

1. Adapter 接收外部原始消息
2. Gateway 把协议数据转换为标准事件 Envelope
3. Controller 执行来源识别、信任分级、Workspace 命中、事件域判定
4. 事件进入对应事件流
5. Runtime / Agent / 审计链路消费事件
6. 如需回写，由 Gateway 再翻译回原始协议

## 幂等与去重

事件系统必须假设重复投递会发生。

因此至少要有：

- 基于 `eventId` 的消费幂等
- 基于 `correlationId` 的链路聚合
- Adapter 入站去重策略
- Runtime 唤醒去重与短时间窗口防抖

尤其是设备侧、Webhook 重试、网络抖动场景下，如果不做幂等，很容易把一次真实变化变成一连串“幻觉风暴”。

对 `P0` / `P1` / `P2` 消息事件，还应增加：

- 基于 `eventId + targetSessionId` 的插入幂等
- 基于 `correlationId` 的恢复锚点聚合
- 基于短时间窗口的重复消息折叠
- 基于 `priority` 的排队 / 延迟 / 打断判定去重

## 重放与补偿

事件总线不应只是“来了就过”。为了支持调试、补偿、审计与恢复，至少要考虑：

- 指定时间窗口或游标后的事件重放
- 只对某个 Workspace 域重放
- 只对某类订阅者开放重放权限
- 重放时附加 `replayed=true` 或等价元数据，避免与实时事件混淆

重放不是默认能力，而是受治理和审计约束的高级能力。

## 死信与失败隔离

并不是所有事件都能成功消费。

因此需要死信隔离对象或等价机制，覆盖：

- 无法通过权限校验的事件
- 无法路由到目标 Workspace 的事件
- 被订阅者反复消费失败的事件
- payload 不合法或版本不兼容的事件
- 超出预算、频率或安全策略限制的事件

死信事件不应悄悄丢弃，而应：

- 可查询
- 可审计
- 可告警
- 在有限条件下可人工修复后重放

## MQTT 智能家居示例

一个典型链路如下：

1. 智能家居设备通过 MQTT 连接到 Gateway 的 MQTT Adapter
2. Adapter 订阅设备 Topic
3. 设备发出告警或状态变化
4. Gateway 转换为统一事件，例如 `device.sensor.triggered`
5. Controller 判定其属于特定 Workspace 域
6. Runtime 中某个 Agent 已订阅该类事件
7. 订阅命中后，该 Agent 被直接唤醒
8. Agent 触发通知、工作流、宿主原生能力或其他设备联动

关键点不在于“接了 MQTT”，而在于：外部变化被纳入统一治理事件流，而不是要求 Agent 一直轮询设备状态。

## 与可观测性的关系

事件系统本身必须可观测。至少要能看到：

- 事件吞吐量
- 事件路由命中率
- 订阅命中率
- 唤醒成功率
- 死信数量
- 重放次数
- 事件端到端延迟

否则系统出了问题时，只会得到一句经典回答：

> “理论上它应该已经收到事件了。”

而“理论上”通常是线上事故最不靠谱的同事。

## Workspace 日志系统与事件总线的关系

日志系统不应独立于事件系统另起炉灶。

更稳的设计是：

- **Agent 执行日志** 由 Runtime 在本地执行链路上产生
- **全局事件日志** 由统一事件总线产生与沉淀
- **Workspace 聚合日志** 由日志投影层对前两者进行汇总、聚合和告警推导

可以把它理解为：

```text
Agent (local execution) ---> Runtime ---> Event Bus
	|                       |
   Agent Log Buffer       Event Log Buffer
	|                       |
	+------> Log Storage / Projection <+
```

其中：

- Runtime 负责把局部执行痕迹写入 Agent 日志缓冲区
- Event Bus 负责承载和广播统一事件
- 日志存储 / 投影层负责查询、聚合、索引、保留和 UI 视图支撑

## 日志分层建议

### 1. Agent 执行日志

用于记录单个 Agent 的局部执行轨迹，例如：

- 节点状态变化
- 工具调用
- 资源绑定
- 异常 / 错误
- 执行结果摘要

建议由 Runtime 写入本地缓冲区，再异步持久化到日志存储。

### 2. 全局事件日志

用于记录所有经过统一总线的关键事件，例如：

- `task_assigned`
- `task_taken`
- `task_done`
- `resource_locked`
- `resource_released`
- `alert`
- `exception`

这类日志天然适合作为跨 Agent 协作分析的主时间轴。

### 3. Workspace 聚合日志

用于把局部日志与事件日志压缩成适合 UI 与告警的聚合视图，例如：

- 某个时间窗口内的 Agent 活跃度
- 节点完成率 / 失败率
- 事件密度热点
- 高风险异常聚合

注意：这类对象不是原始事件替代品，而是面向驾驶舱的派生投影。

## 存储建议

日志存储建议分层考虑：

- **结构化主存储**：优先 PostgreSQL；若后续写入量和查询模式需要，也可扩展 MongoDB
- **检索优化层**：可选 Elasticsearch，用于高吞吐、多维过滤与全文检索
- **热缓冲 / 队列**：Runtime 本地 buffer 或事件日志 buffer，用于削峰和异步落盘

V1 阶段不必一开始就强依赖 Elasticsearch，但数据模型应预留：

- 时间区间查询
- 按 `workspace / agent / task / eventType` 过滤
- 关键字段索引

## 查询与回溯原则

日志系统建议至少支持：

- 单 Agent 时间轴查询
- Workspace 事件时间轴查询
- 按节点 / 任务 / Agent / 资源的关联查询
- 历史回放与复盘查询

同时应遵循：

- 事件驱动优先：agent-to-agent 消息优先转化为事件痕迹
- 可追踪、可回溯：支持复杂 TaskMap / DAG 的复盘
- 聚合优先展示：默认不是原始日志 dump
- Workspace 隔离：默认仅允许本 Workspace 可见

## 第一阶段最小落地建议

V1 不必一开始就做成通用流处理平台，但至少应完成：

- 统一事件 Envelope
- `global` / `workspace` 两级事件域
- 1 条外部事件驱动链路（例如 MQTT 或 Webhook）
- 1 条内部事件驱动链路（例如任务完成或 Agent 完成任务）
- 1 条 Agent 订阅并直接唤醒链路
- 基础死信隔离与审计记录

如果本阶段要把消息事件中断能力也纳入第一批能力，建议最小追加：

- 支持 `P1` 当前轮插入级消息
- 支持 Runtime 记录最小 `ResumeAnchor`
- 支持 Agent 处理完打断消息后恢复原任务

`P0` 可以作为后续增强能力，但其权限、频控与审计必须先写清楚再开放。
