# 事件系统与事件总线

## 定位

事件系统不是 `PuddingController` 的一个可选附件，而是整个协作网络的基础协作面。

如果说：

- `PuddingController` 负责治理、路由与控制
- `PuddingGateway` 负责协议接入与边界转换
- `PuddingRuntime` 负责执行与唤醒

那么事件总线负责把它们连成一个真正的事件驱动系统。

一句话原则：**万物皆事件**。

但这句话并不等于“所有事件都可以被任何人看到或消费”。事件始终受 Workspace、权限、审批、信任等级与治理策略约束。

## 为什么必须有统一事件总线

没有统一事件总线时，系统容易退化为：

- Adapter 收到消息后直接特判调用某个 Controller 分支
- Runtime 只能通过轮询发现外部变化
- Agent 只能在收到显式消息时工作，无法订阅环境变化
- 审计、回放、重放、补偿都缺乏统一入口

统一事件总线的价值在于：

- 让外部协议世界统一翻译成平台事件世界
- 让内部状态变化也采用统一事件模型
- 让 Agent、Workflow、Runtime、审计链路消费同一类对象
- 让订阅、重放、死信、审计、回溯都具备统一语义

更重要的是，它直接回应了很多现有个人智能体系统的问题：

- 只有心跳，没有事件队列
- Agent 靠轮询等待任务，延迟高且浪费 token
- 外部系统接入后只能走特判逻辑，难以演进
- GitHub、MQTT、Webhook、WebSocket 这类真实企业触发器很难成为一等输入

Pudding 选择的方向是：**event-driven AI runtime**，而不是“带心跳的聊天机器人宿主”。

## 非目标

Pudding 的事件系统并不以“替代所有外部系统 API”或“做一个通用消息中间件产品”为目标。

它的目标是：

- 为 Agent Runtime 提供统一事件语义
- 为 Workspace 提供可治理的事件命名空间
- 为多智能体协作提供低延迟、低 token 浪费的唤醒机制
- 为企业集成提供稳定协议适配层

## 事件源分类

### 外部事件

来自 Gateway Adapter 或边界接入层：

- MQTT 设备状态变化
- Webhook 回调
- HTTP 渠道消息
- WebSocket 会话消息
- Email 入站邮件
- 嵌入式 Runtime 宿主上送的原生状态事件

### 内部事件

来自 Controller、Runtime、Workflow 或系统基础设施：

- `task.completed`
- `agent.completed`
- `agent.failed`
- `runtime.heartbeat`
- `runtime.capacity.changed`
- `approval.requested`
- `approval.completed`
- `memory.candidate.created`
- `workflow.step.completed`
- `audit.record.created`

## 事件域

事件不是天然全局广播。平台至少需要两层事件域：

- `global`：平台级治理、节点健康、跨 Workspace 协作、基础设施事件
- `workspace`：绝大多数业务事件、渠道事件、Agent 协作事件

默认原则：

- 业务事件优先进入 `workspace` 域
- 只有治理、基础设施或明确跨域协作的事件才进入 `global` 域
- 从 `workspace` 升级到 `global` 必须是显式策略，而不是默认行为

## 统一事件 Envelope

建议统一事件对象至少包含：

- `eventId`：事件唯一标识
- `eventType`：事件类型，采用稳定命名规范
- `scope`：`global` 或 `workspace`
- `workspaceId`：Workspace 域事件的边界标识
- `source`：来源 Adapter、Runtime、Agent 或系统模块
- `subject`：关联对象，例如设备、任务、Session、Agent
- `payload`：结构化负载
- `trustLevel`：来源信任等级
- `correlationId`：跨链路关联 ID
- `causationId`：导致本事件的上游事件 ID
- `occurredAt`：事件发生时间
- `publishedAt`：事件进入总线时间
- `version`：事件契约版本

## 事件命名规范

事件类型建议统一采用小写点分命名：

- `device.sensor.triggered`
- `device.alarm.triggered`
- `task.completed`
- `task.failed`
- `agent.completed`
- `runtime.heartbeat`
- `workflow.step.completed`
- `approval.requested`

命名原则：

- 先领域，后对象，再动作
- 动作使用稳定动词过去式或状态式
- 不把底层协议名写进业务事件名，避免出现 `mqtt.xxx.received` 这类协议泄漏型命名
- 协议信息保留在 `source` 与 `payload` 元数据里，而不是污染事件语义

## 订阅模型

事件总线上的订阅者不应只有基础设施组件，也包括：

- Runtime 内部组件
- Agent
- sub_agent
- Workflow Worker
- 审计链路
- 调试/可观测性组件
- 高风险审批与治理模块

订阅声明至少需要包含：

- 订阅者身份
- 允许可见的事件域
- 感兴趣的事件类型或模式
- 过滤条件
- 是否允许直接唤醒
- 消费失败策略
- 是否允许重放

## 直接唤醒语义

平台的关键目标之一，是让 Agent 从“轮询驱动”转向“事件驱动”。

因此，订阅命中后不应只产生一条日志，而应支持：

- 恢复休眠中的 Agent
- 补齐恢复执行所需的上下文
- 根据订阅策略触发 Workflow、Task 或原生能力调用
- 在必要时进入审批或人工确认链路

职责边界：

- Controller 负责判定“这个订阅是否合法”
- Runtime 负责落实“命中后如何恢复并执行”

## 路由模型

典型路由链：

1. Adapter 接收外部原始消息
2. Gateway 把协议数据转换为标准事件 Envelope
3. Controller 执行来源识别、信任分级、Workspace 命中、事件域判定
4. 事件进入对应事件流
5. Runtime / Agent / 审计链路消费事件
6. 如需回写，由 Gateway 再翻译回原始协议

## 幂等与去重

事件系统必须假设重复投递会发生。

因此至少要有：

- 基于 `eventId` 的消费幂等
- 基于 `correlationId` 的链路聚合
- Adapter 入站去重策略
- Runtime 唤醒去重与短时间窗口防抖

尤其是设备侧、Webhook 重试、网络抖动场景下，如果不做幂等，很容易把一次真实变化变成一连串“幻觉风暴”。

## 重放与补偿

事件总线不应只是“来了就过”。为了支持调试、补偿、审计与恢复，至少要考虑：

- 指定时间窗口或游标后的事件重放
- 只对某个 Workspace 域重放
- 只对某类订阅者开放重放权限
- 重放时附加 `replayed=true` 或等价元数据，避免与实时事件混淆

重放不是默认能力，而是受治理和审计约束的高级能力。

## 死信与失败隔离

并不是所有事件都能成功消费。

因此需要死信隔离对象或等价机制，覆盖：

- 无法通过权限校验的事件
- 无法路由到目标 Workspace 的事件
- 被订阅者反复消费失败的事件
- payload 不合法或版本不兼容的事件
- 超出预算、频率或安全策略限制的事件

死信事件不应悄悄丢弃，而应：

- 可查询
- 可审计
- 可告警
- 在有限条件下可人工修复后重放

## MQTT 智能家居示例

一个典型链路如下：

1. 智能家居设备通过 MQTT 连接到 Gateway 的 MQTT Adapter
2. Adapter 订阅设备 Topic
3. 设备发出告警或状态变化
4. Gateway 转换为统一事件，例如 `device.sensor.triggered`
5. Controller 判定其属于特定 Workspace 域
6. Runtime 中某个 Agent 已订阅该类事件
7. 订阅命中后，该 Agent 被直接唤醒
8. Agent 触发通知、工作流、宿主原生能力或其他设备联动

关键点不在于“接了 MQTT”，而在于：外部变化被纳入统一治理事件流，而不是要求 Agent 一直轮询设备状态。

## 与可观测性的关系

事件系统本身必须可观测。至少要能看到：

- 事件吞吐量
- 事件路由命中率
- 订阅命中率
- 唤醒成功率
- 死信数量
- 重放次数
- 事件端到端延迟

否则系统出了问题时，只会得到一句经典回答：

> “理论上它应该已经收到事件了。”

而“理论上”通常是线上事故最不靠谱的同事。

## Workspace 日志系统与事件总线的关系

日志系统不应独立于事件系统另起炉灶。

更稳的设计是：

- **Agent 执行日志** 由 Runtime 在本地执行链路上产生
- **全局事件日志** 由统一事件总线产生与沉淀
- **Workspace 聚合日志** 由日志投影层对前两者进行汇总、聚合和告警推导

可以把它理解为：

```text
Agent (local execution) ---> Runtime ---> Event Bus
	|                       |
   Agent Log Buffer       Event Log Buffer
	|                       |
	+------> Log Storage / Projection <+
```

其中：

- Runtime 负责把局部执行痕迹写入 Agent 日志缓冲区
- Event Bus 负责承载和广播统一事件
- 日志存储 / 投影层负责查询、聚合、索引、保留和 UI 视图支撑

## 日志分层建议

### 1. Agent 执行日志

用于记录单个 Agent 的局部执行轨迹，例如：

- 节点状态变化
- 工具调用
- 资源绑定
- 异常 / 错误
- 执行结果摘要

建议由 Runtime 写入本地缓冲区，再异步持久化到日志存储。

### 2. 全局事件日志

用于记录所有经过统一总线的关键事件，例如：

- `task_assigned`
- `task_taken`
- `task_done`
- `resource_locked`
- `resource_released`
- `alert`
- `exception`

这类日志天然适合作为跨 Agent 协作分析的主时间轴。

### 3. Workspace 聚合日志

用于把局部日志与事件日志压缩成适合 UI 与告警的聚合视图，例如：

- 某个时间窗口内的 Agent 活跃度
- 节点完成率 / 失败率
- 事件密度热点
- 高风险异常聚合

注意：这类对象不是原始事件替代品，而是面向驾驶舱的派生投影。

## 存储建议

日志存储建议分层考虑：

- **结构化主存储**：优先 PostgreSQL；若后续写入量和查询模式需要，也可扩展 MongoDB
- **检索优化层**：可选 Elasticsearch，用于高吞吐、多维过滤与全文检索
- **热缓冲 / 队列**：Runtime 本地 buffer 或事件日志 buffer，用于削峰和异步落盘

V1 阶段不必一开始就强依赖 Elasticsearch，但数据模型应预留：

- 时间区间查询
- 按 `workspace / agent / task / eventType` 过滤
- 关键字段索引

## 查询与回溯原则

日志系统建议至少支持：

- 单 Agent 时间轴查询
- Workspace 事件时间轴查询
- 按节点 / 任务 / Agent / 资源的关联查询
- 历史回放与复盘查询

同时应遵循：

- 事件驱动优先：agent-to-agent 消息优先转化为事件痕迹
- 可追踪、可回溯：支持复杂 TaskMap / DAG 的复盘
- 聚合优先展示：默认不是原始日志 dump
- Workspace 隔离：默认仅允许本 Workspace 可见

## 第一阶段最小落地建议

V1 不必一开始就做成通用流处理平台，但至少应完成：

- 统一事件 Envelope
- `global` / `workspace` 两级事件域
- 1 条外部事件驱动链路（例如 MQTT 或 Webhook）
- 1 条内部事件驱动链路（例如任务完成或 Agent 完成任务）
- 1 条 Agent 订阅并直接唤醒链路
- 基础死信隔离与审计记录
