# 协作网络与治理

## 协作网络模型

系统天然支持多用户、多渠道、多 Workspace、多 Agent。用户先进入渠道，再由 Gateway 和 Controller 映射到 Workspace、Session 与 Agent 网络。

## Workspace 作为治理边界

- 一个 Workspace 可绑定多个 ChannelBinding。
- 一个 Workspace 可容纳多个 AgentTemplate 与 Workflow。
- Agent 默认必须属于某个 Workspace，不能是无归属的全局 Agent。
- MemoryPolicy、PluginBinding、服务暴露策略默认归属 Workspace。

## Session 作为运行时边界

- Session 是执行隔离边界。
- Runtime 持有 Session 热状态。
- 默认禁止跨 Session、跨 Workspace、跨 Organization 自动共享记忆与上下文。

## 渠道与信任分级

- 高信任渠道：本机管理员、内部受控系统。
- 中信任渠道：内部群、论坛、Issue 系统。
- 低信任渠道：公开群、服务台、开放入口。

公开或低信任输入默认只进入受限服务会话，不直接污染长期记忆，不自动放行高风险能力。

Gateway Adapter 在进入 Controller 主流程前，应先完成来源声明、能力声明、事件类型标记和基础信任分级。不同 Adapter 即使都映射到同一 Workspace，也不应天然继承相同信任等级。

## 治理与安全

- Ask、CreateTask、ExecuteTool、WriteMemory、TriggerSwarm 必须拆开授权。
- 高风险动作要经过审批链、审计链或人工确认。
- 审计 Agent 必须尽量只接触结构化、脱敏、受限视图。
- 每个 Workspace 至少应拥有 1 个审计 Agent。
- Adapter 插件的安装、启用、升级、熔断与冻结必须纳入控制面审计。
- 嵌入式桌面宿主通过 Adapter 暴露的原生能力，必须同时受模板权限、Workspace 策略和审批链约束。

## 关键基础设施

- Workspace 知识库：目录文件、RAG、向量检索、知识提升。
- 统一存储层：NFS、对象存储、跨 Runtime 共享产物。
- 知识图谱：Workspace 共享的结构化知识资产。
