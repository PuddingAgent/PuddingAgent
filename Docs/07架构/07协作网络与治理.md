# 协作网络与治理

## 协作网络模型

系统天然支持多用户、多渠道、多 Workspace、多 Agent。用户先进入渠道，再由 Gateway 和 Controller 映射到 Workspace、Session 与 Agent 网络。

但 Pudding 要解决的不是“很多 Agent 同时存在”，而是“很多 Agent 如何稳定协作”。

协同的核心问题始终包括：

- 谁决策
- 谁执行
- 谁监督
- 谁汇报
- 谁保留共享记忆与状态痕迹

## Workspace 作为治理边界

- 一个 Workspace 可绑定多个 ChannelBinding。
- 一个 Workspace 可容纳多个 AgentTemplate 与 Workflow。
- Agent 默认必须属于某个 Workspace，不能是无归属的全局 Agent。
- MemoryPolicy、PluginBinding、服务暴露策略默认归属 Workspace。
- Workspace 事件域默认也是治理边界的一部分；大多数业务事件只应在所属 Workspace 内可见、可订阅、可消费。

## Session 作为运行时边界

- Session 是执行隔离边界。
- Runtime 持有 Session 热状态。
- 默认禁止跨 Session、跨 Workspace、跨 Organization 自动共享记忆与上下文。

## 渠道与信任分级

- 高信任渠道：本机管理员、内部受控系统。
- 中信任渠道：内部群、论坛、Issue 系统。
- 低信任渠道：公开群、服务台、开放入口。

公开或低信任输入默认只进入受限服务会话，不直接污染长期记忆，不自动放行高风险能力。

Gateway Adapter 在进入 Controller 主流程前，应先完成来源声明、能力声明、事件类型标记和基础信任分级。不同 Adapter 即使都映射到同一 Workspace，也不应天然继承相同信任等级。

## 事件治理

- 平台遵循“万物皆事件”的建模方式，但不是所有事件都默认公开给所有 Agent。
- 事件的发布、订阅、转发、重放、升级为任务，必须受 Workspace 策略约束。
- 外部事件进入系统后，必须先完成协议转换、来源识别、信任分级和 Workspace 命中。
- Agent 只能订阅其模板权限、Workspace 策略和审批链允许的事件类型。
- 心跳、健康检查、容量变化等基础设施事件可进入全局域，但业务消费应谨慎限制，避免把治理噪声扩散到业务链路。
- 任务完成、Agent 完成任务、审批结果、设备告警等高价值事件，应支持被特定 Agent 订阅并触发直接唤醒。

## 协同机制探索

当前常见的多智能体协同模式大致有三类：

1. **Leader 模式**
	由一个 Leader Agent 负责拆解、派发、收口和复核。

2. **投票 / 联邦模式**
	多个 Agent 分别给出意见，再由聚合器或仲裁器收敛结果。

3. **自发模式**
	不预设固定领导者，依赖局部规则、环境变化和共享痕迹驱动协作。

Pudding 不应只支持其中一种模式，而应把它们视为可切换的协同协议。

### 蜂群 / 蚁群启发

现实中的蚂蚁和蜜蜂并不依赖复杂中心协议，但它们仍能形成稳定协同。其启发不在于“复制自然界行为”，而在于学习这些原则：

- **局部规则优先**：个体不需要掌握全局图，只需要遵循清晰的小规则。
- **事件痕迹驱动**：环境中的痕迹、状态、告警、资源变化，会成为后续个体的触发器。
- **轻协议胜过重计划**：在复杂环境中，过重的协调协议反而容易成为瓶颈。
- **角色可以动态形成**：谁离信息最近、谁最有能力处理，就可能临时成为局部协调者。

对 Pudding 的意义是：

- 协同不应只依赖单一 Leader。
- 事件流本身可以成为协作痕迹。
- Workspace 内共享的任务状态、事件日志、审计记录、记忆候选，可以形成一种“数字信息素”。
- Agent 可以根据局部事件和策略触发协作，而不是永远等待中心调度器逐项发令。

这意味着 Pudding 的协同模型应同时支持：

- 显式 Leader 调度
- 审批/投票式收敛
- 基于事件痕迹的自发协作

## 治理与安全

- Ask、CreateTask、ExecuteTool、WriteMemory、TriggerSwarm 必须拆开授权。
- 高风险动作要经过审批链、审计链或人工确认。
- 审计 Agent 必须尽量只接触结构化、脱敏、受限视图。
- 每个 Workspace 至少应拥有 1 个审计 Agent。
- Adapter 插件的安装、启用、升级、熔断与冻结必须纳入控制面审计。
- 嵌入式桌面宿主通过 Adapter 暴露的原生能力，必须同时受模板权限、Workspace 策略和审批链约束。
- 事件订阅关系本身也必须可审计，避免 Agent 在无显式授权的情况下长期监听高敏感事件。

## 关键基础设施

- Workspace 知识库：目录文件、RAG、向量检索、知识提升。
- 统一存储层：NFS、对象存储、跨 Runtime 共享产物。
- 知识图谱：Workspace 共享的结构化知识资产。
