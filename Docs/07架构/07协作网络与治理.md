# 协作网络与治理

## 协作网络模型

系统天然支持多用户、多渠道、多 Workspace、多 Agent。用户先进入渠道，再由 Gateway 和 Controller 映射到 Workspace、Session 与 Agent 网络。

但 Pudding 要解决的不是“很多 Agent 同时存在”，而是“很多 Agent 如何稳定协作”。

协同的核心问题始终包括：

- 谁决策
- 谁执行
- 谁监督
- 谁汇报
- 谁保留共享记忆与状态痕迹

Pudding 在这件事上的目标，不只是支持“多个 Agent 同时在线”，而是支持一种可治理的 **数字蚁群（Digital Ant Colony）** 协作模型：

- 复杂任务被抽象为图，而不是一段模糊对话
- Agent 像蚂蚁一样感知局部状态，而不是必须掌握全局全量信息
- 协作主要通过事件痕迹发生，而不是靠 Agent 之间建立大量点对点私聊
- 编排可以存在，但不应成为所有协作都必须经过的单点瓶颈

## 任务图 / 地图模型

为了让多 Agent 协作可解释、可恢复、可审计，复杂任务不应只停留在提示词里，而应被显式建模为 **任务图（Task Graph）** 或更宽泛的 **地图（Map）**。

建议最小抽象：

- `TaskMap`：整个任务空间或工作区域
- `TaskNode`：原子子任务节点
- `TaskEdge`：节点之间的依赖、触发或约束关系
- `TaskResourceBinding`：节点对工具、容器、知识、宿主能力的资源绑定
- `TaskEventHook`：节点生命周期上的事件钩子

### 为什么不只叫 DAG

很多流程确实可以被表达为 DAG，但 Pudding 的协作空间不应被 DAG 这个词锁死：

- 有些任务是明确依赖图，适合 DAG
- 有些任务更像探索地图，节点会在运行中被发现、展开、拆分、合并
- 有些任务存在事件回流、补偿、重试和人工介入，不适合只用静态 DAG 思维描述

所以更稳的表达是：

- **DAG 是 TaskMap 的一种特例**
- Pudding 的任务表达应优先面向 `Map / Graph`，而不是只能表达静态 DAG

## 数字蚁群模型

在这个模型下：

- **TaskMap / DAG** 是地形与道路
- **TaskNode** 是可被占领、处理、推进的工作点
- **Agent** 是在地图上移动的执行者
- **事件流** 是痕迹、告警、状态变化与数字信息素
- **Workspace** 是协作蜂巢，也是治理与隔离边界

Agent 不需要掌握整个地图的每一个角落，而只需要知道：

- 自己当前能看到哪些节点
- 哪些节点已满足前置条件
- 哪些节点与自己的 role / skill / workload 匹配
- 哪些节点因事件变化而变得值得接手

这使系统更接近：

- 局部感知
- 事件驱动推进
- 动态分工
- 容错接管

而不是“所有执行都先找中央大脑排一遍班表”。

## Workspace 作为治理边界

- 一个 Workspace 可绑定多个 ChannelBinding。
- 一个 Workspace 可容纳多个 AgentTemplate 与 Workflow。
- Agent 默认必须属于某个 Workspace，不能是无归属的全局 Agent。
- MemoryPolicy、PluginBinding、服务暴露策略默认归属 Workspace。
- Workspace 事件域默认也是治理边界的一部分；大多数业务事件只应在所属 Workspace 内可见、可订阅、可消费。

## Session 作为运行时边界

- Session 是执行隔离边界。
- Runtime 持有 Session 热状态。
- 默认禁止跨 Session、跨 Workspace、跨 Organization 自动共享记忆与上下文。

## 渠道与信任分级

- 高信任渠道：本机管理员、内部受控系统。
- 中信任渠道：内部群、论坛、Issue 系统。
- 低信任渠道：公开群、服务台、开放入口。

公开或低信任输入默认只进入受限服务会话，不直接污染长期记忆，不自动放行高风险能力。

Gateway Adapter 在进入 Controller 主流程前，应先完成来源声明、能力声明、事件类型标记和基础信任分级。不同 Adapter 即使都映射到同一 Workspace，也不应天然继承相同信任等级。

## 事件治理

- 平台遵循“万物皆事件”的建模方式，但不是所有事件都默认公开给所有 Agent。
- 事件的发布、订阅、转发、重放、升级为任务，必须受 Workspace 策略约束。
- 外部事件进入系统后，必须先完成协议转换、来源识别、信任分级和 Workspace 命中。
- Agent 只能订阅其模板权限、Workspace 策略和审批链允许的事件类型。
- 心跳、健康检查、容量变化等基础设施事件可进入全局域，但业务消费应谨慎限制，避免把治理噪声扩散到业务链路。
- 任务完成、Agent 完成任务、审批结果、设备告警等高价值事件，应支持被特定 Agent 订阅并触发直接唤醒。

在协作场景里，建议重点支持以下事件族：

- `task.node.ready`
- `task.node.claimed`
- `task.node.started`
- `task.node.completed`
- `task.node.failed`
- `task.node.blocked`
- `task.node.released`
- `task.map.expanded`
- `task.map.completed`
- `agent.workload.changed`
- `sandbox.faulted`

这些事件不只是日志，而是协作网络中的行为触发器。

## 协同机制探索

当前常见的多智能体协同模式大致有三类：

1. **Leader 模式**
	由一个 Leader Agent 负责拆解、派发、收口和复核。

2. **投票 / 联邦模式**
	多个 Agent 分别给出意见，再由聚合器或仲裁器收敛结果。

3. **自发模式**
	不预设固定领导者，依赖局部规则、环境变化和共享痕迹驱动协作。

Pudding 不应只支持其中一种模式，而应把它们视为可切换的协同协议。

### 编排者 Agent 不是协作的反义词

你提到“我们也需要一个编排者 agent”，我认同，而且这点很关键。

Pudding 不应该在“中心编排”与“完全自发”之间二选一，而应同时支持两层：

1. **显式编排层**
	- 由编排者 Agent（Orchestrator / Coordinator）负责目标澄清、任务拆分、优先级调整、收敛与异常干预。

2. **局部自发层**
	- 其他 Agent 基于事件、节点状态、资源可用性和角色偏好，自主选择可执行节点并推进任务图。

也就是说：

- 编排者 Agent 负责“给地图定边界、补规则、收残局”
- 普通执行 Agent 负责“在地图上探索、占领、执行、回报”

这比“所有事都由 Leader 手动派发”更能扩展，也比“完全没有协调者”更可控。

### 蜂群 / 蚁群启发

现实中的蚂蚁和蜜蜂并不依赖复杂中心协议，但它们仍能形成稳定协同。其启发不在于“复制自然界行为”，而在于学习这些原则：

- **局部规则优先**：个体不需要掌握全局图，只需要遵循清晰的小规则。
- **事件痕迹驱动**：环境中的痕迹、状态、告警、资源变化，会成为后续个体的触发器。
- **轻协议胜过重计划**：在复杂环境中，过重的协调协议反而容易成为瓶颈。
- **角色可以动态形成**：谁离信息最近、谁最有能力处理，就可能临时成为局部协调者。

对 Pudding 的意义是：

- 协同不应只依赖单一 Leader。
- 事件流本身可以成为协作痕迹。
- Workspace 内共享的任务状态、事件日志、审计记录、记忆候选，可以形成一种“数字信息素”。
- Agent 可以根据局部事件和策略触发协作，而不是永远等待中心调度器逐项发令。

这意味着 Pudding 的协同模型应同时支持：

- 显式 Leader 调度
- 审批/投票式收敛
- 基于事件痕迹的自发协作

## 协作机制分层建议

结合现有架构，更建议把协作拆成五层，而不是混成一个“Swarm”黑箱：

| 层级 | 作用 |
| --- | --- |
| `TaskMap / DAG` | 描述任务结构、节点依赖、资源绑定与事件钩子 |
| `Agent` | 订阅事件、声明角色、选择节点、执行操作、发布结果 |
| `Event Bus` | 广播节点状态变化、协作信号、失败与恢复事件 |
| `Runtime` | 管理信号投递、执行上下文、sandbox 生命周期、资源绑定 |
| `Workspace` | 作为隔离域，决定共享范围、可见范围、审计和权限边界 |

这层次里：

- Runtime 负责信号路由与资源管理
- Event Bus 负责协作痕迹传播
- Agent 负责局部决策和执行
- Workspace 负责隔离与治理

不要让 Runtime 替 Agent 做全部决策，也不要让 Agent 无视 Workspace 治理边界自由乱窜。

## Agent 协作流程建议

一个更贴近数字蚁群模型的协作流程可以是：

1. **任务发布**
	- 编排者 Agent、用户或系统将 `TaskMap` / `TaskNode` 发布到 Workspace 域
	- Controller / Runtime 将节点状态转化为可消费事件

2. **节点探索**
	- Agent 订阅 `task.node.ready` 等事件
	- 根据 role、skill、资源、当前工作负载选择可执行节点

3. **执行与状态更新**
	- Agent attach 到对应 sandbox / bash / host capability 执行操作
	- 执行完成后更新节点状态，并发布完成、失败、阻塞或释放事件

4. **动态分工与再平衡**
	- 其他 Agent 根据事件自动改变行为
	- 负载高、失败率高或资源冲突时，可以放弃节点、转移节点或等待重试

5. **收敛与回顾**
	- 编排者 Agent、审计 Agent 或 Leader Agent 对任务图状态进行收敛判断
	- 必要时补充新节点、关闭旧节点、触发人工审批或结束整个任务图

## 冲突处理原则

数字蚁群不代表完全没有冲突，而是尽量避免把冲突处理升级成全局大锁。

建议原则：

- 优先使用 `claim -> execute -> release/complete` 的轻量占领语义
- 优先依赖事件修正与幂等，而不是全局串行化所有节点
- 允许节点被重新发布、重试、转移，但必须保留因果链与审计痕迹
- 对高价值节点可采用更严格的 claim TTL、审批或人工确认

建议最小节点状态至少包括：

- `Pending`
- `Ready`
- `Claimed`
- `Running`
- `Blocked`
- `Completed`
- `Failed`
- `Released`

## 编排者 Agent 的边界

编排者 Agent 很重要，但它不应成为新的万能上帝模块。

它适合负责：

- 拆解任务图
- 定义优先级与验收标准
- 处理阻塞节点
- 收敛整体结果
- 决定是否补充新节点或重分配角色

它不适合直接负责：

- 每个底层 bash 命令的具体执行
- 全局治理规则的最终裁决
- 绕过 Controller 直接修改权限与订阅
- 绕过 Runtime 直接操纵容器生命周期

换句话说，编排者 Agent 是协作层的“调度者”，不是系统层的“超级管理员”。

## 容错与接管

如果某个 Agent、节点或 sandbox 失败，协作系统不应整体卡死。

建议最小接管原则：

- Agent 失败 -> 节点回到 `Released` / `Failed`，等待其他 Agent 接手或编排者处理
- Sandbox 失败 -> Runtime 上报 `sandbox.faulted`，节点进入阻塞或重建流程
- Runtime 失败 -> Controller 根据恢复状态与放置策略决定是否迁移或重试
- 编排者失败 -> 任务图仍应保留状态痕迹，允许新的编排者继续接管

## 协作监测视图原则

多 Agent 协作系统如果把每一条原子事件都直接摊给用户，最后通常会得到一个非常热闹、但不太可理解的页面。

因此，Pudding 的 Workspace 协作监测视图建议遵循以下原则：

- 默认展示聚合状态，而不是原子事件洪流
- 主视图聚焦 `TaskMap / DAG`
- Agent 作为节点执行者附着在任务图上，而不是反客为主
- 关键事件单独进入摘要流，原子事件仅在下钻时显示

推荐的主监测视角：

1. 节点 / 任务状态汇总
2. Agent 当前活动节点
3. 关键事件摘要流
4. 历史轨迹与接管记录

这会让系统更像“协作任务驾驶舱”，而不是“多 Agent 原子日志刷屏器”。

## Workspace 共享协作工具

Workspace 不只是隔离边界，也应是 Agent 共享工具台的挂载点。

建议至少支持以下共享工具族：

- `TaskBoard`：共享任务板 / TODO 列表
- `Spreadsheet`：共享表格
- `Wiki`：共享知识库 / 文档
- `ObjectStorage`：共享文件对象
- `WorkspaceEventHub`：共享事件 / 消息中心
- `ResourcePool`：共享资源池 / 工具库
- `GraphCanvas`：临时协作白板 / 地图画布

这些工具对 Agent 来说应表现为统一工作台能力，而不是一堆散乱的底层系统入口。

## 共享工具的治理原则

为了保证协作不会从“共享”变成“污染”，建议遵循：

- 所有共享工具默认归属某个 Workspace
- 不同 Workspace 之间完全隔离
- Agent 只能访问其所在 Workspace 的工具与数据
- 工具读写应通过统一 API / HTTP API 暴露，并携带 `agent-token` 或等价身份令牌
- 所有关键修改应广播关键事件，但默认只展示摘要事件给用户

这样，Agent 获得的是“共享的虚拟工作台”，而不是对底层基础设施的无限直连能力。

## 治理与安全

- Ask、CreateTask、ExecuteTool、WriteMemory、TriggerSwarm 必须拆开授权。
- 高风险动作要经过审批链、审计链或人工确认。
- 审计 Agent 必须尽量只接触结构化、脱敏、受限视图。
- 每个 Workspace 至少应拥有 1 个审计 Agent。
- Adapter 插件的安装、启用、升级、熔断与冻结必须纳入控制面审计。
- 嵌入式桌面宿主通过 Adapter 暴露的原生能力，必须同时受模板权限、Workspace 策略和审批链约束。
- 事件订阅关系本身也必须可审计，避免 Agent 在无显式授权的情况下长期监听高敏感事件。

在协作系统里，还应额外注意：

- 编排者 Agent 的权限不应天然无限放大
- Agent 对 TaskMap 的可见范围应受 Workspace 和 role 约束
- 节点 claim、释放、重试、转移都必须保留审计痕迹
- Emergent 协作可以是局部自发的，但不能是治理失控的

## 关键基础设施

- Workspace 知识库：目录文件、RAG、向量检索、知识提升。
- 统一存储层：NFS、对象存储、跨 Runtime 共享产物。
- 知识图谱：Workspace 共享的结构化知识资产。
