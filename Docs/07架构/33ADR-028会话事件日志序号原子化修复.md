# ADR-028 Session Event Log 序号分配原子化修复

> 状态：accepted  
> 日期：2026-05-22  
> 范围：`SessionStateManager.AppendAsync`、`session_event_log`、会话事件流持久化  
> 触发日志：`SQLite Error 19: UNIQUE constraint failed: session_event_log.session_id, session_event_log.sequence_num`

## 背景

`data\logs\error\pudding-error-20260522.log` 显示多次写入 `session_event_log` 失败：

```text
UNIQUE constraint failed: session_event_log.session_id, session_event_log.sequence_num
```

唯一约束定义在 [PlatformDbContext.cs](../../Source/PuddingPlatform/Data/PlatformDbContext.cs:284)：

```csharp
e.HasIndex(e => new { e.SessionId, e.SequenceNum }).IsUnique();
```

当前序号分配在 [SessionStateManager.cs](../../Source/PuddingPlatform/Services/SessionStateManager.cs:111)：

```csharp
lock (seqLock)
{
    var maxSeq = db.SessionEventLogs
        .Where(wqe => wqe.SessionId == sessionId)
        .Max(e => (long?)e.SequenceNum) ?? 0;
    seq = maxSeq + 1;

    db.SessionEventLogs.Add(entity);
}

await db.SaveChangesAsync(ct);
```

问题是 `SaveChangesAsync` 在 lock 外。两个并发 append 可以都读到同一个已提交 `maxSeq`，然后分别插入相同 `sequence_num`。第一个提交成功，第二个撞唯一约束。

## 决策

### ADR-028-A：序号分配和持久化必须处在同一个 per-session 异步临界区

把 `ConcurrentDictionary<string, object> _seqLocks` 改为 `ConcurrentDictionary<string, SemaphoreSlim> _seqLocks`。

`AppendAsync` 中对同一 `sessionId` 执行：

```text
WaitAsync(session gate)
  -> create DbContext
  -> read max sequence
  -> insert event
  -> SaveChangesAsync
  -> release gate
```

这样同进程内不会出现同一 session 的并发序号竞争。

### ADR-028-B：保留数据库唯一约束作为最后防线

`UNIQUE(session_id, sequence_num)` 必须保留。应用层负责正常递增，数据库约束负责防止未知并发路径或未来多实例写入造成脏序号。

### ADR-028-C：唯一约束冲突最多重试 3 次

即使加了 per-session `SemaphoreSlim`，仍应对 `DbUpdateException` + SQLite error 19 做有限重试。

重试策略：acquire session gate → read latest max sequence → insert → save。on unique constraint → clear tracked entity → small delay → retry。after 3 failures → throw。

### ADR-028-D：SQLite 是权威事件源，JSONL 和 Channel 必须在 SQLite 成功后执行

现有顺序保留：

```text
1. SQLite insert
2. JSONL backup
3. in-memory channel push
4. stream state transition
5. runtime activity
```

不得在 SQLite 失败后继续写 JSONL 或推送 Channel。

## 代码级变更

见 [SessionStateManager.cs](../../Source/PuddingPlatform/Services/SessionStateManager.cs)：
- `_seqLocks` 类型从 `ConcurrentDictionary<string, object>` 改为 `ConcurrentDictionary<string, SemaphoreSlim>`
- 新增 `AppendSqliteEventAsync` — 封装锁内查询 + 插入 + SaveChangesAsync
- 新增 `AppendSqliteEventWithRetryAsync` — 最多 3 次重试的 DbUpdateException 包装
- 新增 `IsSessionSequenceConflict` — 判定 SQLite error 19 是否为 session_event_log 序号冲突
- `AppendAsync` 调用链替换原有 `lock + SaveChangesAsync`

## 测试

见 [SessionStateManagerSequenceTests.cs](../../Source/PuddingPlatformTests/Services/SessionStateManagerSequenceTests.cs)：
- 测试 1：50 并发同 session append → 序列号连续不重复
- 测试 2：不同 session 并发 → 各自从 1 开始
- 测试 3：SQLite 表中无重复 (session_id, sequence_num)

## 后续增强

如果未来支持多进程或多节点同时写同一个 SQLite 文件，`SemaphoreSlim` 只能解决单进程并发。届时应升级为数据库序列表：

```sql
CREATE TABLE IF NOT EXISTS session_sequence_state (
    session_id TEXT PRIMARY KEY,
    next_sequence_num INTEGER NOT NULL
);
```

用事务原子更新 `next_sequence_num`，再插入 `session_event_log`。当前单宿主形态下，`SemaphoreSlim + unique retry` 是最小、稳定、低风险修复。
