# 34 ADR-033 Admin Chat Composer 浮层重设计

> 状态：**proposed**  
> 日期：2026-05-23  
> 范围：`/admin/chat` 底部 Composer 的 `+` 浮层、运行状态入口与技术指标披露方式  
> 关联：[20AdminChat简约克制界面ADR](20AdminChat简约克制界面ADR.md)、[../Features/AdminChat简约克制设计方案](../Features/AdminChat简约克制设计方案.md)、[../Features/AdminChatComposer浮层重设计方案](../Features/AdminChatComposer浮层重设计方案.md)

---

## 1. 背景

当前 `/admin/chat` 已按 ADR-020 完成一轮“Quiet Chat”收敛：输入区默认只保留输入框、发送/停止、语音和 `+` 更多入口，运行状态从常驻状态栏进入 Popover。

浏览器评审选中的浮层暴露出新的问题：`+` 菜单里同时放了上传附件、上传图片、思考强度、导出对话、运行状态、Token、Provider、ASP/LSP、索引、潜意识、SubAgent 等内容。它在视觉上已经从“更多动作”退化为“杂糅的运行时抽屉”。

这带来四类问题：

1. **语义混杂**：用户点击 `+` 预期是添加附件或选择输入动作，却看到运行状态和技术指标。
2. **层级失衡**：被禁用的未来能力和内部诊断指标占据同一层级，真正可用的 `导出对话` 被埋没。
3. **可读性不足**：窄浮层承载多枚状态组件，图标和短文本难以解释，触达区域也偏小。
4. **与 ADR-020 冲突**：ASP/LSP、索引、潜意识 LLM 等内部状态虽已不常驻，但仍在普通模式的主菜单里可见。

本 ADR 只处理 Composer 浮层的信息架构与前端组件边界，不改变后端 API、SSE 协议、数据库模型和运行时指标来源。

---

## 2. 决策驱动因素

| 驱动因素 | 说明 |
|----------|------|
| 输入动作优先 | `+` 是 Composer 的动作入口，应优先服务“本轮输入能做什么”。 |
| 运行状态可见但不抢占 | 用户需要知道 Agent 是否在工作，但不应在添加附件菜单里阅读内部指标。 |
| 技术指标按需披露 | Token、Provider、ASP/LSP、索引、潜意识、SubAgent 属于详情或开发者信息。 |
| 保持前端局部改造 | 复用现有 `InputArea.tsx`、状态指示器组件和样式 token，避免扩大到消息流或后端。 |
| 可访问性 | Popover 内的可点击项需要明确标签、焦点态、键盘可达和足够点击面积。 |

---

## 3. 方案对比

### 方案 A：只美化现有 Popover

- **做法**：保留当前内容，只调宽、加分组、优化间距。
- **优点**：改动最小。
- **缺点**：没有解决 `+` 菜单语义错误；内部指标仍暴露给普通用户；后续功能增加会继续膨胀。
- **结论**：不采纳。

### 方案 B：拆分“输入动作菜单”和“运行状态详情”（采纳）

- **做法**：`+` Popover 只放输入动作与会话动作；Composer 状态胶囊或状态行承载运行状态入口；技术指标进入“状态详情”二级浮层或开发者模式。
- **优点**：语义清晰；符合 ADR-020 的渐进披露；实现可局部完成；后续附件、图片、思考强度上线时有自然位置。
- **缺点**：需要拆分 `InputArea.tsx` 中的内联 Popover 内容，新增少量组件和样式。
- **结论**：采纳。

### 方案 C：取消 `+` Popover，全部进入 `/` 命令面板

- **做法**：附件、导出、思考强度、状态查询都变成命令。
- **优点**：界面最干净，适合键盘用户。
- **缺点**：发现性差；附件/图片这类直接操作不适合藏进命令；移动端和普通用户成本高。
- **结论**：不采纳。

---

## 4. 决策

### ADR-033-A：`+` 入口只表达“输入与会话动作”

`+` Popover 的主职责限定为：

- 添加到本轮输入：附件、图片、后续文件引用；
- 调整本轮输入参数：思考强度；
- 会话级轻动作：导出对话；
- 未来可用动作的占位说明。

`+` Popover 不再展示：

- Token 明细；
- Provider 余额；
- ASP/LSP；
- 索引状态；
- 潜意识 LLM；
- SubAgent 列表；
- 原始运行时枚举或内部缩写。

### ADR-033-B：运行状态从“菜单内容”升级为“Composer 状态入口”

Composer 在输入框上方或左侧保留一条自然语言状态，例如：

- `就绪`
- `正在整理上下文…`
- `正在调用工具…`
- `正在生成回复…`
- `出错了，可重试`

当有更多信息可看时，状态文本或状态胶囊可点击，打开“运行状态详情”。详情浮层只展示面向用户可理解的摘要：

- 本轮状态；
- Token 使用概览；
- 记忆 / 索引 / 子代理是否参与；
- 错误与恢复动作；
- `打开开发者详情` 入口。

### ADR-033-C：技术指标默认进入开发者详情

ASP/LSP、潜意识 LLM、Provider balance、完整 Token 统计、SubAgent 原始列表默认不出现在普通状态详情第一层。

普通模式可显示经过翻译的摘要：

| 内部指标 | 普通模式文案 |
|----------|--------------|
| ASP/LSP | `上下文服务：可用 / 待机 / 异常` |
| Index | `索引：可用 / 建立中 / 未启用` |
| Subconscious LLM | `后台记忆整理：待机 / 运行中` |
| SubAgent | `子任务：0 个 / 运行中 N 个` |
| Provider balance | `模型服务：可用 / 需配置 / 异常` |

开发者模式仍可看到内部名称、原始状态和完整指标。

### ADR-033-D：禁用项必须降低视觉权重

`上传附件`、`上传图片` 等未开放动作可以保留在 `+` 菜单中，但必须满足：

- 明确标注 `即将开放`；
- 视觉权重低于可用动作；
- 不占据主操作位；
- 不阻塞键盘导航；
- 后续能力上线时可直接切换为可用项。

### ADR-033-E：Popover 必须满足可访问性与响应式基线

默认目标：

- 桌面可点击项高度不小于 36px；
- 触屏断点不小于 44px；
- 相邻交互项间距不小于 8px；
- 图标按钮必须有 `aria-label` 与 Tooltip；
- Popover 支持 Esc 关闭、外部点击关闭、焦点返回触发按钮；
- 窄屏时 Popover 宽度不超过视口减 24px，并允许滚动。

### ADR-033-F：隐藏指标必须转化为“轻反馈”，不能直接消失

上下文、记忆、索引、子代理、后台整理等指示器不应以重型图标常驻，但也不能完全失去存在感。普通模式采用“环境反馈带”表达系统状态：

- Composer 状态行显示当前主状态；
- 状态行右侧可显示 1-4 个低权重状态点或短标签；
- 状态点只表达是否参与、是否运行、是否异常，不表达内部实现名；
- hover / click 后进入状态详情；
- 开发者模式保留完整内部指标。

示例映射：

| 能力 | 普通反馈 | 详情文案 |
|------|----------|----------|
| 上下文 | `上下文` 小标签或 1 个暖棕状态点 | `上下文已整理` / `上下文不足` |
| 记忆 | `记忆` 小标签或紫灰状态点 | `已参考 N 条记忆` |
| 索引 | `索引` 小标签或蓝灰状态点 | `索引可用 / 建立中 / 未启用` |
| 子代理 | `子任务 N` | `N 个子任务运行中` |
| 后台整理 | `后台` 小标签 | `后台记忆整理待机 / 运行中` |

### ADR-033-G：输入焦点反馈必须作用于 Composer 容器

输入状态的视觉反馈不得只画在 `textarea` 底部。Composer 作为一个整体输入容器，需要拥有统一的 focus / typing / loading / error 状态：

- focus：容器边框轻微变深，出现低强度纸面阴影；
- typing：允许一个 150-300ms 的低强度 shimmer / ink-line 动效，但不得持续抢焦点；
- loading：发送按钮切换为停止，状态行显示当前主状态；
- error：容器边框使用低饱和错误色，并提供恢复动作；
- `prefers-reduced-motion` 下关闭 shimmer，只保留边框和颜色变化。

### ADR-033-H：Composer 多行布局必须固定操作区，避免发送按钮被遮挡

Composer 主体必须从“单行 flex”升级为稳定的三段式布局：

1. 左侧工具区：`+`、语音、未来附件入口，固定宽度；
2. 中央输入区：`textarea` 自动增高并内部滚动；
3. 右侧操作区：发送 / 停止按钮固定在右下，不随文本滚动消失。

多行输入时：

- 输入区最大高度建议 132-160px；
- `textarea` 超出后内部滚动；
- 发送按钮始终可见；
- 左侧工具区与右侧发送区不被文本区域覆盖；
- Composer 整体不贴边，保留稳定内边距和圆角。

## 6. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 用户找不到运行状态详情 | 长任务中安全感下降 | 状态行本身可点击，并在生成中显示轻量 `查看状态`。 |
| 开发者认为技术指标被隐藏 | 排障效率下降 | 开发者模式保留完整指标，并从状态详情提供入口。 |
| 新增组件造成小改动扩散 | 回归风险 | 不改状态来源，只拆渲染；优先纯组件重构。 |
| 禁用项仍显得像错误 | 用户困惑 | 使用 `即将开放` 标签和低权重样式，不放在主位。 |
| 移动端 Popover 溢出 | 操作困难 | 使用最大宽度、滚动和触屏点击目标基线。 |

## 7. 验收标准

### 7.1 信息架构

- 点击 `+` 后，第一眼只看到输入动作与会话动作。
- `运行状态` 不再作为 `+` 菜单的一个分区出现。
- ASP/LSP、索引、潜意识、SubAgent 等内部指标不在普通 `+` 菜单中出现。
- `导出对话` 是可用动作，视觉上不被禁用项和指标淹没。

### 7.2 状态披露

- Composer 状态行能表达当前主状态。
- 点击状态入口能查看用户可理解的运行摘要。
- 开发者模式能继续查看完整技术指标。
- 错误状态提供 `重试` 或 `查看详情`。

### 7.3 可访问性

- 所有图标按钮有 Tooltip 和 `aria-label`。
- Popover 可通过键盘打开、关闭和遍历。
- 可点击项桌面高度不小于 36px，触屏不小于 44px。
- 焦点态可见，文字对比度满足 WCAG AA。

### 7.4 视觉

- Popover 宽度、分组和留白稳定，不出现截图中的窄长指标堆叠。
- 禁用项与可用项有清晰差异。
- 常态无发光、扫描线和多点 pulse。
- 与现有暖纸面、低饱和、8px 圆角体系一致。
- 输入 focus 不再表现为突兀的整条紫色底线。
- 多行输入时发送 / 停止按钮始终可见。
- 上下文、记忆、索引、子代理等状态通过低权重反馈带可感知。

---

## 8. 回滚策略

本决策不改变后端契约，回滚可限定在前端：

1. 保留旧 `composerMenuContent` 实现作为临时 fallback；
2. 新组件只消费现有 props，不引入新数据源；
3. 若状态详情入口发现性不足，可提高状态行显著性，而不是把技术指标放回 `+` 菜单；
4. 若开发者排障受影响，默认只在开发者模式显示完整指标。

---

## 9. 结论

采纳 **方案 B：拆分输入动作菜单与运行状态详情**。

`+` 菜单是 Composer 的动作入口，不是运行时仪表盘。运行状态应作为独立、可解释、按需展开的状态入口存在；技术指标继续服务开发者模式，而不是进入普通用户的第一层菜单。
