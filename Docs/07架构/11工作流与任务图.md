# 工作流与任务图

## 定位

工作流在 Pudding 中不是一个“附属功能”，而是平台的一等执行模型。

如果说：

- `PuddingController` 负责控制与治理
- `PuddingRuntime` 负责执行与唤醒
- `PuddingAgent` 负责角色与能力定义
- `Event Bus` 负责协作痕迹传播

那么 **Workflow / TaskMap** 负责把复杂任务表达成可视化、可执行、可恢复、可审计的结构。

这意味着：

- 复杂任务不应只存在于提示词里
- 工作流不应只支持线性顺序执行
- Agent 不应只以“聊天”方式工作，也应成为工作流节点的一类执行单元

## 前端设计参考

在前端或管理页面上，Pudding 可以参考 **FlowGram.AI** 的设计理念来构建工作流编辑体验，重点借鉴：

- 可视化画布
- 节点配置表单
- 变量作用域与变量消费
- 节点与边的结构化保存
- 自由布局 / 固定布局两种编辑方式

但要特别说明：

- **FlowGram 更适合作为前端编辑器与设计态参考**
- **Pudding 的运行时语义、Agent 生命周期、权限治理与事件驱动协作仍应由自身架构决定**

也就是说：

- 前端可以借鉴或集成类似 FlowGram 的画布能力
- 后端运行时与工作流执行引擎仍应服务于 Pudding 的 Controller / Runtime / Workspace 架构

## 工作流的组成

Pudding 的工作流建议至少由以下部分组成：

- `WorkflowDefinition`：工作流定义对象
- `WorkflowNode`：节点定义
- `WorkflowEdge`：边定义
- `WorkflowTrigger`：触发方式定义
- `WorkflowVariable`：变量声明与作用域定义
- `WorkflowRun`：一次实际运行实例
- `WorkflowCheckpoint`：恢复点 / 中间快照

## 节点类型

工作流中的节点不应只是一种“代码块”，而应支持多种执行单元。

建议首批节点类型：

1. **程序逻辑节点**
   - 内置组件
   - Python 脚本
   - Shell / Bash
   - Curl / HTTP 请求
   - 其他可注册程序或插件节点

2. **Agent 节点**
   - 调用长期 Agent
   - 调用临时 Workflow Agent
   - 调用一次性 Agent

3. **事件节点**
   - 事件触发器
   - 事件消费器
   - 事件发布器

4. **控制节点**
   - Start
   - End
   - Condition
   - Loop
   - Retry / Failover
   - SubWorkflow

5. **存储节点**
   - 知识库写入
   - 表格 / 数据表写入
   - 文件输出
   - 审计结果输出

## 工作流触发方式

工作流不应只支持人工点击启动。

建议至少支持以下触发方式：

1. **Agent 直接触发**
   - 某个 Agent 在执行过程中决定启动一个工作流

2. **事件触发**
   - 某个外部或内部事件命中 `WorkflowTrigger`
   - 例如 `device.sensor.triggered`、`task.completed`、`approval.completed`

3. **用户 / 平台触发**
   - 用户在 `PuddingPlatform` 上手动执行工作流
   - 运维、管理员或调试入口手动触发

4. **工作流触发工作流**
   - 一个工作流在某个节点或结束阶段触发另一个工作流
   - 可用于拆分长链路、异步扩展或子流程复用

## 触发器组件与事件回写

工作流不仅可以被触发，也可以反过来成为事件生产者。

建议在工作流中显式支持 `TriggerNode` / `EventEmitterNode` 一类组件，用于：

- 发布业务事件
- 发布审计事件
- 发布节点状态事件
- 触发其他工作流或 Agent 唤醒

这样工作流在 Pudding 中就不只是“执行器”，而是事件网络中的一等参与者。

## 工作流与 TaskMap / DAG 的关系

工作流与 TaskMap 是同一类问题的两个视角：

- **Workflow** 更强调可编排执行链
- **TaskMap** 更强调任务空间、节点探索和协作痕迹
- **DAG** 是其中一种常见但不唯一的表示方式

因此建议：

- 设计态以 `WorkflowDefinition` 表示结构
- 协作态允许映射为 `TaskMap`
- 执行态通过 `WorkflowRun`、节点状态和事件流推进

## 建议的节点状态

工作流节点建议至少支持：

- `Pending`
- `Ready`
- `Claimed`
- `Running`
- `Blocked`
- `Completed`
- `Failed`
- `Released`
- `Skipped`

含义建议：

- `Pending`：已存在但未满足前置条件
- `Ready`：已满足执行条件，等待被领取
- `Claimed`：已被某个 Agent / Executor 认领
- `Running`：正在执行
- `Blocked`：依赖、审批、资源或外部条件未满足
- `Completed`：执行完成
- `Failed`：执行失败
- `Released`：执行者放弃该节点，允许重新接管
- `Skipped`：因条件分支或策略跳过

## 工作流中的 Agent 生命周期

Agent 在工作流中不应只有一种生命周期。

建议至少区分三类：

### 1. 长期 Agent

- 在 Workspace 中长期存在
- 拥有长期记忆
- 可被多个工作流或普通任务重复调用
- 适合审计 Agent、编排者 Agent、领域专家 Agent 等

### 2. 工作流记忆 Agent

- 为某个 `WorkflowRun` 临时创建
- 记忆在整个工作流运行期间有效
- 工作流完成后记忆销毁或归档
- 记忆作用域应与该工作流隔离，默认不污染 Workspace 长期记忆

适合：

- 跨多个节点共享临时上下文
- 承担某次流程中的临时专家角色
- 进行长链路推理，但不把过程记忆长期保留

### 3. 一次性 Agent

- 只为单个节点执行周期创建
- 节点结束后即销毁
- 记忆只在该节点局部有效

适合：

- 一次性分析
- 某个节点的专用临时执行者
- 高隔离、低复用的场景

## Agent 生命周期选择建议

不同节点适合的 Agent 生命周期不同：

- **编排节点**：优先长期 Agent 或工作流记忆 Agent
- **分析节点**：可使用工作流记忆 Agent
- **校验节点**：可使用一次性 Agent 或长期审阅 Agent
- **高风险执行节点**：更适合一次性 Agent + Dedicated Sandbox

建议模板或工作流节点配置支持：

- `agentLifecycle = persistent | workflow-scoped | ephemeral`
- `agentTemplateRef`
- `isolationLevel`
- `memoryPolicyOverride`

## 一个典型链路示例

例如：

`开始 -> 采集 -> 程序/脚本清洗 -> Agent 处理分析 -> 另一个 Agent 校验 -> 存储组件 -> 结束`

其中：

- 若校验 **通过**：进入存储组件并结束
- 若校验 **不通过**：回流到“Agent 处理分析”节点继续修正

这个例子说明：

- 工作流不是只能线性向前
- Agent 节点与程序节点可以混编
- 校验与回流是工作流中的正常能力
- 节点之间既可以由边表达，也可以由事件钩子表达

## Workflow 与 Runtime 的关系

工作流不应直接绕过 Runtime 自己执行。

更合理的边界是：

- `Controller` 负责触发、路由、权限与治理
- `Runtime` 负责节点执行、Agent 组装、sandbox 生命周期和状态回报
- `Workflow Engine` 可以作为控制面与执行面的桥接能力存在

建议原则：

- 工作流定义可由 Platform 管理
- 工作流运行可由 Controller 调度
- 节点执行由 Runtime 落实
- Agent 节点、脚本节点、组件节点都统一回到 Runtime 执行模型下

## 工作流变量与作用域

工作流中的变量不只是“某个 JSON 字段”，而是跨节点传值和约束的重要基础。

建议至少区分：

- `workflow input`
- `node output`
- `workflow temp variables`
- `workspace shared refs`
- `agent memory references`

作用域建议：

- 节点输出默认向后可见
- 分支内变量默认受分支作用域限制
- 工作流记忆 Agent 的上下文应与变量作用域配合，而不是绕开变量系统随意共享

## 设计态与运行态分离

借鉴 FlowGram 的一个重要启发是：

- **设计态** 关注节点、边、表单、变量、画布表达
- **运行态** 关注执行引擎、状态机、恢复、权限、资源与审计

Pudding 也应明确区分这两层：

### 设计态

- 在 Platform / Admin 中编辑工作流
- 使用画布、节点表单、变量面板、属性面板
- 保存为 YAML / JSON 结构化定义

### 运行态

- Controller 接收触发请求
- Runtime 执行节点
- Event Bus 推进节点状态变化
- 审计系统记录全过程

## YAML 作为第一阶段表达

第一阶段建议：

- 后端以 YAML 作为工作流定义主格式
- 前端画布可加载 / 保存同构结构
- 后续再逐步增强为更丰富的图形化编辑与子画布能力

建议原则：

- YAML 是权威配置表达
- 图形画布是更友好的编辑视图
- 两者应可双向映射

## 前端管理页面建议

`PuddingPlatformAdmin` 中建议至少包含：

- 工作流列表页
- 工作流设计器画布
- 节点属性表单
- 变量面板
- 触发器配置页
- 运行记录 / 调试页
- 子工作流引用与依赖关系视图

布局形态可参考 FlowGram：

- 自由布局适合探索性流程和任务图
- 固定布局适合清晰的步骤式流程
- 节点表单用于配置节点数据与校验
- 变量面板用于查看上下游变量与作用域

## 第一阶段最小落地建议

V1 不必一次把工作流系统做成“可视化 BPM 平台”，但至少应完成：

- 1 种 YAML 工作流定义格式
- 4 类基本节点：开始、结束、程序/脚本、Agent
- 1 类条件分支
- 1 类事件触发器
- 1 类存储节点
- 支持一个工作流触发另一个工作流
- 支持至少两种 Agent 生命周期：长期 Agent、一次性 Agent
- 支持节点状态回流与失败重试
