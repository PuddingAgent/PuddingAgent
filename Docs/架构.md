# Pudding Agent Network — 架构总览

> **2026-05-02 架构简化**：从多服务分布式模型简化为单进程 P2P 模型。每个 Agent 是一个独立可执行程序，内嵌 Web UI、Controller、Runtime、SQLite。Agent 之间通过 P2P 发现与直连通信。

## 产品定位

Pudding 不是一个需要 Nginx + PostgreSQL + Redis + RabbitMQ + MinIO 全家桶才能跑起来的"企业平台"。

它是一个**双击即用的 AI 代理程序**。

> **Pudding = 可 P2P 组网的 AI Agent 进程**

- 双击启动，浏览器打开即用
- 一个进程 = 一个 Agent（内嵌 Web UI + LLM 对话 + 记忆 + 工具）
- 多个 Agent 之间通过 P2P 协议自动发现、直连通信、协作完成任务
- 数据库只依赖 SQLite，零外部基础设施
- 仍支持 Docker 部署（单容器暴露端口即可）

```
│  ← P2P →  其他 Agent 节点                                  │
└────────────────────────────────────────────────────────────┘
```

## 多 Agent 协作：P2P 网络模型

Agent 不是孤立进程。当多个 Pudding Agent 运行在同一局域网或可互通网络中时：

- **自动发现**：基于 mDNS 或 DHT 发现对等节点
- **直连通信**：Agent 之间直接建立 HTTP/gRPC 连接，无需中心控制器
- **事件广播**：Agent 将任务完成、状态变更等事件广播给感兴趣的节点
- **数字蚁群**：每个 Agent 像蚂蚁一样感知局部状态，协作推进任务图

这就是原来的 Controller/Runtime 两层架构的 P2P 实现——每个 Agent 同时具备控制面和执行面能力，协作去中心化。

```

- 可以用 Docker 单容器部署：docker run -p 8080:8080 pudding-agent
- 可以在 Linux 服务器上 systemd 托管
```

## 技术选型

| 组件 | 技术 | 原因 |
|---|---|---|
| 后端框架 | ASP.NET Core | C# 生态，高性能 |
| 数据库 | SQLite（通过 EF Core） | 零配置，单文件，足够个人/小团队使用 |
| 前端 | React/TypeScript，打包嵌入后端 | 单进程部署 |
| P2P 发现 | mDNS + UDP 广播（局域网）/ DHT（广域网） | 零配置自动发现 |
| 节点通信 | HTTP/gRPC 直连 | 简单可靠 |
| LLM | 直接调用 API（OpenAI 兼容） | 不经过中间代理 |
| 连接器 | `IPuddingConnector` 接口（DI 注册） | 替代旧 `IPuddingGatewayAdapter`，双向通道管理 |

> **连接器迁移说明**：旧 `IPuddingGatewayAdapter` 保留兼容，新旧接口在 DI 中并存。已有 Adapter 逐步迁移为 Connector。连接器详细设计见 [Tasks/task36-event-trigger-and-subagent.md](Tasks/task36-event-trigger-and-subagent.md) §0。

## 不再需要的基础设施

以下全部废弃，由程序自身替代：

| 废弃 | 替代方案 |
|---|---|
| Nginx | ASP.NET Core 内置 Kestrel 直接对外 |
| PostgreSQL | SQLite |
| Redis | 内存缓存 + SQLite |
| RabbitMQ | P2P 直连事件广播 |
| MinIO | 本地文件系统 + SQLite BLOB |
| Docker Compose 多容器 | 可选：单容器或裸进程 |

```text

├── PuddingAgent/          ← 主程序入口（单进程宿主）
│   ├── WebUI/             ← 内嵌前端（React 构建产物）

│   ├── Connectors/        ← 连接器模块（Webhook/MQTT/邮箱等协议通道）
│   ├── Runtime/           ← Agent 执行（LLM/工具/记忆/Skill/子代理）
│   ├── P2P/               ← P2P 网络层（发现/直连/事件广播）
│   └── Data/              ← SQLite 数据访问（EF Core）
├── PuddingCore/           ← 共享抽象与协议（ILlmGateway/IPuddingConnector等）
├── PuddingPlatformAdmin/  ← 前端源码（构建后嵌入 PuddingAgent）
└── PuddingCode/           ← 旧 CLI 原型（保留过渡）
```

## 关键边界

- 一个进程 = 一个 Agent，一个 Agent = 一个身份
- Agent 拥有自己的 SQLite 数据库、记忆、配置
- Agent 之间平等，通过 P2P 协议协作，无中心节点
- 用户在任一 Agent 的 Web UI 上可以管理自己的 Agent，也可以看到对等网络中的其他 Agent
- 工具在 Agent 进程内直接执行
- 高风险操作需用户确认

## 场景（Scene/Workspace）与 Agent 关系模型

> **2026-05-03 设计确定**：Workspace 保留作为"用户场景分组"概念，面向用户称为"场景"。Agent 模板简化为全局模板库，Agent 从模板创建。

### 设计决策

| 维度 | 旧定位（平台思维） | 新定位（个人助手） |
|------|---------------------|---------------------|
| 用户第一眼看到 | Workspace 列表（管理面板） | Chat 对话界面 |
| Workspace 的角色 | 必须创建的顶层组织 | 场景分组（工作/生活/创意），用户可切换 |
| Agent 管理 | Workspace 下两级模板+实例 | 全局模板库 → 场景内创建 Agent 实例 |
| 权限/成员/治理 | 暴露在 UI 中 | 隐藏（V2+ 再考虑） |
| 冻结/解冻 | 暴露在 UI 中 | API 保留，UI 隐藏（V1 单进程无意义） |

### 数据模型（后端不变）

```
用户 (单用户，V1)
  │
  ├── 场景A (Workspace: "工作")
  │   ├── Agent: 助手 (默认，继承自"通用助手"模板)
  │   ├── Agent: 邮件管理 (继承自"邮件处理"模板)
  │   └── Agent: 会议纪要
  │
  ├── 场景B (Workspace: "生活")  
  │   ├── Agent: 生活助手 (默认)
  │   └── Agent: 智能家居 (继承自"Home Assistant"模板，含 MQTT 连接器)
  │
  └── 场景C (Workspace: "开源项目")
      ├── Agent: 代码助手 (默认)
      └── Agent: 代码审查 (继承自"代码审查"模板)
```

### 原则

1. **用户直接面对对话** — 打开就是聊天界面，顶栏切换场景和 Agent
2. **每个进程自带 `default` 场景** — 对用户透明，无需手动配置即可使用
3. **场景是 Agent 的容器** — 不同场景的 Agent 有独立记忆和上下文，互不混杂
4. **全局 Agent 模板库** — 模板跨场景复用，创建 Agent 时从模板选择
5. **设置页是后台** — 场景管理、Agent 管理、模板库均在 ⚙️ 设置中，对话页不暴露管理复杂度

## Agent 能力分层模型

> **2026-05-02 新增**：基于 Anthropic/Hermes/OpenClaw 等业界标杆分析，定义 Pudding Agent 的六层能力模型及 V1 优先级。

```
┌─────────────────────────────────────────────────┐
│  L5  自动化层    Cron调度 · 事件触发 · 工作流管道   │  ← P2
├─────────────────────────────────────────────────┤
│  L4  接入层      多渠道Connector · Web UI · CLI    │  ← P1
├─────────────────────────────────────────────────┤
│  L3  多智能体层  子代理 · Orchestrator · 路由分发  │  ← P1
├─────────────────────────────────────────────────┤
│  L2  工具与技能层  工具注册/执行 · Skill系统 · MCP  │  ← P0
├─────────────────────────────────────────────────┤
│  L1  记忆与上下文  会话记忆 · 长期记忆 · 上下文管理  │  ← P0
├─────────────────────────────────────────────────┤
│  L0  基础层      LLM对话循环 · 多模型 · 流式输出   │  ← P0
└─────────────────────────────────────────────────┘
```

## V1 实施优先级总览

| 优先级 | 能力 | 理由 |
|--------|------|------|
| **P0** | L0 基础层 + L1 记忆 + L2 工具/Skill | 这是 Agent 的骨架，没有这些不叫 Agent |
| **P1** | L3 子代理 + L4 连接器 | 差异化竞争力：多 Agent 协作 + 多渠道接入 |
| **P2** | L5 Cron + 工作流 | 锦上添花，实现简单但非 V1 核心 |
| **P3** | 浏览器自动化 | 远期生态扩展 |
