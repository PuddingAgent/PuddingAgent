# Pudding Agent Network — 架构总览

这份文档现在只保留总体定位、分层边界、阅读地图和当前架构共识。

具体模块说明已经拆分到 [07架构](07架构/README.md) 目录下，避免继续把所有细节堆叠在单一文档中。

## 产品定位

Pudding Agent Network 不是单个智能体应用，而是一个以 Workspace 为治理边界、以 Controller + Runtime 为核心、支持多渠道接入、多 Agent 协作、审批审计与记忆治理的 Agent OS 与协作网络平台。

一句更直接的定位是：

> **Pudding = Event-Driven Multi-Agent Operating System**

它的目标不是实现“一个更强的 Agent”，而是实现：

- 多用户、多渠道、多 Workspace 的协作网络
- 多 Agent、Swarm、SubAgent 的受控执行体系
- 以控制面和执行面分离为基础的平台化架构
- 可审计、可审批、可治理、可扩展的系统能力

## 为什么要做这个系统

Pudding 的出发点不是“让 AI 看起来更聪明”，而是解决现有个人智能体系统在工程化落地中的几个硬问题：

1. **多智能体与工作流能力不足**
	现有系统通常擅长单 Agent 对话，不擅长严格遵循任务图、工作流图和执行约束。Pudding 要把 Workflow Engine 提升为一等能力，由系统决定流程，LLM 只作为 reasoning node。

2. **多任务交叉导致上下文污染**
	企业环境中会同时存在项目 A、项目 B、安全测试、财务分析等多条工作线。Pudding 通过 Workspace 把记忆、Agent、工具、工作流和事件域放进同一个命名空间，解决上下文串味问题。

3. **现有智能体大多是轮询驱动而不是事件驱动**
	只有心跳、没有事件队列，会导致响应慢、浪费 token、扩展性差。Pudding 采用统一事件总线，让 Agent 被世界唤醒，而不是持续问“有没有事做”。

4. **与现有系统接入困难**
	企业真实场景需要 MQTT、WebSocket、HTTP、Webhook、GitHub 触发器等协议接入。Pudding 把协议适配做成独立能力，而不是让 Agent 模拟人类点击界面。

5. **缺少可演进的协同机制**
	现有多智能体系统常见 Leader 模式、投票/联邦模式、自发模式，但对协同协议本身思考不够。Pudding 不满足于“多开几个 Agent”，而是探索角色、委派、事件痕迹、局部规则如何产生稳定协作。

## MVP 三大核心模块

如果只保留最核心的 MVP，Pudding 首先必须站稳三根柱子：

1. **Agent Runtime**
	承载 Session、Agent、sub_agent、技能、记忆与唤醒执行。

2. **Event Bus**
	承载外部输入、内部状态变化、订阅、唤醒、审计、死信与重放。

3. **Workspace Memory**
	把 Workspace 升级为 Memory + Agent + Tool + Event 的完整命名空间，而不是一个简单目录或数据库租户标签。

这三者叠起来，才构成 Pudding 的最小工程核心。

## 当前架构决策基线

- 分布式优先：从一开始支持多 Runtime 节点
- Runtime 多承载：每个 Runtime 可承载多个 Session / Agent
- Runtime 支持独立宿主与嵌入宿主：既可以独立进程运行，也可以嵌入其他 C# 桌面软件并被视为可调度节点
- Runtime 为会话权威：Session 热状态与主状态由 Runtime 持有
- Controller 为控制权威：路由、鉴权、审批、审计、调度在 Controller
- Controller 维护全局 Skill / MCP Registry：模板引用全局能力，Runtime 负责装配执行
- Gateway 采用 Adapter Plugin 模式：平台可内置核心接入源，但接入机制本身必须插件化并支持热插拔
- 统一事件总线是系统骨架：内部事件总线建议基于 RabbitMQ，外部协议输入统一转换为平台事件
- 万物皆事件：外部消息、设备告警、任务完成、Agent 完成任务、心跳、审批结果都应统一进入事件流
- Workspace 是治理边界：Agent、Workflow、MemoryPolicy、PluginBinding 默认归属 Workspace
- Runtime 选址由 Controller 决定：支持显式指定 Runtime、按标签偏好选择，缺省优先同 Workspace 共置
- AgentInstance 生命周期由 Runtime 负责落实：至少支持创建、运行、忙碌、睡眠、休眠、归档、销毁等状态与状态迁移
- Runtime 必须支持冻结（stop）语义：可冻结单 Agent，也可冻结整个 Workspace 的全部 Agent，并停止对应 Docker / sandbox 容器
- 冻结 / 恢复链路优先采用全局治理事件广播：Platform 发命令给 Controller，Controller 发布全局冻结 / 恢复事件，Runtime 订阅后只处理自己承载且命中目标 Workspace / Agent 的实例
- 知识库、统一存储层、知识图谱属于 Workspace 共享能力
- 批准属于系统控制链路，而不是业务 Agent 自主决定
- 每个 Workspace 至少 1 个审计 Agent


##  PuddingController
PuddingController 是整个协作网络的控制面宿主。PuddingGateway 是它内部的边界接入模块，而不是长期独立存在的平级产品。

比如：
路由
授权
审批
审计
调度
Adapter 治理
Runtime 节点管理
Session 控制链路

## PuddingPlatform
PuddingPlatform 是上层平台层与系统组合层。它不等于控制面进程，而是承载产品语义、业务模型、服务暴露策略与平台能力编排的上层概念。
比如：

Workspace 产品管理
业务流程
服务目录
运营配置
产品入口
管理后台
用户可见的业务语义



## 顶层分层

- `PuddingCore`：无状态抽象、协议、通用模型
- `PuddingRuntime`：执行面，承载 Session、AgentInstance、技能、记忆、沙箱与受限执行
- `PuddingAgent`：定义层模块，承载 AgentTemplate、画像、默认能力与策略定义；Runtime 依赖它来组装 AgentInstance，但它不持有执行热状态
- `PuddingMemoryEngine`：PuddingRuntime的记忆模块
- `PuddingRuntime` 是独立执行宿主
- `PuddingController`：是PuddingRuntime的控制面（类似于docker的控制面，守护程序负责调度和控制runtime，不承载网站和上层的业务），负责接入、路由、鉴权、审批、调度、审计与治理
- `PuddingGateway`：Controller 内的边界接入模块
- `PuddingGateway` 通过 Adapter Plugin 承载外部系统、渠道与嵌入式节点接入
- `PuddingPlatform`：上层平台层（asp.net core 后台网站，承载业务，用户的进入的管理平台，前端是PuddingPlatformAdmin），负责业务语义、产品能力组合与平台编排


## 关键边界

- Session 权威在 Runtime
- 调度与治理权威在 Controller
- Platform 负责上层业务逻辑，不替代控制面
- Agent 模板属于 Workspace，Agent 实例属于 Session
- AgentTemplate 引用全局 Skill / MCP，Skill / MCP 的注册权威在 Controller
- Agent 生命周期的定义权与执行权分离：模板声明默认策略，Runtime 执行状态迁移，Controller / 用户负责高权限生命周期动作的准入
- 公开或低信任输入默认不能直接污染长期记忆
- 高风险动作必须进入审批链、审计链或人工确认链路
- `PuddingMemoryEngine` 属于 `PuddingRuntime` 内模块，不是独立控制面组件
- 被嵌入的桌面软件可以作为 Runtime 节点向 Controller 注册，并暴露受控原生能力，例如查询软件状态、驱动软件测试或调用宿主原生功能
- Gateway 应优先采用事件驱动 Adapter，而不是把轮询当作默认接入模型
- Agent 可以在 Runtime 支持下订阅感兴趣的事件，并在命中时被直接唤醒，而不是通过轮询等待外部变化
- Sleeping Agent 仍可由 Runtime 按已批准订阅或显式唤醒恢复；Hibernated Agent 不接受普通事件，只能通过显式唤醒进入工作
- Frozen Agent 不接受普通执行、事件唤醒和自动恢复；只有显式恢复后才能重新进入工作
- Archived Agent 会被冻结、回收运行资源并释放渠道绑定；Destroyed Agent 只能从归档态进入，且只能由用户请求
- 事件虽然无处不在，但事件可见性与消费权限仍以 Workspace、权限与审批为约束
- Sandbox 是抽象层，Docker 只是首批实现；隔离级别支持 `none` / `workspace` / `dedicated`
