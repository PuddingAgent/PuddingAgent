# Pudding Agent Network — 架构总览

> **2026-05-02 架构简化**：从多服务分布式模型简化为单进程 P2P 模型。每个 Agent 是一个独立可执行程序，内嵌 Web UI、Controller、Runtime、SQLite。Agent 之间通过 P2P 发现与直连通信。

## 产品定位

Pudding 不是一个需要 Nginx + PostgreSQL + Redis + RabbitMQ + MinIO 全家桶才能跑起来的"企业平台"。

它是一个**双击即用的 AI 代理程序**。

> **Pudding = 可 P2P 组网的 AI Agent 进程**

- 双击启动，浏览器打开即用
- 一个进程 = 一个 Agent（内嵌 Web UI + LLM 对话 + 记忆 + 工具）
- 多个 Agent 之间通过 P2P 协议自动发现、直连通信、协作完成任务
- 数据库只依赖 SQLite，零外部基础设施
- 仍支持 Docker 部署（单容器暴露端口即可）

## 单 Agent 内部结构

```
┌─────────────────── Pudding Agent（单进程）────────────────┐
│                                                            │
│  浏览器 → localhost:8080                                   │
│  ┌──────────────────────────────────────────────────┐     │
│  │  内嵌 Web UI（管理界面 + 对话界面）                 │     │
│  ├──────────────────────────────────────────────────┤     │
│  │  Controller 逻辑（路由/鉴权/调度/审计，同进程）     │     │
│  ├──────────────────────────────────────────────────┤     │
│  │  Runtime（LLM 对话/工具执行/记忆/Skill 装配）     │     │
│  ├──────────────────────────────────────────────────┤     │
│  │  SQLite（配置/会话/记忆/审计，单文件数据库）        │     │
│  ├──────────────────────────────────────────────────┤     │
│  │  P2P 网络层（节点发现/直连通信/事件广播）           │     │
│  └──────────────────────────────────────────────────┘     │
│                                                            │
│  ← P2P →  其他 Agent 节点                                  │
└────────────────────────────────────────────────────────────┘
```

## 多 Agent 协作：P2P 网络模型

Agent 不是孤立进程。当多个 Pudding Agent 运行在同一局域网或可互通网络中时：

- **自动发现**：基于 mDNS 或 DHT 发现对等节点
- **直连通信**：Agent 之间直接建立 HTTP/gRPC 连接，无需中心控制器
- **事件广播**：Agent 将任务完成、状态变更等事件广播给感兴趣的节点
- **数字蚁群**：每个 Agent 像蚂蚁一样感知局部状态，协作推进任务图

这就是原来的 Controller/Runtime 两层架构的 P2P 实现——每个 Agent 同时具备控制面和执行面能力，协作去中心化。

## 对最终用户的体验

```
用户视角：只有一个东西

1. 下载 pudding-agent（单个可执行文件，约 50MB）
2. 双击运行
3. 浏览器打开 http://localhost:8080
4. 开始对话

如果想多 Agent 协作：
5. 在另一台机器上也运行 pudding-agent
6. 它们自动发现对方
7. 在 UI 中可以看到对等节点，分配任务

高级用户：
- 可以用 Docker 单容器部署：docker run -p 8080:8080 pudding-agent
- 可以在 Linux 服务器上 systemd 托管
```

## 技术选型

| 组件 | 技术 | 原因 |
|---|---|---|
| 后端框架 | ASP.NET Core | C# 生态，高性能 |
| 数据库 | SQLite（通过 EF Core） | 零配置，单文件，足够个人/小团队使用 |
| 前端 | React/TypeScript，打包嵌入后端 | 单进程部署 |
| P2P 发现 | mDNS + UDP 广播（局域网）/ DHT（广域网） | 零配置自动发现 |
| 节点通信 | HTTP/gRPC 直连 | 简单可靠 |
| LLM | 直接调用 API（OpenAI 兼容） | 不经过中间代理 |

## 不再需要的基础设施

以下全部废弃，由程序自身替代：

| 废弃 | 替代方案 |
|---|---|
| Nginx | ASP.NET Core 内置 Kestrel 直接对外 |
| PostgreSQL | SQLite |
| Redis | 内存缓存 + SQLite |
| RabbitMQ | P2P 直连事件广播 |
| MinIO | 本地文件系统 + SQLite BLOB |
| Docker Compose 多容器 | 可选：单容器或裸进程 |

## 解决方案结构

```text
Source/
├── PuddingAgent/          ← 主程序入口（单进程宿主）
│   ├── WebUI/             ← 内嵌前端（React 构建产物）
│   ├── Controller/        ← 控制逻辑（路由/鉴权/调度）
│   ├── Runtime/           ← Agent 执行（LLM/工具/记忆）
│   ├── P2P/               ← P2P 网络层
│   └── Data/              ← SQLite 数据访问
├── PuddingCore/           ← 共享抽象与协议
├── PuddingPlatformAdmin/  ← 前端源码（构建后嵌入 PuddingAgent）
├── PuddingCode/           ← 旧 CLI 原型（保留过渡）
└── PuddingCodeCLI/        ← 旧 CLI 入口（保留过渡）
```

## 关键边界

- 一个进程 = 一个 Agent，一个 Agent = 一个身份
- Agent 拥有自己的 SQLite 数据库、记忆、配置
- Agent 之间平等，通过 P2P 协议协作，无中心节点
- 用户在任一 Agent 的 Web UI 上可以管理自己的 Agent，也可以看到对等网络中的其他 Agent
- 工具在 Agent 进程内直接执行
- 高风险操作需用户确认

## 当前状态

处于从 PuddingCode 原型向单进程 Pudding Agent 的过渡期。

已废弃的概念：Controller/Runtime/Platform/Gateway 分进程部署、Docker Compose 多容器、RabbitMQ/Redis/MinIO 依赖。

保留的概念：Workspace、Agent Template、Skill/MCP、记忆引擎、事件驱动、数字蚁群——这些仍然有效，只是实现方式从分布式服务变为单进程内模块 + P2P 协作。
