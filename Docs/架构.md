# Pudding Agent Network — 架构文档

我们正在实现一个类似于 Claude Code / OpenClaw 的智能体系统，与同类项目不同的是，我们天然支持**多智能体 + 子智能体**架构，所有 Agent 在统一网关上运行并受网关管理。

当前架构决策基线：

- **分布式优先**：从一开始支持多 Runtime 节点
- **Runtime 多承载**：每个 Runtime 可承载多个 Session / Agent
- **Platform 与 Runtime 混合协议**：控制走 HTTP/gRPC，事件与订阅走 MQTT / WebSocket
- **插件默认受信任**：当前阶段偏内部环境与快速集成
- **Runtime 为会话权威**：Session 热状态与主状态由 Runtime 持有
- **租户主模型**：组织 > 空间/项目空间 > 会话
- **公共用户归属渠道**：公共用户先挂渠道，再映射到会话
- **内置邮箱渠道**：平台内置支持 Email Channel
- **渠道接入插件化**：渠道机制必须插件化，以便后续扩展更多渠道

---

## 设计目标

- 支持多用户、多渠道、多 Runtime 的 Agent 协作网络
- 支持公共用户接入，而不仅仅是单人本机助手模式
- 支持组织级租户、多空间隔离、会话级执行隔离
- 支持一个工作空间容纳多个 Agent，且 Agent 必须隶属于某个工作空间
- 支持多 Agent、Swarm、SubAgent 的编排、隔离与审计
- 支持显式工作流、事件驱动自动化、跨 Agent 任务交接
- 支持后续把 Platform、Runtime、Web 前端分别独立部署
- 支持后续引入租户治理、计费、契约管理与插件生态

---

## 解决方案结构总览

```
PuddingAgentNetwork.slnx
Source/
├── PuddingCore/       ← 核心类库：接口、模型、Swarm、Skills、Tools
├── PuddingRuntime/    ← 运行时宿主（独立 exe，可独立部署）
├── PuddingAgent/      ← Agent 定义类库（待实现）

├── PuddingWeb/        ← Web 前端（PuddingPlatform 的前端，前后端分离）
├── PuddingCoreTests/  ← PuddingCore 单元测试
└── PuddingCLITests/   ← PuddingCLI 单元测试
```

## 各项目职责

### PuddingCore（核心类库）

所有项目共享的核心定义，目标框架 `net10.0`，NuGet 依赖：`CliWrap`。

定位：**提供能力，不持有长期运行状态**。

#### 分层结构

| 目录 | 职责 | 命名空间 |
|------|------|---------|
| `Abstractions/` | 接口与枚举定义 | `PuddingCode.Abstractions` |
| `Models/` | 数据模型（record/enum） | `PuddingCode.Models` |
| `Core/` | 核心实现（LLM 网关、Tool 注册、权限守卫） | `PuddingCode.Core` |
| `Swarm/` | 多 Agent 协作（消息总线、会话编排、锁协调） | `PuddingCode.Swarm` |
| `Skills/` | 可扩展技能插件系统 | `PuddingCode.Skills` |
| `Tools/` | 内置工具实现（文件、Shell） | `PuddingCode.Tools` |

#### 核心抽象

- **`ILlmGateway`** — LLM 调用入口，实现：`OpenAiLlmGateway`
- **`IAgentOrchestrator`** — 单 Agent 编排（思考 → 工具调用 → 输出事件流）
- **`ISwarmOrchestrator`** — 多 Agent Swarm 编排
- **`ISwarmMessageHub`** — Agent 间消息总线
- **`ISwarmTransport`** — 传输层（当前实现：`FileSwarmTransport`）
- **`IWorkerManager`** — Worker（子 Agent）生命周期管理
- **`ICentralLockManager`** — 跨 Agent 协调锁（防止竞态）
- **`IContractManager`** — Agent 任务契约管理
- **`IToolRegistry`** — 工具注册与分发
- **`ISkillRegistry`** — 技能插件注册（基于 `[PuddingSkill]` Attribute）
- **`IOutputDistiller`** — LLM 输出精馏（降噪/摘要）
- **`IGitSnapshot`** — Git 工作区快照

说明：

- `PuddingCore` 继续保留 Skill、Tool、MCP、Plugin 所需的**抽象、协议和通用模型**
- 但基础 Skill 的装配、MCP 的连接会话、Provider 绑定后的运行逻辑，应下沉到 `PuddingRuntime`
- 插件系统的注册、装配、启停和租户级管理，不放在 `PuddingCore`

#### Agent 事件流

```
ThinkingEvent → ToolCallEvent → ToolResultEvent → AnswerEvent
                                                 ↘ ErrorEvent
ReasoningEvent（推理链）
StreamingAnswerEvent（流式输出）
```

#### Swarm 消息模型

```
SwarmMessage（含 SwarmMessagePriority）
  └─ 通过 SwarmMessageHub 路由至各 SwarmNode
       └─ SwarmOrchestrator 编排各 WorkerRole
            └─ WorkerManager 管理 WorkerInfo 实例
```

### PuddingRuntime（运行时宿主）

`PuddingRuntime` 不再只被定义为一个类库层，而是一个**独立的运行时宿主程序**。它负责真正承载 Agent、会话、记忆与运行时能力，并且可以被单独部署。

定位：**数据平面 / 执行平面**。

形态建议：

- 物理形态：独立 `exe`
- 宿主模型：`.NET Generic Host`
- 部署形态：分布式优先，支持远程节点、多实例集群

说明：

- Runtime 负责“真正执行”Agent 工作负载
- Platform 不再把 Runtime 简单视为内部对象，而是把它视为可注册、可发现、可调度的运行节点
- 这种关系更接近“Platform 管 Runtime”，类似 Docker 控制 Runtime/Engine 的思路
- 一个 Runtime 默认可以承载多个 Session / Agent / Worker，而不是一 Runtime 只对应一个 Agent
- Agent 的具体运行实例属于某个会话，但 Agent 模板与工程定义应归属于空间/项目空间

建议放入该层的内容：

- `SubconsciousEngine`：潜意识/双模型运行时
- `MemoryIndexer`：记忆索引与召回
- `TodoListStore`：任务清单状态
- `ReviewQueueStore`：审阅队列状态
- 基础 `SkillRuntime`：内置 Skill 装配、发现、注册、执行上下文
- 基础 `McpRuntime`：MCP Server 连接、会话、能力发现、调用转发
- 基础 `PluginRuntimeSupport`：插件执行上下文、能力桥接、生命周期回调支持
- 后续的 `AgentRuntime`、`SessionRuntime`、`MemoryRuntime`

#### Runtime 的部署与注册

- Runtime 可以独立启动
- Runtime 启动后向 Platform 注册自身能力、状态、版本、健康信息
- Platform 根据策略把渠道/会话/Agent 调度到某个 Runtime
- 一个 Platform 可以管理多个 Runtime

#### Runtime 的承载模型

- 一个 Runtime 可以同时承载多个 Session
- 一个 Session 内可以包含多个 Agent / Swarm / SubAgent
- Runtime 负责这些 Session 的实际执行与隔离
- Platform 不直接持有会话执行状态，而是通过控制协议管理 Runtime

#### Runtime 的职责边界

- 管理单个 Agent 的短期上下文窗口
- 管理长期记忆、召回、压缩、写回候选
- 管理会话级状态，如 Todo、Review、临时工作记录
- 管理单个会话内的 Agent 运行态与会话隔离
- 作为 Session 的权威宿主，持有会话主状态与热状态
- 负责“自我管理”相关逻辑，例如自检、记忆维护、上下文裁剪策略
- 负责基础 Skill 的运行时注册、加载、执行上下文注入
- 负责 MCP Client 生命周期、Server 会话、工具能力挂载
- 为插件提供运行时支持，例如上下文注入、能力桥接、调用宿主 API
- 承载实际 Agent 实例、Worker、Swarm 执行流程
- 上报自身健康状态、资源状态、负载状态给 Platform
- 对外暴露可被 CLI、Gateway、Manager 复用的运行时服务

#### Runtime 内建议子模块

| 子模块 | 职责 |
|------|------|
| `AgentRuntime` | 单 Agent 会话运行时，封装上下文、记忆、工具与技能调用 |
| `MemoryRuntime` | 长期记忆、索引、召回、压缩维护 |
| `SessionRuntime` | Todo、Review、临时工作态、会话元数据、会话隔离 |
| `SkillRuntime` | 基础 Skill 注册、发现、执行、上下文注入 |
| `McpRuntime` | MCP Server 连接管理、能力发现、调用路由 |
| `PluginRuntimeSupport` | 插件执行上下文、宿主能力访问、运行时桥接 |

#### Runtime 不负责的事

- 不负责多 Runtime 全局调度
- 不负责渠道接入边界
- 不负责全局认证鉴权
- 不负责租户级治理与平台策略
- 不负责公共用户入口治理与全局计费决策

这些职责由 `PuddingPlatform` 负责。

#### 不属于 Runtime 的职责

- 不负责 LLM 基础协议抽象，这属于 `PuddingCore`
- 不负责 Skill Attribute、Skill 抽象协议本身，这属于 `PuddingCore`
- 不负责 MCP 协议模型与底层抽象本身，这属于 `PuddingCore`
- 不负责插件的注册中心、安装、启停、版本与装配决策，这属于 `PuddingGateway`
- 不负责具体 Agent 人设与能力声明，这属于 `PuddingAgent`
- 不负责进程托管、路由、权限权威控制，这属于 `PuddingGateway`
- 不负责终端交互与展示，这属于 `PuddingCLI`

#### 分层原则

```
PuddingCore     = 无状态能力与抽象
PuddingRuntime  = 独立运行时宿主、执行平面、Agent 承载节点
PuddingAgent    = Agent 定义与角色模板
PuddingPlatform = ASP.NET Core 控制面、接入层、调度与治理中心
```

### PuddingAgent（Agent 定义类库）

定义具体 Agent 行为、系统提示词、能力声明。

定位：**描述 Agent 是谁，而不是 Agent 现在处于什么状态**。

职责：

- 定义 Agent 的角色、人设、System Prompt 模板
- 定义可用能力、技能组合、工具白名单
- 定义不同 Agent 的默认运行策略

说明：

- `PuddingAgent` 决定“这个 Agent 默认启用哪些 Skill / MCP 能力”
- `PuddingRuntime` 决定“这些 Skill / MCP 能力如何真正加载与运行”
- `PuddingAgent` 可以声明 Agent 需要哪些插件能力，但不负责插件装配

当前为占位状态（待实现）。

### PuddingPlatform（ASP.NET Core 平台宿主）

`PuddingPlatform` 是整个系统的主宿主程序，建议直接采用 **ASP.NET Core** 形态。

定位：**控制平面 / 管理平面**。

形态建议：

- 物理形态：一个可执行程序
- 宿主框架：`ASP.NET Core` / `.NET Generic Host`
- 承载内容：API、WebSocket、后台服务、平台服务、Gateway 模块

说明：

- 这里不再把 Gateway 视为单独项目名
- `PuddingGateway` 改为 `PuddingPlatform` 内部模块名
- Runtime 作为独立 exe 存在，Platform 通过控制协议管理 Runtime
- 这种关系更接近“平台控制运行时”，而不是“平台内部直接 new 运行时对象”

#### Platform 与 Runtime 控制协议

采用**混合模式**：

- 控制面：`HTTP/gRPC`
- 事件、订阅、广播：`MQTT / WebSocket`

原则：

- 配置、调度、注册、健康检查走强类型控制协议
- 状态流、事件流、订阅广播走消息协议
- Runtime 不要求和外部用户直接通信，优先通过 Platform 统一中转

职责：
- 管理 Runtime 节点注册、发现、健康检查、容量感知
- 调度 Agent、Session、Swarm 到具体 Runtime
- 暴露 API（HTTP/gRPC）供 CLI 和 Web 前端连接
- 提供消息路由与订阅能力，支持 `MQTT`、`WebSocket`
- 负责 Swarm 消息路由与协调锁的权威节点
- 负责会话创建、会话隔离、租户/用户边界隔离
- 负责用户认证、鉴权、渠道权限控制
- 管理插件注册、安装、启停、版本与装配
- 按 Agent、工作区、策略决定装配哪些插件
- 负责公共用户入口治理、租户边界、计费策略与审计归集
- 守护进程

#### 租户主模型

当前采用：

```
Organization
  └─ Space / Workspace / ProjectSpace
    └─ Session
      └─ Agent Runtime Instance
```

含义：

- `Organization` 是最高租户边界
- `Space` 是组织内部的工作空间/项目空间边界
- `Session` 是执行隔离边界
- `Agent` 的运行实例属于 Session
- `Agent` 的模板、能力定义、装配策略通常属于 Space

#### 资源归属原则

默认归属如下：

- `Session` 归属于 `Space`
- `Agent 模板` 归属于 `Space`
- `插件装配策略` 归属于 `Space`
- `计费账户` 归属于 `Organization` 或 `Space`
- `记忆库` 归属于 `Space` 或 `Session`，不默认跨空间共享

其中：

- `Agent` 更接近“工作空间项目资源”
- `Session` 更接近“运行态资源”
- `Memory` 是需要最严格定义边界的资源之一

#### PuddingGateway（平台内模块）

`PuddingGateway` 不再作为独立项目名存在，而是 `PuddingPlatform` 内部的边界接入模块。

职责：

- 作为平台对外的统一入口
- 处理 HTTP / WebSocket / MQTT 等接入协议
- 承接认证、鉴权后的消息入口与出口
- 将外部请求路由到具体渠道、会话、Agent 网络

#### Platform 与 Runtime 的关系

```
PuddingPlatform = 控制面
PuddingRuntime  = 执行面
```

Platform 负责：

- 管理 Runtime 节点
- 下发任务、会话、配置、插件装配策略
- 汇总状态、审计、指标、权限决策
- 管理全局租户、公共用户入口与计费规则

Runtime 负责：

- 实际运行 Agent
- 实际承载 Session 上下文与执行状态
- 实际调用 Skill / MCP / Plugin / Tool
- 作为 Session 主状态权威

可以类比为：

- Platform 更像“控制台 / 编排层”
- Runtime 更像“被平台管理的执行引擎”

#### 渠道（Channel）模型

参考 OpenClaw 的思路，这里引入**渠道**概念。

渠道表示“谁通过什么入口与 Gateway 通信”，而不是单纯的连接对象。Gateway 需要天然支持多用户，因此所有外部请求都必须先落到渠道，再进入会话和 Agent 网络。

公共用户不直接成为平台内部租户主体，而是**先附着在渠道上**，再由 Platform 根据策略映射到某个组织/空间/会话。

同时，渠道不应被写死为平台内部的一组固定 if/else 分支。平台需要区分：

- **内置渠道**：例如 CLI、HTTP/WebSocket、Email
- **扩展渠道**：例如飞书、企业微信、Telegram、Slack、论坛、工单系统、定制企业接入

也就是说，平台可以内置少量核心渠道，但**渠道接入机制本身必须插件化**。

渠道可能包括：

- 本机管理员渠道
- 组织内部用户渠道
- 内置邮箱渠道（Email）
- 第三方集成渠道
- 不受信任匿名渠道
- 自动化系统渠道（Webhook / Bot / External App）
- 公共用户渠道（例如客服、服务台、开放问答入口）

其中内置邮箱渠道用于：

- 接收组织邮箱、服务邮箱、共享邮箱、工单邮箱中的邮件消息
- 将邮件线程映射到 `ServiceSession` 或任务型会话
- 按收件地址、发件人、主题规则、工作空间配置路由到目标 Agent

邮箱渠道建议具备的附加维度：

- `MailboxAddress`
- `ThreadKey`
- `SenderIdentity`
- `SubjectRoutingRule`
- `AttachmentPolicy`

#### 渠道插件化原则

目标：

- 后续新增渠道时，不需要修改 `PuddingPlatform` 的核心路由主干
- 不同渠道只需要实现统一的接入、身份解析、消息标准化、回传能力
- 平台统一处理鉴权、会话映射、Agent 路由、审计与审批

建议抽象：

```
ChannelProvider
  ├─ ChannelType
  ├─ BindAsync(...)
  ├─ NormalizeInboundAsync(...)
  ├─ ResolveIdentityAsync(...)
  ├─ SendReplyAsync(...)
  └─ GetCapabilities()
```

建议最小能力：

- 接收外部消息并转换为标准化 `InboundChannelMessage`
- 解析渠道身份，映射为平台可识别用户上下文
- 把平台回复发送回原渠道
- 暴露渠道能力，例如是否支持线程、附件、引用消息、回执、富文本

边界：

- `ChannelProvider` 负责“怎么接进来、怎么发回去”
- `PuddingPlatform` 负责“接进来后是谁、能进哪个 Workspace、路由给哪个 Agent、是否允许执行”
- 渠道插件不直接拥有平台治理权

每个渠道至少绑定以下维度：

- `ChannelId`
- `OwnerType`（个人、组织、第三方、匿名）
- `Identity`
- `TrustLevel`
- `AuthPolicy`
- `PermissionScope`
- `Transport`（CLI / HTTP / WebSocket / MQTT）

#### 公共用户接入原则

- 公共用户默认先属于某个渠道
- 渠道再决定该用户能进入哪个组织、空间或会话
- 公共用户不直接持有平台内部高权限身份
- 对客服、开放服务台、面向不定向用户的入口，渠道是首要安全边界

说明：

- 邮箱渠道同样适用于公共入口和半信任入口
- 来自外部邮箱域名的消息默认应视为中低信任输入
- 邮件附件、转发链和引用历史都应进入污染检测与审计范围

#### Gateway 内建议子模块

| 子模块 | 职责 |
|------|------|
| `ChannelManager` | 渠道注册、渠道状态、渠道权限策略 |
| `AuthGateway` | 用户认证、令牌校验、身份映射 |
| `AuthorizationService` | 渠道级、会话级、Agent 级鉴权 |
| `SessionRouter` | 会话路由、会话隔离、会话寻址 |
| `MessageRouter` | Agent 间消息路由、订阅分发、广播 |
| `TransportHost` | MQTT / WebSocket / HTTP 等接入承载 |
| `PluginManager` | 插件装配、启停、版本、租户策略 |
| `RuntimeManager` | Runtime 注册、发现、调度、健康检查、资源感知 |
| `BillingManager` | 计费策略、用量归集、渠道/租户/会话成本统计 |
| `ContractCoordinator` | Swarm 契约登记、分发、校验与执行跟踪 |
| `ChannelPluginHost` | 渠道插件发现、加载、生命周期管理、能力注册 |
| `PuddingGateway` | 平台内边界接入模块，封装统一入口与协议暴露 |

#### 路由与隔离原则

- Gateway 是唯一外部入口，所有消息先到 Gateway，再路由到具体会话或 Agent
- 会话天然隔离，不同用户、不同渠道、不同组织的上下文默认不可见
- Agent 默认不能直接暴露给外部网络，必须通过 Gateway 代理
- 外部连接只绑定渠道，渠道再绑定到会话和权限范围
- MQTT / WebSocket 只是传输协议，不直接拥有授权语义，授权仍由 Gateway 统一判断

#### 认证与鉴权原则

- 认证回答“你是谁”
- 鉴权回答“你能访问哪个渠道、会话、Agent、工具、插件”
- 渠道权限高于传输协议权限
- 会话权限高于单条消息内容
- 不受信任渠道默认只能访问受限能力和隔离沙箱会话

边界：

- Platform 管理 Agent 的生死、资源、权限、路由
- Runtime 管理 Agent 的上下文、记忆和自我状态，以及实际执行
- Platform 管理插件“是否装、装哪些、何时启停”
- Runtime 负责插件“装上后怎么跑、怎么拿上下文、怎么访问能力”
- Platform 管理“谁可以进入网络、进入哪个会话、向谁发消息”
- Gateway 模块负责“从哪里进来、通过什么协议进来、如何进入平台”
- Platform 负责任务下发到哪个 Runtime
- Runtime 不直接承担平台治理职责

#### 插件信任模型

当前阶段采用：**插件默认受信任**。

适用前提：

- 主要部署在受控环境
- 插件来源以官方或内部插件为主
- 目标是先降低集成成本和平台复杂度

后续可演进到：

- 官方插件默认高信任
- 第三方插件低信任
- 不受信任插件进入沙箱执行

当前为占位状态（Hello World），待实现。

## 协作网络模型

### 多用户与渠道

系统天然支持多用户，用户并不直接连接某个 Agent，而是先进入某个渠道，再由 Gateway 将渠道映射到会话和权限域。

```
User / Org / Third-Party / Untrusted
    └─ Channel
      └─ Gateway Module
        └─ Platform
          └─ Runtime
            └─ Session
              └─ Agent / Swarm / SubAgent
```

含义：

- 用户是身份主体
- 渠道是接入视角
- Gateway Module 是协议入口
- Platform 是控制与治理宿主
- Runtime 是实际执行宿主
- 会话是隔离边界
- Agent 是执行单元

### 组织、空间、会话三层模型

当前推荐的租户模型为：

```
Organization
  └─ Space
       └─ Session
```

边界含义：

- `Organization`：最高治理与计费边界
- `Space`：项目、业务域、团队协作边界
- `Session`：一次具体交互或执行任务的运行边界

建议：

- Agent 模板属于 `Space`
- Session 运行实例属于 `Space`
- 公共用户通过 `Channel` 进入某个 `Space`
- Runtime 承载 `Session`，而不是直接承载整个 `Organization`

### 工作空间与 Agent 归属

在当前文档里，`Space / Workspace / ProjectSpace` 可以视为一类概念，后续命名可统一，但语义应保持稳定。

核心原则：

- 同一个工作空间内可以容纳多个 Agent
- Agent 必须属于某个工作空间，不能成为“无归属的全局 Agent”
- Agent 模板、默认能力、插件装配策略、工作流绑定关系，默认都属于工作空间
- 同一个工作空间可以挂接多个渠道
- 每个渠道绑定关系必须声明该渠道可见哪些 Agent，必要时可声明默认 Agent
- Session 是运行时实例边界，Workspace 是配置与治理边界

建议结构：

```
Organization
  └─ Workspace
  ├─ Channel Binding A
  ├─ Channel Binding B
       ├─ Agent Template A
       ├─ Agent Template B
       ├─ Workflow A
       ├─ Workflow B
       ├─ Memory Policy
       └─ Runtime Sessions
```

边界含义：

- `Workspace` 决定接入了哪些渠道
- `Workspace` 决定有哪些 Agent 可以存在
- `Workspace` 决定不同渠道可以看到哪些 Agent，以及默认路由到哪个 Agent
- `Workspace` 决定这些 Agent 能参与哪些工作流
- `Runtime` 根据 Workspace 中的模板与策略创建实际 Agent 实例
- 普通 Agent 可以长期存在，子智能体由运行时按需临时创建

### 会话隔离

会话是运行时隔离的第一单位。

每个会话至少包含：

- 会话 ID
- 所属渠道
- 所属组织 / 空间 / 用户
- 会话级上下文与记忆引用
- 允许访问的 Agent 集合
- 允许访问的 Skill / MCP / Plugin 集合
- 所属 Runtime

隔离规则：

- 默认禁止跨会话共享上下文
- 默认禁止跨渠道读取会话状态
- 默认禁止跨空间共享记忆
- 默认禁止跨组织共享记忆
- 仅 Gateway 可执行跨会话路由或授权代理
- Runtime 只处理“本会话内”的状态，不自行越权访问其他会话
- Runtime 持有本会话主状态，Platform 只持有控制与索引视图

#### 记忆边界原则

记忆是当前最容易做错的边界之一，因此默认原则必须保守：

- Session Memory：仅属于当前 Session
- Space Memory：属于当前 Space，可被该 Space 内授权 Agent 使用
- Organization Memory：仅在组织级显式授权时共享
- 默认禁止跨 Space / 跨 Organization 自动召回记忆

如果未来需要共享记忆，应通过显式策略和审计进行开放，而不是默认联通。

### 工作流模型

系统应支持**显式工作流**，而不仅仅是自由对话式 Agent 协作。

这里的工作流是：

- 由管理员或租户显式配置的任务流
- 由事件、定时器、上游任务结果、用户行为或外部消息触发
- 由工作流引擎负责推进，而不是完全依赖单个 Agent 自发决定

典型例子：

- `搜集 -> 整理 -> 报告`
- 第一个 Agent 负责搜集
- 第二个 Agent 负责整理
- 第三个 Agent 负责输出报告
- 每个 Agent 按工作流节点接收输入并输出结果
- Workflow Engine 根据节点状态驱动下一步执行

工作流要求：

- Workflow 必须属于某个 Workspace
- Workflow 节点必须能绑定到一个或多个 Agent 模板
- Workflow 必须支持显式输入、输出、状态、失败、重试、交接
- Workflow 必须可以被事件触发，而不是只能手工启动

建议模型：

```
Workflow
  └─ Step
      ├─ BoundAgentTemplate
      ├─ InputContract
      ├─ OutputContract
      ├─ RetryPolicy
      └─ NextStepRule
```

边界结论：

- Agent 负责完成节点任务
- Workflow Engine 负责推进任务流
- Platform 负责注册、调度、审计 Workflow
- Runtime 负责执行 Workflow 节点所需的 Agent 实例

### 事件模型

系统应具备统一事件总线，至少支持以下事件来源：

- `ScheduledEvent`：定时任务事件
- `EnvironmentEvent`：环境事件，例如文件变化、构建结果、监控指标
- `ExternalEvent`：外部事件，例如 WebSocket、Webhook、MQTT 消息
- `AgentEvent`：智能体事件，例如唤醒其他 Agent、发送消息、交接任务
- `WorkspaceEvent`：工作空间事件，例如配置更新、工作流发布、策略变化
- `UserEvent`：用户行为触发，例如提问、审批、确认执行、关闭任务
- `GlobalEvent`：全局事件，例如平台策略更新、系统维护广播、全局告警

事件要求：

- 所有事件必须带来源、作用域、可信级别、载荷类型
- 事件必须能够路由到 Workflow、Agent、Interceptor、Trigger
- 事件必须可审计、可过滤、可订阅
- 不同作用域的事件默认不能越级广播

建议统一模型：

```
PlatformEvent
  ├─ EventId
  ├─ EventType
  ├─ Source
  ├─ Scope (Global / Organization / Workspace / Session / Agent)
  ├─ TrustLevel
  ├─ Payload
  └─ CorrelationId
```

### 消息路由与订阅

Gateway 负责整个协作网络中的消息路由与订阅分发。

支持的接入协议：

- `WebSocket`：适合交互式前端、CLI、实时控制面板
- `MQTT`：适合 Agent 网络、设备、低耦合集成、事件订阅

路由能力包括：

- 用户到 Gateway 的入站消息路由
- Platform 到 Runtime 的控制消息路由
- Gateway 到会话的定向投递
- 会话内 Agent 到 Agent 的消息路由
- 事件订阅与广播
- 渠道级主题过滤与访问控制

建议主题模型：

```
channels/{channelId}/events
sessions/{sessionId}/events
sessions/{sessionId}/agents/{agentId}/inbox
sessions/{sessionId}/agents/{agentId}/outbox
```

其中：

- 主题结构由 Gateway 统一定义
- 渠道是否可订阅、是否可发布，由 Gateway 鉴权决定
- Agent 不直接暴露底层 MQTT / WebSocket 细节

建议进一步支持：

- 工作流订阅事件后自动启动
- Agent 订阅其他 Agent 或 Workflow 的完成事件
- 触发器订阅高风险操作事件，执行拦截或审计

## 关键应用场景

以下场景用于反向约束架构设计，确保平台不是只适合“单用户 IDE 助手”，而是真正适合多用户、多渠道、公开环境下的 Agent 网络。

### 场景 1：租户通过渠道与工作空间指定 Agent 通信

场景：

- 某个组织用户通过渠道进入平台
- 渠道被映射到该组织下某个 `Space`
- 该 `Space` 指定了一个或多个对外服务 Agent
- 用户只与这些被工作空间显式暴露的 Agent 通信

架构要求：

- `Space` 必须能声明“哪些 Agent 对哪些渠道可见”
- `Space` 必须支持一个工作空间挂接多个渠道
- 默认不是所有 Agent 都能被渠道访问
- 某个渠道可以声明默认 Agent，也可以声明多个可见 Agent
- 渠道只拿到 Agent 的“服务接口”，而不是 Agent 的内部运行上下文
- 平台必须支持按 `Space` 配置 Agent 暴露策略

边界结论：

- Agent 属于 `Space` 资源
- 渠道访问的是 `Space` 对外暴露的 Agent 能力
- Gateway/Platform 负责把渠道请求路由到正确的 Runtime Session

### 场景 2：Agent 进入公开群，面对不受信任用户

场景：

- 渠道例如飞书、Slack、论坛、公开群
- Agent 被邀请进入一个公开群
- 群内既有租户自己的成员，也有外部不受信任人员
- 外部人员可能发起提示词攻击、越权诱导、上下文污染、记忆投毒

这类场景下，**群本身不能被视为可信上下文**。

架构要求：

- 公开群渠道必须具备 `TrustLevel`
- 不受信任消息不能直接进入长期记忆
- 不受信任消息不能自动影响 Space 级记忆或 Agent 策略
- 群消息应先经过渠道级清洗、过滤、审计，再进入会话
- Agent 在公开群中应运行在“受限服务模式”，而不是完整内部助手模式

防护要求：

- **提示词攻击隔离**：用户输入与系统提示、Agent 提示严格分层
- **上下文污染隔离**：群消息默认只进入当前 Session 工作上下文，不自动进入 Space Memory
- **记忆写入审查**：公开群来源内容进入长期记忆前必须经过策略或人工确认
- **能力降级**：公开群 Agent 默认禁用高风险 Tool、内部插件、跨空间检索
- **身份区分**：同一群内的组织成员与外部用户应能在身份层面被区分

边界结论：

- 不受信任用户不应直接拥有 Space 内部身份
- 公开群是高风险渠道，不是高信任会话
- 公开群消息应优先挂载到“渠道隔离会话”或“受限服务会话”

### 场景 3：公司内部群或论坛，Agent 负责答疑、派发任务、执行任务

场景：

- 渠道是公司内部 IM 群、论坛、Issue 系统、工单系统
- Agent 既可能回答问题，也可能把问题转成 Issue
- Agent 还可能基于 Issue 派发任务、启动 Swarm、执行自动化流程

这类场景属于**半信任到高信任环境**，但仍不能等同于完全本机交互。

架构要求：

- 渠道需要支持“问答模式”和“任务模式”两种工作形态
- Agent 不能仅仅回答问题，还要能把消息转成结构化任务
- Platform 需要支持从渠道消息到 `Issue / Task / Contract / Swarm` 的映射
- Runtime 需要支持把任务型消息切换到执行态 Session

建议模式：

- **问答模式**：只允许读型能力、知识检索、低风险 Skill
- **任务派发模式**：允许创建 Issue、创建 Todo、触发审批
- **执行模式**：需要更高权限，可能需要人工确认、契约确认或审批流

边界结论：

- 同一个渠道内也应区分“消息问答”和“任务执行”两类会话
- 任务执行不应因为渠道可信就自动放权
- 内部群/论坛适合触发任务，但不应绕过 Platform 的鉴权、契约和审计

### 三个场景共同推导出的设计要求

#### 1. 渠道必须分级信任

至少应支持：

- 高信任渠道：本机管理员、内部受控系统
- 中信任渠道：组织内部群、论坛、Issue 系统
- 低信任渠道：公开群、面向客户的服务台、开放入口

#### 2. Agent 必须有“服务暴露面”概念

Agent 不能默认对所有渠道可见。应至少支持：

- Agent 是否对某个 Space 可见
- Agent 是否对某个渠道可见
- Agent 是否是某个渠道的默认服务 Agent
- Agent 在该渠道下启用哪些能力
- Agent 在该渠道下是否允许写记忆、派发任务、调用高风险工具

#### 3. 会话必须区分服务会话与执行会话

建议区分：

- `ServiceSession`：面向外部问答、公开群、客服、论坛
- `ExecutionSession`：面向内部任务执行、Swarm、Tool 调用、自动化流程

原则：

- 低信任输入优先进入 `ServiceSession`
- 只有经过策略确认后，才升级或派生出 `ExecutionSession`

#### 4. 记忆写入必须受来源约束

建议按来源控制：

- 本机/高信任渠道：可按策略自动写入 Space Memory
- 内部群/论坛：可写入候选记忆，需筛选
- 公开群/不受信任渠道：默认只保留在 Session，禁止自动写入长期记忆

#### 5. 问答与任务执行必须拆开授权

用户能“问问题”不代表用户能“让 Agent 执行任务”。

至少应区分：

- `Ask` 权限
- `CreateTask` 权限
- `TriggerSwarm` 权限
- `ExecuteTool` 权限
- `WriteMemory` 权限

### 4. 工作流与任务交接必须成为一等公民

系统不能只支持“消息对话”，还必须支持：

- 显式 Workflow 定义
- Agent 到 Agent 的任务交接
- 任务完成事件驱动下一个节点
- 从问答会话派生任务执行会话

### 5. 子智能体必须是临时执行资源

子智能体不是长期租户资源，而是临时执行资源。应满足：

- 由父 Agent 或 Workflow 按需创建
- 默认继承创建者权限边界，而不是提升权限
- 生命周期受任务约束
- 任务完成后自动销毁
- 通过事件把结果回传给父 Agent 或 Workflow Engine

### 6. 权限必须同时作用于 Agent 与执行环境

不仅要限制 Agent 能做什么，还要限制 Agent 在什么环境里做。

至少应支持：

- 模板级权限
- 会话级权限
- Tool/Plugin 级权限
- 文件系统/网络/进程级运行环境限制
- 按模板选择受限执行环境，例如容器、沙箱、隔离目录

## 跨工作区共享模型

默认原则：**跨工作区共享必须显式授权，而不是默认联通**。

允许共享的对象可以包括：

- 消息
- 任务
- 文件
- 工作流输出
- 结构化知识或经过批准的记忆

约束要求：

- 共享必须经过 Platform 授权
- 共享必须带来源 Workspace 与目标 Workspace
- 共享必须可审计、可撤销、可设置有效期
- 默认应通过“导出/发布”或“代理访问”模式完成，而不是直接开放底层内存或文件视图

建议模式：

- `ShareMessage`
- `ShareTask`
- `ShareArtifact`
- `PublishKnowledge`

边界结论：

- Workspace 之间可以协作，但不应直接混合上下文
- 跨工作区共享更像受控交换，而不是共享内存

## 权限与受限环境

考虑提示词污染、工具滥用、越权访问和外部集成风险，Runtime 必须能够根据 Agent 模板创建不同权限等级的执行实例。

要求：

- Agent 模板定义默认权限画像
- Runtime 根据模板创建对应权限的 Agent 实例
- 高风险 Agent 必须运行在更强隔离环境中
- 受限环境可以是 Docker、进程沙箱、隔离工作目录、受限网络策略

### 权限主体

当前建议把权限拆到三个主要主体：

- `User`：决定谁可以发起、批准、确认执行
- `WorkspaceRole`：决定该用户在当前 Workspace 内拥有多大审批权与操作范围
- `AgentTemplate`：决定该 Agent 天生“可以做什么”

说明：

- 渠道负责携带身份线索，例如第三方 IM 的用户标识
- 但渠道本身不是最终审批权威
- 审批应由 Platform 依据用户身份、Workspace 角色和策略完成，而不是由 Agent 自行判断

### 默认权限来源

当前建议：**Agent 实例的实际权限取三者交集**。

即：

- AgentTemplate 允许的能力
- Workspace 策略允许的能力
- 当前触发用户被授予的能力

最终取交集，而不是并集。

原则：

- 任一层不允许，则实际执行时视为不允许
- 冲突时取最严格，默认拒绝
- Agent 不得因上下文诱导而突破模板或策略边界

### 提问、任务与执行的权限分离

当前明确要求：

- 提问
- 创建任务
- 执行任务
- 批准执行

必须是彼此分离的权限，而不是默认联通。

说明：

- 用户能发消息，不代表用户能让 Agent 执行危险动作
- 用户能创建任务，不代表用户能批准真正落地执行
- Agent 能生成执行方案，不代表 Agent 能自己批准执行

### 平台审批规则

当前建议以下条件同时满足时，才允许高风险动作进入执行阶段：

- Agent 本身具备对应模板权限
- 当前会话中的消息来源可映射为有权限的用户
- Workspace 策略允许该用户在该上下文下发起此动作
- 如有要求，用户完成显式确认
- 如有要求，审计 Agent 审核通过

典型高风险批准流程可以是：

1. Agent 生成待执行命令或动作清单
2. Platform 根据策略判断是否允许自动批准
3. 若要求显式批准，则要求用户在限定时间内发送确认消息
4. 若配置审计 Agent，则先进入审计问答与审计判定
5. 若审计 Agent 拒绝，则升级为人工批准或直接拒绝

### 显式确认机制

对于高风险执行，建议支持显式确认令牌。

例如：

- Agent 生成操作摘要与风险说明
- Platform 生成一次性确认码
- 用户必须在限定时间内回发明确确认消息
- 只有匹配确认码后，才能进入执行

原则：

- 确认动作必须和原始请求、用户身份、时间窗口绑定
- 不能用模糊自然语言替代明确批准

### 沙箱映射原则

以下动作当前建议默认进入受限环境或更强隔离等级：

- 文件写入
- 外部网络访问
- Shell / 进程执行
- 跨工作区文件共享
- 插件加载
- 公开群触发的任何执行动作

说明：

- 是否进入哪一级沙箱，应由 AgentTemplate 和 Workspace 策略共同决定
- 公开群来源即便触发低风险执行，也应优先落在更保守的运行环境

建议权限维度：

- 可访问的文件范围
- 可访问的网络范围
- 可调用的 Tool 范围
- 可装载的 Plugin 范围
- 可写入的记忆范围
- 可触发的 Workflow / Swarm 范围

建议执行环境等级：

- `TrustedRuntimeProfile`
- `RestrictedRuntimeProfile`
- `SandboxRuntimeProfile`

边界结论：

- 权限不只是逻辑鉴权，也包括物理执行环境约束
- 相同 Workspace 下的不同 Agent 可以拥有不同运行权限与不同沙箱等级
- 平台审批权威高于 Agent 自我判断
- 高风险执行必须绑定明确身份、明确批准和明确运行环境

## 拦截器与触发器

系统需要同时支持：

- **拦截器（Interceptor）**：在 Agent 操作真正执行前介入
- **触发器（Trigger）**：在 Agent 操作或事件发生后触发后续行为

典型场景：

- Agent 发消息前先经过内容过滤
- Agent 调用 Tool 前先经过权限审查
- Agent 写入记忆前先经过污染检测
- Agent 完成任务后触发后续 Workflow
- Agent 空闲或上下文膨胀时触发记忆回收
- 一个监督 Agent 可以订阅另一个 Agent 的关键行为并做审查

拦截器要求：

- 支持拒绝、修改、审计、降级、转人工
- 支持按渠道、Workspace、Agent 模板、事件类型挂载
- 支持由规则引擎或监督 Agent 执行

触发器要求：

- 支持订阅事件源
- 支持触发 Workflow、唤醒 Agent、生成任务、执行清理动作
- 支持异步执行与失败重试

建议模型：

```
Interceptor
  ├─ MatchRule
  ├─ Scope
  ├─ Action (Allow / Deny / Modify / Escalate / Audit)
  └─ Executor (Rule / Agent / Workflow)

Trigger
  ├─ EventFilter
  ├─ Scope
  ├─ Target (Workflow / Agent / Hook)
  └─ RetryPolicy
```

## 心跳、休眠、唤醒与 Agent 生命周期

参考 OpenClaw 的思路，Agent 不应只在收到用户消息时被动执行，还应支持心跳驱动、自主休眠和事件唤醒。

### 心跳

要求：

- Agent 可以按心跳周期检查待办、状态、订阅事件、心跳文件任务
- 心跳周期可以固定，也可以自适应调整
- 平台需要感知 Agent 心跳状态，防止失联与重复调度

建议：

- `HeartbeatPolicy` 属于 Agent 模板或 Workflow 配置
- Runtime 根据当前负载、待办量、优先级动态调整心跳间隔

### 休眠

要求：

- Agent 在任务完成、长时间空闲、等待上游结果时可进入休眠
- 休眠的目标是节约 Token、降低无意义轮询、释放执行资源
- 休眠状态必须可被 Platform 观测

### 唤醒

唤醒来源可以包括：

- 用户消息
- Workflow 下一步触发
- 其他 Agent 的交接消息
- 子智能体完成任务后的回调事件
- 定时器或外部事件

原则：

- 唤醒必须带原因、来源和上下文载荷
- 唤醒不应绕过权限检查
- 休眠中的 Agent 可以只恢复到受限上下文，而不是无条件恢复完整上下文

### 普通智能体与子智能体

普通智能体：

- 可持久存在
- 隶属于某个 Workspace
- 可以被重复唤醒并持续承担角色职责

子智能体：

- 临时性存在
- 由父 Agent 或 Workflow 创建
- 默认继承创建者的权限边界
- 完成任务后自动销毁
- 通过事件把结果、产物或状态交回给父 Agent 或 Workflow

### 典型运行场景

建议支持以下流程：

- 主 Agent 接到任务后，把探索工作下发给探索型子智能体或普通 Agent
- 主 Agent 自主进入休眠或低频心跳状态
- 子智能体完成任务后发布完成事件，并携带结果消息
- 事件系统唤醒主 Agent
- 主 Agent 恢复上下文，接收结果并继续推进任务或交付用户

## 上下文与自我管理归属

为了解决“Agent 的上下文、自我管理到底谁负责”的问题，新的结论是：

- `PuddingCore` 负责无状态基础能力
- `PuddingRuntime` 是独立可部署的执行宿主，负责上下文、记忆、自我管理、会话状态，以及 Skill / MCP / Plugin 的运行支持
- `PuddingAgent` 负责 Agent 定义
- `PuddingPlatform` 负责控制面、认证鉴权、渠道接入、调度治理、插件装配
- `PuddingGateway` 作为 `PuddingPlatform` 内模块，负责边界接入与协议入口
- `PuddingCLI` 只负责交互，不再承担运行态核心职责
- `PuddingWeb` 作为 `PuddingPlatform` 的前端，采用前后端分离架构

当前明确的关键边界：

- **Session 权威在 Runtime**
- **调度与治理权威在 Platform**
- **接入与协议边界在 Gateway Module**
- **Agent 身份与默认能力定义在 PuddingAgent**
- **租户边界为 Organization > Space > Session**
- **记忆默认不跨 Space / Organization 自动共享**
- **公开群和不受信任渠道默认只进入受限服务会话**
- **问答授权与任务执行授权必须分离**
- **同一个 Workspace 可容纳多个 Agent，且 Agent 必须属于某个 Workspace**
- **工作流推进由 Workflow Engine 负责，Agent 只负责节点执行**
- **子智能体默认继承创建者权限，并在任务完成后销毁**
- **权限边界必须映射到受限执行环境，而不只是逻辑鉴权**

换句话说：

- Agent 的“脑子”放在 `PuddingRuntime`
- Agent 的“基础 Skill 和 MCP 接线”也放在 `PuddingRuntime`
- Agent 的“插件是否启用、装配哪些”由 `PuddingPlatform` 决定
- Agent 的“插件运行支持”放在 `PuddingRuntime`
- Agent 的“身份”放在 `PuddingAgent`
- Agent 的“执行宿主”放在 `PuddingRuntime`
- Agent 的“调度中心”放在 `PuddingPlatform`
- Agent 网络的“认证入口、渠道边界、消息路由边界”在 `PuddingPlatform` 内的 `PuddingGateway` 模块

## Swarm 契约模型

契约在这里不应只被理解为“输入输出 schema”，而应被理解为：**Workflow 或 Step 在执行前的约束、批准、失败处理与责任边界定义**。

### 契约适用层级

当前建议支持两层契约：

- `WorkflowContract`：定义整个 Workflow 的总目标、总体约束、审批要求、全局失败策略
- `StepContract`：定义单个 Step 的输入、输出、风险级别、执行门槛、失败策略

原则：

- Workflow 和 Step 都可以定义失败处理策略
- Step 契约优先约束具体执行动作
- Workflow 契约负责定义整体治理边界

### 契约的核心内容

一个契约至少应包含：

- 适用范围：Workflow 或 Step
- 输入约束
- 输出约束
- 可调用能力范围
- 风险级别
- 审批要求
- 失败策略
- 审计要求

建议模型：

```
Contract
  ├─ Scope (Workflow / Step)
  ├─ InputContract
  ├─ OutputContract
  ├─ CapabilityPolicy
  ├─ RiskLevel
  ├─ ApprovalPolicy
  ├─ FailurePolicy
  └─ AuditPolicy
```

### 哪些动作必须经过契约或确认

当前明确需要契约门槛的动作包括：

- 跨 Agent 任务交接
- 触发 Swarm
- 调用高风险 Tool
- 跨工作区共享

说明：

- 普通低风险问答可以不强制契约
- 但一旦从问答升级到执行、派发、共享、跨 Agent 协作，就应进入契约治理范围

### 失败处理策略

默认策略：

- **直接失败并终止当前动作**

同时应允许显式配置以下策略：

- 重试
- 触发失败处理流程
- 唤醒上游 Agent 重新决策
- 交由专门的错误处理 Agent 兜底
- 发送消息请求人工接管
- 丢弃并记录审计事件

建议模型：

```
FailurePolicy
  ├─ OnError (FailFast / Retry / Escalate / FallbackAgent / HumanHandoff / Drop)
  ├─ RetryLimit
  ├─ FallbackTarget
  └─ NotifyTarget
```

### 人工介入门槛

当前建议强制人工介入的场景包括：

- 跨工作区写数据
- 高风险工具执行
- 公开群输入触发执行任务
- 记忆写入升级到 Space 或 Organization

说明：

- 是否强制人工介入，仍应允许通过策略调整
- 但默认应站在保守侧，而不是默认自动执行

### 审计 Agent

系统应支持专门的 `AuditAgent` 模型。

职责：

- 只负责审计、批准、拒绝、质询
- 不直接参与被污染上下文的业务执行
- 可依据平台策略对高风险动作进行独立审查

原则：

- 审计 Agent 应尽量接触结构化、脱敏、受限视图
- 不应直接继承业务 Agent 的全部上下文
- 审计 Agent 可以作为自动批准或自动拒绝链路的一部分

### 契约批准方

当前建议的批准来源包括：

- 工作空间管理员
- 组织管理员
- 触发任务的有权限用户
- 平台策略自动批准
- 审计 Agent 根据策略自动批准或拒绝

结论：

- 契约批准不一定总是人工完成
- 但自动批准必须来自明确的平台策略或审计链路，而不是 Agent 自行越权放行

### 契约与工作流的关系

建议执行顺序如下：

1. 事件触发 Workflow 或 Step
2. Workflow Engine 读取对应契约
3. 平台执行审批、审计、权限检查
4. 满足条件后由 Runtime 创建或唤醒 Agent 实例执行
5. 失败时按 Step 或 Workflow 的失败策略处理

边界结论：

- Agent 不应自己解释自己是否有权执行高风险动作
- 契约校验权威应在 Platform
- Runtime 负责落实已批准的执行约束

## Runtime 节点模型

Runtime 在逻辑上应尽量保持**同构宿主**，不把业务语义写死在某个特殊节点里。

也就是说：

- Runtime 程序本身不应被设计成一组功能完全不同的专有节点
- 差异主要来自部署环境、可用资源、安装的能力包和允许运行的 Agent 类型
- Agent 的差异来自模板、任务、权限与策略，而不是来自“这个 Runtime 是另一种产品”

### 节点可见能力

尽管 Runtime 逻辑上应尽量同构，Platform 仍需要感知节点当前可承载的现实差异，例如：

- 可用模型或 Provider
- 可用工具与插件集合
- 硬件能力，例如 CPU、GPU、内存
- 网络可达性
- 安全等级或沙箱等级
- 所在地域或机房

这些不是为了把 Runtime 做成异构产品，而是为了做**现实部署感知调度**。

### 调度优先级

当前建议：**优先固定绑定，再在可用节点中匹配安全与资源条件**。

意味着：

- Session 一旦落到某个 Runtime，默认尽量不迁移
- Platform 应优先维护会话与节点的稳定绑定关系
- 调度时先判断该节点是否仍满足安全、资源、策略要求
- 不应为了微小收益频繁搬迁正在运行的 Session

这与会话权威在 Runtime 的设计是一致的。

### 绑定关系

当前建议：

- `Session` 与 `Runtime` 默认是一种稳定绑定关系
- `Agent` 的运行实例跟随所属 Session 所在节点
- Workflow 若没有显式切换需求，也应尽量在同一 Runtime 内推进

原则：

- Session 热状态优先本地保持
- 不鼓励频繁跨节点迁移上下文
- 固定绑定优先于“全局最优调度”

### 节点失效策略

当前建议默认策略：

- **等待节点恢复，不自动迁移**

原因：

- Session 主状态权威在 Runtime
- 如果没有成熟的热迁移或状态复制机制，自动迁移很容易造成状态损坏、上下文不一致或权限失真

因此：

- 短期优先做稳定恢复，而不是激进漂移
- 后续如果实现了显式可迁移状态模型，再考虑受控迁移

### Platform 对 Runtime 心跳的观测要求

Platform 应至少掌握：

- 节点健康状态
- 当前承载的 Session 数
- 资源占用情况
- 可用沙箱能力
- 可用插件/工具集合
- 待处理队列长度

说明：

- 这些信息主要用于容量感知、风控判断和运维观测
- 不意味着 Platform 要频繁干预每个 Session 的具体执行

### 调度边界结论

- Runtime 逻辑上尽量同构
- 差异来自环境、资源、策略与可用能力
- Session 默认固定绑定到某个 Runtime
- 节点故障默认优先等待恢复，而不是自动漂移
- 未来若要支持迁移，需要单独设计会话状态复制与迁移契约

## 租户角色模型

当前建议把租户角色分成以下一等角色：

- `OrganizationAdmin`
- `WorkspaceAdmin`
- `Member`
- `Auditor`
- `PublicUser`
- `ThirdPartySystemAccount`

### 角色职责

`OrganizationAdmin`：

- 负责组织级策略、全局限制、上层治理覆盖
- 可以覆盖下级 Workspace 的高层策略

`WorkspaceAdmin`：

- 负责当前 Workspace 内的 Agent、Workflow、插件、审批策略配置
- 负责 Workspace 级高风险操作批准

`Member`：

- 负责日常提问、任务发起、在授权范围内使用 Agent 能力
- 默认不具备高风险批准权

`Auditor`：

- 负责审计、质询、拒绝或建议批准高风险动作
- 不以业务执行为目标，而以治理和风控为目标

`PublicUser`：

- 只拥有面向公开入口的受限交互权限
- 默认不是内部租户成员

`ThirdPartySystemAccount`：

- 作为受限机器人账号存在
- 用于外部系统、Webhook、Issue 系统、自动化集成接入

### 公共用户的默认权限

当前建议公共用户默认可拥有：

- 只能提问
- 查看公开回复
- 上传资料供分析
- 触发低风险任务

限制：

- 公共用户不能直接批准执行
- 公共用户触发的低风险任务，应委托给受限 Agent 执行
- 这类 Agent 应优先运行在更强沙箱中，例如 Docker 或其他受限环境
- 公共用户对应的服务会话应尽量绑定到该用户，并在长时间无活动后销毁

### Workspace 管理员的默认批准权

当前建议以下批准权默认属于 `WorkspaceAdmin`：

- 批准高风险工具执行
- 批准跨工作区共享
- 批准记忆升级到 Space 或 Organization
- 批准 Swarm 执行
- 批准插件启用

说明：

- 这些批准权仍可以被组织级策略收紧
- 默认应允许 Workspace 在本地治理自己的 Agent 网络，但不能突破组织级红线

### 组织与 Workspace 的优先级

当前建议：

- **OrganizationAdmin 的治理权高于 WorkspaceAdmin**

含义：

- 组织级策略可以覆盖 Workspace 策略
- Workspace 可以做局部放行或局部治理，但不能突破组织级禁止项

### 第三方系统账号

当前建议把第三方系统账号建模为：

- **受限机器人账号**

含义：

- 它不是普通用户，也不是单纯渠道概念
- 它有独立身份，但默认能力受限
- 它应只拥有为接入场景所需的最小权限集合

适用场景：

- Issue 系统机器人
- Webhook 触发器
- 外部论坛同步账号
- 企业内部自动化集成账号

### 角色模型边界结论

- 公共用户默认只拥有受限交互权，不直接拥有内部治理权
- 高风险批准权默认放在 WorkspaceAdmin，并受 OrganizationAdmin 覆盖
- Auditor 是独立治理角色，不应混入普通业务执行链路
- 第三方系统账号应被视为受限机器人主体，而不是普通成员

---

## 计费模型

当前建议采用：**组织级总账 + 多维度成本追踪 + 分层预算控制**。

### 计费主体

当前默认：

- **费用最终归属于 Organization**

含义：

- 组织承担平台总成本
- Workspace、渠道、会话、Agent 等更多用于成本归集、分析和预算控制，而不是最终法律计费主体

### 成本追踪维度

当前建议至少按以下维度归集成本：

- Workspace
- 渠道
- 会话
- Agent 模板
- Provider / Model
- 插件 / Tool 调用

说明：

- 这样既能看组织总账，也能回看具体是哪个 Workspace、哪个入口、哪种 Agent、哪类工具消耗了成本
- 后续也便于做成本优化、限流和配额策略

### 公共入口成本归属

对于公开群、客服、论坛等公共入口，当前建议：

- **默认由所属 Organization 统一承担成本**

原因：

- 公共入口本质上是组织对外提供的服务能力
- 不适合一开始就把成本强行回溯给不稳定外部用户
- 组织统一承担更符合治理与预算模型

### 预算控制层级

当前建议优先支持以下三层预算：

- `OrganizationBudget`
- `WorkspaceBudget`
- `WorkflowBudget` 或 `TaskBudget`

含义：

- Organization 控总盘子
- Workspace 控团队或项目消耗
- Workflow / Task 控单次执行上限，避免某个自动化流程失控

### 超预算处理策略

当前默认建议：

- **优先降级到低成本模型或只读模式**

说明：

- 不必一超预算就完全拒绝所有后续动作
- 可以先退到低成本 Provider、低能力模式、只读模式、禁止高风险工具模式
- 如果仍需继续高成本执行，再进入人工批准或更高层预算判断

建议策略链：

1. 发现预算逼近或超出阈值
2. Platform 尝试切换到低成本模型或受限模式
3. 若任务仍无法完成，则请求批准或终止后续高成本执行

### 计费模型边界结论

- 组织是默认计费主体
- 平台需要做细粒度归集，而不仅是 Provider 账单透传
- 公共入口成本默认由组织承担
- 预算控制应至少覆盖组织、Workspace、Workflow/Task 三层
- 超预算默认优先降级，而不是直接把系统打死

---

## Plugin 信任演进

当前阶段虽然仍以“插件默认受信任”为基础方向，但治理模型已经需要开始具备**来源分级、审批链、能力边界与失信隔离**。

### 插件来源分级

当前建议至少区分以下来源：

- `OfficialPlugin`
- `OrganizationInternalPlugin`
- `WorkspacePrivatePlugin`
- `ThirdPartyPlugin`

说明：

- 来源分级不只是展示标签，而应直接影响默认启用策略、审批要求和沙箱等级

### 默认启用策略

当前建议：

- **官方插件和组织内部插件默认可启用**
- **第三方插件必须经过审批后才能进入 Workspace**

说明：

- Workspace 私有插件应至少被视为需要本地治理的插件来源
- 后续可以再细化为“默认禁用但管理员可快速放行”之类的中间态

### 插件权限边界

当前最需要受控的插件能力包括：

- 文件访问
- 网络访问
- 进程执行
- 读取记忆
- 写入记忆
- 访问平台管理接口

原则：

- 插件不能因为被装配就自动获得宿主全部能力
- 插件权限应单独声明、单独批准、单独审计
- 插件权限应与 AgentTemplate、Workspace 策略、运行沙箱共同取交集

### 插件失信处理

当前建议默认策略：

- **立即停用并隔离**

适用场景：

- 插件被判定为高风险
- 插件发生异常越权行为
- 插件被审计链路标记为不可信

说明：

- 停用后应保留审计记录
- 需要时可进一步触发人工调查、Workspace 告警或组织级封禁

### 第三方插件审批方

当前建议以下主体可以参与第三方插件审批：

- `OrganizationAdmin`
- `WorkspaceAdmin`
- `Auditor`

说明：

- 普通成员不应直接放行第三方插件
- 自动批准应是后续策略能力，而不应成为当前默认基线

### Plugin 治理边界结论

- 插件必须按来源分级，而不是统称“插件”
- 第三方插件默认不应自动进入 Workspace
- 插件权限需要被单独约束与审计
- 插件一旦失信，默认立即停用并隔离

## 核心数据模型

当前建议先把最核心的数据实体稳定下来，再推进代码实现。否则 Platform、Runtime、Workflow、审批链很容易在实现过程中反复返工。

### AgentTemplate 与 AgentInstance

当前建议：

- **一个 AgentTemplate 可以派生多个普通 Agent 和多个临时子智能体**

含义：

- `AgentTemplate` 负责定义角色、人设、默认能力、权限画像、心跳策略、沙箱要求
- `AgentInstance` 负责承载具体运行态
- 普通 Agent 与子智能体都来自模板，但生命周期不同

建议模型：

```
AgentTemplate
  ├─ TemplateId
  ├─ WorkspaceId
  ├─ RoleProfile
  ├─ CapabilityPolicy
  ├─ RuntimeProfile
  ├─ HeartbeatPolicy
  └─ DefaultMemoryPolicy

AgentInstance
  ├─ AgentId
  ├─ TemplateId
  ├─ SessionId
  ├─ RuntimeId
  ├─ AgentKind (Persistent / Child)
  ├─ State
  └─ ParentAgentId?
```

### Workflow 模型

当前建议：

- Workflow 先作为一等实体存在
- **短期先不做版本化，只保留当前生效定义**

说明：

- 这能降低第一阶段实现复杂度
- 后续需要灰度发布、历史回放、审计追踪时，再引入 WorkflowVersion

建议模型：

```
Workflow
  ├─ WorkflowId
  ├─ WorkspaceId
  ├─ Name
  ├─ TriggerPolicy
  ├─ FailurePolicy
  └─ Steps[]

WorkflowStep
  ├─ StepId
  ├─ WorkflowId
  ├─ BoundTemplateId
  ├─ InputContract
  ├─ OutputContract
  ├─ RetryPolicy
  └─ NextStepRule
```

### Session 类型

当前建议至少把以下类型建模为正式 Session：

- `ServiceSession`
- `ExecutionSession`
- `AuditSession`
- `ChildSession`

含义：

- `ServiceSession`：面向外部问答、公开入口、客服、论坛等受限交互
- `ExecutionSession`：面向真正执行任务、调用工具、触发 Workflow / Swarm
- `AuditSession`：面向审计 Agent 或审批链的独立治理会话
- `ChildSession`：面向子智能体临时执行任务的短生命周期会话

建议模型：

```
Session
  ├─ SessionId
  ├─ SessionType
  ├─ WorkspaceId
  ├─ ChannelId
  ├─ RuntimeId
  ├─ OwnerUserId?
  ├─ ParentSessionId?
  ├─ State
  └─ PermissionSnapshot
```

### 记忆对象

当前建议最少包含以下对象：

- `SessionMemory`
- `WorkspaceMemory`
- `OrganizationMemory`
- `MemoryCandidate`
- `MemoryPolicy`

含义：

- `SessionMemory`：只属于当前 Session
- `WorkspaceMemory`：属于当前 Workspace，可按授权供多个 Agent 使用
- `OrganizationMemory`：组织级共享记忆，需要显式授权
- `MemoryCandidate`：待审核、待筛选、待提升的候选记忆
- `MemoryPolicy`：控制哪些来源、哪些 Agent、哪些事件可以写入或召回

建议模型：

```
MemoryRecord
  ├─ MemoryId
  ├─ MemoryScope (Session / Workspace / Organization / Candidate)
  ├─ OwnerId
  ├─ SourceType
  ├─ TrustLevel
  ├─ ContentRef
  └─ Status

MemoryPolicy
  ├─ PolicyId
  ├─ WorkspaceId
  ├─ WriteRule
  ├─ RecallRule
  ├─ PromotionRule
  └─ AuditRule
```

### 审批记录

当前建议 `Approval` 至少具备以下关键字段：

- 发起人或请求来源
- 审批人
- 目标动作
- 确认码或令牌
- 过期时间
- 审计结论

建议模型：

```
ApprovalRecord
  ├─ ApprovalId
  ├─ RequestSource
  ├─ RequestedBy
  ├─ ApprovedBy?
  ├─ TargetAction
  ├─ ConfirmationCode?
  ├─ ExpiresAt
  ├─ AuditResult?
  └─ Status
```

### 数据模型边界结论

- AgentTemplate 与 AgentInstance 必须分离建模
- Workflow 短期先不做版本化，优先把执行链路跑通
- Session 至少要区分服务、执行、审计、子智能体四类
- Memory 需要同时建模正式记忆与候选记忆
- ApprovalRecord 是高风险执行和人工确认链路的核心对象

## 第一版实现路线

当前建议第一版目标不是一次性做完整平台，而是先跑通一条最小闭环主链路。

### 第一阶段目标

当前建议第一版优先跑通：

- **渠道 -> ServiceSession -> 指定 Agent 问答**

原因：

- 这是所有公开入口、内部群、客服、论坛场景的共同起点
- 先把接入、身份映射、会话路由、Agent 服务暴露面跑通，后续 Workflow 和审批链才能稳定接上

### 第一版暂缓能力

当前建议第一版暂时不做：

- 跨工作区共享
- Workflow 版本化
- 多 Runtime 分布式调度
- 第三方插件审批自动化
- 复杂预算策略

说明：

- 这些能力都重要，但不适合压进第一版最小闭环
- 第一版应优先验证接入、权限、会话、Agent 调度和低风险任务链路

### 最先实现的宿主

当前建议：

- **优先落地 PuddingPlatform**

原因：

- 第一版主链路是“渠道进入平台，再映射为会话和 Agent 服务”
- 如果没有 Platform，后续 Runtime、CLI、Workflow 很难形成统一治理入口

### 第一版持久化策略

当前建议：

- **先采用本地文件或 SQLite**

适合先落地的数据包括：

- Workspace 配置
- AgentTemplate
- Session 索引
- Workflow 定义
- ApprovalRecord
- 审计记录

说明：

- 第一版重点是验证模型与闭环，不是先上重型基础设施
- 等实体和行为稳定后，再考虑完整数据库与分布式存储

### 第一版成功标准

当前建议至少满足以下条件，才算真正跑通：

- 渠道消息能路由到指定 Agent
- Agent 能创建并完成一个低风险任务
- 高风险动作能进入审批链
- Workflow 能跑完 2 到 3 个步骤

### 建议里程碑

`Milestone 1`：Platform 接入骨架

- 建立 Channel -> User -> Workspace -> AgentTemplate 路由链
- 建立 ServiceSession
- 打通指定 Agent 的低风险问答

`Milestone 2`：低风险任务闭环

- Agent 从问答升级为低风险任务创建
- Runtime 能执行受限任务
- 审计与事件记录可见

`Milestone 3`：审批链接入

- 高风险动作进入 ApprovalRecord
- 支持确认码与显式批准
- 支持审计 Agent 或人工批准

`Milestone 4`：Workflow 最小闭环

- 支持 2 到 3 步 Workflow
- 支持 Agent 交接
- 支持失败事件与简单失败策略

### 第一版实现边界结论

- 第一版先做 Platform 控制入口，而不是先做完整 Runtime 集群
- 第一版先跑 ServiceSession 和指定 Agent 服务暴露面
- 第一版先验证低风险任务和审批链，再放大到复杂 Workflow
- 第一版持久化优先选择本地文件或 SQLite，降低验证成本

---

## 第一版接口面

为了让第一版尽快形成最小闭环，Platform 需要先暴露一组足够小但完整的入口与查询接口。

### 首批入口协议

当前建议第一版优先支持：

- `CLI through Platform API`
- `HTTP API`
- `WebSocket`

说明：

- CLI 不再直连运行时内部对象，而应通过 Platform API 进入系统
- HTTP API 适合管理、查询、批准、集成接入
- WebSocket 适合实时消息、流式输出、会话状态推送
- MQTT 和 Webhook 可以放到后续阶段

### Session 创建方式

当前建议：

- **收到渠道消息时自动创建或复用 Session**

含义：

- 用户不需要先手工创建 Session 才能开始交互
- Platform 根据渠道、用户、Workspace、Agent 路由规则自动决定是否复用已有会话或新建会话

### Agent 寻址方式

当前建议：

- **渠道先绑定 Workspace，再由渠道绑定规则和路由规则共同选择 Agent**

说明：

- 第一版不要求消息直接携带 AgentId
- 也不建议一开始退化成“整个系统只有一个默认 Agent”
- 路由应由 Platform 依据渠道、Workspace、渠道绑定的默认 Agent/允许 Agent 集合，以及服务暴露策略完成

### 第一版批准入口

当前建议优先支持以下批准入口：

- 渠道回复确认码
- Platform HTTP API
- CLI 命令批准

说明：

- 这三种入口足够覆盖公开入口、平台管理、开发调试三类场景
- Web 控制台和审计 Agent 自动批准可以后续补上

### 第一版查询接口

当前建议第一版至少提供以下查询能力：

- 查询 Session 状态
- 查询 Agent 状态
- 查询 Approval 待办
- 查询 Workflow 执行状态
- 查询审计事件

说明：

- 没有这些查询面，第一版即便“能跑”，也很难真正运维和调试

### 第一版接口边界结论

- 第一版入口以 HTTP、WebSocket 和 Platform API 为主
- Session 应自动创建或复用，而不是要求人工预创建
- Agent 寻址权应在 Platform 路由层，而不是暴露给外部调用方自由指定
- 批准入口先覆盖渠道、HTTP、CLI 三条链路
- 状态查询接口必须和执行链路同时存在

---

## Workspace 配置模型

为了避免第一版后续在配置层反复返工，Workspace 需要先明确最小配置对象集合。

### Workspace 最小配置对象

当前建议一个 Workspace 至少应配置以下对象：

- `AgentTemplate`
- `ChannelBinding`
- `Workflow`
- `MemoryPolicy`
- `PermissionPolicy`
- `BudgetPolicy`
- `PluginBinding`

说明补充：

- `ChannelBinding` 绑定的是“渠道实例/渠道插件产物”到某个 Workspace 的治理关系
- 后续新增渠道时，优先新增 `ChannelProvider` 插件，而不是扩充 Workspace 模型本身

说明：

- 这组对象基本覆盖了服务暴露、路由、执行、权限、记忆、预算和插件装配七条主线

### ChannelBinding 粒度

当前建议：

- **ChannelBinding 先绑定到 Workspace，并声明该渠道可见/默认的 Agent，再由路由规则最终选择 Agent**

含义：

- 渠道不必被硬编码为只能绑一个 AgentTemplate
- 渠道先落到 Workspace 治理边界内
- 但渠道绑定本身应至少能声明：默认 Agent、允许 Agent 集合、渠道级能力限制
- 具体路由由 Platform 在这些约束内动态决定

建议最小模型：

```
ChannelBinding
  ├─ ChannelBindingId
  ├─ WorkspaceId
  ├─ ChannelType
  ├─ ChannelIdentity
  ├─ DefaultAgentTemplateId?
  ├─ AllowedAgentTemplateIds[]
  ├─ TrustLevel
  └─ ChannelPolicy
```

### 内置邮箱渠道

第一版建议把 Email 作为内置渠道之一。

建议能力：

- 绑定一个或多个邮箱地址到某个 Workspace
- 为该邮箱渠道声明默认 Agent 或允许 Agent 集合
- 按发件人、收件地址、主题规则路由到不同 Agent
- 将邮件线程映射到 `ServiceSession`
- 记录附件、转发、引用内容的审计与信任等级

建议最小模型：

```
EmailChannelConfig
  ├─ MailboxAddress
  ├─ WorkspaceId
  ├─ DefaultAgentTemplateId?
  ├─ AllowedAgentTemplateIds[]
  ├─ SenderTrustPolicy
  ├─ SubjectRoutingRules[]
  └─ AttachmentPolicy
```

说明：

- Email 是平台内置渠道之一，但仍建议通过统一 `ChannelProvider` 抽象接入
- 这样后续新增 IM、论坛、工单、Webhook 渠道时，可以复用同一套渠道插件宿主机制

### 第一版路由规则

当前建议第一版路由主要依据以下输入：

- 渠道来源
- 用户身份或角色
- 消息类型，例如问答、任务、审批
- 关键词或意图分类

说明：

- 这些输入足以支持一个 Workspace 内多个 Agent 的服务暴露与基本分流
- 预算状态和 Runtime 负载可在后续迭代加入更高级调度规则

### 配置来源

当前建议第一版采用：

- **Workspace YAML + Platform 持久化**

含义：

- Workspace 可以通过 YAML 定义可读、可版本管理的结构化配置
- Platform 负责把运行所需索引、状态和派生结果持久化下来

### 配置生效方式

当前建议第一版：

- **手动 reload 后生效**

说明：

- 先避免自动热更新带来的不确定性
- 等配置对象和执行语义稳定后，再考虑更复杂的热更新策略

### Workspace 配置边界结论

- Workspace 是第一版最重要的治理配置边界
- 一个 Workspace 应支持多个 ChannelBinding
- ChannelBinding 应先绑定 Workspace，并可声明默认 Agent 与允许 Agent 集合
- 渠道接入机制本身必须插件化，避免平台核心代码被渠道种类污染
- 第一版路由先靠渠道、身份、消息类型和意图分类
- 配置先采用 YAML 加 Platform 持久化混合模式
- 配置变更先手动 reload，降低运行期复杂度

## 可观测性与测试基线

第一版如果只有执行链路而没有审计、指标、可读错误和调试查询，后续很难真正运维。因此可观测性不能等到第二阶段再补。

### 首批审计事件

当前建议第一版必须记录以下审计事件：

- 渠道消息进入
- Session 创建或复用
- Agent 路由决策
- 工具执行
- 审批请求与结果
- Workflow Step 开始与完成
- 记忆写入或提升

说明：

- 这组事件覆盖了接入、路由、执行、审批、工作流和记忆五条关键链路

### 首批运行指标

当前建议第一版至少统计以下指标：

- 请求量或消息量
- Session 数量
- Agent 活跃数
- 审批等待数
- Workflow 成功数与失败数
- 模型或工具成本
- Runtime 负载

说明：

- 这组指标足以支持第一版的运行观测、成本观测和容量判断

### 首批测试重点

当前建议第一版测试优先覆盖：

- 路由规则测试
- 权限与审批链测试
- Workflow 2 到 3 步执行测试
- 公开群受限模式测试
- 记忆边界测试

说明：

- 这些测试都直接对应当前最容易出安全问题或行为漂移的地方

### 错误呈现方式

当前建议第一版错误优先通过以下方式暴露：

- HTTP 或 CLI 可读错误
- 审计事件流
- Platform 日志

说明：

- 第一版先把错误讲清楚，比先做复杂告警系统更重要

### 首批调试查询能力

当前建议第一版至少支持以下调试查询：

- 查看某条消息最终路由到哪个 Agent
- 查看某个高风险动作为何被拒绝
- 查看某个 Workflow 卡在哪一步
- 查看某个 Session 的权限快照
- 查看某次记忆写入为何被拦截

### 可观测性边界结论

- 第一版必须把审计事件、运行指标、可读错误和调试查询同时做起来
- 调试查询要直接对应路由、审批、Workflow、权限和记忆五个高风险点
- 没有这些能力，第一版即便功能可用，也不算真正可运维

## 第一版项目落地映射

当前已经可以把第一版的最小能力映射回具体项目，避免后续实现时再把职责重新堆回 CLI。

### PuddingPlatform 首批模块

当前建议第一版优先在 `PuddingPlatform` 落地：

- `ChannelManager`
- `ChannelPluginHost`
- `SessionRouter`
- `AuthorizationService`
- `ApprovalService`
- `WorkflowEngine`
- `AuditStore`

说明：

- 这组模块已经足够支撑渠道接入、渠道插件宿主、会话路由、权限检查、审批链、最小 Workflow 和审计闭环

### PuddingRuntime 首批模块

当前建议第一版优先在 `PuddingRuntime` 落地：

- `SessionRuntime`
- `AgentRuntime`
- `SkillRuntime`
- `MemoryRuntime`
- `SandboxExecutor`

说明：

- 第一版先不强推 `ChildAgentRuntime` 独立成型
- 可以先让子智能体能力在已有 SessionRuntime / AgentRuntime 语义内落地，再逐步抽出

### PuddingCLI 第一版职责

当前建议第一版 `PuddingCLI` 主要负责：

- 通过 Platform API 发起会话
- 查看 Session / Approval / Workflow 状态
- 执行批准操作
- 调试路由与审计查询

说明：

- CLI 不再承担本地直接运行 Agent 的主要职责
- CLI 是 Platform 的一个管理和调试入口，而不是新的控制面

### PuddingAgent 第一版起步方式

当前建议：

- **先只放少量内置模板定义**

说明：

- 第一版没必要先做复杂模板仓库
- 先有几个能稳定跑通路由、问答、低风险任务和审批链的内置模板即可

### 第一条垂直切片

当前建议最先编码的完整切片是：

- **渠道消息 -> 路由 -> ServiceSession -> 指定 Agent 回复**

原因：

- 这是所有入口和后续能力的共同底盘
- 这条切片一旦跑通，审批链、Workflow、公开群受限模式都能逐步叠加上去

### 项目落地边界结论

- 第一版控制入口和治理逻辑先放在 PuddingPlatform
- 第一版执行能力先收敛在 PuddingRuntime
- PuddingCLI 只做入口、查询、批准和调试
- PuddingAgent 先从少量内置模板开始
- 第一条必须先跑通的垂直切片是“渠道消息到指定 Agent 回复”

## 第一条垂直切片验收标准

当前建议把第一条垂直切片定义为：

- **CLI 通过 Platform API 发送消息 -> Platform 路由 -> 自动创建或复用 ServiceSession -> Runtime 中指定 Agent 返回真实 LLM 回复**

### 首切片输入来源

当前建议第一版先从：

- `CLI through Platform API`

开始。

原因：

- 这条链路最利于开发和调试
- 同时又已经验证了 CLI、Platform、Runtime 三者之间的真实边界

### 首切片最少必须完成的路由动作

当前建议至少完成以下步骤：

- 命中 Workspace
- 命中 AgentTemplate
- 自动创建或复用 `ServiceSession`
- 把消息投递到 Runtime
- 记录审计事件

说明：

- 少其中任何一项，这条切片都不能算真正跑通

### 首切片回复能力

当前建议：

- **必须走真实 LLM 回复**

说明：

- 第一条主链路不应只返回固定模板或模拟数据
- 否则后续对权限、审计、成本、错误处理的验证都会失真

### 首切片失败处理

如果消息没有命中任何 Agent，当前建议：

- **返回明确错误并记录审计事件**

说明：

- 第一版不建议静默失败
- 也不建议自动回退到某个默认 Agent，避免路由语义变得含糊

### 首切片完成标准

当前建议只有同时满足以下条件，才算该切片完成：

- 从 CLI 发消息，能收到指定 Agent 的真实回复
- Session 状态可查询
- 路由决策可查询
- 审计事件可查询
- Runtime 中 Agent 状态可查询

### 首切片边界结论

- 第一条切片必须是真实链路，不接受纯模拟闭环
- 路由、Session、审计和 Runtime 状态必须都可观测
- Agent 未命中时必须明确报错，不做隐式兜底
