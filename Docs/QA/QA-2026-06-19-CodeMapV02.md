## 3. P1 严重问题

### 3.1 ❌ ProjectMapTool.AppendTreeText — 错误的变量比较

**文件**: [ProjectMapTool.cs](Source/PuddingRuntime/Tools/BuiltIns/CodeIntelligence/ProjectMapTool.cs#L214)  
**严重度**: P1（逻辑错误 — 输出误导性截断标记）

```csharp
// 第 214 行附近
if (file.Types.Count > 0)
{
    var typeDesc = string.Join(", ", file.Types.Take(MaxTopLevelTypes));
    if (node.TotalFileCount > MaxTopLevelTypes) typeDesc += "...";  // ❌ 错误比较
    sb.AppendLine($"{fileIndent}{filePrefix}{file.Name}    {typeDesc}");
}
```

**问题**: `node.TotalFileCount` 是**目录中的 .cs 文件数**（上限50），`MaxTopLevelTypes` 是**每个文件显示的类型数上限**（值为3）。这两个量纲完全不同。当前逻辑会导致：只要目录中超过3个 .cs 文件，该目录下**每个文件**的类型列表都会显示 `...`，即使该文件实际只有1个类型。

**正确逻辑**：`...` 应表示"此文件的类型被截断"，需在 `GetFileTopTypesAsync` 中记录原始类型数，或在 `FileEntry` 中新增 `TotalTypeCount` 字段。

**修复建议**:
```csharp
// FileEntry 中增加
public int TotalTypeCount { get; set; }

// GetFileTopTypesAsync 中记录
fileEntry.TotalTypeCount = visitor.RootNodes.Count;
fileEntry.Types = visitor.RootNodes.Take(MaxTopLevelTypes).Select(...).ToList();

// AppendTreeText 中
if (file.TotalTypeCount > MaxTopLevelTypes) typeDesc += "...";
```

### 3.2 ❌ 路径遍历风险 — CodeOutlineTool 和 ProjectMapTool 均无沙箱约束

**严重度**: P1（安全性 — 潜在路径遍历）

**CodeOutlineTool.ResolveFullPath**:
```csharp
private static string ResolveFullPath(string filePath)
{
    if (Path.IsPathRooted(filePath))
        return filePath;
    var cwd = Directory.GetCurrentDirectory();
    return Path.GetFullPath(Path.Combine(cwd, filePath));
}
```

- 接受绝对路径直接返回，无任何验证
- 相对路径 `..\..\..\Windows\System32\drivers\etc\hosts` 可逃逸工作区
- `Path.GetFullPath` 会解析但不阻止遍历

**ProjectMapTool.ResolveRootPath** 同理。

**影响**: Agent 可通过这些工具读取工作区外的任意文件/目录（进程权限范围内）。虽标记为 ReadOnly，但仍泄露信息。

**修复建议**:
1. 定义工作区根目录（从 `ToolExecutionContext` 获取或通过配置注入）
2. 解析后的绝对路径必须 `StartsWith(workspaceRoot)`
3. 不在工作区内的路径返回 `Fail("(文件不在工作区范围内: ...)")`

### 3.3 ⚠️ CodeSummaryTool — 依赖注入服务未注册时静默失败不完整

**文件**: [CodeSummaryTool.cs](Source/PuddingRuntime/Tools/BuiltIns/CodeIntelligence/CodeSummaryTool.cs#L45-L46)  
**严重度**: P1（可观测性）

```csharp
if (_queryService is null)
    return Fail("Code query tools are not available: ICodeQueryService is not registered.");
```

✅ 对 `_queryService` 有 null 检查。但 `_resolver` 为 null 时没有检查，会传递 null 给 `CodeQueryToolHelper.ResolveAndEnsureProjectIdAsync`，该方法正确处理了 null（返回 null），但 `projectId` 为 null 会怎样？

检查 `CodeSymbolSearchRequest` 构造函数：`ProjectId` 是可选参数（`string?`），所以 null 是合法的。但后续 `GetCallersAsync(sym.ProjectId, ...)` 如果 `sym.ProjectId` 为 null 会怎样？`ICodeQueryService.GetCallersAsync` 第二个参数是 `string projectId`（非可空），但调用时 `sym.ProjectId` 是 `CodeSymbolRecord.ProjectId` 类型需要确认。

**建议**: 确认 `CodeSymbolRecord.ProjectId` 的类型；若可能为 null，需防御。

## 4. P2 改进建议

### 4.1 CodeOutlineTool — 错误消息泄露内部路径

```csharp
return Fail($"(文件不存在: {filePath})");
return Fail($"(无法读取文件: {ex.Message})");
```

错误消息可能包含完整绝对路径，暴露服务器目录结构。

**建议**: 对生产环境使用脱敏后的路径（仅保留文件名或工作区相对路径）。

### 4.2 CodeSummaryTool — 符号匹配逻辑简陋

```csharp
var match = symbols[0];  // 总是取第一个结果
```

`SearchSymbolsAsync` 返回的结果排序可能不可靠（取决于索引器实现）。对于模糊匹配，第一个结果不一定是最佳匹配。

**建议**: 
- 优先精确名称匹配
- 若没有精确匹配，在文本输出中标注"最佳匹配"提示

### 4.3 ProjectMapTool — 潜在性能问题

- 每个 .cs 文件执行 `File.ReadAllTextAsync` + `CSharpSyntaxTree.ParseText` + visitor 遍历
- `MaxFilesPerDir=50`、`depth≤3`，但大型项目可能仍有数百文件
- Roslyn 语法解析虽快，但数百次累积可能超时（默认工具超时通常 30-60s）

**建议**: 
- 考虑对单个文件解析增加 `CancellationToken` 超时子令牌
- 在文本输出中标注总扫描文件数和耗时
- 考虑增加 `MaxTotalFiles` 全局上限（如 200）

### 4.4 CodeSummaryTool — ExtractDocComment 使用 DescendantNodes() 可能昂贵

```csharp
var nodes = root.DescendantNodes()
    .Where(n => { ... })
    .ToList();
```

对大文件（数千行）遍历全部节点并做 `GetLineSpan()` 可能耗时。

**建议**: 利用行号范围提前结束遍历，或使用 `ChildNodes()` 逐层缩小范围。

### 4.5 三个新工具 — 缺少关键链路日志

根据项目要求"在每个功能的关键链路上增加日志"，这三个工具均无 `SimpleLogger` 日志记录。对于：
- 文件不存在 / 读取失败（CodeOutlineTool, ProjectMapTool）
- 符号搜索为空 / 调用链查询失败（CodeSummaryTool）
- 目录扫描跳过（权限不足等）（ProjectMapTool）

**建议**: 在 `catch` 块中加入 `SimpleLogger` 日志。
