## P0 — 阻断

### P0-1: 前后端 JSON 字段命名不匹配（功能完全失效）

**文件**: `Source/PuddingPlatform/Controllers/Api/WorkspaceNotificationsController.cs` L49
**影响**: 前端所有通知字段读取为 `undefined`，未读计数、Toast、会话刷新全部失效。

**原因**:
后端控制器手动调用 `System.Text.Json.JsonSerializer.Serialize(notification)` **未传选项**，默认使用 PascalCase：
```json
{"Type":"...", "SessionId":"...", "WorkspaceId":"..."}
```
但前端 `WorkspaceNotification` 类型期望 camelCase：
```ts
ev.type, ev.sessionId, ev.workspaceId  // → 全部为 undefined
```

**修复**: 第 49-51 行改用已存在的 `ServerSentEventFrame.Json()` 方法（该工厂方法使用 `JsonSerializerDefaults.Web` 即 camelCase）：
```csharp
// 替换：
var data = System.Text.Json.JsonSerializer.Serialize(notification);
await WriteSseAsync(Response,
    new ServerSentEventFrame(notification.Type, data), ct);

// 改为：
var frame = ServerSentEventFrame.Json(notification.Type, notification);
await WriteSseAsync(Response, frame, ct);
```

## P1 — 严重

### P1-1: SSE 断线重连逻辑死代码

**文件**: `useChatState.ts` L930-937
**影响**: SSE 连接因网络波动断开后不会自动重连，用户无感知丢失通知。

`scheduleReconnect` 函数在第 930 行定义，但**从未被调用**。
`subscribeWorkspaceNotifications` 内部的 `fetch` 在断连时 `.catch` 静默吞掉异常，没有触发重连回调。

**修复建议**: 
- 方案A：让 `subscribeWorkspaceNotifications` 接收 `onDisconnect` 回调，断连时通知调用方
- 方案B：在 `.catch` 中调用 `scheduleReconnect`（需将该函数传递给 api 层）
- 方案C（推荐）：`subscribeWorkspaceNotifications` 返回 cleanup 并在 `finally`/`.catch` 处通过返回的 Promise 链通知
