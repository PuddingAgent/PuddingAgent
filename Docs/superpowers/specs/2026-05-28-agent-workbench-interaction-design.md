## 1. 产品定位

Pudding 不应把语音能力做成聊天输入框里的一个麦克风按钮。只要用户开始说话、Agent 开始发声，产品形态就从“文本聊天”变成了一个实时协作工作台。用户会期待 Agent 能听见、等待、打断、回应、朗读，并且能清楚表现自己当前在做什么。

推荐方向是 **Agent 工作台**：

- Workspace 仍然是组织项目、Agent、工具、文件和自动化的顶层空间。
- Agent 工作台是面向一次具体任务或会话的深度交互界面。
- Chat 仍然存在，但它变成工作台里的事件时间线，不再是整个产品隐喻。
- 文本、语音、摄像头、文件、命令、Agent 提及和自动化是并列的意图输入方式。
- Agent 的状态应以“存在感”呈现，而不是只能从气泡消息里猜测。

这个方向是当前聊天页面与长期多 Agent OS 形态之间的过渡层。它不会过早把界面做成完整桌面系统，但会为语音、视觉、虚拟形象和事件驱动编排留出明确位置。

## 2. 设计目标

1. 语音是一等交互模式。
   用户可以进入语音会话、连续说话、停止、取消、打断 Agent、回放 Agent 语音，并始终看见清楚的听取与播放状态。

2. Agent 需要可见的存在感。
   界面必须表现 Agent 是空闲、聆听、思考、说话、阻塞、使用工具、委派给其他 Agent，还是等待用户确认。

3. 事件可解释，但默认不噪音化。
   工具调用、语音采集、ASR、TTS、摄像头帧、Agent 交接和消息投递都应该能成为结构化事件，但默认阅读体验仍然保持安静。

4. 文本输入仍然高效。
   语音不是替代键盘，而是与键盘共享上下文的另一个主模式。模式切换要快，语义一致。

5. 首期先完成语音输入与语音输出。
   摄像头和虚拟形象需要在界面结构里预留位置，但不把首期施工复杂度扩大到完整实时音视频。

## 3. 产品语言

Pudding 的设计语言延续“静谧的本地智能”：

- 冷静、专业、面向工作。
- 足够紧凑，适合重复使用，而不是演示型 AI 页面。
- 更接近工作台、控制室或书斋，而不是聊天玩具。
- 语音反馈要明确，但不戏剧化。
- 虚拟形象首先是状态载体和陪伴感，其次才是动画资产。

推荐术语：

- `Agent 工作台`：页面级产品概念。
- `意图控制台`：多模态输入区域。
- `会话时间线`：消息与事件投影区域。
- `Agent 感知栏`：语音、摄像头、虚拟形象、运行状态区域。

## 4. 信息架构

工作台由四个稳定区域组成：

```text
┌─────────────────────────────────────────────────────────────────────┐
│ 顶部栏：Workspace / Agent / Session / Health                         │
├───────────────┬───────────────────────────────────────┬─────────────┤
│ 左侧导航       │ 中央会话时间线                         │ 右侧感知栏   │
│               │                                       │             │
│ Sessions      │ 消息                                  │ Agent 形象   │
│ Artifacts     │ 语音转写                              │ 麦克风状态   │
│ Agents        │ 工具与工作流事件                       │ 语音输出     │
│ Tools         │ 用户确认                              │ 摄像头状态   │
│ Automations   │                                       │ 活跃工具     │
├───────────────┴───────────────────────────────────────┴─────────────┤
│ 意图控制台：文本 / 语音 / 摄像头 / 文件 / 命令 / 发送策略              │
└─────────────────────────────────────────────────────────────────────┘
```

### 左侧导航

作用：切换上下文，而不是承载语音交互。

内容：

- 当前 workspace 下的会话。
- 相关文件与产物。
- 参与 Agent。
- 常用工具与自动化。

左侧导航应保持紧凑，避免抢占当前任务的注意力。

### 中央会话时间线

作用：记录“发生了什么”。

内容：

- 用户文本消息。
- 用户语音转写，带语音来源标记。
- Assistant 回复。
- Assistant 语音回复与回放控制。
- 工具调用、Agent 交接、错误、人工确认。
- 可折叠的技术细节，用于调试和审计。

时间线默认应该像自然会话一样可读，只有用户需要时才展开执行细节。

### 右侧 Agent 感知栏

作用：让 Agent 的实时状态可见。

内容：

- Agent 虚拟形象或紧凑状态块。
- 正在聆听、思考、说话、阻塞等状态。
- 当前语音输入状态。
- 当前语音播放状态。
- 摄像头权限与画面状态。
- 活跃工具或子 Agent 委派状态。
- 本地隐私提示。

语音输出和实时状态属于这里，不应隐藏在输入框内部。

### 底部意图控制台

作用：捕获用户下一步意图。

内容：

- 模式切换：键盘、语音、摄像头、文件、命令。
- 语音会话控制：开始、暂停、停止、取消、发送。
- 文本兜底与转写编辑。
- 发送策略：手动发送、静音后自动发送、按住说话。
- 附件与上下文控制。

底部控制台是语音输入的主交互界面，但不应只是图标按钮排布。

## 5. 核心交互模型

### 键盘模式

键盘模式用于精确输入和编辑。

流程：

1. 用户输入文字。
2. 用户可附加文件、提及 Agent 或调用工具。
3. 消息提交到事件/消息系统。
4. 时间线记录用户消息与 Agent 回复。

键盘模式只需要保留清晰的语音模式入口，不需要塞入完整语音界面。

### 语音模式

语音模式是一个会话面板。

流程：

1. 用户切换到语音模式。
2. 意图控制台展开为语音会话面板。
3. 若需要，浏览器请求麦克风权限。
4. UI 进入“准备聆听”状态。
5. 用户开始录音。
6. UI 显示音量/波形、已录制时长和部分转写。
7. 用户停止，或静音策略停止录音。
8. 若启用手动确认，用户可编辑/确认转写。
9. 消息携带 `inputMode=voice` 元数据发送。
10. Agent 回复进入时间线。
11. 若开启语音输出，Agent 发声，右侧感知栏显示播放状态。

语音模式需要自己的节奏：

- 准备聆听
- 正在聆听
- 正在转写
- 等待确认
- 正在发送
- Agent 思考中
- Agent 说话中
- 完成

### 语音输出

语音输出不是一个扬声器按钮。

规则：

- Agent 语音回复应作为本轮回复的一部分展示在时间线中。
- 右侧感知栏显示当前播放状态：生成中、说话中、暂停、停止、被打断。
- 用户可以回放最近一次语音回复。
- 当用户开始说话而 Agent 正在播放语音时，界面要支持打断语义。
- 如果 TTS 不可用，界面降级为文本回复，并明确提示语音输出不可用。

### 摄像头模式

摄像头模式是后续同级输入模式，首期只预留结构。

流程：

1. 用户切换到摄像头模式，或将摄像头与语音组合使用。
2. UI 请求摄像头权限。
3. 预览出现在意图控制台或右侧感知栏。
4. 用户捕获快照、开启低频帧发送，或附加当前画面。
5. 图片/视频帧事件携带元数据发送。
6. 时间线显示本轮使用了哪些视觉输入。

关键原则：摄像头是 Agent 的感知能力，不只是“上传图片”的另一种入口。

### 虚拟形象

虚拟形象是实时状态显示。

推荐状态：

- `idle`
- `listening`
- `hearing`
- `thinking`
- `speaking`
- `interrupted`
- `using_tool`
- `delegating`
- `blocked`
- `error`

虚拟形象不替代文字状态。动画增强氛围，但状态标签才是权威信息。

## 6. UI 状态机

### 语音采集状态

```text
Unavailable
  -> PermissionNeeded
  -> Ready
  -> Listening
  -> Paused
  -> Transcribing
  -> ReviewTranscript
  -> Submitted
  -> Error
```

界面行为：

- `PermissionNeeded`：展示一个主要动作，请求麦克风权限。
- `Ready`：主要动作是开始聆听。
- `Listening`：显示停止、取消、音量/波形、已录制时长、部分转写。
- `ReviewTranscript`：允许编辑、发送、丢弃、重新录制。
- `Submitted`：面板收敛成提交回执，时间线保留转写。
- `Error`：解释权限、设备或后端问题，并提供恢复动作。

### 语音播放状态

```text
Unavailable
  -> Ready
  -> Generating
  -> Speaking
  -> Paused
  -> Interrupted
  -> Completed
  -> Error
```

界面行为：

- `Generating`：右侧感知栏显示 Agent 正在准备语音。
- `Speaking`：显示播放中状态和停止动作。
- `Interrupted`：时间线保留被打断的回复，并标记中断。
- `Completed`：保留回放最近回复的入口。

### Agent 运行状态

```text
Idle
  -> ReceivingInput
  -> Thinking
  -> UsingTool
  -> Delegating
  -> RespondingText
  -> RespondingVoice
  -> WaitingForUser
  -> Blocked
  -> Error
```

这些状态应优先来自运行时事件投影。仅在过渡期可以用 UI 本地状态兜底。

## 7. 事件投影契约

UI 应消费稳定投影，而不是从孤立组件状态里拼产品语义。

推荐投影：

- `RoomTimelineProjection`
- `ParticipantPresenceProjection`
- `MessageDeliveryProjection`
- `ExecutionTraceProjection`
- `VoiceSessionProjection`
- `VisionSessionProjection`
- `AvatarStateProjection`
- `SystemHealthProjection`

首版最小映射：

| UI 区域 | 投影 | 必需字段 |
| --- | --- | --- |
| 时间线消息 | `RoomTimelineProjection` | message id、sender、content、modality、timestamps |
| 语音输入面板 | `VoiceSessionProjection` | capture state、transcript、elapsed time、error |
| 语音播放 | `VoiceSessionProjection` | playback state、response id、elapsed time、error |
| 虚拟形象 | `AvatarStateProjection` | state、active Agent id、last transition time |
| 右侧健康状态 | `SystemHealthProjection` | mic available、speaker available、backend connected |
| 工具/事件细节 | `ExecutionTraceProjection` | step type、status、label、expandable payload |

首期实现可以把已有浏览器本地运行时事件适配成这些结构。设计目标仍然是投影驱动 UI。

## 8. 首屏布局

首屏应该立即传达“这是一个实时 Agent 工作台”，而不是普通聊天页。

推荐层级：

1. 顶部栏：workspace、Agent、session health。
2. 中央：会话时间线。
3. 右侧：Agent 感知栏与语音状态。
4. 底部：带模式切换的意图控制台。

语音面板应从底部意图控制台向上展开，而不是在正常使用中打开模态框。模态框只适合权限修复、设备设置、摄像头详细预览等场景。

## 9. 语音优先控制台

语音模式激活时，控制台变为专用面板：

```text
┌───────────────────────────────────────────────────────────────┐
│ 语音模式                                                       │
│ Agent 可以通过当前浏览器听见你。                                │
│                                                               │
│ [ 实时音量/波形 ]                         00:12                │
│                                                               │
│ 部分转写                                                       │
│ “帮我总结今天的任务，然后...”                                  │
│                                                               │
│ [开始/停止] [取消] [发送转写] [重新录制]                         │
│ 发送策略：手动确认 | 静音后自动发送 | 按住说话                  │
└───────────────────────────────────────────────────────────────┘
```

设计要点：

- 主按钮随状态变化。
- 次级动作稳定且紧凑。
- 聆听时展示部分转写。
- 手动确认模式下，最终转写可以编辑。
- 设备与权限细节可访问，但不占据主视觉。

## 10. Agent 感知栏

右侧感知栏示意：

```text
┌──────────────────────────┐
│ Agent                    │
│ [avatar]                 │
│ 正在聆听                 │
│ 麦克风：启用             │
│ 扬声器：就绪             │
│ 摄像头：关闭             │
│                          │
│ 当前活动                 │
│ - 正在转写语音           │
│ - 等待用户发送           │
│                          │
│ 最近语音回复             │
│ [回放] [停止]            │
└──────────────────────────┘
```

设计要点：

- 感知栏跨输入模式持久存在。
- 语音输出控制位于这里。
- 窄屏时可折叠为顶部状态条。
- 虚拟形象动画跟随状态，但文本标签仍然权威。

## 11. 会话时间线

时间线事件类型：

- 用户文本
- 用户语音转写
- 用户摄像头快照/帧批次
- Assistant 文本
- Assistant 语音
- 工具调用
- Agent 交接
- 系统权限/错误
- 人工确认

语音转写展示规则：

- 最终转写作为用户消息展示。
- 标记来源为 `Voice`。
- 原始/部分转写只在需要时放入详情。
- 如果用户编辑了转写，消息正文使用编辑后的内容，原始转写保留在元数据中。

Assistant 语音展示规则：

- 如果文本和语音同时生成，时间线展示一条 Assistant 回复，并附带音频控制。
- 除非文本和语音内容实质不同，否则不要重复生成两条消息。

## 12. 权限、隐私与失败状态

语音和摄像头必须有明确、克制的隐私提示。

必备状态：

- 浏览器未授予麦克风权限。
- 麦克风设备不可用。
- 麦克风正在采集。
- 音频输出不可用。
- TTS 不可用。
- 摄像头未授权。
- 摄像头正在采集。
- 后端实时服务不可用。

UI 绝不能在未采集时暗示 Agent 正在听。

对于本地优先的产品信任感，需要展示：

- `麦克风正在此浏览器中启用`
- `摄像头关闭`
- `音频在本地播放`
- 如果使用远端实时服务，展示对应连接状态。

## 13. 响应式行为

桌面端：

- 三栏工作台。
- 右侧 Agent 感知栏默认可见。
- 语音面板在底部控制台内展开。

平板端：

- 左侧导航折叠。
- 右侧感知栏保留或变成侧抽屉。
- 语音面板仍然内联。

移动端：

- 时间线和意图控制台优先。
- Agent 感知栏压缩为顶部状态条。
- 语音模式激活时可占据屏幕下半部分。

## 14. 首期施工范围

首期应该包含：

- 工作台级模式切换：键盘、语音。
- 内联语音会话面板。
- 浏览器麦克风采集状态。
- 根据可用能力展示真实或过渡期转写。
- 携带语音元数据提交消息。
- 最近一次 Agent 回复的语音输出控制。
- 右侧 Agent 感知栏，覆盖聆听、思考、说话、空闲、错误。
- 时间线中的语音转写消息展示。
- 权限和不可用状态的降级体验。

首期不包含：

- 完整 WebRTC Qwen-Omni 接入。
- 摄像头流式 UI。
- 多 Agent 桌面窗口。
- 复杂虚拟形象骨骼或 Live2D 级能力。
- 生产级波形分析；简单音量条足够时不应过度施工。

## 15. 验收标准

产品验收：

- 用户能立刻理解这里是 Agent 工作台，而不是只看见文本聊天。
- 用户能开始语音输入，而不是寻找一个小麦克风图标。
- 用户能看见 Agent 是否正在聆听、思考或说话。
- 用户能停止或取消语音采集。
- 用户能查看、编辑或发送语音转写。
- 语音输出可用时，用户能回放或停止最近一次 Agent 语音回复。
- 权限失败可理解、可恢复。

界面验收：

- 语音控件没有被埋在文本输入框里。
- 右侧感知栏稳定表达 Agent 存在感。
- 时间线默认可读，不变成系统日志。
- 文本模式仍然高效。
- 界面符合 Pudding 安静、专业、面向工作的设计语言。

架构验收：

- 语音输入和语音输出状态能表达为事件投影。
- 时间线条目携带 modality 元数据。
- 虚拟形象状态可由运行时事件驱动，而不是写死动画开关。
- 摄像头和未来 Qwen-Omni 实时会话有明确扩展点。

## 16. 实施阶段

### Phase 1：语音工作台外壳

- 在当前聊天界面外层建立工作台语义。
- 在意图控制台增加键盘/语音模式切换。
- 增加内联语音会话面板。
- 增加右侧 Agent 感知栏，表达语音输入/输出状态。
- 时间线增加语音来源标记和元数据展示。

### Phase 2：运行时接入

- 将浏览器麦克风采集接入 `VoiceSessionProjection`。
- 接入 ASR 或转写来源。
- 接入 Agent 语音输出播放。
- 将 Assistant 回复生命周期映射到虚拟形象状态。
- 增加权限、设备、后端错误恢复流程。

### Phase 3：视觉与虚拟形象扩展

- 增加摄像头模式，作为同级输入方式。
- 增加摄像头预览与画面提交。
- 将摄像头状态映射到 `VisionSessionProjection`。
- 基于 `AvatarStateProjection` 升级虚拟形象动画。

### Phase 4：多 Agent OS 表面
