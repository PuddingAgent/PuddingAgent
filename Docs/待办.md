## 最优先需要补的缺口（P0）

这些不是“锦上添花”，而是继续落地时很容易卡住实现分工的地方。

### 1. 身份与权限模型还不完整

现在文档里已经有：

- 用户
- Workspace
- Agent
- `agent-token`
- 审批
- 事件订阅权限

但是**谁以什么身份访问什么接口**，还没形成完整模型。

目前缺的关键点：

- **主体类型定义**  
  至少要区分：
  - human user
  - workspace admin
  - platform admin
  - agent instance
  - runtime node
  - gateway adapter
  - system service

- **令牌体系**
  - 用户 token
  - `agent-token`
  - runtime registration token / node secret
  - adapter credential
  - 临时 delegation token

- **权限粒度**
  现在有“能不能做”的概念，但还没细化到：
  - 能否查看某 Workspace 日志
  - 能否释放某节点
  - 能否触发 workflow
  - 能否订阅某类事件
  - 能否访问某个共享工具目录

- **RBAC / ABAC 的边界**
  目前文档偏“策略存在”，但还没明确：
  - 角色模型
  - 资源模型
  - scope
  - policy evaluation 次序

### 2. Workspace 生命周期还没正式定义

现在 Workspace 是治理边界，但**它自身的状态机还不完整**。

建议至少补清楚：

- `Creating`
- `Active`
- `Frozen`
- `Draining`
- `Archived`
- `Deleted`
- `Restoring`

还需要明确这些动作的系统后果：

- Freeze 后：
  - 是否拒绝新事件？
  - 是否允许查看日志？
  - 是否允许继续读知识库？
  - 正在运行的 workflow 怎么处理？

- Archive 后：
  - 事件是否停止订阅？
  - 对象存储怎么处理？
  - 记忆是否转冷存储？

- Delete / Restore：
  - 是软删还是硬删？
  - 是否支持导出再恢复？

这个点不补，后面治理、审计、存储、驾驶舱、审批都容易各写各的版本。

### 3. Workflow / TaskMap 的执行语义还不够“工程化”

你们已经把 Workflow 设计得很好了，但还偏“结构设计”，**还缺执行语义合同**。

尤其要补：

- 一个节点是 **at-most-once**、**at-least-once** 还是“幂等可重放”
- `Claimed -> Running -> Released` 的超时和续租规则
- 节点失败时：
  - 自动重试几次
  - 谁能接管
  - 什么情况下进入人工干预
- 补偿 / rollback 是节点级还是 workflow 级
- workflow-to-workflow 触发时，失败因果链怎么串起来
- 一个 workflow run 的最终状态是什么集合：
  - `Succeeded`
  - `PartiallySucceeded`
  - `Failed`
  - `Cancelled`
  - `Compensated`
  - `Blocked`

现在这些内容有“建议”，但还没到“可以直接写执行引擎”的程度。

### 4. Controller 无状态，但一致性策略还没被写清楚

总览里说了 `PuddingController` 需要无状态，方向是对的。  
但只要你引入：

- PostgreSQL
- Redis
- RabbitMQ
- Runtime heartbeats
- 审批
- 事件重放
- Workflow 状态推进

就一定会碰到**跨存储一致性**问题。

目前还缺的架构点：

- 是否采用 **Outbox / Inbox** 模式
- 事件发布与数据库写入的原子性怎么保证
- Runtime 上报状态与 Controller 状态表之间如何防重
- 重放事件时如何避免重复推进 workflow 状态
- 审批结果与执行恢复之间如何保证幂等

这个点如果不先立原则，后面会变成经典分布式事故收藏夹。

## 很重要但可以排在第二梯队的缺口（P1）

### 5. 组织 / 租户 / 团队模型还不完整

现在 `Workspace` 很清楚，但再往上一层的组织关系还比较模糊。

建议明确：

- Organization
- Team
- Project / Domain / Environment
- Workspace 与它们的从属关系

因为后面你一定会碰到这些问题：

- 一个团队管理多个 Workspace
- 不同 Workspace 共享某些能力，但数据隔离
- 模板能否跨 Workspace 复用
- 审计人和管理员的可见范围怎么划分

现在文档基本是 `Workspace` 第一层，但**租户层 / 组织层没正式建模**。

### 6. Secret / Key / 凭据治理缺口

你已经明确：

- LLM API Key 不该进 Runtime
- Controller 统一管供应商路由

这是很好的基线。  
但还缺：

- secret 的存储策略
- rotation
- masking / redaction
- 审计日志中如何脱敏
- Runtime 获取临时凭据的方式
- Agent 是否能拿到临时 object-storage upload url
- adapter secret 如何发放与回收

换句话说：

> 你们已经定义了“谁不应该碰 key”，但还没完整定义“key 怎么活着、怎么轮换、怎么失效”。

### 7. 配置治理与 Schema 演进没收口

当前已经有：

- YAML
- 持久化配置
- reload
- registry
- template refs

但还缺几个关键工程点：

- 配置 schema 版本
- migration 策略
- backward compatibility
- 配置变更审计
- 配置变更是否即时生效或延迟生效
- Workspace 级 override 与全局默认值合并规则

尤其你们文档里很多是“建议最小字段”，说明结构方向有了，但**配置演进策略**还没有统一模型。

### 8. 插件 / Skill / MCP 的供应链治理还不够完整

你们已经设计了注册中心，但还缺：

- 签名 / 来源校验
- 兼容性矩阵
- 依赖冲突处理
- 安装 / 升级 / 回滚
- 风险等级如何和审批链联动
- 哪些插件只允许某些 Runtime 标签使用
- 插件运行失败后的熔断与降级

当前文档里更多是“Registry + 引用 + 执行”，但**插件生命周期治理**还没成为单独架构点。

### 9. 资源与配额模型还比较散

现在能看到：

- Runtime 负载
- token 使用
- 审批
- 事件吞吐
- 隔离级别

但还缺一个统一的 **Quota / Budget / Priority** 模型。

建议后续补清楚：

- Workspace token budget
- Agent budget
- Workflow budget
- 事件速率限制
- Sandbox 数量上限
- 并发节点数上限
- 高优先级 workflow 抢占规则
- 低优先级 Agent 是否可被挂起

否则后面会变成：

- 观测系统看得到
- Controller 也知道超了
- 但不知道该怎么裁决

## 第三梯队但对长期演进很重要的缺口（P2）

### 10. 记忆治理还缺“生命周期与合规”维度

你们已经有：

- `PuddingMemoryEngine`
- candidate / formal memory
- recall / compact / boundary

但还缺一些更长期的制度：

- retention policy
- memory TTL
- forget / purge
- 人工标记永不提升 / 永久删除
- PII / 敏感内容脱敏
- legal hold / export

如果以后真走企业落地，这块迟早会成为关键。

### 11. 高可用 / 灾备模型还没正式定义

文档里有分布式优先、多 Runtime、多 Controller 方向，但还缺：

- Controller 多实例下的 leaderless / coordination 策略
- RabbitMQ / PostgreSQL 故障后的最小恢复路径
- Runtime 断联多久视为失联
- Workspace 恢复策略
- Event replay 和恢复点的顺序保证
- 冷启动/热恢复时如何避免重复唤醒

这块目前是“意识到要做”，但不是“设计清楚了”。

### 12. 对外 API 面的正式契约仍然不足

现在任务文档和架构文档里已经出现很多 API 意图，但仍缺一个统一的：

- Platform API contract
- Controller API contract
- Runtime node protocol contract
- Agent-facing HTTP API contract
- 查询 API contract
- 驾驶舱聚合 API contract

你们现在已经到了一个阶段：  
**光有概念和数据模型不够了，需要一层明确的接口契约分册。**

### 13. 人工干预 / 人机协同策略还可以再明确定义

你们已经提到：

- pause
- cancel
- release
- reassign
- approval

但还缺一个清晰的 **human intervention model**：

- 哪些动作只能人做
- 哪些动作 Agent 可建议、人确认
- 干预是否会创建控制事件
- 干预和 workflow 状态迁移如何绑定
- 谁能 override 编排者
- 审计 Agent 与业务 Agent 的冲突如何裁决
