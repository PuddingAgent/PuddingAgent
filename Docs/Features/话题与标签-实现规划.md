## 二、实现规划

### Phase 1: 消息ID查询能力 (后端基础) — 优先级 P0

#### 1.1 新增 API: 按消息ID查询单条消息

**文件**: `SessionEventsController.cs`

```
GET /api/sessions/{sessionId}/messages/by-id/{messageId}
```

返回: `ChatMessageDto` 或包含 `{ id, role, content, createdAt, sessionId }` 的完整消息。

**实现思路**: 查询 `ChatMessages` 表按 `Id`，或查询 `session_event_log` 按 `id`（视哪个作为消息ID）。

#### 1.2 QuerySessionLogsTool 新增 `by_message_id` action

**文件**: `QuerySessionLogsTool.cs`

新增 action:
```json
{
  "action": "by_message_id",
  "workspace_id": "default",
  "message_id": 123
}
```

返回该消息的完整转录内容（含日期）。

#### 1.3 IRawSessionLogService 新增接口方法

**文件**: `IRawSessionLogService.cs`

```csharp

    string workspaceId,
```

### Phase 2: 话题索引 (#开头检测) — 优先级 P0

#### 2.1 数据库: 新增 MessageTopics 表

**文件**: `Program.cs` (DDL) + `MessageTopicEntity.cs` (新实体)

```sql
CREATE TABLE IF NOT EXISTS message_topics (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    message_id      INTEGER NOT NULL,       -- FK → ChatMessages.Id 或 session_event_log.id
    topic_title     TEXT    NOT NULL,        -- # 后面的标题文本
    session_id      TEXT    NOT NULL,
    workspace_id    TEXT    NOT NULL,
    created_at      TEXT    NOT NULL,        -- ISO 8601
    UNIQUE(message_id)
);
```

#### 2.2 后端: 消息入口检测

**位置**: 消息发送处理链路（`ChatApiController.SendMessage` 或 `AgentExecutionService`）

**逻辑**:
1. 收到用户消息 content
2. 检查 `content.TrimStart().StartsWith("#")`
3. 如果是，提取第一行（到 `\n` 或 `\r\n`），去掉 `#` 前缀，trim → topicTitle
4. 消息持久化后，获取 messageId
5. 写入 `message_topics` 表

#### 2.3 后端: 话题搜索 API

```
GET /api/topics/search?workspaceId=default&q=java线程池&limit=20
```

返回匹配的话题列表（含 message_id, topic_title, session_id, created_at）。

#### 2.4 前端: 无需额外改动

`#` 开头的消息对用户是自然输入习惯，不需要 UI 变化。

## 四、关键设计决策

### 决策1: 消息ID 用哪个表的主键？

**建议**: 使用 `ChatMessages.Id`（long 类型）。
- 理由: `ChatMessages` 是面向 UI 的消息表，`session_event_log` 是诊断事件日志
- `ChatMessages` 的 `Id` 更稳定，事件日志可能被截断或仅保留流水帧
- 若需要关联到原始事件，可通过 `session_id + created_at` 反查

### 决策2: 话题存储位置

**建议**: 独立 `message_topics` 表。
- 理由: 话题是可选的（只有 `#` 开头的消息才有），不应放在事件日志 JSON 中增加查询复杂度
- 独立表支持高效的话题搜索索引

### 决策3: 历史搜索模态窗的后端

**建议**: 新增 `MessageSearchController` + 专用 `POST /api/messages/search`。
- 理由: 职责清晰，不耦合 Agent 诊断工具
- 底层复用 `RawSessionLogService.GrepFtsAsync()` 的 Lucene 引擎
