# Admin Chat 简约克制设计方案

> 日期：2026-05-18  
> 页面：`/admin/chat`  
> 目标：保持“静谧书斋式 AI Agent”，降低调试感与视觉噪音  
> ADR：[20AdminChat简约克制界面ADR](../07架构/20AdminChat简约克制界面ADR.md)

---

## 1. 设计目标

本方案用于指导 `/admin/chat` 下一轮 UI 优化。

核心目标：**让用户像在一张安静书桌前与私人 AI 助手对话，而不是进入一个运行时调试仪表盘。**

因此，本轮优化遵循三条原则：

1. **做减法**：不新增默认可见能力，优先隐藏、折叠、弱化。
2. **答案优先**：Agent 最终回复是视觉主角，过程只作为辅助说明。
3. **可信优先**：任何内部变量泄漏都比视觉瑕疵更严重。

---

## 2. 当前问题诊断

从截图和当前实现看，页面的能力基础是好的：

- 米色背景与浅紫用户气泡有“书斋 + 轻品牌”的基础；
- Runtime Timeline 体现了 Agent 可解释性；
- 底部输入区能力完整；
- DevPanel 能支持开发调试；
- 组件已从早期单文件拆分为 `ChatLayout`、`ChatMain`、`MessageGroup`、`InputArea` 等。

但默认体验存在明显噪音：

| 问题 | 用户感受 | 优先级 |
|------|----------|--------|
| 思考摘要出现 `undefined用户问...` | 系统内部拼接泄漏，信任受损 | P0 |
| Runtime Timeline 默认暴露过多 | 像看调试日志，不像聊天 | P0 |
| 底部状态栏图标过多 | 输入框主入口被稀释 | P1 |
| 紫色 glow / neural / breathing 动效偏多 | 书斋气质被科幻调试感打断 | P1 |
| Glass / blur 层层叠加 | 文字清晰度与纸面质感下降 | P1 |
| Token、模型、潜意识、索引等状态常驻 | 高级信息挤占普通用户视野 | P1 |

## 3. 目标体验

### 3.1 默认第一屏

默认第一屏只保留三个主区域：

```text
┌─────────────────────────────────────────────────────────┐
│ Pudding · 当前工作区 · 当前 Agent                         │  ← 弱上下文
├─────────────────────────────────────────────────────────┤
│                                                         │
│  用户消息                                                │
│                                      ┌───────────────┐   │
│                                      │ 什么是记忆图书馆 │   │
│                                      └───────────────┘   │
│                                                         │
│  Agent 状态：正在查找相关记忆…                            │  ← 单一主状态
│                                                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │ Agent 最终回答                                     │  │
│  │ ...                                               │  │
│  └───────────────────────────────────────────────────┘  │
│  已参考 3 条记忆 · 查看过程                              │  ← 弱提示，可展开
│                                                         │
├─────────────────────────────────────────────────────────┤
│ 输入你的问题或任务…                         [停止/发送]   │
└─────────────────────────────────────────────────────────┘
```

### 3.2 用户应感知到的层级

1. **我说了什么**：用户气泡清晰但不刺眼。
2. **Agent 是否在工作**：一条人话状态即可。
3. **Agent 答案是什么**：白色纸面卡片，最易阅读。
4. **过程细节在哪里**：有入口，但不默认展开。
5. **我能做什么**：发送、停止、重试、复制，其他进入更多。

## 4. 设计原则

### 4.1 安静优先

- 页面静止 3 秒后，除当前运行任务外，不应有明显可见动画。
- 同屏持续循环动画不超过 1 个。
- 状态反馈优先使用文案、边框、轻微透明度变化，而不是发光或扫描线。

### 4.2 渐进披露

默认显示：

- 当前 Agent / Workspace；
- 用户消息；
- 单一主状态；
- 最终答案；
- 输入框与发送/停止；
- 错误恢复动作。

默认隐藏：

- 完整 Runtime Timeline；
- thinking 原文；
- 工具原始输出；
- Token 明细；
- Provider / 模型细节；
- Dev raw events；
- 潜意识 LLM、索引、ASP/LSP 等内部状态。

### 4.3 可信度门禁

用户可见文本不得包含：

- `undefined`
- `null`
- `NaN`
- 原始 prompt 拼接片段
- 原始 JSON / payload
- 内部枚举值
- 未清洗工具输出

失败时使用安全 fallback：

- `正在整理上下文…`
- `正在查找相关信息…`
- `正在生成回复…`
- `处理过程暂不可用`

### 4.4 纸面而非玻璃

- 主内容卡片采用暖白纸面，而非大面积玻璃拟态。
- `backdrop-filter` 只保留在 sticky 输入区或浮层。
- 卡片阴影低强度，强调“纸张层级”而非“悬浮科技面板”。

### 4.5 紫色只做点缀

紫色使用场景：

- 用户消息气泡；
- 输入框 focus；
- 当前运行状态的小型强调；
- Pudding 品牌少量点缀。

不得用于：

- 大面积背景；
- 多个状态同时高亮；
- 常态发光；
- 长时间循环动画。

## 5. 视觉系统

### 5.1 色彩

| Token 角色 | 建议 | 用途 |
|------------|------|------|
| Page background | 暖米色 | 页面底色，稳定与柔和。 |
| Surface primary | 暖白 / 纸白 | Agent 答案、输入区、浮层主面。 |
| Surface subtle | 更浅暖灰 | hover、折叠摘要、弱区域。 |
| Text primary | 深灰 / 近黑 | 正文与标题。 |
| Text secondary | 大地棕灰 | 时间、元信息、弱提示。 |
| Accent purple | 低饱和浅紫 | 用户气泡、focus、当前状态。 |
| Runtime memory | 紫灰 | 记忆检索，不发光。 |
| Runtime tool | 蓝灰 | 工具调用，不用高饱和青色。 |
| Success | 灰绿 | 成功摘要。 |
| Error | 低饱和红 | 错误边框与恢复操作。 |

### 5.2 字体与字号

延续当前中文友好字体栈，并控制层级：

| 场景 | 字号 | 行高 | 说明 |
|------|------|------|------|
| 正文 / Markdown | 14-15px | 1.65-1.75 | 长文阅读优先。 |
| 消息元信息 | 12px | 1.4 | 时间、Agent 名称。 |
| Runtime 摘要 | 12px | 1.5 | 弱可见，不抢答案。 |
| 状态栏 / 技术指标 | 11px | 1.4 | 默认隐藏或弱化。 |
| 按钮 | 13-14px | 1.4 | 清晰可点击。 |

### 5.3 圆角与阴影

| 元素 | 圆角 | 阴影 |
|------|------|------|
| 用户气泡 | 8-10px | 无或极弱 |
| Agent 答案卡 | 10-12px | `0 1px 6px rgba(0,0,0,.04)` |
| 输入区 | 12px 顶部圆角 | 仅 sticky 层级阴影 |
| Popover / 详情浮层 | 10-12px | 轻微阴影 |
| Timeline 摘要 | 6-8px | 无 |

### 5.4 动效预算

| 场景 | 允许动效 | 时长 |
|------|----------|------|
| 新消息进入 | opacity + translateY 轻微入场 | 150-240ms |
| Timeline 节点出现 | opacity | 150-220ms |
| 输入 focus | 边框 / 底线颜色过渡 | 150-200ms |
| 当前生成 | 一个低强度状态点或光标闪烁 | 循环允许 |
| 错误 | 一次性边框/背景提示 | 150-240ms |

禁用或默认关闭：

- particle flow；
- neural pulse；
- 大面积 glow；
- 持续 breathe；
- 扫描线；
- glitch shake；
- 多个指示器同时 pulse。

## 6. 组件设计

### 6.1 顶部 Header

当前涉及：`ChatMain.tsx`、`styles.header`。

目标：从“多个选择器 + session id + icon”收敛为“上下文胶囊”。

建议：

- 保留 Pudding logo；
- Workspace / Agent 可继续选择，但视觉弱化；
- Session ID 默认隐藏到 tooltip 或详情菜单；
- 控制台和开发者模式入口保持右侧低权重；
- 窄屏下优先保留 Agent 名称，隐藏 Workspace 长文本。

### 6.2 消息流

当前涉及：`MessageList.tsx`、`MessageGroup.tsx`、`MessageItem.tsx`、`styles.timeline*`。

目标：最终答案成为视觉主角。

建议：

- 用户气泡保留右对齐浅紫，但降低饱和度；
- Agent 答案卡采用暖白纸面，减少 blur；
- message actions 默认 hover 或完成后弱显示，避免常驻；
- Token 用量默认隐藏到 tooltip / 详情；
- 长 Markdown 保持舒适行宽，避免 `maxWidth` 过宽导致阅读疲劳。

### 6.3 Runtime Timeline

当前涉及：`MessageGroup.tsx`、`styles.timeline`、`styles.runtimeState*`。

目标：从“时间线主舞台”改为“执行边注”。

默认态：

```text
正在查找相关记忆…
```

完成态：

```text
已参考 3 条记忆 · 已调用 1 个工具 · 查看过程
```

展开态：

```text
过程详情
- 记忆检索：完成，3 条结果
- 工具调用：完成，0.8s
- 生成回答：完成
```

实现建议：

- 当前运行节点展开，历史节点折叠；
- thinking 文本默认不展示原文，只展示安全摘要；
- 工具输出超过 120-200 字必须折叠；
- 完成后 Timeline 自动收束；
- `undefined/null/NaN` 清洗在摘要生成函数中完成。

### 6.4 输入区

当前涉及：`InputArea.tsx`、`styles.consoleDock`、`styles.consoleStatusBar`。

目标：输入框是主入口，Console 能力退为次级。

默认显示：

- 输入框；
- 发送 / 停止；
- 一个状态胶囊：`就绪` / `生成中` / `出错`；
- 一个“更多”入口。

默认隐藏到更多/详情：

- 附件、图片、语音；
- 思考强度；
- 导出、设置；
- Token 明细；
- Provider balance；
- ASP/LSP、索引、潜意识 LLM、SubAgent 指示器。

文案建议：

| 当前 | 建议 |
|------|------|
| `Agent Console` | 不显示标题，或称“输入” |
| `Runtime 就绪` | `就绪` |
| `正在组织思路` | `正在整理上下文…` |
| `正在调用工具` | `正在处理任务…` / `正在调用工具…` |
| `执行受阻` | `出错了，可重试` |
| `交给我吧。Enter 发送...` | `输入你的问题或任务…` |

### 6.5 状态与错误

必须显示：

- 生成中：`停止` 按钮；
- 错误：人话说明 + `重试`；
- 工具失败：说明是否影响最终回答；
- 高风险动作：确认弹窗或显著提示。

错误文案格式：

```text
生成失败，可以重试。
[重试] [查看详情]
```

工具失败文案格式：

```text
记忆检索失败，但仍可继续生成回答。
[重试检索] [继续生成] [查看详情]
```

### 6.6 开发者模式

当前涉及：`DevPanel.tsx`、`ChatMain.tsx`。

策略：开发者模式保留完整可观测性，但不污染默认态。

开发者模式可显示：

- Raw SSE events；
- 完整 Runtime Timeline；
- Token 明细；
- 上下文层级；
- Provider / 模型 / session id；
- 内部状态枚举。

普通模式不显示这些内容。

## 7. 分阶段实施建议

### Phase P0：可信度与默认折叠

目标：先修用户信任和第一眼噪音。

| 任务 | 文件 | 验收 |
|------|------|------|
| 清洗 thinking / memory 摘要，禁止 `undefined/null/NaN` | `MessageGroup.tsx` | 截图中的 `undefined用户问...` 不再出现。 |
| Runtime Timeline 默认改为单一主状态 | `MessageGroup.tsx` | 普通对话只显示一条主状态。 |
| 完成后过程信息自动收束为 1 行摘要 | `MessageGroup.tsx` | 答案视觉权重高于过程。 |
| Token 用量默认隐藏到 tooltip / 详情 | `MessageGroup.tsx` | 不再常驻占位。 |
| 错误态增加恢复动作 | `MessageGroup.tsx` / `InputArea.tsx` | 出错可见且可重试。 |

### Phase P1：视觉降噪

目标：把界面从“科技调试台”收敛为“暖纸面聊天”。

| 任务 | 文件 | 验收 |
|------|------|------|
| 降低紫色使用面积与饱和度 | `styles.ts` | 紫色仅点缀，常态面积 ≤ 5%。 |
| 关闭常态 glow / particle / breathe / scan 动效 | `styles.ts` | 静止 3 秒后页面几乎不动。 |
| 主卡片从 glass 改为 paper | `styles.ts` | Agent 答案清晰、纸面感更强。 |
| 统一 hover / focus 反馈 | `styles.ts` | 无布局位移，无放大。 |
| Runtime 状态色降饱和 | `styles.ts` | 状态像批注，不像警报灯。 |

### Phase P2：输入区重构

目标：让输入区成为安静主入口。

| 任务 | 文件 | 验收 |
|------|------|------|
| 输入区默认只展示输入、发送/停止、状态胶囊、更多 | `InputArea.tsx` | 默认外露图标 ≤ 5。 |
| 技术状态进入更多菜单或详情浮层 | `InputArea.tsx` | 状态栏不再像 IDE。 |
| 文案用户化 | `InputArea.tsx` | 不再强调 Console / Runtime 技术词。 |
| 窄屏自适应 | `InputArea.tsx` / `styles.ts` | 消息流与输入框优先。 |

### Phase P3：统一 Design Tokens

目标：将 Chat、登录、后台控制台统一为 Pudding 风格。

| 任务 | 文件 | 验收 |
|------|------|------|
| 建立 Surface / Text / Border / Accent / Runtime token 分层 | `styles.ts` 或全局 token | 色彩职责清晰。 |
| 定义 motion token | `styles.ts` | 动效可控且可禁用。 |
| 补充 `prefers-reduced-motion` | 全局样式 | 减少动画系统级可用。 |

## 8. 验收清单

### 8.1 第一眼

- [ ] 第一屏只出现顶部上下文、消息流、输入区三块主区域。
- [ ] 普通对话同时只有一个主状态。
- [ ] 答案卡片比 Runtime 过程更醒目。
- [ ] 页面无明显持续发光、扫描线、粒子、呼吸动画。

### 8.2 可信度

- [ ] 用户可见界面不存在 `undefined`。
- [ ] 用户可见界面不存在 `null`。
- [ ] 用户可见界面不存在 `NaN`。
- [ ] 不显示原始 prompt 拼接片段。
- [ ] 不显示未清洗 JSON / payload。
- [ ] 摘要失败时使用安全 fallback 或隐藏。

### 8.3 可操作性

- [ ] 生成中有明确停止按钮。
- [ ] 错误状态有 `重试` 或其他恢复动作。
- [ ] “查看过程”入口可展开完整 Timeline。
- [ ] 开发者模式可查看技术细节。
- [ ] 默认模式不影响普通聊天效率。

### 8.4 视觉与可访问性

- [ ] 正文对比度 ≥ 4.5:1。
- [ ] 图标按钮有 tooltip / aria-label。
- [ ] 焦点态可见。
- [ ] 桌面点击目标 ≥ 36px，触屏目标尽量 ≥ 44px。
- [ ] 支持 `prefers-reduced-motion`。
- [ ] 窄屏下侧栏、状态栏、Timeline 自动折叠或隐藏。

---

## 9. 非目标

本轮不做：

- 新消息分支；
- 消息编辑历史；
- 消息引用；
- 新后端 API；
- 新 SSE 协议；
- 新数据库字段；
- 完整虚拟滚动重构；
- 大规模品牌重绘。

这些能力可继续参考 [ChatUIRedesign](ChatUIRedesign.md) 的长期规划，但不进入本轮“简约克制”MVP。

---

## 10. 交付判断

当用户重新打开 `/admin/chat` 时，理想感受应是：

> “这是一个安静可靠的 AI 工作台。它告诉我它正在做什么，但不会把内部机器声都倒给我看。”

如果页面看起来仍像 Runtime Debugger、IDE 状态栏或紫色神经网络控制台，则本轮优化未完成。
