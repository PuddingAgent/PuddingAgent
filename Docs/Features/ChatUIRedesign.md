# Admin Chat 页面 UI 重设计方案

> **日期**: 2026-05-05
> **参考**: Discord / WhatsApp / Slack / ChatGPT / Cursor / Claude Code
> **现状**: 单文件 1045 行 index.tsx，消息渲染含 legacy/structured 双模式，无头像在气泡内，无右键菜单，无分支/编辑/引用

---

## 一、现状评估

### 1.1 当前优劣势

| 维度 | 现状 | 评分 |
|------|------|------|
| **信息密度** | 消息卡片宽敞，padding 大，适合阅读但信息密度低 | ★★☆ |
| **视觉层次** | 依赖 antd Card + List，无明显分层（无侧栏/频道列表） | ★★☆ |
| **动画** | 仅有全局 fadeIn/slideUp/shake，聊天页未充分利用 | ★☆☆ |
| **头像** | 仅顶部选择器有，气泡内无头像 | ★☆☆ |
| **交互反馈** | hover 显示复制/删除按钮，无更多微交互 | ★★☆ |
| **Agent 特性** | 有 structured 模式（时间线+推理+步骤卡+答案），但展开/折叠不直观 | ★★★ |
| **右键菜单** | 无 | ☆☆☆ |
| **消息引用** | 无 | ☆☆☆ |
| **编辑历史** | 无 | ☆☆☆ |
| **代码结构** | 单文件 1045 行，状态散落在 useState 中，无组件拆分 | ★☆☆ |

### 1.2 与 Discord/WhatsApp 的核心差距

| Discord 特性 | Pudding 现状 |
|-------------|-------------|
| 紧凑列表 + 小字号 | 大卡片 + 宽松间距 |
| 头像在每条消息左侧 | 无 |
| hover 高亮消息背景 | 无 |
| 右键菜单（引用/编辑/删除） | 无 |
| 消息分组（同一人连续消息合并头像） | 无 |
| 频道侧栏 | 仅有 Session 下拉列表 |
| emoji reaction | 无 |

## 二、方案对比（3 个方案）

### 方案 A：Discord 风格 — 紧凑列表 + 侧栏

**布局**：
```
┌──────────┬──────────────────────────────────────┐
│ 左侧栏   │  聊天区                               │
│          │                                      │
│ Session  │  ┌─ 头像 用户名 时间 ──────────────┐  │
│ 列表     │  │  消息内容（紧凑）                 │  │
│ (200px)  │  └────────────────────────────────┘  │
│          │  ┌─ 头像 用户名 时间 ──────────────┐  │
│ • Sess1  │  │  消息内容                        │  │
│ • Sess2  │  └────────────────────────────────┘  │
│ • Sess3  │                                      │
│          │  ┌────────────────────────────────┐  │
│ [+新建]  │  │  输入区                         │  │
└──────────┴──────────────────────────────────────┘
```

**消息气泡**：左侧 40px 圆形头像 + 用户名 + 时间戳 + 紧凑内容
**头像策略**：同人连续消息合并头像（仅第一条显示）
**动画**：新消息 slideUp + fadeIn，hover 背景变灰
**右键菜单**：复制 / 引用 / 删除 / 重新执行 / 加入记忆

**优点**：信息密度高，符合 IM 用户习惯，头像提升人格感
**缺点**：Agent 的结构化输出（步骤卡/推理链）在紧凑布局下可能难以展示

---

### 方案 B：ChatGPT/Claude 风格 — 宽气泡 + 功能面板

**布局**：
```
┌─────────────────────────────────────────────────┐
│  聊天区（全宽）                                   │
│                                                  │
│  ┌──────────────────────────────────────────┐   │
│  │  👤 Agent 名称                            │   │
│  │  回复内容（Markdown 渲染）                 │   │
│  │  ┌─ 推理链（可折叠）───────────────────┐  │   │
│  │  │ Step 1: 调用 API ✓                  │  │   │
│  │  │ Step 2: 数据清洗 ✓                  │  │   │
│  │  │ Step 3: 生成报告 ⏳                 │  │   │
│  │  └────────────────────────────────────┘  │   │
│  │  [👍👎] [📋复制] [🔄重试]               │   │
│  └──────────────────────────────────────────┘   │
│                                                  │
│  ┌──────────────────────────────────────────┐   │
│  │  输入区 + 命令面板（/）                    │   │
│  └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
```

**消息气泡**：宽卡片，头像在左上角，内容区完整渲染 Markdown + 步骤卡
**头像策略**：每条消息都带头像（Agent 人格关键识别点）
**动画**：打字机效果 + 步骤卡逐条出现 + 完成脉冲
**右键菜单**：ChatGPT 风格的复制/重新生成 + Agent 专属操作

**优点**：充分发挥 Agent 结构化输出优势，推理链/步骤卡展示清晰
**缺点**：信息密度略低，长对话滚动量大

---

### 方案 C：混合分层 — 紧凑列表 + 展开式 Agent 面板（**推荐**）

**核心思路**：日常对话用紧凑列表（方案 A），Agent 执行过程自动展开为功能面板（方案 B）

**布局**：
```
┌──────────┬──────────────────────────────────────────┐
│ 左侧栏   │  聊天区                                   │
│ Session  │                                           │
│ 列表     │  ┌─ 头像 用户名 时间 ───────────────┐    │
│          │  │  "帮我分析这段日志"               │    │
│          │  └─────────────────────────────────┘    │
│          │  ┌─ 头像 Agent ─────────────────────┐    │
│          │  │  🔍 正在分析...                   │    │
│          │  │  ┌─ 执行面板（自动展开）───────┐  │    │
│          │  │  │ 📍Step 1: 读取日志  ✓ 0.3s │  │    │
│          │  │  │ 📍Step 2: 解析格式  ✓ 0.5s │  │    │
│          │  │  │ 📍Step 3: 异常检测  ⏳ ... │  │    │
│          │  │  │ [⏸暂停] [⏹停止] [🔄重试]  │  │    │
│          │  │  └────────────────────────────┘  │    │
│          │  └─────────────────────────────────┘    │
│          │                                           │
│          │  ┌─ / 命令面板 ──────────────────────┐   │
│          │  │  [输入...]            [📎] [发送] │   │
│          │  └──────────────────────────────────┘   │
└──────────┴──────────────────────────────────────────┘
```

**动态行为**：
- 文字回复：紧凑气泡，头像在左
- 工具调用：自动展开为步骤面板（animating）
- 完成后：面板收起到摘要行，可点击重新展开
- 引用/分支：内联引用卡片 + 分支指示器

**优点**：兼顾信息密度和 Agent 特性，无信息过载
**缺点**：实现复杂度最高，需要精细的状态管理

### 3.3 头像系统设计

| 场景 | 样式 | 大小 |
|------|------|------|
| 聊天消息 | 圆形 + emoji/首字母 + 背景色 | 32px |
| 消息组首条 | 同上 + 用户名 + 时间戳 | 32px |
| 消息组后续 | 仅 32px 占位（无头像/名称） | 32px |
| 侧栏 Session | 小圆形 + 首字母 | 24px |
| 顶栏选择器 | 略大圆形 + emoji + 名称 | 28px |

**消息分组规则**：
- 同一发送者 + 间隔 < 5 分钟 → 合并为一组
- 组内仅第一条显示头像和名称
- 不同类型（user/assistant/system）强制分组

### 3.4 动画系统增强

新增动画定义：

```css
/* 消息入场 */
@keyframes messageIn {
  from { opacity: 0; transform: translateY(12px); }
  to { opacity: 1; transform: translateY(0); }
}

/* 步骤卡入场（交错延迟） */
@keyframes stepIn {
  from { opacity: 0; transform: translateX(-8px); }
  to { opacity: 1; transform: translateX(0); }
}

/* 打字光标 */
@keyframes cursorBlink {
  0%, 100% { opacity: 1; }
  50% { opacity: 0; }
}

/* 思考中脉冲 */
@keyframes thinkingPulse {
  0% { opacity: 0.6; }
  50% { opacity: 1; }
  100% { opacity: 0.6; }
}

/* hover 浮起 */
@keyframes hoverLift {
  from { transform: translateY(0); box-shadow: none; }
  to { transform: translateY(-1px); box-shadow: 0 2px 8px rgba(0,0,0,0.08); }
}

/* 完成勾 */
@keyframes checkPop {
  0% { transform: scale(0); }
  60% { transform: scale(1.2); }
  100% { transform: scale(1); }
}

/* 新消息提醒脉冲（未读红点） */
@keyframes unreadPulse {
  0% { box-shadow: 0 0 0 0 rgba(22,119,255,0.4); }
  70% { box-shadow: 0 0 0 6px rgba(22,119,255,0); }
  100% { box-shadow: 0 0 0 0 rgba(22,119,255,0); }
}
```

**动画应用场景**：
- 消息入场：`messageIn 0.3s ease-out`，仅对新出现消息
- 步骤卡：`stepIn 0.2s ease-out`，每步交错延迟 0.1s
- 打字光标：`cursorBlink 1s infinite`（流式输出时）
- 思考中：`thinkingPulse 1.5s infinite`（Agent 等待 LLM 时）
- Hover：`hoverLift`（消息卡片 hover 时）
- 步骤完成：`checkPop 0.3s ease-out`（步骤由 ⏳→✓ 时）
- 新消息通知：`unreadPulse 1.5s`（滚动到底部按钮激活时）

### 3.7 可编辑历史与分支

**数据模型**（依赖 ADR-013 的 Message ParentId 链）：

```typescript
interface MessageEditRecord {
  editId: string;
  originalMessageId: string;
  editedContent: string;
  editedAt: number;
  branchType: 'MAIN' | 'RETRY' | 'FORK';
}
```

**UI 交互**：
1. 双击用户消息 → 变为可编辑文本框 → Enter 保存，Esc 取消
2. 编辑后显示 "已编辑" 标记 + 原始内容可查看
3. "从这里创建分支" → 弹出分支命名 → 新建分支会话
4. 分支切换器显示在当前消息上方

## 四、底层架构需求

在开始 UI 工作前，以下后端能力不可缺少：

| 能力 | 现有状态 | 需要开发 | 关联任务 |
|------|---------|---------|---------|
| 消息 ParentId 链 | ADR-013 设计中 | ✅ Phase 1 | task-20260505-008 |
| 消息编辑/版本 | ❌ 无 | ✅ 需要 | 新任务 |
| 消息引用（Quote） | ❌ 无 | ✅ 需要（DB 字段 + API） | 新任务 |
| 会话分支 | ADR-013 设计中 | ✅ Phase 3 | task-20260505-026 |
| 右键菜单操作 API | ❌ 无 | ✅ 需要（重新执行/加入记忆/固定上下文） | 新任务 |
| 命令面板后端 | ❌ 无 | ⚠️ 前端工具注册表即可 | UI 侧 |
| 消息分组（头像合并） | ❌ 无 | ⚠️ 纯前端逻辑 | UI 侧 |
| 虚拟滚动 | ❌ 无 | ✅ 需要（react-virtuoso） | 新任务 |

### 需要新增的后端任务

1. **Message Edit API**：PUT /api/messages/{id} → 保存编辑记录，设置 edited=true
2. **Message Quote API**：创建消息时支持 quotedMessageId 字段
3. **Message Rerun API**：POST /api/chat/rerun → 从指定消息重新执行
4. **Pin Context API**：POST /api/sessions/{id}/context/pin/{messageId}

## 五、实施分阶段

### Phase UI-1：基础重构（P0）
- 拆分 index.tsx → 多组件（MessageList/MessageItem/InputArea/SessionSidebar）
- 头像系统：消息气泡内添加头像 + 消息分组
- 紧凑布局：减小 padding/gap，提升信息密度
- 动画：消息入场 + 步骤卡交错 + typing 光标

### Phase UI-2：交互增强（P1）
- 右键菜单（所有选项）
- 命令面板（/ 触发）
- 内联操作按钮（👍📋🔄📌）
- Hover 高亮 + 微动效
- 未读提示（新消息提醒）

### Phase UI-3：Agent 特性（P1）
- 可编辑历史 + 从编辑点重新生成
- 消息分支可视化 + 切换
- 上下文面板（可折叠）
- 拖拽文件/消息到输入框
- 执行控制面板（暂停/停止/跳过）

### Phase UI-4：高级功能（P2）
- 虚拟滚动（大数据量）
- 多选消息批量操作
- Emoji reaction
- 消息搜索结果高亮
