# 会话层统一投影与前端观察者模型设计方案

> 日期：2026-06-14  
> 页面：`/admin/chat`  
> ADR：[ADR-050 会话层统一投影与前端观察者模型](../07架构/51ADR-050会话层统一投影与前端观察者模型ADR.md)

---

## 1. 目标

把 `/admin/chat` 从“前端理解多种后端事件并自行合并”的模型，迁移为“前端观察会话层统一投影”的模型。

目标状态：

- 后端会话层负责记录、合并、排序和去重所有用户可见消息。
- 前端只发送命令、订阅投影、渲染投影。
- 浏览器离线、刷新、重连后，可以从会话层追赶最新状态。
- Agent 消息、系统错误、子代理卡片、工具摘要的顺序和终态由后端统一保证。

---

## 2. 当前问题

### 2.1 前端承担了过多协议知识

当前前端需要理解：

- `metadata` 如何绑定 turn。
- `delta/thinking/tool_call/tool_result` 如何追加。
- `error/done/cancelled` 如何关闭 loading。
- `subagent.spawned/subagent.completed` 如何生成卡片。
- replay 和 SSE 的 cursor 何时推进。
- 本地 pending 和后端投影何时互相覆盖。

这些逻辑集中在 `useChatState.ts`、`MessageList.tsx` 和多个状态组件中，导致修复一个场景时容易影响另一个场景。

### 2.2 后端事实已经存在，但没有统一 UI 合约

当前已有事实层：

- `session_event_log`
- `ChatMessages`
- `sub_agent_runs`
- `message_deliveries`
- Agent status / conversation projection

但 UI 合约不够统一。前端有时读 projection，有时读 messages，有时读 replay，有时直接合并 runtime event。

### 2.3 用户可见错误没有统一物化

例如 LLM Unauthorized 和 session fuse 已经写入 `session_event_log`，也可能写入 `ChatMessages`，但前端仍可能因为事件映射或投影刷新时机表现为空白或无反馈。

---

## 3. 目标架构

```text
Runtime / Tools / SubAgent / Message Fabric
        |
        v
session_event_log + ChatMessages + sub_agent_runs + deliveries
        |
        v
ConversationProjectionReducer
        |
        v
ConversationProjectionStore
        |
        +--> GET snapshot / replay
        +--> SSE projection stream
        |
        v
Frontend ConversationProjectionStore
        |
        v
MessageList / ComposerStatus / SubAgentIndicator
```

前端不再直接消费 Runtime 内部事件。低层事件仍可通过诊断面板查看，但普通消息流只渲染会话投影。

---

## 4. 后端设计

### 4.1 数据模型

建议定义统一 DTO：

```csharp
public sealed record ConversationProjectionFrame(
    string SessionId,
    long SequenceNum,
    string ProjectionId,
    int Revision,
    string Kind,
    string Status,
    DateTimeOffset RecordedAt,
    object Payload);
```

建议定义投影项：

```csharp
public sealed record ConversationProjectionItem(
    string ProjectionId,
    int Revision,
    string Kind,
    string Status,
    DateTimeOffset CreatedAt,
    DateTimeOffset UpdatedAt,
    string? MessageId,
    string? RunId,
    string? SourceId,
    string? SourceName,
    object Payload);
```

首批 `Kind`：

| Kind | 用途 |
|------|------|
| `user_message` | 用户输入、语音转写确认后的用户消息、外部用户消息。 |
| `agent_message` | Agent 回复、失败回复、熔断回复。 |
| `system_notification` | 系统错误、权限拒绝、provider 配置错误、会话状态提示。 |
| `sub_agent_run` | 子代理生命周期卡片。 |
| `process_summary` | 工具、思考、上下文、记忆、索引等过程摘要。 |
| `status_marker` | 可选的短状态标记，如正在排队、正在等待用户确认。 |

### 4.2 Reducer 输入

`ConversationProjectionReducer` 首阶段从以下来源读取：

- `session_event_log`：metadata、delta、thinking、tool、error、done、cancelled、subagent 事件。
- `ChatMessages`：稳定用户/agent 转录。
- `sub_agent_runs`：子代理终态、输出摘要、失败原因。
- `message_deliveries`：队列、投递、ack 状态。

Reducer 不应该依赖前端状态。任何一次 reducer 运行都应能从事实层重新构建同样的 projection。

### 4.3 Reducer 规则

基本规则：

1. `metadata` 建立 message/run/source 关系，但不直接渲染。
2. `delta` 聚合到对应 `agent_message` 草稿。
3. `error` 归并为失败的 `agent_message` 或 `system_notification`。
4. `done` 覆盖对应 `agent_message` 的终态和最终内容。
5. `subagent.*` 归并为同一个 `sub_agent_run` projection item。
6. `ChatMessages` 是已物化的转录，优先用于稳定快照；事件层用于补齐过程和运行中状态。
7. 同一 `messageId/runId/projectionId` 的后续事件只能产生 revision，不创建重复投影项。

错误可见性规则：

- Provider Unauthorized：`system_notification(status=failed)`，并关联当前 `messageId`。
- Session fuse：`agent_message(status=failed)` 或 `system_notification(status=failed)`，内容使用 fuse summary。
- Tool failed：默认进入 `process_summary(status=failed)`；如果导致整轮失败，同时更新 `agent_message(status=failed)`。

### 4.4 API

新增或演进接口：

```http
GET /api/sessions/{sessionId}/conversation?after={sequence}&limit={limit}
GET /api/sessions/{sessionId}/conversation/snapshot
GET /api/sessions/{sessionId}/conversation/stream?after={sequence}
```

响应建议：

```json
{
  "sessionId": "session-1",
  "items": [],
  "frames": [],
  "cursor": 120,
  "hasMore": false,
  "snapshot": true
}
```

当 `after` 太旧或 projection store 已压缩时，后端返回 snapshot + cursor，前端直接替换当前会话视图。

### 4.5 物化策略

阶段 1：实时构建 projection。

- 查询时从事实层构建。
- SSE 仍由现有会话 append 通道触发，但输出 projection frame。
- 优点是改动小，不新增表。

阶段 2：物化 projection 表。

建议表：

- `conversation_projection_items`
- `conversation_projection_frames`
- `conversation_projection_cursors`

物化后可降低大历史会话的 replay 成本，也能更稳定支持多浏览器追赶。

## 5. 前端设计

### 5.1 新状态模型

新增轻量 store：

```ts
interface ConversationProjectionState {
  sessionId: string | null;
  cursor: number;
  items: ConversationProjectionItem[];
  streamStatus: 'idle' | 'connecting' | 'connected' | 'reconnecting' | 'offline' | 'error';
  pendingCommands: PendingCommand[];
}
```

前端组件只读 `items`，不再读低层 event。

### 5.2 渲染模型

`MessageList` 从 projection item 构建 view model：

```text
user_message          -> UserMessageBubble
agent_message         -> AgentMessageBubble
system_notification   -> SystemNoticeRow
sub_agent_run         -> SubAgentCard
process_summary       -> MessageProcessSummary / Inspector
status_marker         -> ComposerStatusLine / inline status
```

`SubAgentIndicator` 只显示 projection 聚合后的 running/failed/completed 统计，不再自行读取子代理事件。

### 5.3 Optimistic UI

前端可以在 `POST message` 后插入临时 `pendingCommands`，但规则必须很窄：

- 只用于用户刚发送的文本可见性。
- 不推断 agent 回复。
- 不维护本地队列事实。
- 一旦收到后端 projection 中相同 `clientCommandId` 或同一 message ack，立即移除 pending。

### 5.4 删除或下沉的前端逻辑

目标是逐步删除：

- `messageIdToTurnIdRef`
- `activeMessageIdsRef`
- 低层 `event.type` 业务 switch
- `subAgentCards` 前端合并器
- 本地 interaction queue 事实状态
- 根据 raw `error/done/delta` 修改消息气泡的普通 UI 逻辑

保留：

- 滚动位置。
- 输入框草稿。
- pending command 临时提示。
- stream reconnect 状态。
- 开发者诊断中的 raw event inspector。

## 6. 迁移计划

### Phase 1：后端 Projection Snapshot

目标：

- 新增 `ConversationProjectionService`。
- 从现有 `ChatMessages + session_event_log + sub_agent_runs` 构建 snapshot。
- 确保 LLM Unauthorized、session fuse、sub-agent terminal 都可见。

验收：

- 刷新页面后可见最后一次错误/终态。
- 重复发送同一句不会导致 projection 丢失。

### Phase 2：Projection Stream

目标：

- 提供 `conversation/stream?after=cursor`。
- 后端输出 projection frame，不输出 raw runtime event。
- 支持断线后 after cursor 追赶。

验收：

- 断开前端再恢复，不丢 agent 输出和系统错误。
- `sequenceNum` 单调递增。

### Phase 3：前端 Store 收敛

目标：

- 前端新增 `ConversationProjectionStore`。
- `MessageList` 改为渲染 projection item。
- 旧 `useChatState` 的 raw event 分发退到诊断模式。

验收：

- 普通消息流代码不再 switch `delta/thinking/tool_call/error/done`。
- 子代理卡片由 projection item 渲染。

### Phase 4：物化与性能优化

目标：

- 必要时新增 projection 表。
- 支持 snapshot compaction。
- 对长会话做分页和窗口化。

验收：

- 长会话滚动和输入不卡顿。
- Projection reducer 指标可查询。

## 7. 测试计划

后端测试：

- 从 `ChatMessages` 构建 user/agent message。
- 从 LLM error event 构建 `system_notification`。
- 从 session fuse done/error 构建失败终态。
- 从 `subagent.spawned/completed` 构建单个 `sub_agent_run`，不重复。
- event sequence 乱序输入时按 `sequenceNum/recordedAt` 输出稳定顺序。

前端测试：

- projection item 渲染为对应 row。
- 重复发送同一句时 optimistic pending 可见，projection 到达后被覆盖。
- 离线后 snapshot 替换旧状态。
- 子代理 item 刷新后顺序不变。
- system notification 刷新后仍可见。

集成测试：

- 发送消息 -> projection 出现 user pending -> agent running -> agent succeeded/failed。
- 模拟 provider Unauthorized -> UI 显示系统错误或失败消息。
- 模拟子代理完成 -> 卡片出现在后端记录的顺序位置。

---

## 8. 风险与约束

- 不能一次性删除旧 `/replay` 和 `/messages`，诊断和兼容仍依赖它们。
- Projection reducer 必须保留 raw evidence reference，便于从 UI 跳到诊断证据。
- 多浏览器场景下，projection 是共享事实；输入框草稿、滚动位置和 inspector 打开状态仍是浏览器本地状态。
- `ChatMessages` 当前缺少稳定 `MessageId/RunId` 字段，短期可从 event payload 关联，中期应补 schema 以提高幂等性。

---

## 9. Dev 施工建议

首个施工任务不要直接重写整个前端。建议先做后端闭环：

1. 新增 projection DTO 和 service。
2. 用现有 session `4cb706c1f28d4a96a4363d01087cceb2` 的 LLM Unauthorized/fuse 事件做回归样本。
3. 提供 snapshot API。
4. 给前端加一个只读 projection inspector 或隐藏开关，验证 projection 与现有消息流一致。
5. 再切 `MessageList` 数据源。

这样能先把事实源稳定下来，再减少前端状态代码，避免边设计边改 UI 导致新的滚动和空白问题。
