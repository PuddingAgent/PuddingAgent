## 1. 问题背景

Agent 记忆是整个 AI 行业最被低估的难题之一。当前的记忆系统（Task 08）已定义了短期/工作/长期三层记忆结构，但缺少一个**自动化、后台运行的记忆处理引擎**。现有方案依赖显式指令触发记忆写入，无法实现以下能力：

- **自动观察与提取**：从对话中自动识别值得记住的事实
- **记忆提纯**：去除冗余废话，浓缩关键信息，不损失信息熵
- **智能索引**：自动打标签、分类、建立记忆间的关联
- **记忆验证**：召回前验证记忆的可靠性（人类不会盲目信任自己的记忆）
- **遗忘机制**：不重要/过时的记忆自动衰减和清理

### 1.1 四个核心难题

记忆引擎本质上需要回答四个问题：

| 难题 | 描述 | 示例 |
|------|------|------|
| **什么值得记住？** | 区分"闲聊"和"关键事实" | "今天天气不错" → 遗忘；"PostgreSQL 部署在 192.168.1.100:5432" → 记住 |
| **记忆粒度多细？** | 记一句话还是记一个段落？ | "数据库是 PG" vs "2025年5月选型 PG 15，因为事务支持好，部署在 K8s 集群 node3" |
| **偏好如何演进？** | 用户的偏好会变，旧偏好不能永远占位 | 用户去年喜欢 React，今年转向 Vue——记忆需更新而非叠加 |
| **大规模检索怎么做？** | 10万条记忆中如何毫秒级找到"那个 PostgreSQL 的部署地址" | PostgreSQL 可能掺杂在无数上下文里，需要精确地从废话海洋中捞出那条关键信息 |

### 1.2 反思先行：操作前回忆

> **核心洞察**：Agent 在执行操作前，应该先**反思和回忆**，而不是盲目执行。

**场景**：用户说"启动 PostgreSQL 数据库"。

没有记忆的 Agent：
```
用户: 启动 PostgreSQL
Agent: 好的，我用哪个地址？端口多少？密码是什么？
```

有记忆引擎的 Agent：
```
用户: 启动 PostgreSQL
Agent: [潜意识召回] → 找到记忆："PG 部署在 192.168.1.100:5432，用户 admin"
Agent: 好的，正在连接 192.168.1.100:5432...
```

**多数据库消歧**：如果当前环境有多个数据库（PG、MySQL、MongoDB），Agent 必须根据上下文推断用户指的是哪一个——这需要记忆引擎提供**上下文感知的消歧能力**。

### 1.3 闲聊 vs 关键事实

记忆引擎的核心判断力：**不是所有对话都值得记住**。

```
对话流 ─────────────────────────────────────────────
│ "早上好"  "今天天气不错"  "你吃了吗"              │ → 闲聊，遗忘
│ "对了，我们生产环境 PG 部署在 10.0.1.50"           │ → 关键事实，记住！
│ "端口 5432，用户名 admin，密码在 vault"            │ → 关键事实，记住！
│ "那你今天想做什么"  "随便聊聊"                     │ → 闲聊，遗忘
└──────────────────────────────────────────────────┘
                                    ↓ 潜意识提纯后
┌──────────────────────────────────────────────────┐
│ 生产 PostgreSQL: 10.0.1.50:5432, user=admin       │
│ 密码: vault://prod-pg-password                    │
│ (最后提及: 2026-05-02, 置信度: 0.95, 重要性: 0.9) │
└──────────────────────────────────────────────────┘
```

**核心理念**：借鉴人类认知模型——显意识负责对话与任务执行，**潜意识在后台负责观察、反思、处理记忆和召回记忆**。就像人类每天晚上做梦时梳理记忆，不是每个字都记住，而是选择性地浓缩、摘要化或遗忘。

## 2. 研究目标

设计一个**轻量、快速、后台异步运行的潜意识记忆引擎（Subconscious Memory Engine）**，与主模型隔离，负责：

1. **观察**显意识的对话流
2. **提取**值得记住的事实、决策、偏好
3. **提纯**记忆（去冗余、浓缩、摘要化），保证信息熵不丢失
4. **标签索引**（图书馆式登记造册）
5. **三维评分**（置信度/重要性/持久性）
6. **冲突检测**与自动衰减
7. **记忆验证**（先验证可靠性再召回）
8. **关键记忆**独立存储，支持快速上下文插入

## 4. 小模型选型方向

### 需求约束

| 约束 | 要求 |
|------|------|
| **速度** | 毫秒级响应，不阻塞主对话 |
| **资源** | 单进程内运行，CPU 友好 |
| **隔离** | 与主 LLM 推理完全隔离 |
| **能力** | 文本分类、摘要、实体提取、语义相似度 |

### 候选方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **ONNX 小模型** (如 Phi-3-mini 量化版) | 本地运行、低延迟、免费 | 需要 ONNX Runtime，内存占用 |
| **规则引擎 + 模板** | 极快、零依赖 | 灵活性差，无法语义理解 |
| **轻量嵌入模型** (如 all-MiniLM-L6-v2) | 向量化快，适合相似度+RAG | 只能编码，不能生成 |
| **API 小模型** (如 DeepSeek-V4-Pro-lite) | 能力强，免部署 | 网络延迟，费用 |
| **混合方案**：规则 + 嵌入 + 小模型 fallback | 分层处理，成本可控 | 复杂度增加 |

**推荐方向**：混合方案——简单分类用规则，相似度用本地嵌入模型，摘要/提纯用轻量 API。

## 5. 五维记忆评分模型

> **核心洞察**：记忆评价不只是"记住"，更是"知道什么该记住"。越重要的事情记忆越深，昨天的比去年的更清晰——但我们比人类更优秀，因为我们有完整的会话日志可以回溯。

每条长期记忆附带五个维度评分（0.0 ~ 1.0）：

| 维度 | 含义 | 作用 |
|------|------|------|
| **置信度 (Confidence)** | 这条记忆有多可靠 | 低于阈值不召回，需翻日志验证 |
| **重要性 (Importance)** | 对用户/AI 的价值 | 决定存储优先级，高重要性记忆更难衰减 |
| **新鲜度 (Freshness)** | 距今多久，越近越高 | 昨天的事情比去年清晰——时间加权 |
| **持久性 (Persistence)** | 抗衰减能力（人工可标记保护） | 重要决策/教训持久性高，琐事低 |
| **关键词匹配 (Keyword Match)** | 与当前查询的相关度 | 召回时实时计算，影响排序权重 |

### 综合召回评分

```
recall_score = w1×confidence + w2×importance + w3×freshness + w4×persistence + w5×keyword_match
```

召回时按 `recall_score` 降序返回，top-K 注入上下文。

### 重要洞察：人类 vs Agent 记忆

| | 人类 | Pudding Agent |
|------|------|------|
| 琐事遗忘 | 铅笔盒颜色，10年后记不住 | 可翻会话日志（完整归档），不需要记住 |
| 重要记忆 | 骑自行车骨折了，获得教训 | 提纯后存入长期记忆，长期可召回 |
| 回忆方式 | 靠联想，模糊 | 混合检索 + 日志回溯，精确 |

> **关键设计原则**：Agent 不需要记住每句话——有完整的会话日志。记忆引擎的职责是从废话中提取金子，只记住"在某年某地做了某事，结果如何，收获什么，待办什么"。

## 5.1 衰减机制：AI 驱动的自适应摘要化

> **核心洞察**：衰减不是降低分数，而是不断摘要化。不定义硬编码规则——让潜意识 AI 根据提示词自行判断何时压缩、压缩多少、保留什么。

### 设计哲学：不定义规则，定义提示词

**硬编码方式（不采用）**：
```
if days >= 1: compress to 1KB
if days >= 7: compress to 500 chars
if days >= 30: compress to 200 chars
...
```
问题：刻板、不区分内容重要性、无法适应不同场景。

**AI 驱动方式（采用）**：
```
潜意识 AI 收到提示词：
  "这是一条记忆，当前新鲜度评分 0.3，距今 45 天。
   请判断是否需要压缩，如果需要，压缩到什么程度。
   注意：如果内容是关键决策或教训，即使时间久也应保留较多细节。
   如果是琐事或闲聊残留，可以激进压缩。"
→ AI 自行决定输出长度和保留内容
```

### AI 决策上下文

每次衰减触发时，潜意识小模型会收到完整上下文：

```
[系统提示词]
你是记忆管理助手。你的任务是决定记忆是否需要压缩，以及压缩到什么程度。
不要机械套用时间规则，要根据内容本身的重要性灵活判断。

[当前记忆]
{记忆内容}

[记忆元数据]
- 置信度: 0.92
- 重要性: 0.85 (高)
- 新鲜度: 0.25 (45天前)
- 持久性: 0.8 (用户标记为重要)
- 关键词: PostgreSQL, 数据库, 生产环境
- 历史: 已压缩 2 次

[判断准则]
1. 高重要性的记忆（>0.7）：即使时间久也保留较多细节
2. 低重要性的记忆（<0.3）：可以激进压缩，甚至建议遗忘
3. 决策类记忆（选型、架构）：保留理由和上下文
4. 事实类记忆（IP、端口）：压缩但不丢失核心数据
5. 闲聊残留：直接遗忘
6. 用户明确标记"不压缩"：跳过

[输出格式]
返回 JSON：{
  "action": "compress" | "keep" | "forget",
  "compressed_content": "压缩后的内容（如果 action=compress）",
  "reason": "判断理由，一句话"
}
```

### 触发机制

不硬编码时间线。潜意识 AI 在以下时机主动评估：

- 会话空闲时（`session.idle` 事件）
- 新会话开始时（检查旧记忆是否需要压缩）
- 记忆数量超过阈值时（防止存储膨胀）
- 用户主动触发"整理记忆"

AI 会收到所有待评估记忆的列表，自行决定优先级和处理顺序。

### 压缩不丢失可回溯性

每次压缩输出保留指向上一级版本的引用，形成链：
```
v3 (精华) → 引用 v2 → 引用 v1 (原始提取)
```
需要时可以沿链回溯到更详细的版本，或直接翻原始会话日志。

## 5.2 分层检索策略：AI 自适应排序

> **核心洞察**：不硬编码"30天=活跃，30天~1年=沉淀"。让 AI 根据记忆内容和查询意图，自行决定哪些记忆优先返回、哪些降权保留。

### 设计方式：提示词驱动检索

召回时，潜意识 AI 收到：

```
[系统提示词]
你是记忆检索助手。根据用户的查询，从候选记忆中选出最相关的，
并按优先级排序返回。

[用户查询]
"我当初为什么选 PostgreSQL？"

[候选记忆列表]
1. [活跃] 2026-04-28: "PG 部署在 10.0.1.50:5432" (新鲜度 0.9, 重要性 0.6)
2. [沉淀] 2025-11-15: "选 PG 不选 MySQL，因为事务支持、JSON 查询、扩展生态..."
   (新鲜度 0.1, 重要性 0.9)
3. [沉淀] 2025-08-03: "MongoDB 测试失败，文档模型不适合我们的 OLTP 场景"
   (新鲜度 0.05, 重要性 0.8)
4. [活跃] 2026-05-01: "今天天气不错" (新鲜度 0.95, 重要性 0.1)

[排序原则]
- 相关性 > 新鲜度：即使很旧，如果直接回答用户问题，优先级应更高
- 不要因为"天气不错"更新鲜就排在"选型决策"前面
- 沉淀知识在正确的查询面前应该浮现到顶端
- 闲聊类即使新鲜度高也排在最后或直接排除
```

### 三层逻辑概念（非硬编码边界）

| 层 | 概念 | AI 的行为 |
|------|------|---------|
| **活跃层** | 近期、高频访问、高重要性 | AI 默认优先检索，但不排除沉淀层 |
| **沉淀层** | 历史决策、经验教训、旧偏好 | 正确查询来临时浮现，平时安静 |
| **归档层** | 完整日志，仅标签索引 | AI 认为需要翻原始日志时触发回溯 |

**关键**：层与层之间没有硬编码的时间边界。一条 3 年前的记忆如果重要性极高，AI 可以把它排在活跃层前面返回。一条昨天的闲聊即使新鲜度满分，AI 也应该把它沉底。

> **设计哲学**：沉淀的知识永远不会被删除。它只是安静地待在底层，在正确的问题来临时浮现——就像你不会忘记 10 年前骨折的教训，即使你已经不记得那天穿了什么颜色的衣服。

## 5.3 输入法式预测网络（可选实验特性）

> **灵感来源**：输入法根据用户的输入习惯，自动预测下一个字。用户用得越多，预测越准。记忆引擎同理——**对话越多、标签树越完善，神经网络的训练数据越充足，预测越精准**。

### 冷启动到热运行的渐进过程

```
阶段 1: 冷启动 (0 条记忆，新用户)
┌─────────────────────────────────────┐
│ 无标签树，无训练数据                 │
│ 预测网络 = 不启用                    │
│ 回退：纯关键词匹配 + 五维评分排序    │
└─────────────────────────────────────┘
    │ 用户使用中...
    ▼
阶段 2: 预热期 (标签树开始生长，记忆 > 50 条)
┌─────────────────────────────────────┐
│ 标签树已有结构，潜意识 AI 已打标分类 │
│ 训练数据开始累积                    │
│ 预测网络 = 离线训练模式（收集样本）  │
│ 此时仍用确定性检索，网络在后台学习   │
└─────────────────────────────────────┘
    │ 样本数量 > 阈值（如 2000 次"提问→记忆"配对）
    ▼
阶段 3: 热运行 (标签树完善，样本充足)
┌─────────────────────────────────────┐
│ 神经网络已训练完成                   │
│ 预测网络 = 在线推理模式              │
│ 用户打字时 → 实时预测 → 预加载记忆   │
│ 猜对 → 毫秒级响应                    │
│ 猜错 → 降级确定性检索，写入负样本    │
│ 持续在线学习，越用越准               │
└─────────────────────────────────────┘
```

### 核心思路

用户说了一句话 → 潜意识 AI 不仅召回相关记忆，还**预判用户下一步可能要什么**，提前加载到活跃层。

```
用户输入: "帮我查一下那个数据库的..."
                              ↑ 说到一半
┌─────────────────────────────────────────┐
│ 预测网络激活（热运行阶段）               │
│ 根据历史模式：                          │
│   "数据库" + "查" → 70% 是查连接信息    │
│   "数据库" + "配置" → 20% 是查配置参数  │
│   "数据库" + "状态" → 10% 是查运行状态  │
│                                         │
│ 预加载：PG 连接信息、MySQL 连接信息、     │
│         Redis 连接信息                  │
└─────────────────────────────────────────┘
```

### 实现方式

**轻量神经网络（运行时动态训练）**：

```
输入层: 用户当前输入的嵌入向量 + 上下文关键词
    ↓
隐藏层: 小型全连接网络（可配置，如 64-32-16）
    ↓
输出层: 候选记忆标签的概率分布
```

**训练数据自动生成**：用户的每一次"提问 → 最终需要的记忆"构成一个训练样本。无需人工标注。

```
样本示例（由潜意识 AI 自动记录）：
  输入特征: ["查", "数据库", "地址"]
  期望输出: tag="基础设施/数据库/PostgreSQL/连接信息"
  
  输入特征: ["为什么", "选型", "当时"]
  期望输出: tag="决策/技术选型/数据库"

负样本（预测错误时自动记录）：
  输入特征: ["数据库", "备份"]
  预测输出: tag="连接信息"  ✗
  实际输出: tag="运维/备份策略"  ← 修正
```

### 为什么必须先有标签树？

标签树的层级结构直接作为神经网络的**输出分类空间**：

```
没有标签树 → 输出空间是扁平的 tag 列表，预测发散
有标签树   → 输出空间是树节点路径，预测聚焦
              "数据库/PostgreSQL/连接信息" 比 "#postgresql" 精确得多
```

标签树越完善（节点拆分合理、命名准确），神经网络的分类边界越清晰，预测越准。

### 约束（必须满足）

| 约束 | 要求 |
|------|------|
| **性能** | 预测延迟 < 500ms，异步处理，不能拖慢主流程 |
| **资源** | 模型文件 < 100MB，推理内存 < 500MB |
| **冷启动** | 新用户无标签树/训练数据时，网络不启用，退化为纯关键词匹配 |
| **渐进激活** | 记忆 < 50 条 → 关闭；记忆 50-200 条 → 收集样本；记忆 > 200 条 → 在线推理 |
| **可关闭** | 用户可禁用预测，仅用确定性检索 |
| **隔离** | 预测失败不影响正确召回结果 |

### 预测与确定性检索的关系

```
用户查询
    │
    ├─→ [预测网络 热运行?] ──是──→ 预加载候选记忆（"猜你要什么"）
    │       │ 否                         ↓
    │       │                     如果猜对了 → 毫秒级响应
    │       │                     如果猜错了 → 写入负样本，降级 ↓
    │       └─────────────────────────────────────────────→
    │                                                      ↓
    └─→ [确定性检索] → 五维评分排序（标准流程）←──────────┘
                         ↓
                    最终返回结果
```

预测是**加速层**，不是替代层。猜错了不影响确定性检索的正确结果。用户用得越多，标签树越完善，训练样本越充足，预测越准——就像输入法，用一个月比第一天准得多。

> **可选性**：此特性为实验性质，不作为 V1 必须交付项。在性能受限的环境中可以不启用。

## 6. 动态标签树：AI 自组织的知识图谱

> **核心洞察**：标签不应该是扁平的列表或预设的分类。潜意识 AI 应该像生物学家建立物种树一样，从用户的对话中**动态生成层次化的标签树**，每条记忆挂在树的某个节点下。

### 6.1 设计哲学：不预设，自生长

**预设分类方式（不采用）**：
```
├── 数据库
│   ├── PostgreSQL
│   └── MySQL
├── 前端
│   ├── React
│   └── Vue
└── 部署
    ├── Docker
    └── K8s
```
问题：预设的分类无法覆盖所有场景，用户的知识领域程序员无法预知。

**AI 自组织方式（采用）**：
潜意识 AI 每次提取记忆时，评估这条记忆应该放在标签树的哪个位置。如果现有树没有合适的节点，AI 可以**创建新分支**。如果某个节点下的记忆太多，AI 可以**自动拆分**。

### 6.2 标签树的动态演化

软件框架需要每天或者每隔一周自动的调用潜意识的记忆AI整理标签树，调用的频率取决于对话的频率和记忆的增长速度（阈值，计数器）。AI 会根据当前标签树的结构和新记忆的内容，决定是否需要调整树的结构。

```
初期（10 条记忆）:
记忆
├── 项目
│   └── Pudding
│       ├── 数据库 (3条)
│       └── 前端 (2条)
└── 个人偏好 (1条)

中期（100 条记忆，自动拆分）:
记忆
├── 项目
│   ├── Pudding
│   │   ├── 数据库
│   │   │   ├── PostgreSQL (5条)  ← AI 发现 PG 记忆太多，自动拆分
│   │   │   └── Redis (2条)
│   │   ├── 前端
│   │   │   ├── React (3条)
│   │   │   └── Vite (2条)
│   │   └── 部署
│   │       ├── Docker (4条)
│   │       └── K8s (2条)
│   └── 其他项目
├── 技术决策 (8条)  ← AI 发现"决策"类记忆跨项目重复出现，提升为一级分类
├── 个人偏好 (5条)
└── 经验教训 (3条)
```

### 6.3 AI 操作标签树的能力

潜意识 AI 通过提示词获得以下"手术"能力：

```
[系统提示词]
你是记忆标签树的管理员。标签树是一个层次化的索引结构，每条记忆挂在一个叶子节点下。

你可以执行以下操作：
1. **挂载**：将新记忆挂到最合适的现有节点下
2. **创建节点**：如果没有合适的节点，创建新分支
3. **拆分**：某个节点下记忆 > N 条（默认 20），评估是否需要拆分子类别
4. **合并**：两个节点的记忆高度相似，合并为一个
5. **提升**：某个子节点频繁被跨父节点引用，提升为一级节点
6. **重命名**：节点的名称不再准确描述其下记忆，建议新名称
7. **归档**：某个节点下的记忆全部衰减到沉淀层，标记为[低频]

[当前标签树]
{树的 JSON 结构}

[新记忆]
{记忆内容 + 标签 + 元数据}

[输出格式]
返回 JSON：{
  "action": "mount" | "create" | "split" | "merge" | "promote" | "rename" | "archive",
  "target_path": "记忆/项目/Pudding/数据库/PostgreSQL",
  "reason": "判断理由"
}
```

### 6.4 标签树作为检索加速器

```
用户查询: "PostgreSQL 连接地址"
          │
          ▼
┌─────────────────────┐
│ 树定位              │  ← O(log n) 快速定位到 "数据库/PostgreSQL" 子树
│ 不需要扫描全部记忆    │
└────────┬────────────┘
         ▼
┌─────────────────────┐
│ 子树内五维评分排序   │  ← 只在 5 条候选中排序，不是 10 万条
└────────┬────────────┘
         ▼
┌─────────────────────┐
│ 返回 + 兄弟节点补充   │  ← 顺便带上 Redis、MySQL 的连接信息（它们在同一父节点下）
└─────────────────────┘
```

### 6.5 标签 vs 标签树的区别

| | 扁平标签 | 动态标签树 |
|------|---------|----------|
| 结构 | `#database #postgresql` | `数据库/PostgreSQL/连接信息` |
| 关系 | 平等，无层级 | 父子、兄弟、跨分支引用 |
| 索引效率 | O(n) 全扫描 | O(log n) 树定位 |
| 维护者 | 人工打标签 | AI 自动组织 |
| 适应性 | 新领域需要人工加标签 | AI 自动生长新分支 |


标签树的节点还支持指针（地址），可以快速索引到与之相关的记忆或者其他标签。
记忆本身，也许标签和指针都不止一个，标签是分类用的，指针是关联用的。比如一条记忆可能挂在"数据库/PostgreSQL"下，但同时有一个指针指向"技术决策/选型"，表示它也是一个重要的选型决策。

## 8. 潜意识 Agent 交互协议

### 8.1 显意识 → 潜意识（被动观察）

潜意识持续监听以下事件（通过进程内事件总线）：

- `conversation.turn.completed` — 一轮对话完成
- `task.status.changed` — 任务状态变更
- `tool.executed` — 工具执行完成
- `session.idle` — 会话进入空闲（触发深度反思）

### 8.2 潜意识 → 显意识（主动推送/阻塞召回）

| 场景 | 模式 | 说明 |
|------|------|------|
| 记忆刷写 | 异步非阻塞 | 后台写入，不打断主流程 |
| 记忆提纯 | 异步非阻塞 | 空闲时批处理 |
| 记忆召回 | **阻塞子代理** | 新会话开始或显式回忆时，等待召回完成 |
| 冲突提醒 | 异步推送 | 发现矛盾记忆时通知显意识 |
| 定期清理 | 异步后台 | 低优先级定时任务 |

### 8.3 反思先行：操作前自动回忆（关键设计）

> **核心机制**：Agent 在执行任何操作前，先走"回忆 → 消歧 → 行动"流程，而不是盲目执行。

#### 标准交互流程

```
用户指令
    │
    ▼
┌─────────────────────┐
│ Step 1: 解析意图     │  用户说"启动 PostgreSQL"
│ 提取：动词=启动      │
│       目标=PostgreSQL│
└────────┬────────────┘
         ▼
┌─────────────────────┐
│ Step 2: 阻塞召回     │  [潜意识同步返回]
│ 关键词=PostgreSQL    │  找到 3 条相关记忆：
│ 关键词=数据库        │  • 生产 PG: 10.0.1.50:5432
│ 关键词=启动          │  • 测试 PG: localhost:5433
│                      │  • 开发 MySQL: localhost:3306
└────────┬────────────┘
         ▼
┌─────────────────────┐
│ Step 3: 消歧        │  当前上下文是"生产环境"？
│ 活跃层优先           │  → 是 → 选中 10.0.1.50
│ 上下文匹配           │  → 否 → 反问用户确认
└────────┬────────────┘
         ▼
┌─────────────────────┐
│ Step 4: 执行        │  连接 10.0.1.50:5432
│ 附带记忆上下文       │  用户 admin，密码从 vault 取
└─────────────────────┘
```

#### 多目标消歧策略

当检索返回多个匹配记忆时：

| 消歧手段 | 说明 |
|---------|------|
| **上下文推断** | 当前对话提到"生产环境" → 自动选生产 PG |
| **新鲜度加权** | 最近访问过的数据库优先 |
| **重要性比较** | 标记为"关键"的记忆优先 |
| **反问确认** | 以上都无法确定时，主动问"您是指 10.0.1.50 的生产库还是 localhost 的测试库？" |

最重要的是，确认记忆与实际现实情况的匹配程度。比如如果用户说"连接 PG"，但记忆里最新的 PG 连接信息是半年前的，潜意识应该标记"可能过时"，并在执行前提醒用户确认。
另外，可以通过只读操作，直接确认，比如读取目录或者Linux下的文件，如记忆里面是XX，但是实际情况是XXX，潜意识可以直接确认记忆过时了。



### 8.4 闲聊过滤：记住该记的，忘掉该忘的

> **核心判断**：不是所有用户说的话都值得进入记忆系统。闲聊应被遗忘，关键事实应被提取。

```
对话分类器（潜意识第一道关卡）
    │
    ├─ 闲聊类 → 直接丢弃
    │   "早上好"、"今天天气不错"、"你吃了吗"
    │   "哈哈"、"有意思"、纯情感表达
    │
    ├─ 上下文类 → 暂存会话日志，不进入长期记忆
    │   "帮我看看这个文件"、"继续上次的话题"
    │   当前任务的中间步骤、命令行输出（可能有一大堆低价值信息）
    │
    └─ 关键事实类 → 提取并进入记忆管道
        • 基础设施信息："PG 部署在 10.0.1.50:5432"
        • 决策记录："选 PG 不选 MySQL，因为事务支持"
        • 用户偏好："我喜欢用 dark theme"
        • 约束条件："预算不能超过 ¥5000/月"
        • 待办事项："下周一前需要完成数据库迁移"
        • 经验教训："上次直接删表导致宕机，先备份"
```

上面的规则不是定义好的，而是AI根据用户的对话内容和历史记忆动态学习调整的。比如如果用户经常说"今天天气不错"但从不提及技术细节，AI 可能会学会把"天气不错"这类话归类为闲聊，自动过滤掉。相反，如果用户经常在技术讨论中提到"上次直接删表导致宕机"，AI 会学会把这类包含"上次"、"导致"的句子归类为重要经验，自动提取进入记忆管道。

#### 分类启发式规则

| 信号 | 倾向 |
|------|------|
| 含 IP/端口/路径/密码/配置 | **记住** |
| 含"不要"/"禁止"/"必须" | **记住**（约束） |
| 含"上次"/"之前"/"当时" | **记住**（经验） |
| 纯寒暄/表情/短感叹 | **遗忘** |
| 中间步骤/临时文件/调试输出 | 记日志，不入长期记忆 |
| 含日期+行动+结果 | **记住**（事件） |
| 含技术选型/架构决策 | **记住**（决策） |


上面的规则不是定义好的，而是AI根据用户的对话内容和历史记忆动态学习调整的。比如如果用户经常说"今天天气不错"但从不提及技术细节，AI 可能会学会把"天气不错"这类话归类为闲聊，自动过滤掉。相反，如果用户经常在技术讨论中提到"上次直接删表导致宕机"，AI 会学会把这类包含"上次"、"导致"的句子归类为重要经验，自动提取进入记忆管道。

|------|------|
