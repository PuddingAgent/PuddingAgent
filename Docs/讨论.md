## 1. 设计目标

下一代 Coding Agent 的目标不是"帮人写代码"，而是 **自动化完成软件工程闭环**：

```text
需求理解 → 任务拆分 → 架构设计 → 代码生成 → 单测生成
    → 运行 → 调试 → 回归验证 → 重构优化 → 质量评估
```

> 当前市面所有工具（Claude Code、Cursor、OpenCode）基本只覆盖第 4 步——代码生成。

## 2. 行业现状：8 个结构性瓶颈

### 2.1 文本级处理，无结构化代码理解

| 现象 | 根因 |
| --- | --- |
| 重命名只能暴力文本替换，不区分类名/注释/字符串 | Agent 看到的是"文本"而非 AST 语法树或语义图 |
| 无法做 IDE 级别的批量重命名 | 缺少 LSP、类型系统、符号表、引用分析 |

> **本质：** 大模型是概率预测器，没有真正的结构化代码理解能力。

### 2.2 外挂拼接式生态，无法开箱即用

| 现象 | 根因 |
| --- | --- |
| 需要配置 SKILL、MCP、LSP 等大量外部组件 | Agent = 模型 + Prompt 工程 + 工具拼接，而非完整系统 |
| 工具调用失败时无回滚机制 | MCP/Tool/LSP 是"外挂补丁"，不是原生能力 |

### 2.3 单模型依赖，成本与能力失衡

| 现象 | 根因 |
| --- | --- |
| 旗舰模型聪明但贵，便宜模型干不了复杂任务 | 只能绑定一个主模型，无智能路由和任务分解 |
| 用大模型做格式化/重命名等简单任务，成本浪费 | 缺少多模型协作调度框架 |

### 2.4 概率修补式调试，修一个炸一堆

| 现象 | 根因 |
| --- | --- |
| 修一个 Bug 引入更多 Bug | 调试流程 = 修改 → 运行 → 看报错 → 再猜一次 |
| 改 A 影响 B，局部正确但全局错误 | 缺少执行路径分析、数据流分析、影响面评估 |

### 2.5 无项目意识，容易跑偏

| 现象 | 根因 |
| --- | --- |
| 无法自我推进项目，需要人工不断纠正 | 记忆仅为最近几万 Token 上下文，无长期状态 |
| 不知道当前阶段、核心文件、架构约束 | 缺少目标管理、阶段计划、任务图 |

### 2.6 幻觉与记忆丢失

| 现象 | 根因 |
| --- | --- |
| 虚构不存在的函数/库，忘记几轮前的约定 | 无真正状态存储，每轮都是重算概率 |
| 修改已确认的接口 | Token 窗口有限，旧约定被冲掉 |

### 2.7 代码质量默认"能跑即可"

| 现象 | 根因 |
| --- | --- |
| 缺少可维护性、解耦、扩展性考虑 | 模型优化目标 = 生成"看起来正确的文本"，而非工程质量 |
| 需要不断人工约束才能产出高质量代码 | 缺少自动化质量评估与重构反馈环 |

### 2.8 依赖人工纠偏

> 以上 7 个瓶颈叠加的结果：Agent 每一步都可能偏离，必须人工介入。

### 瓶颈一览

| # | 瓶颈 | 优先级 |
| --- | --- | --- |
| 1 | 语义理解弱（文本级处理） | 🔴 高 |
| 2 | 无真正 AST 操作能力 | 🔴 高 |
| 3 | 无多模型协作 | 🟡 中 |
| 4 | 无智能调度 | 🟡 中 |
| 5 | 无长期项目记忆 | 🔴 高 |
| 6 | 无结构化调试能力 | 🟡 中 |
| 7 | 无代码质量评估体系 | 🟢 低 |
| 8 | 依赖人工纠偏（综合结果） | — |

## 3. 总体架构：6 层结构

```text
┌─────────────────────────────────────────────────────────────┐
│  Layer 1. 项目状态机层 (Project Brain)                        │
│  — 长期目标、阶段管理、架构约束、任务图                          │
├─────────────────────────────────────────────────────────────┤
│  Layer 2. 任务规划与调度层 (Planner)                          │
│  — 需求拆分、子任务生成、执行顺序编排                            │
├─────────────────────────────────────────────────────────────┤
│  Layer 3. 多模型协作层 (Model Router)                         │
│  — 任务分类、成本感知路由、动态模型切换                          │
├─────────────────────────────────────────────────────────────┤
│  Layer 4. 代码理解层 (Semantic Engine)                        │
│  — AST、依赖图、调用图、类型系统、符号引用                       │
├─────────────────────────────────────────────────────────────┤
│  Layer 5. 执行与验证层 (Execution Loop)                       │
│  — 沙盒运行、自动单测、编译验证、递归修复                        │
├─────────────────────────────────────────────────────────────┤
│  Layer 6. 质量与安全评估层 (QA System)                        │
│  — 复杂度评分、耦合检测、覆盖率、安全扫描                        │
└─────────────────────────────────────────────────────────────┘
```

### 各层解决的瓶颈映射

| 层 | 解决的瓶颈 |
| --- | --- |
| Layer 1 项目状态机 | #5 无项目意识、#8 依赖人工纠偏 |
| Layer 2 任务规划 | #5 无项目意识、#4 无智能调度 |
| Layer 3 多模型协作 | #3 单模型依赖、#4 无智能调度 |
| Layer 4 语义引擎 | #1 语义理解弱、#2 无 AST 操作 |
| Layer 5 执行验证 | #6 无结构化调试 |
| Layer 6 质量评估 | #7 无质量评估体系 |

## 4. 5 个核心突破方向

### 4.1 项目状态机 — 解决"跑偏"

**思路：** 每个项目维护一份持久化的工程状态图，Agent 每一步必须"读取状态 → 更新状态 → 才允许改代码"。

**状态模型示例 (`.pudding/state.json`)：**

```json
{
  "stage": "implementation",
  "architecture_locked": true,
  "modules": {
    "core-abstractions": "done",
    "local-tools": "done",
    "llm-gateway": "done",
    "cli-repl": "done",
    "lsp-integration": "in-progress",
    "git-snapshot": "planned"
  },
  "constraints": [
    "must use .NET 10",
    "core library zero third-party dependency (except CliWrap)",
    "OpenAI compatible API protocol"
  ],
  "risks": [
    "AOT compatibility with Spectre.Console"
  ]
}
```

**关键行为：**
- Agent 动手前读取状态，确保不违背已锁定的架构
- 每完成子任务自动更新状态，保持全局进度可追踪
- 支持回滚到任意状态节点

### 4.2 语义代码引擎 — 解决暴力替换

**思路：** LLM 只做决策，不做执行。结构化代码操作交给确定性引擎。

**正确的重命名流程：**

```text
LLM 决策: "将 Class1 重命名为 AgentOrchestrator"
    │
    ▼
语义引擎执行:
    1. 通过 LSP 定位 symbol (Class1)
    2. 通过 find_references 找到所有引用点
    3. 在 AST 层面修改（非文本替换）
    4. 重新生成代码
    5. 验证编译通过
```

**必须内置的能力：**

| 能力 | 实现方式 | 用途 |
| --- | --- | --- |
| AST 解析 | Roslyn (C#) / Tree-sitter (多语言) | 结构化理解代码 |
| 类型系统 | LSP Protocol | 类型安全的重构 |
| 符号引用图 | Roslyn Workspace API | 精确定位所有引用 |
| 调用图 | LSP call hierarchy | 影响面评估 |

### 4.3 多模型调度 — 解决成本与能力失衡

**思路：** 建立任务路由器，根据任务复杂度自动选择最适合的模型。

**路由策略：**

| 任务类型 | 推荐模型级别 | 示例 |
| --- | --- | --- |
| 格式化 / 重命名 / Lint | 小模型（本地或低成本 API） | Qwen-Coder、DeepSeek-Coder |
| 简单函数生成 / 单测 | 中模型 | GPT-4o-mini、DeepSeek-Chat |
| 复杂算法 / 架构决策 | 大模型 | Claude Sonnet/Opus、GPT-4o |
| 复杂逻辑验证 | 推理模型 | o3、DeepSeek-R1 |

**高级模式 — Model Ensemble：**
- 多模型交叉校验：生成模型出代码，推理模型验证逻辑，审计模型检查安全
- 成本预估：路由前估算 Token 消耗，选择性价比最优的模型

### 4.4 自我调试闭环 — 解决"修一个炸一堆"

**思路：** 修改必须最小化，每次改动自动触发验证，失败则自动回滚重试。

**完整调试流程：**

```text
Agent 生成修改计划（非直接改代码）
    │
    ▼
定位目标函数（通过语义引擎）
    │
    ▼
最小化 Patch（只改目标函数，保证调用链不变）
    │
    ▼
自动生成/运行单测
    │
    ▼
  ┌─ 通过 → 提交修改，更新状态
  │
  └─ 失败 → 分析报错堆栈
              │
              ▼
         定位函数级问题（非文件级）
              │
              ▼
         差分分析 + 影响面评估
              │
              ▼
         最小修改重试（最多 N 轮）
              │
              ▼
         回归测试 + 对比覆盖率
```

**关键约束：**
- 不允许重写整个文件，必须以函数为粒度 Patch
- 每次修改前创建 Git 快照，失败可一键回滚
- 递归修复上限（如 5 轮），超出则中止并报告人工

### 4.5 质量评估层 — 解决"能跑即可"

**思路：** 代码生成后自动评分，低于阈值则自动重构，而非直接交付。

**评估指标：**

| 指标 | 工具/方法 | 阈值建议 |
| --- | --- | --- |
| 圈复杂度 (Cyclomatic Complexity) | Roslyn Analyzer | 单函数 ≤ 10 |
| 耦合度 (Coupling Score) | 依赖图分析 | 模块间依赖 ≤ 5 |
| 测试覆盖率 (Test Coverage) | dotnet test + coverlet | ≥ 70% |
| 死代码 (Dead Code) | Roslyn Unused Code Analyzer | 0 |
| 安全漏洞 | Security Analyzer / SAST | 0 Critical |
| 代码重复率 | 结构化比对 | ≤ 5% |

**自动重构触发：**
```text
生成代码 → 质量评分 → 低于阈值？
    │                         │
    ├─ 是 → 自动重构 → 重新评分（最多 3 轮）
    │
    └─ 否 → 交付
```

## 5. 核心设计原则

### LLM 决策，确定性引擎执行

| 角色 | LLM（概率性） | 确定性引擎 |
| --- | --- | --- |
| 重命名 | 决定"将 X 重命名为 Y" | LSP/Roslyn 执行符号级替换 |
| 重构 | 生成"修改计划" | AST 操作 + Patch Generator |
| 调试 | 分析报错、制定修复方案 | 沙盒运行 + 单测验证 |
| 架构设计 | 输出架构方案文档 | 项目状态机校验约束合规 |

> **原因：** LLM 直接写代码永远不可控。执行层必须是可验证的确定性系统。

## 6. 映射到 PuddingAssistant 路线图

将 5 个突破方向对应到 PuddingAssistant 的现有架构和开发阶段：

### 与现有架构的对应关系

| 突破方向 | PuddingAssistant 对应模块 | 当前状态 |
| --- | --- | --- |
| 项目状态机 | `.pudding/state.json` + 状态管理服务 | 📋 规划中 |
| 语义代码引擎 | `ISenseAbility` (LSP Client) | 📋 规划中 (D05) |
| 多模型调度 | `/model` 指令 → Model Router | 📋 规划中 |
| 自我调试闭环 | Self-Healing Loop (Map.md §2.3) | 📋 规划中 |
| 质量评估层 | QA System (新增) | 📋 规划中 |

### 建议实现顺序

| 优先级 | 方向 | 理由 |
| --- | --- | --- |
| **P0** | 语义代码引擎 (LSP) | 基础能力，其他方向都依赖结构化代码理解 |
| **P1** | 自我调试闭环 | 直接提升 Agent 输出质量，减少人工纠偏 |
| **P1** | 项目状态机 | 解决最常见的"跑偏"问题 |
| **P2** | 多模型调度 | 优化成本，当前可先用单模型跑通 |
| **P2** | 质量评估层 | 锦上添花，可后期集成 Roslyn Analyzer |
