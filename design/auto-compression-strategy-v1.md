## 1. 问题定义

当前缓存命中率仅 **38%**，远低于目标值（≥60%，最优 80~90%）。每次 LLM 调用重复消耗 Token，无有效压缩机制。上下文窗口利用率缺乏实时计量和自动触发。

## 2. 设计目标

| 目标 | 当前值 | 目标值 | 最优值 |
|------|--------|--------|--------|
| 缓存命中率 | 38% | ≥60% | 80~90% |
| 上下文使用率触发 | 无 | 80% 自动触发 | — |
| 压缩方式 | 手动 | 自动（心跳集成） | — |
| 压缩模型 | — | Flash 模型 | — |

## 3. 压缩策略

### 3.1 触发机制

```
触发条件 (满足任一即触发):
├── 硬触发: 上下文窗口使用率 ≥ 80%（来自滑动窗口熔断）
├── 心跳触发: 每 N 次心跳自动评估（N=10，约 10 分钟）
└── 手动触发: 用户主动调用 /compress
```

### 3.2 压缩范围

```
压缩边界:
├── 起点: 上次压缩标记位（或会话起点）
├── 终点: 当前消息序列号
└── 保留: 最近 5 条用户/Agent 消息不参与压缩
```

### 3.3 压缩输出格式

```markdown
> 压缩范围提示：本次会话压缩只收录当前 session 窗口内
> 消息 Sequence X-Y（N 条）。更早的消息已移出 active 上下文。

<compact_summary>
## 用户目标
[一句话核心目标]

## 已完成事项
[本轮压缩范围内已完成的关键事项，3-5 条]

## 关键决策
[本轮做出的重要决策，含简要理由]

## 涉及文件和代码位置
[结构化列表：文件路径 + 改动摘要]

## 工具调用与重要输出
[关键工具调用及其结果摘要]

## 错误、阻塞与修复
[遇到的错误、阻塞点、及修复方案]

## 当前工作状态
[一句话当前进展]

## 明确的下一步
[接下来要做什么，1-3 条]

## 保留的用户偏好和约束
- [从记忆中提取的关键偏好]
</compact_summary>
```

### 3.4 压缩管线

```
原始消息序列
  │
  ├─→ [阶段1] 范围标记
  │     └─→ 确定压缩起点/终点，保留最近 5 条
  │
  ├─→ [阶段2] Flash 模型摘要
  │     ├─→ 模型: deepseek/deepseek-v4-flash
  │     ├─→ 输入: 被压缩消息序列（结构化提示词）
  │     └─→ 输出: compact_summary XML 块
  │
  ├─→ [阶段3] 质量检查
  │     ├─→ 检查: 是否有下一步？
  │     ├─→ 检查: 任务状态是否保留？
  │     └─→ 检查: 用户偏好是否保留？
  │
  ├─→ [阶段4] 注入
  │     ├─→ 替换被压缩消息为 compact_summary
  │     └─→ 更新压缩标记位
  │
  └─→ [阶段5] 记录
        ├─→ 记录压缩事件到 decision_log
        └─→ 更新缓存命中率指标
```

## 4. 心跳集成

### 4.1 心跳自检项

每次心跳新增检查项：

```
心跳自检清单 (更新版):
1. 缓存命中率检查 — ≥60%? 低于则预警
2. 上下文窗口检查 — ≥80%? 触发熔断+压缩
3. 工具状态检查 — 关键工具在线?
4. goal.md 状态检查 — 正常可读?
5. 记忆可用性检查 — 记忆图书馆正常?
6. 上次压缩时间检查 — 超过 30 分钟强制评估
7. 系统问题登记 — 异常记录+证据
```

### 4.2 心跳中的压缩评估

```csharp
// 伪代码
async Task HeartbeatAsync()
{
    // 1. 采样上下文使用率
    var usage = await monitor.SampleAsync();

    // 2. 熔断评估
    var state = fuse.Evaluate(usage, trend);

    if (state == FuseState.Red)
    {
        // 3. 触发自动压缩
        await compression.CompactAsync();

        // 4. 恢复
        await recovery.RecoverAsync();
    }

    // 5. 定期评估（每 10 次心跳）
    if (heartbeatCount % 10 == 0)
    {
        await compression.EvaluateAsync(); // 即使未达 80% 也检查
    }

    // 6. 更新指标
    await UpdateCacheHitRateMetric();
}
```

## 5. 缓存命中率优化方向

| 方向 | 优先级 | Jieba 依赖 | 说明 |
|------|:---:|:---:|------|
| 压缩后摘要缓存 | P0 | ❌ | compact_summary 作为固定前缀，天然缓存友好 |
| 记忆裁剪 L5/L6 | P1 | ✅ | 依赖 Jieba + FTS5，等修复后推进 |
| 重复查询消除 | P2 | ❌ | 对相同 goal.md/记忆查询增加短期缓存 |
| 前缀匹配优化 | P2 | ❌ | Canonicalization 方案的一部分 |

## 8. 验收标准

- [ ] 上下文使用率 ≥80% 时自动触发压缩
- [ ] 压缩后缓存命中率 ≥60%
- [ ] compact_summary 包含所有必需字段
- [ ] 最近 5 条消息在压缩后保留
- [ ] 心跳中可见压缩事件日志
- [ ] 压缩不影响 Agent 任务连续性
